# Implementation Patterns & Consistency Rules

These patterns ensure that all code generated by Claude Opus 4.6 across different sessions is consistent, compatible, and follows the same conventions. Any deviation from these patterns should be flagged and corrected.

## Naming Patterns

### Database Naming (Prisma Schema)

| Element | Convention | Example |
|---------|-----------|---------|
| Model names | PascalCase, singular | `Customer`, `CustomerInvoice`, `JournalEntry` |
| Field names | camelCase | `customerId`, `transactionDate`, `createdAt` |
| Relation fields | camelCase, descriptive | `customer`, `invoiceLines`, `createdByUser` |
| Enum names | PascalCase | `InvoiceStatus`, `PaymentMethod` |
| Enum values | UPPER_SNAKE_CASE | `DRAFT`, `APPROVED`, `POSTED`, `CANCELLED` |
| Table names (@@map) | snake_case, plural | `@@map("customer_invoices")` |
| Column names (@map) | snake_case | `@map("customer_id")` |
| Index names | `idx_{table}_{columns}` | `@@index([customerId, status], map: "idx_customer_invoices_customer_status")` |

**Example:**
```prisma
model CustomerInvoice {
  id              String         @id @default(uuid())
  invoiceNumber   String         @unique @map("invoice_number")
  customerId      String         @map("customer_id")
  customer        Customer       @relation(fields: [customerId], references: [id])
  status          InvoiceStatus  @default(DRAFT)
  totalAmount     Decimal        @map("total_amount") @db.Decimal(19, 4)
  currencyCode    String         @default("GBP") @map("currency_code") @db.VarChar(3)
  transactionDate DateTime       @map("transaction_date") @db.Date
  createdAt       DateTime       @default(now()) @map("created_at")
  updatedAt       DateTime       @updatedAt @map("updated_at")
  createdBy       String         @map("created_by")
  updatedBy       String         @map("updated_by")

  lines           CustomerInvoiceLine[]

  @@map("customer_invoices")
  @@index([customerId, status], map: "idx_customer_invoices_customer_status")
}
```

### API Naming

| Element | Convention | Example |
|---------|-----------|---------|
| URL paths | kebab-case, plural nouns | `/api/v1/customer-invoices`, `/api/v1/stock-movements` |
| URL parameters | camelCase | `/api/v1/customers/:customerId` |
| Query parameters | camelCase | `?statusFilter=DRAFT&sortBy=createdAt` |
| Request body fields | camelCase | `{ "customerId": "...", "transactionDate": "..." }` |
| Response body fields | camelCase | `{ "invoiceNumber": "INV-00001" }` |
| HTTP headers | Standard casing | `Authorization`, `Content-Type`, `X-Tenant-Id` |

**URL Pattern:**
```
GET    /api/v1/{module}/{resource}           # List
POST   /api/v1/{module}/{resource}           # Create
GET    /api/v1/{module}/{resource}/:id       # Get one
PATCH  /api/v1/{module}/{resource}/:id       # Update
DELETE /api/v1/{module}/{resource}/:id       # Soft delete
POST   /api/v1/{module}/{resource}/:id/approve  # Status transition
POST   /api/v1/{module}/{resource}/:id/post     # Status transition
```

Module prefixes: `/api/v1/finance/...`, `/api/v1/ar/...`, `/api/v1/ap/...`, `/api/v1/sales/...`, `/api/v1/purchasing/...`, `/api/v1/inventory/...`, `/api/v1/crm/...`, `/api/v1/hr/...`, `/api/v1/manufacturing/...`, `/api/v1/reporting/...`, `/api/v1/admin/...`

| Element | Convention | Example |
|---------|-----------|---------|
| Files (TypeScript) | kebab-case | `customer-invoice.service.ts`, `create-invoice.schema.ts` |
| Files (React components) | kebab-case | `invoice-form.tsx`, `customer-list.tsx` |
| React components | PascalCase | `InvoiceForm`, `CustomerList`, `AiChatPanel` |
| Functions | camelCase | `createInvoice`, `getCustomerBalance`, `approveInvoice` |
| Variables | camelCase | `invoiceTotal`, `customerName`, `isApproved` |
| Constants | UPPER_SNAKE_CASE | `MAX_INVOICE_LINES`, `DEFAULT_PAYMENT_TERMS` |
| Types/Interfaces | PascalCase | `CustomerInvoice`, `CreateInvoiceInput`, `InvoiceListResponse` |
| Enums | PascalCase (name), UPPER_SNAKE_CASE (values) | `enum InvoiceStatus { DRAFT, APPROVED }` |

## Structure Patterns

### File Suffixes

All files use descriptive suffixes to indicate their role:

| Suffix | Purpose | Example |
|--------|---------|---------|
| `.service.ts` | Business logic | `invoice.service.ts` |
| `.repository.ts` | Database access | `invoice.repository.ts` |
| `.routes.ts` | Fastify route definitions | `invoice.routes.ts` |
| `.schema.ts` | Zod validation schemas | `invoice.schema.ts` |
| `.test.ts` | Tests (co-located) | `invoice.service.test.ts` |
| `.types.ts` | TypeScript type definitions | `invoice.types.ts` |
| `.hook.ts` | React hooks (if multiple per file) | `use-invoice.hook.ts` |
| `.tsx` | React components | `invoice-form.tsx` |

## Format Patterns

### API Response Format

**Success Response:**
```typescript
// Single item
{ "data": { "id": "...", "invoiceNumber": "INV-00001", ... } }

// List with pagination
{
  "data": [ { ... }, { ... } ],
  "pagination": {
    "cursor": "abc123",
    "hasMore": true,
    "total": 156
  }
}

// Action result
{ "data": { "id": "...", "status": "APPROVED" }, "message": "Invoice approved successfully" }
```

**Error Response:**
```typescript
{
  "error": {
    "code": "PERIOD_LOCKED",           // Machine-readable error code
    "message": "Cannot modify transactions in a locked financial period",  // Human-readable
    "details": {                        // Optional additional context
      "periodStart": "2026-01-01",
      "periodEnd": "2026-01-31"
    },
    "statusCode": 422                   // HTTP status code
  }
}
```

**Error Code Convention:** UPPER_SNAKE_CASE, module-prefixed for domain errors:
- `AUTH_INVALID_CREDENTIALS`, `AUTH_TOKEN_EXPIRED`, `AUTH_MFA_REQUIRED`, `AUTH_MFA_INVALID`, `AUTH_MFA_SETUP_REQUIRED`
- `AR_INVOICE_ALREADY_APPROVED`, `AR_CREDIT_LIMIT_EXCEEDED`
- `FIN_PERIOD_LOCKED`, `FIN_UNBALANCED_JOURNAL`
- `INV_INSUFFICIENT_STOCK`, `INV_SERIAL_REQUIRED`
- `VALIDATION_ERROR` (generic, with field-level details)
- `NOT_FOUND`, `USER_NOT_FOUND`, `FORBIDDEN`, `RATE_LIMITED`, `MFA_ALREADY_ENABLED`

### Date/Time Formats

| Context | Format | Example |
|---------|--------|---------|
| JSON API responses | ISO 8601 string | `"2026-02-15T10:30:00.000Z"` |
| Date-only fields (transaction date) | ISO 8601 date | `"2026-02-15"` |
| Prisma schema | `DateTime` or `@db.Date` | `transactionDate DateTime @db.Date` |
| Display (UI) | UK format via Intl | `15/02/2026`, `15 Feb 2026` |
| Stored in DB | UTC always | Timezone conversion in presentation layer only |

### Money Handling

| Rule | Implementation |
|------|---------------|
| All amounts as Decimal | `Prisma.Decimal` in code, `DECIMAL(19,4)` in DB |
| Never use `number` for money | Always `Decimal` type from Prisma |
| Rounding | Round at the last step, not intermediate calculations |
| Display formatting | `Intl.NumberFormat('en-GB', { style: 'currency', currency: 'GBP' })` |
| Calculations | Use Decimal arithmetic methods: `.plus()`, `.minus()`, `.times()`, `.dividedBy()` |

## Communication Patterns

### Event Naming

| Convention | Format | Example |
|-----------|--------|---------|
| Event name | `{entity}.{action}` (lowercase, dot-separated) | `invoice.created`, `invoice.approved`, `payment.received` |
| Action verbs | Past tense (events describe what happened) | `created`, `updated`, `approved`, `posted`, `cancelled` |
| Payload | Typed interface, includes entity ID + relevant IDs | `{ invoiceId, customerId, amount, currency }` |

### Logging

| Level | Use Case | Example |
|-------|----------|---------|
| `error` | Unrecoverable failures, exceptions | Database connection failure, HMRC API error |
| `warn` | Recoverable issues, business rule violations | Credit limit exceeded, rate limit approaching |
| `info` | Business events, state transitions | Invoice approved, payment received, user login |
| `debug` | Detailed execution flow | SQL query, AI tool call, cache hit/miss |

## Process Patterns

### Error Handling (Backend)

**Custom error classes hierarchy:**
```typescript
// Base
class AppError extends Error {
  constructor(public code: string, message: string, public statusCode: number, public details?: Record<string, unknown>) {}
}

// Domain errors (422)
class DomainError extends AppError { constructor(code: string, message: string, details?) { super(code, message, 422, details); } }

// Auth errors (401/403)
class AuthError extends AppError { ... }

// Not found (404)
class NotFoundError extends AppError { ... }

// Validation errors (400)
class ValidationError extends AppError { ... }
```

**Fastify error handler:**
```typescript
fastify.setErrorHandler((error, request, reply) => {
  if (error instanceof AppError) {
    return reply.status(error.statusCode).send({ error: { code: error.code, message: error.message, details: error.details } });
  }
  // Unexpected errors → 500, log full stack
  logger.error({ err: error, correlationId: request.id }, 'Unhandled error');
  return reply.status(500).send({ error: { code: 'INTERNAL_ERROR', message: 'An unexpected error occurred' } });
});
```

### Error Handling (Frontend)

- React Query `onError` callbacks for API errors
- Global error boundary for unexpected crashes
- Toast notifications for user-facing errors
- Error messages from API error.message (never raw stack traces)

### OKFlag State Machine Pattern

Every transactional entity follows this universal pattern:

```typescript
// Generic state machine service
class DocumentStateMachine<T extends { status: DocumentStatus }> {
  async approve(id: string, userId: string): Promise<T> {
    return this.transition(id, 'DRAFT', 'APPROVED', userId, async (doc) => {
      // Entity-specific side effects (GL posting, stock update, etc.)
      await this.onApprove(doc);
    });
  }

  async post(id: string, userId: string): Promise<T> {
    return this.transition(id, 'APPROVED', 'POSTED', userId, async (doc) => {
      await this.onPost(doc);
    });
  }

  // ... unapprove, cancel, etc.
  // Subclasses override onApprove(), onPost() for entity-specific side effects
}
```

## RBAC Permission Patterns

> These patterns replace the previous `createRbacGuard({ minimumRole })` approach.
> Design doc: `docs/plans/2026-02-19-granular-rbac-access-groups-design.md`

### Permission Guard Pattern

All routes use `createPermissionGuard(resourceCode, action?)` instead of `createRbacGuard()`:

```typescript
// OLD (deprecated — do not use):
fastify.get('/sales/orders', {
  preHandler: createRbacGuard({ minimumRole: 'STAFF' })
}, handler);

// NEW — granular permission guard:
fastify.get('/sales/orders', {
  preHandler: createPermissionGuard('sales.orders.list', 'view')
}, handler);

fastify.post('/sales/orders', {
  preHandler: createPermissionGuard('sales.orders.list', 'new')
}, handler);

fastify.patch('/sales/orders/:id', {
  preHandler: createPermissionGuard('sales.orders.detail', 'edit')
}, handler);

fastify.delete('/sales/orders/:id', {
  preHandler: createPermissionGuard('sales.orders.detail', 'delete')
}, handler);
```

**Guard implementation location:** `apps/api/src/core/rbac/permission.guard.ts`

**Guard behaviour:**
1. SUPER_ADMIN bypass — if `UserCompanyRole.role = SUPER_ADMIN`, skip permission check
2. Load user's effective permissions from `PermissionService` (cached)
3. Check `canAccess` for the resource; if `false` → 403
4. If `action` specified, check corresponding flag (`canNew`, `canView`, `canEdit`, `canDelete`); if `false` → 403
5. Attach resolved permissions to request context for downstream use

### Field-Level Visibility Hook

Response hook that strips hidden fields and annotates read-only fields:

```typescript
// Applied as onSend hook on routes with field-level control:
fastify.get('/sales/orders/:id', {
  preHandler: createPermissionGuard('sales.orders.detail', 'view'),
  onSend: filterFieldsByPermission('sales.orders.detail')
}, handler);
```

**Hook implementation location:** `apps/api/src/core/rbac/field-filter.hook.ts`

**Hook behaviour:**
1. Load `AccessGroupFieldOverride` entries for user's groups + resourceCode (from cache)
2. Merge visibility per field (most permissive wins: VISIBLE > READ_ONLY > HIDDEN)
3. Strip `HIDDEN` fields from response JSON `data` object
4. Add `_fieldMeta` object to response with read-only markers for frontend

**Response format with field metadata:**
```json
{
  "data": { "orderNumber": "SO-00001", "customerName": "Acme Ltd", "totalExVat": 1500.00 },
  "_fieldMeta": { "totalExVat": "readOnly" }
}
```

### Permission Cache Pattern

All permission lookups go through `PermissionService` which caches resolved permissions in Redis:

```typescript
// Service location: apps/api/src/core/rbac/permission.service.ts
class PermissionService {
  // Cache key: permissions:{userId}:{companyId}
  // TTL: 60 seconds
  // Stores: merged permission matrix + field overrides for all resources

  async getEffectivePermissions(userId: string, companyId: string): Promise<EffectivePermissions>;
  async hasPermission(userId: string, companyId: string, resourceCode: string, action?: Action): Promise<boolean>;
  async getFieldVisibility(userId: string, companyId: string, resourceCode: string, fieldPath: string): Promise<FieldVisibility>;

  // Invalidation triggers (call these from admin endpoints):
  async invalidateUser(userId: string, companyId: string): Promise<void>;
  async invalidateGroup(accessGroupId: string, companyId: string): Promise<void>;
  async invalidateAll(companyId: string): Promise<void>;
}
```

**Invalidation triggers:**
- Access group permission edited → `invalidateGroup()`
- User assigned/removed from group → `invalidateUser()`
- Resource table changed → `invalidateAll()`

### Resource Registration Pattern

Each module epic seeds its resources into the `Resource` table. Resources are also declared in `company-defaults.json` for new company seeding:

```typescript
// In module's Prisma migration (e.g., E16 Sales epic):
// Migration SQL inserts resources for the module:
INSERT INTO resources (id, code, name, module, type, sort_order)
VALUES
  (uuid(), 'sales.orders.list', 'Sales Orders', 'sales', 'PAGE', 100),
  (uuid(), 'sales.orders.detail', 'Sales Order Detail', 'sales', 'PAGE', 110),
  (uuid(), 'sales.quotes.list', 'Sales Quotes', 'sales', 'PAGE', 200),
  (uuid(), 'sales.quotes.detail', 'Sales Quote Detail', 'sales', 'PAGE', 210),
  (uuid(), 'sales.reports.revenue', 'Revenue Report', 'sales', 'REPORT', 900);
```

**Resource code convention:** `{module}.{entity}.{view}` — e.g., `sales.orders.list`, `finance.journals.detail`, `system.users.list`

**Resource type convention:**
- `PAGE` — list and detail views of transactional/master data
- `REPORT` — read-only reporting views
- `SETTING` — configuration pages (company settings, module settings)
- `MAINTENANCE` — administrative maintenance operations (recalculate, rebuild, etc.)

### Default Data File Pattern

A JSON file at `packages/db/default-data/company-defaults.json` is imported on company creation. This replaces hardcoded seed functions with a declarative, editable data file:

```typescript
// On company creation:
async function seedCompanyDefaults(companyId: string, db: PrismaClient): Promise<void> {
  const defaults = await loadDefaults('company-defaults.json');

  // 1. Upsert resources (global, not company-scoped)
  for (const resource of defaults.resources) {
    await db.resource.upsert({ where: { code: resource.code }, create: resource, update: resource });
  }

  // 2. Create access groups for this company
  for (const group of defaults.accessGroups) {
    const created = await db.accessGroup.create({
      data: { companyId, code: group.code, name: group.name, description: group.description, isSystem: true, ... }
    });
    // 3. Create permissions for each group
    for (const perm of group.permissions) {
      await db.accessGroupPermission.create({ data: { accessGroupId: created.id, ...perm } });
    }
    // 4. Create field overrides for each group
    for (const override of group.fieldOverrides) {
      await db.accessGroupFieldOverride.create({ data: { accessGroupId: created.id, ...override } });
    }
  }

  // 5. Seed VAT codes, payment terms, number series, currencies
  for (const vatCode of defaults.vatCodes) { /* upsert */ }
  for (const term of defaults.paymentTerms) { /* upsert */ }
  // ... etc
}
```

**File supports:** resources, accessGroups (with permissions + fieldOverrides), vatCodes, paymentTerms, numberSeries, currencies. Future industry variants (retail, manufacturing) will be separate files.

## Enforcement Guidelines

**All AI agents (Claude Opus 4.6 sessions) MUST:**

1. Follow the naming conventions exactly — no exceptions, no "creative" alternatives
2. Use the module structure template for every new module
3. Create co-located tests for every service file
4. Use Zod schemas for ALL API input validation — never trust raw input
5. Use the AppError hierarchy for ALL error responses — never throw raw Error
6. Use the repository pattern — services never import PrismaClient directly
7. Use the event bus for ALL cross-module side effects — never call another module's service directly
8. Use Decimal type for ALL monetary values — never use `number`
9. Follow the API response format — { data: T } for success, { error: {...} } for failure
10. Include correlationId in ALL log entries
11. Add audit trail entries for ALL financial entity mutations
12. Add `isActive` field to ALL reference/lookup entities — use it in LOV queries, not in detail/report queries
13. Never DELETE a reference entity that has foreign key dependents — deactivate it instead
14. Use `createPermissionGuard(resourceCode, action)` on ALL routes — never use deprecated `createRbacGuard()`
15. Apply `filterFieldsByPermission(resourceCode)` on detail/record endpoints that have sensitive fields
16. Register all new pages/reports/settings as resources in `company-defaults.json` and in Prisma migration

**Anti-Patterns (NEVER do these):**

- Direct Prisma calls in route handlers (use repository → service → route)
- `number` type for monetary values
- Raw string concatenation for SQL queries
- `any` type (use `unknown` and narrow)
- Mutable state in services (all services are stateless)
- Cross-module direct service calls (use events)
- Hardcoded error messages without error codes
- `console.log` (use structured logger)
- Storing secrets in code or committed files
- Deleting reference entities instead of deactivating (use `isActive: false`)
- Filtering by `isActive` in detail/report queries (only filter in LOV/dropdown queries)
- Using `createRbacGuard({ minimumRole })` — deprecated, use `createPermissionGuard(resourceCode, action)` instead
- Hardcoded permission checks against UserRole enum for page/action access — use Access Group permissions
- Adding pages/reports without registering them as resources in `company-defaults.json`
