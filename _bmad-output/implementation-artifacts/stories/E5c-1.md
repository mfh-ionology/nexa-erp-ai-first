# Story E5c.1: Automation Engine & Schema

Status: done

## Story

As a **developer**,
I want an automation execution engine with database schema so that AI agents can run goal-oriented workflows on schedules, respond to events, and chain together,
so that the AI transitions from a reactive assistant into a proactive operations engine that runs autonomous workflows.

## Acceptance Criteria

1. **AC-1: Prisma schema migration** — GIVEN the Prisma schema WHEN `prisma migrate dev` runs THEN the 6 tables (`ai_automations`, `ai_automation_steps`, `ai_automation_runs`, `ai_automation_step_runs`, `ai_prompt_variables`, `ai_automation_schedules`) are created with proper indexes, foreign keys (to `AiAgent`, `AiPrompt`, `User`, `CompanyProfile`), self-referential relations (`chainFromId`, `chainNextId`), and all columns as specified in the epic appendix

2. **AC-2: Scheduled automation execution** — GIVEN an automation with trigger type `SCHEDULED` and cron `0 7 * * 1-5` WHEN the scheduler runs THEN the automation executes at 7:00 AM Monday-Friday in the configured timezone, with `nextRunAt` updated after each execution

3. **AC-3: Event-triggered automation** — GIVEN an automation with trigger type `EVENT` and event `InvoiceOverdue` WHEN the event fires THEN the automation is triggered with the event payload as input, and a new `AiAutomationRun` record is created with `triggeredBy: 'event'`

4. **AC-4: Sequential step execution with I/O piping** — GIVEN an automation with multiple ordered steps WHEN it executes THEN each step runs sequentially (by `stepOrder`), with the output of step N resolved and piped as input to step N+1 via the `inputConfig`/`outputConfig` variable bindings

5. **AC-5: Goal-oriented agent execution** — GIVEN an automation step with an assigned agent WHEN the step executes THEN the agent runs autonomously using `DynamicContextService.assembleAutonomous()` (module knowledge + skill instructions + step input — no user memories or screen context), multi-turn with tools up to `maxTurns`, and produces structured JSON output

6. **AC-6: Automation chaining** — GIVEN an automation with `chainNextId` WHEN it completes successfully THEN the chained automation is triggered with the output of the current automation as input, and the chained run's `triggeredBy` is set to `'chain'`

7. **AC-7: Circular chain detection** — GIVEN an automation with `chainNextId` WHEN saving via API THEN the system validates that no circular chain exists (traverses the chain graph using `chainNextId` pointers and rejects with a 422 validation error if a cycle is detected). Maximum chain depth: 10. If depth exceeds 10, reject with 422

8. **AC-8: Immutable run records** — GIVEN an automation run WHEN it completes (success or failure) THEN an immutable `ai_automation_runs` record is created with `status`, `startedAt`, `completedAt`, `totalTokens`, `totalCost`, `result` (final step output) or `error`. The record has no `updatedAt` (append-only)

9. **AC-9: Immutable step run records** — GIVEN an automation step run WHEN it completes THEN an immutable `ai_automation_step_runs` record is created with step-level metrics: `agentId`, `modelId`, `inputTokens`, `outputTokens`, `latencyMs`, `turns`, `input` (resolved), `output`, `status`. Append-only (no `updatedAt`)

10. **AC-10: Prompt variable resolution (DB_FIELD)** — GIVEN a prompt variable with source `DB_FIELD` WHEN resolved THEN the variable value is fetched from the specified database table and field, scoped by `companyId`, supporting relation traversal (e.g., `customer.primaryContact.email`)

11. **AC-11: AUTONOMOUS mode context assembly** — GIVEN the automation's context assembly WHEN an agent step executes THEN it uses E5b's `DynamicContextService.assembleAutonomous()` in AUTONOMOUS mode with ~3000 token budget (base ~500 + knowledge ~500 + skill instructions ~1000 + automation input data ~1000). No user memories. No screen context

12. **AC-12: Mandatory parameter validation** — GIVEN a tool call (query or action) with required parameters defined in its `inputSchema` WHEN the AI proposes calling that tool AND any required parameter is missing THEN the orchestrator MUST NOT execute the tool — instead it returns a clarification request listing all missing required fields with their descriptions (Project Context §19)

13. **AC-13: Optional parameter passthrough** — GIVEN a tool call where all required parameters are present AND some optional parameters are omitted THEN the orchestrator executes the tool normally — optional parameters use service-layer defaults

14. **AC-14: Nested schema validation** — GIVEN a tool with nested required parameters (e.g., `conditions[].operator`, `lines[].quantity`) WHEN the AI proposes a call with array items missing nested required fields THEN the orchestrator blocks execution and requests the missing nested fields

15. **AC-15: AUTONOMOUS mode param failure** — GIVEN an automation step (AUTONOMOUS mode) where a required tool parameter cannot be resolved from the variable binding system THEN the step FAILS with status `FAILED` and error `UNRESOLVABLE_REQUIRED_PARAM: {paramName}` — the step must never proceed with partial data

16. **AC-16: Batch parameter gathering** — GIVEN multiple required parameters are missing from a proposed tool call WHEN the orchestrator prompts the user (interactive mode) THEN it asks for ALL missing values in a single prompt (batch gathering), not one at a time

17. **AC-17: Token budget enforcement** — GIVEN an automation with `maxTokenBudget` (default 50,000) WHEN the cumulative token usage across all steps exceeds the budget THEN the run is terminated gracefully with status `CANCELLED` and error `TOKEN_BUDGET_EXCEEDED`

18. **AC-18: Duration budget enforcement** — GIVEN an automation with `maxDurationMs` (default 300,000ms / 5 min) WHEN the run duration exceeds the budget THEN the run is terminated gracefully with status `CANCELLED` and error `DURATION_BUDGET_EXCEEDED`

19. **AC-19: Circuit breaker** — GIVEN an automation that has failed 3 consecutive times WHEN the circuit breaker triggers THEN the automation's schedule is auto-paused (`isPaused = true`), an `ai.automation.paused` event is emitted, and admins are notified

20. **AC-20: Events emitted** — GIVEN automation lifecycle transitions WHEN they occur THEN the corresponding typed events are emitted via the event bus: `ai.automation.triggered` (on start), `ai.automation.completed` (on success), `ai.automation.failed` (on failure), `ai.automation.paused` (on circuit breaker trigger)

21. **AC-21: Automation CRUD API** — GIVEN the automation CRUD endpoints WHEN called by an authenticated admin THEN automations can be created, listed (with pagination + filters), retrieved, updated, and soft-deleted. All operations scoped by `companyId`

22. **AC-22: Run history API** — GIVEN the run history endpoints WHEN called THEN automation runs can be listed (paginated, filterable by date range and status), and individual runs can be retrieved with their step runs. Results load within 500ms (NFR2)

### Task 4: Implement automation execution engine (AC: #4, #5, #8, #9, #11, #17, #18) ✅

- [x] 4.1 Create `apps/api/src/ai/automation/automation-executor.ts`:
  - `AutomationExecutor` class — main execution engine
  - `execute(automationId: string, input?: Json, triggeredBy: string)`:
    1. Load automation definition with steps (ordered by `stepOrder`)
    2. Create `AiAutomationRun` record with status `PENDING`, then update to `RUNNING`
    3. For each step (sequentially):
       a. Resolve input variables (from DB, previous step output, constants, system vars)
       b. Build agent context using `DynamicContextService.assembleAutonomous()` — pass `moduleKey` from agent config, `inputData` from resolved variables
       c. Execute agent autonomously via orchestrator (multi-turn, up to `step.maxTurns`)
       d. Capture structured output
       e. Create `AiAutomationStepRun` record (append-only)
       f. Feed output to next step's input config
    4. On success: update run status to `COMPLETED`, store `result` (final step output)
    5. On failure: update run status to `FAILED`, store `error`
    6. Emit appropriate event (`ai.automation.completed` or `ai.automation.failed`)
    7. If `chainNextId`: trigger next automation (AC-6)
    8. If `notificationConfig`: send notification to configured users/roles

- [x] 4.2 Implement token budget enforcement:
  - Track cumulative `inputTokens + outputTokens` across all step runs
  - Before each step: check if remaining budget allows execution
  - If exceeded mid-step: terminate gracefully, status `CANCELLED`, error `TOKEN_BUDGET_EXCEEDED`
  - Compare against `automation.maxTokenBudget` (default 50,000)

- [x] 4.3 Implement duration budget enforcement:
  - Track elapsed time from run `startedAt`
  - Before each step: check if remaining time allows execution
  - If exceeded: terminate gracefully, status `CANCELLED`, error `DURATION_BUDGET_EXCEEDED`
  - Compare against `automation.maxDurationMs` (default 300,000ms)

- [x] 4.4 Implement AUTONOMOUS mode agent execution:
  - Create `apps/api/src/ai/automation/autonomous-agent-executor.ts`
  - Uses existing `DynamicContextService.assembleAutonomous()` for context (~3000 tokens: base + module knowledge + skill instructions + input data — NO user memories, NO screen context)
  - Multi-turn execution loop: send goal + input → receive response → if tool calls, validate params (Task 5), execute tools → loop until agent signals completion or `maxTurns` reached
  - Capture final structured JSON output for next step piping
  - On tool call: use existing `QueryExecutor` (reads) and `ActionExecutor` (writes) — but in AUTONOMOUS mode, writes that require user approval must FAIL (no user to approve)

- [x] 8.2 Implement circular chain detection in automation CRUD service:

### Key Design Decisions

1. **Append-only run/step-run records**: `AiAutomationRun` and `AiAutomationStepRun` have NO `updatedAt` field. Status transitions create new records or use a single initial insert with status updates via raw SQL `UPDATE` (since Prisma `update()` requires `updatedAt` on `@updatedAt` fields). Since these don't have `@updatedAt`, standard Prisma `update()` works fine for status changes within the same record.

2. **AUTONOMOUS vs INTERACTIVE mode**: The param validator handles both modes. Interactive: prompts user for missing params. Autonomous: fails the step. The `AutomationExecutor` always runs in AUTONOMOUS mode.

3. **Self-referential FK pattern for chaining**: `AiAutomation` has both `chainFromId` and `chainNextId` pointing to itself. Use named Prisma relations: `chainFrom AiAutomation? @relation("AutomationChainFrom", ...)` and `chainNext AiAutomation? @relation("AutomationChainNext", ...)`.

4. **companyId scoping**: `AiAutomation` is the only automation table with a direct `companyId` column. All queries on runs/step-runs must join through the automation to enforce tenant isolation.

5. **Financial write guardrails in AUTONOMOUS mode**: Since there is no user to approve financial writes (per IMP-005), any tool call that triggers `ActionExecutor` with a financial write operation must FAIL the step with an appropriate error. The guardrail chain (Architecture §6.9) handles this.
