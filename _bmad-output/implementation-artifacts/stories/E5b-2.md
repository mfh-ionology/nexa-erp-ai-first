# Story E5b-2: Skills Registry, Tool Framework & Dynamic Context Assembly

**Epic:** E5b — AI Co-Pilot Intelligence — Memory, Skills & Dynamic Context
**Status:** done
**Dependencies:** E5 (AI Orchestration — complete), E5b-1 (Memory Storage & Retrieval API — complete)
**Complexity:** X-Large (14 tasks)

---

## User Story

As a developer, I want a database-driven skills registry, tool execution framework, module knowledge base, entity trigger registry, and dynamic context assembler so that the AI Co-Pilot automatically gains capabilities, domain understanding, and entity awareness as modules are added, without requiring code changes to the AI system.

---

## Acceptance Criteria

### AC1: Enhanced AiSkill Prisma Model (Schema & Migration)
**GIVEN** the enhanced AiSkill model
**WHEN** migration runs
**THEN** the existing `ai_skills` table gains new columns: `module_key`, `pack_key`, `trigger_phrases[]`, `negative_triggers[]`, `context_required[]`, `orchestration_pattern`, `parameters` (Json), `examples` (Json), `priority`, `version`
**AND** E5's `instructions` column is renamed to `skill_content`
**AND** E5's `keywords` column is renamed to `trigger_phrases`
**AND** skills remain system-wide (NO companyId) — tenant overrides use `ai_skill_overrides`

### AC2: AiSkillContext Model
**GIVEN** the new AiSkillContext model
**WHEN** migration runs
**THEN** the `ai_skill_contexts` table is created with `skill_id` (FK), `context_key`, `context_query` (Text), `token_budget`, `cache_ttl_seconds`, and `is_required`

### AC3: AiModuleKnowledge Model
**GIVEN** the new AiModuleKnowledge model
**WHEN** migration runs
**THEN** the `ai_module_knowledge` table is created with `module_key`, `knowledge_type` (OVERVIEW, ENTITIES, WORKFLOWS, BUSINESS_RULES, FAQ, TERMINOLOGY), `title`, `content` (Text), `priority`, `is_active`
**AND** this table is system-wide (no companyId) — module knowledge is the same for all tenants

### AC4: AiSkillOverride Model
**GIVEN** the new AiSkillOverride model
**WHEN** migration runs
**THEN** the `ai_skill_overrides` table is created with `skill_id` (FK), `company_id` (FK), `is_active` (override enable/disable), `trigger_phrases_override` (String[]?), `priority_override` (Int?)
**AND** a unique constraint exists on `[skill_id, company_id]`

### AC5: AiEntityTrigger Model
**GIVEN** the new AiEntityTrigger model
**WHEN** migration runs
**THEN** the `ai_entity_triggers` table is created with `module_key`, `trigger_word`, `entity_type`, `search_endpoint`, `display_field`, `subtitle_field` (optional), `scope_by` (optional), `icon` (optional), `priority`, `is_active`
**AND** the table is system-wide (no companyId)
**AND** a unique constraint exists on `[module_key, trigger_word]`

### AC6: Meta-Router (L0) Intent Classification
**GIVEN** the meta-router (L0)
**WHEN** a user message arrives
**THEN** the system loads all active module pack summaries (~200 tokens) and classifies the intent to the correct module

### AC7: Module Pack (L1) Skill Selection
**GIVEN** the module pack (L1)
**WHEN** a module is identified
**THEN** all skills for that module are loaded with trigger phrases (~500 tokens)
**AND** the module's OVERVIEW knowledge is loaded (~200 tokens)
**AND** the best-matching skill is selected using trigger phrase matching + negative trigger exclusion
**AND** tenant skill overrides are applied (disabled skills excluded, custom trigger phrases used)

### AC8: Skill Activation (L2) Full Loading
**GIVEN** a selected skill (L2)
**WHEN** the skill is activated
**THEN** its full `skill_content` (instructions), `parameters`, and `examples` are loaded (~300 tokens)
**AND** relevant entity/business rule knowledge for the skill's context is loaded (~300 tokens)

### AC9: QueryExecutor — READ Tool Execution
**GIVEN** the QueryExecutor service
**WHEN** a READ tool is called by the AI (e.g., `get_aging_report`)
**THEN** the executor: (a) looks up the registered handler by tool name, (b) enforces companyId scoping on all queries, (c) checks RBAC permission (user must have read access to the module), (d) executes the handler and returns structured JSON, (e) truncates results if they exceed the tool's token budget (default: 2000 tokens)
**AND** no user confirmation is required for reads

### AC10: ActionExecutor — WRITE Tool Execution (E5 Extension)
**GIVEN** the ActionExecutor service (from E5)
**WHEN** a WRITE tool is called by the AI (e.g., `create_invoice`)
**THEN** the existing E5 flow applies: ActionPlanner stages proposal → guardrails check → user confirmation required → ActionExecutor executes in transaction

### AC11: Tool Registration at Module Startup
**GIVEN** a module's tool definitions
**WHEN** the module plugin initialises
**THEN** it registers BOTH query handlers (`queryExecutor.registerHandler('get_aging_report', handler)`) AND action handlers (`actionExecutor.registerHandler('CREATE_INVOICE', handler)`) using the registry pattern
**AND** tool definitions (name, description, inputSchema) are stored in the `@nexa/ai-tools` package

### AC12: Skill-to-Tool Resolution
**GIVEN** the skill routing chain
**WHEN** a skill is activated (L2)
**THEN** the skill's `requiredTools` are resolved from the tool registry and passed to the AI Gateway as the `tools` array for that turn
**AND** only the tools the active skill needs are sent — not all registered tools

### AC13: Dynamic Context Assembly — INTERACTIVE Mode
**GIVEN** the dynamic context assembler in INTERACTIVE mode (chat sessions)
**WHEN** building a session's system prompt
**THEN** it assembles: base system prompt + user memories (≤2000 tokens) + active skill chain L0→L1→L2 (≤1000 tokens) + module knowledge (≤500 tokens) + user permissions + current screen context
**AND** the total is within a budget of ~5000 tokens

### AC14: Dynamic Context Assembly — AUTONOMOUS Mode
**GIVEN** the dynamic context assembler in AUTONOMOUS mode (E5c automations)
**WHEN** building an agent's context
**THEN** it assembles: base system prompt + module knowledge (≤500 tokens) + skill instructions (≤1000 tokens) + automation-specific input data (from step config)
**AND** NO user memories and NO screen context are included (no user session)
**AND** the total budget is ~3000 tokens

### AC15: Module Seed Script Pattern
**GIVEN** a seed script for a module
**WHEN** the module's migration runs
**THEN** the skill pack, module knowledge, tool definitions, AND entity triggers are all automatically registered

### AC16: Skills API Endpoint
**GIVEN** the `GET /ai/skills` endpoint
**WHEN** called with `?moduleKey=ar`
**THEN** only skills for that module are returned, ordered by priority, with any tenant overrides applied

### AC17: Knowledge API Endpoint
**GIVEN** the `GET /ai/knowledge?moduleKey=ar&type=ENTITIES` endpoint
**WHEN** called
**THEN** it returns structured knowledge about that module's entities including fields, status flows, and business rules

### AC18: Entity Triggers API Endpoint
**GIVEN** the `GET /ai/entity-triggers?moduleKey=ar` endpoint
**WHEN** called
**THEN** it returns all entity triggers for that module, used by the frontend chat textbox for autocomplete

  1. Look up registered handler by tool name — return `TOOL_NOT_FOUND` error if not registered
  2. Enforce companyId scoping — pass companyId into the handler (handler must scope all queries)
  3. Check RBAC permission via `permissionService.hasModuleAccess(userId, companyId, moduleKey, 'read')` — return `PERMISSION_DENIED` error if denied

- [x] 3.5 Add graceful error handling — tool execution errors are caught and returned as structured error results (never throw)

**Implementation notes:**
- No user confirmation required for reads (unlike ActionExecutor which requires confirmation for writes)
- The QueryExecutor mirrors ActionExecutor's pattern but is simpler: no guardrails chain, no transaction wrapping
- The `moduleKey` for RBAC lookup is resolved from the ToolRegistry's `ToolDefinition.moduleKey`
- The existing `ActionExecutor` handles WRITE tools — this story does NOT modify ActionExecutor

### Task 5: Skill Router — L1 Module Pack Loader
**AC:** #7
**Files:**

**Subtasks:**
- [x] 5.1 Implement `loadModulePack()` (L1):
  - Load all `ai_skills` where `moduleKey = param AND isActive = true`, ordered by priority DESC
  - Load all `ai_skill_overrides` where `companyId = param AND skillId IN (loaded skills)`
  - Merge overrides: if override `isActive === false`, exclude skill; if `triggerPhrasesOverride` is non-empty, replace skill's trigger phrases; if `priorityOverride` is set, override priority

- [x] 5.2 Implement `selectSkill()` from a loaded module pack:
  - Score each skill against the user message using trigger phrase matching
  - For each skill: compute a match score by counting how many trigger phrases appear in the user message (case-insensitive, normalised)
  - Apply negative trigger exclusion: if any negative trigger phrase matches, set score to 0
  - Return the highest-scoring skill, or null if no skill scores above threshold (0.3)
- [x] 5.3 Define types:
  ```typescript
  interface ModulePack {
    moduleKey: string;
    skills: SkillSummary[];
    overview: string;
    tokenCount: number;
  }

  interface SkillSummary {
    id: string;
    name: string;
    triggerPhrases: string[];
    negativeTriggers: string[];
    contextRequired: string[];
    priority: number;
  }

  interface SelectedSkill {
    id: string;
    name: string;
    moduleKey: string;
    confidence: number;
  }
  ```

**Implementation notes:**
- Trigger phrase matching uses normalised word overlap: each trigger phrase is split into words, and we count how many of those words appear in the user message. The score is `matchedWords / totalWords` averaged across matched trigger phrases.
- Negative triggers use exact phrase matching (substring check) — if the user message contains the negative trigger phrase, the skill is excluded.
- Tenant overrides are sparse — most skills will have no override for a given company.

  1. Start with base system prompt (~500 tokens)
  2. Inject module knowledge for the specified module (≤500 tokens)
  3. Inject skill instructions for the specified skill (≤1000 tokens)
  4. Inject automation-specific input data from step config (≤1000 tokens)
  5. NO user memories, NO screen context (no user session)
  6. Validate total doesn't exceed ~3000 tokens
  7. Return `AssembledContext`

## Dev Notes

1. **Migration Column Rename**: The `instructions`→`skill_content` and `keywords`→`trigger_phrases` renames MUST use raw SQL `ALTER TABLE ai_skills RENAME COLUMN` in the migration. Prisma's default behaviour is to drop and recreate columns, which destroys data. Use `prisma migrate dev --create-only`, edit the SQL, then apply.

2. **Backwards Compatibility**: The orchestrator changes in Task 9 are optional — if `DynamicContextService` is null, the orchestrator works exactly as before. This is critical for testing and staged rollout.

3. **Trigger Phrase Matching**: L1 skill selection uses deterministic keyword matching, NOT an LLM call. This keeps latency <50ms for skill routing. The LLM is only called once (after context is assembled) for the actual response generation.

4. **Token Budget**: Strictly enforce the token budgets documented in the epic: memories ≤2000, skills ≤1000, knowledge ≤500, total ≤5000 (interactive) / ≤3000 (autonomous). Over-budget context wastes money and can degrade response quality.

5. **Graceful Degradation (IMP-006)**: Every new service must handle failures gracefully. If the SkillRouter fails, fall back to the generic chat-router agent. If the QueryExecutor fails, return a structured error. If module knowledge is missing, proceed without it. Never crash the orchestrator.

6. **Existing E5 ActionExecutor**: This story does NOT modify the `ActionExecutor` class. WRITE tools continue to use the existing E5 action framework. The new `QueryExecutor` is only for READ tools.

7. **Seed Pattern for Future Epics**: The seed templates created in Task 13 document the "4-seed pattern" from project-context.md §14b. Every future business module epic (E14 Finance, E17 AR, etc.) will follow this pattern to register their AI capabilities.

8. **No UI in This Story**: E5b-2 is entirely backend. The management UI for skills, knowledge, and entity triggers is in E5b-5. The inline entity mention UI is in E5b-7.
