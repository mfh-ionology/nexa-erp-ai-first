# Story E5d-1: Knowledge Base Schema & RAG Pipeline

**Epic:** E5d — AI Knowledge Evolution & Cross-Tenant Intelligence
**Status:** done
**Dependencies:** E5b-4 (Hybrid Search & Memory Intelligence — done), E5 (AI Orchestration — done), E3b (Platform API — done)
**Complexity:** X-Large (10 tasks)
**FRs:** FR4 (contextual AI), FR6 (AI-powered analytics) | **NFRs:** NFR2 (CRUD <500ms)

---

## User Story

As a **developer**,
I want a per-tenant knowledge base with document upload, chunking, embedding, and RAG retrieval so that the AI can answer questions using tenant-specific knowledge,
so that each tenant's AI Co-Pilot can draw on their company's documented processes, terminology, and rules — becoming more useful and accurate over time.

---

## Acceptance Criteria

### AC1: Tenant Knowledge Schema Migration
**GIVEN** the Prisma schema and migration
**WHEN** `prisma migrate dev` runs
**THEN** 5 tenant-side tables are created:
  - `ai_knowledge_articles` — tenant knowledge documents with category, source, confidence score
  - `ai_knowledge_chunks` — chunked fragments with individual embeddings for RAG retrieval
  - `ai_training_examples` — curated input/output pairs for few-shot guidance
  - `ai_correction_log` — immutable log of user corrections to AI responses
  - `ai_learning_signals` — aggregated per-skill learning metrics
**AND** each table includes `company_id` foreign key (multi-company scoping per project-context.md §1)
**AND** `ai_knowledge_chunks` has an `embedding vector(1536)` column with HNSW index (reusing pgvector extension from E5b-4)
**AND** `ai_knowledge_chunks` has a `search_vector tsvector` column with GIN index and auto-update trigger (matching E5b-4's pattern)
**AND** composite indexes are created: `(company_id, category, is_active)`, `(company_id, source)`, `(company_id, correction_type)`, `(company_id, skill_key)`, etc. per the epic's proposed Prisma models
**AND** unique constraints are created: `(article_id, chunk_index)` on chunks, `(company_id, skill_key, signal_date)` on learning signals
**AND** the migration uses `prisma migrate dev --create-only` first, raw SQL appended for vector/tsvector columns, then applied (per Prisma Migration Rules)

### AC2: Document Upload & Processing Pipeline
**GIVEN** a `POST /ai/knowledge` endpoint
**WHEN** called with a document (text, markdown, or PDF content)
**THEN** the document is stored as an `AiKnowledgeArticle` with:
  - `source` = `ADMIN_UPLOADED`
  - `confidenceScore` = 1.0 (admin-uploaded articles are highest confidence)
  - `isConfirmed` = true
  - `category` = one of: `BUSINESS_PROCESS`, `TERMINOLOGY`, `INDUSTRY_RULES`, `CUSTOM_FIELDS`, `HISTORICAL_PATTERN`
**AND** the document is chunked via the ChunkingService (target: ~500 tokens per chunk, ~50 token overlap)
**AND** each chunk is embedded using E5b's shared `EmbeddingService` (via AI Gateway, batch-capable — do NOT create a separate embedding service)
**AND** chunks are stored in `ai_knowledge_chunks` with their embeddings
**AND** the full pipeline (store + chunk + embed + index) completes within 5 seconds for documents under 10 pages
**AND** `companyId` scoping is enforced — articles are only created for the authenticated user's active company
**AND** events `ai.knowledge.articleCreated` are emitted per event catalog §17

### AC3: Document Chunking Service
**GIVEN** a document's text content
**WHEN** the ChunkingService processes it
**THEN** it uses recursive text splitting with configurable parameters:
  - Target chunk size: ~500 tokens (~2000 characters at 4 chars/token)
  - Overlap: ~50 tokens (~200 characters)
  - Split boundaries: paragraph > sentence > word (never mid-word)
**AND** each chunk's `tokenCount` is stored (estimated at 4 chars/token, matching `DynamicContextService.estimateTokens()`)
**AND** markdown headers are preserved at chunk boundaries where possible (a chunk inherits its parent heading)
**AND** the chunking is deterministic — same input always produces same chunks
**AND** empty or whitespace-only chunks are discarded

### AC4: RAG Retrieval with Confidence-Weighted Re-Ranking
**GIVEN** a user's AI query during a chat session
**WHEN** the RAG pipeline runs (via `KnowledgeRagService`)
**THEN** it performs vector similarity search using E5b's shared `VectorSearchService.similaritySearch()` — do NOT create a separate vector search implementation
**AND** it retrieves the top-K most relevant chunks (K=5, configurable)
**AND** retrieved chunks are re-ranked by confidence-weighted scoring:
  - `finalScore = similarity * confidenceWeight`
  - Confidence weights by source: `ADMIN_UPLOADED` = 1.0, `PLATFORM_SUGGESTED` (accepted) = 0.9, `AI_GENERATED` (confirmed) = 0.8, `AI_GENERATED` (unconfirmed) = 0.5
**AND** the re-ranked chunks are injected into the AI context within a ~1000 token budget
**AND** the RAG injection is formatted as:
  ```
  <tenant_knowledge>
  ## Relevant Knowledge for This Query
  - [CATEGORY] Title: chunk content...
  </tenant_knowledge>
  ```
**AND** if no relevant chunks are found (all below similarity threshold 0.5), no knowledge section is injected (graceful absence)
**AND** the search is scoped by `company_id` — a tenant NEVER sees another tenant's knowledge (R-002 mitigation)

### AC5: Knowledge Usage Tracking
**GIVEN** a knowledge article's chunk is retrieved and used in an AI response
**WHEN** the RAG pipeline returns results
**THEN** the parent article's `lastUsedAt` is updated to the current timestamp
**AND** the parent article's `usageCount` is incremented by 1
**AND** the event `ai.knowledge.articleUsed` is emitted per event catalog §17
**AND** usage tracking is fire-and-forget — it does not block the AI response (non-critical side effect)

### AC6: Integration with DynamicContextService
**GIVEN** the existing `DynamicContextService.assembleInteractive()` method
**WHEN** a user chat session assembles context
**THEN** a new step is added between memories (step 2) and skill routing (step 3):
  - Step 2.5: Knowledge RAG retrieval (~1000 tokens)
  - Calls `KnowledgeRagService.retrieveRelevantKnowledge(userMessage, companyId)`
  - Injects the `<tenant_knowledge>` block into the system prompt
**AND** the `BUDGET_INTERACTIVE.knowledge` is increased from 500 to 1000 tokens (matching epic spec: ~1000 tokens for knowledge)
**AND** the `tokenBreakdown` correctly reports knowledge tokens separately from skill/module knowledge
**AND** the existing module knowledge (`aiModuleKnowledge` table from E5b) continues to work via skill routing — the new tenant knowledge is an additional, separate section
**AND** if RAG retrieval fails, the context assembly continues without knowledge (graceful degradation per IMP-006)

### AC7: Knowledge CRUD API
**GIVEN** the REST API endpoints under `/ai/knowledge-articles`
**WHEN** authenticated users call them
**THEN** the following endpoints are available:
  - `GET /ai/knowledge-articles` — list articles for current company, filtered by `category`, `source`, `isActive`, ordered by category + confidence score (permission: `ai.chat` view)
  - `GET /ai/knowledge-articles/:id` — get single article with chunk count (permission: `ai.chat` view)
  - `POST /ai/knowledge-articles` — create article + trigger chunking/embedding pipeline (permission: ADMIN role)
  - `PATCH /ai/knowledge-articles/:id` — update article content/metadata; if content changes, re-chunk and re-embed (permission: ADMIN role)
  - `DELETE /ai/knowledge-articles/:id` — soft-delete article (set `isActive = false`); cascading soft-delete of chunks (permission: ADMIN role)
**AND** all endpoints enforce `companyId` scoping — only the current company's articles are accessible
**AND** cross-tenant access attempts return 404 (not 403, to avoid information leakage)
**AND** the existing `knowledge.routes.ts` (E5b module knowledge) is NOT modified — the new endpoints use a different path prefix (`/ai/knowledge-articles` vs `/ai/knowledge`)

### AC8: Knowledge Article Categories
**GIVEN** the category system for knowledge articles
**WHEN** articles are created or filtered
**THEN** the following categories are supported:
  - `BUSINESS_PROCESS` — documented SOPs, workflows, approval chains
  - `TERMINOLOGY` — company-specific terms, abbreviations, jargon
  - `INDUSTRY_RULES` — sector-specific regulations, conventions, standards
  - `CUSTOM_FIELDS` — meanings and usage rules for custom/configurable fields
  - `HISTORICAL_PATTERN` — recurring patterns detected from historical data
**AND** categories are validated on create/update (must be one of the defined values)
**AND** the `GET /ai/knowledge-articles` endpoint supports filtering by one or multiple categories

### AC9: Knowledge Article Sources
**GIVEN** the source tracking system for knowledge articles
**WHEN** articles are created
**THEN** the following sources are tracked:
  - `ADMIN_UPLOADED` — manually uploaded by a tenant admin (confidence: 1.0)
  - `AI_GENERATED` — auto-generated by the correction loop (E5d-2, confidence: 0.5 unconfirmed, 0.8 confirmed)
  - `PLATFORM_SUGGESTED` — pushed from the platform knowledge base (E5d-4, confidence: 0.9 when accepted)
  - `CORRECTION_DERIVED` — directly created from a specific correction (E5d-2, confidence: 0.8)
**AND** `sourceRef` stores the originating ID (platform article ID, correction log ID, etc.) for traceability
**AND** source is immutable after creation — it cannot be changed via PATCH

### AC10: Seed Data for Default Knowledge Categories
**GIVEN** the automation seed or tenant provisioning
**WHEN** a new tenant is created or E5d migrations are applied to an existing tenant
**THEN** no default knowledge articles are seeded (knowledge is tenant-specific and starts empty)
**BUT** the knowledge category enum values are documented and validated in the schema
**AND** the seed script verifies that the 5 tables exist and are queryable

### Task 5: Knowledge RAG Retrieval Service (AC: #4) ✅

- [x] 5.1 Create `apps/api/src/ai/knowledge-rag.service.ts` with:
  - Constructor receives: `PrismaClient`, `Logger`, `VectorSearchService` (from E5b-4), `EmbeddingService` (from E5b-4), `KnowledgeArticleService`
  - Reuses E5b's `VectorSearchService.similaritySearch()` — do NOT create separate vector search logic

- [x] 5.2 Implement `retrieveRelevantKnowledge(query: string, companyId: string, opts?: RagOptions): Promise<RagResult>`:
  - `RagOptions`: `{ topK?: number, tokenBudget?: number, minSimilarity?: number }`
  - Defaults: `topK = 5`, `tokenBudget = 1000`, `minSimilarity = 0.5`
  - Algorithm:
    1. Generate query embedding via `EmbeddingService.generateEmbedding(query)`
    2. Call `VectorSearchService.similaritySearch(queryEmbedding, 'ai_knowledge_chunks', { company_id: companyId }, topK * 2, minSimilarity)` — fetch extra for re-ranking
    3. Join chunk results back to parent articles (to get `confidenceScore`, `source`, `category`)
    4. Apply confidence-weighted re-ranking: `finalScore = similarity * confidenceWeight`
    5. Select top-K chunks that fit within `tokenBudget`
    6. Fire-and-forget: call `KnowledgeArticleService.trackUsage()` for used articles
  - Returns `RagResult`: `{ chunks: RetrievedChunk[], totalTokens: number, articleIds: string[] }`

- [x] 5.3 Implement confidence weight mapping:
  ```typescript
  function getConfidenceWeight(source: string, isConfirmed: boolean): number {
    switch (source) {
      case 'ADMIN_UPLOADED': return 1.0;
      case 'PLATFORM_SUGGESTED': return 0.9; // accepted from vendor
      case 'AI_GENERATED': return isConfirmed ? 0.8 : 0.5;
      case 'CORRECTION_DERIVED': return 0.8;
      default: return 0.5;
    }
  }
  ```

- [x] 5.4 Implement context formatting:
  - Format retrieved chunks as XML-tagged knowledge block:
    ```
    <tenant_knowledge>
    ## Relevant Knowledge for This Query
    - [TERMINOLOGY] "VAT code 3" means reverse charge for EU purchases
    - [BUSINESS_PROCESS] PO approval for >£5000 requires Finance Director
    </tenant_knowledge>
    ```
  - Truncate to stay within token budget
  - Return empty string if no relevant chunks found

- [x] 5.5 Handle graceful degradation:
  - If embedding generation fails: fall back to BM25-only via `VectorSearchService.keywordSearch()` on `ai_knowledge_chunks`
  - If VectorSearchService is unavailable: return empty result (no knowledge injected)
  - If no chunks match: return empty result
  - Never throw — always return a valid `RagResult` (may be empty)
