# Story E2b.1: Admin can view the system resource registry

Status: done

## Story

As an **administrator**,
I want to view all controllable pages, reports, settings, and maintenances registered in the system,
So that I can understand what resources are available for permission configuration.

## Acceptance Criteria

1. GIVEN an administrator is authenticated WHEN they request GET /system/resources THEN they receive a list of all resources with code, name, module, type, sortOrder, parentCode, and isActive AND resources can be filtered by module, type, search, and isActive
2. GIVEN a new database is seeded WHEN the seed script runs THEN all resources defined in company-defaults.json are upserted into the Resource table AND each resource has a unique dot-notation code (e.g., `system.users.list`)
3. GIVEN a non-admin user WHEN they request GET /system/resources THEN they receive a 403 Forbidden response

## Dev Notes

### Critical Architecture: Resource Table is GLOBAL (No companyId)

The Resource table is one of the very few exceptions to the "companyId on every table" rule from project-context.md §1. Resources represent system-wide metadata (all pages, reports, settings, maintenances that CAN be permissioned). They are shared across all companies in a tenant database.

**DO NOT add `companyId` to the Resource model or any Resource query.** This is explicitly specified in:
- Data Models §3.1: "Resource table: global (not company-scoped), unique code"
- Architecture §3b: "Resource registry table as the single source of truth"

### API Contract Compliance (api-contracts.md §3.12)

```
GET /system/resources
Auth: ADMIN
FR Ref: FR81, FR227

Query Parameters:
  module: string — Filter by module (e.g., "sales", "finance", "system")
  type: ResourceType — Filter by type: PAGE, REPORT, SETTING, MAINTENANCE
  search: string — Full-text search across code, name, description
  isActive: boolean — Filter active/inactive (default: true)

Response 200:
{
  "success": true,
  "data": [
    {
      "id": "uuid",
      "code": "sales.orders.list",
      "name": "Sales Orders",
      "module": "sales",
      "type": "PAGE",
      "parentCode": null,
      "sortOrder": 100,
      "icon": null,
      "description": "Sales order list view",
      "isActive": true
    }
  ],
  "meta": { "total": 42 }
}
```

### Resource Code Convention

Dot-notation hierarchy: `{module}.{entity}.{view}` where:
- module: `system`, `finance`, `sales`, `purchasing`, `inventory`, `crm`, `hr`, `manufacturing`, `reporting`
- entity: `users`, `orders`, `invoices`, `access-groups`, etc.
- view: `list`, `detail`, `post`, `approve` (maps to ResourceType — mostly PAGE)

Examples: `system.users.list`, `system.users.detail`, `system.company-profile.detail`, `finance.journal-entries.list`

### Existing Code Patterns to Follow

**Route Plugin Pattern** (from `apps/api/src/modules/system/user.routes.ts`):
```typescript
import { type FastifyInstance } from 'fastify';
import fp from 'fastify-plugin';
import { createRbacGuard } from '../../core/rbac/index.js';
import { UserRole } from '@nexa/db';
import { sendSuccess } from '../../core/utils/response.js';

async function resourceRoutes(fastify: FastifyInstance): Promise<void> {
  fastify.get('/resources', {
    onRequest: createRbacGuard({ minimumRole: UserRole.ADMIN }),
    schema: { querystring: listResourcesQuerySchema },
  }, async (request, reply) => {
    const result = await listResources(prisma, request.query);
    sendSuccess(reply, result.data, result.meta);
  });
}

export const resourceRoutesPlugin = fp(resourceRoutes, { name: 'resource-routes' });
```

- Set `createdBy`/`updatedBy` from `ctx.userId` on mutations

### Previous Implementation Learnings (from reverted E2b)

The previous E2b implementation (since reverted) provided these learnings:
1. **In-memory permission cache with Map works for MVP** — Redis interface planned for scale, but Map is sufficient
2. **Fastify 5 requires getter/setter for reference-type request decorators** — if decorating `request` with objects, use `{ getter() { return val; }, setter(v) { val = v; } }` pattern
3. **Zod `.default([])` makes arrays always truthy** — no need for null checks when using default empty arrays

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: NOT applicable for Resource table (global). BUT all other E2b tables (AccessGroup, etc.) in later stories WILL require companyId
- **i18n**: Error messages should use translation key pattern. For MVP, use hardcoded English strings but structure them as keys (e.g., `errors.forbidden`, `errors.not_found`)
- **Audit**: Resource upserts during seeding don't need audit events (system operation). User-facing mutations in later stories WILL need event emission
- **Attachments/Notes/Tasks**: Not applicable for Resource entity (system metadata)
