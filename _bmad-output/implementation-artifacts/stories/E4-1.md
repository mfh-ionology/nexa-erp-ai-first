# Story E4.1: Translation Key System

Status: done

## Story

As a **developer**,
I want a centralised translation key system with `t()` helper and English locale files,
so that all user-facing strings are internationalisation-ready from day one.

## Acceptance Criteria

1. GIVEN the web application is loaded WHEN any UI component renders a user-facing string THEN it uses the `t('namespace.key')` helper, never a hardcoded string
2. GIVEN an English locale file exists at `packages/i18n/locales/en/` WHEN a translation key is resolved THEN the English text is returned
3. GIVEN a translation key that does not exist in the current locale WHEN the `t()` helper is called THEN it falls back through the chain: user locale -> company locale -> `en`, and logs a missing-key warning in development mode
4. GIVEN the React application WHEN `t()` is called with interpolation parameters THEN variables are substituted correctly (e.g., `t('validation.required', { field: t('field.customerName') })` returns "Customer Name is required")
5. GIVEN a developer adds a new UI component WHEN they use a hardcoded English string instead of `t()` THEN the ESLint rule `no-raw-text` flags it as an error

## Dev Notes

### Architecture Context

This is a **Tier 1 infrastructure story** that creates the foundational i18n system all subsequent epics depend on. E6 (Web Frontend Shell) will consume the `I18nProvider` when bootstrapping the React app. Backend i18n (E4.2) and formatting (E4.3) build on this package.

**Critical constraint: `apps/web` is currently a stub** (empty `index.ts`, no React runtime). The i18n package (`packages/i18n`) must be self-contained with its own tests. The React Provider component will be consumed by `apps/web` when E6 bootstraps it with Vite + React 19. Do NOT attempt to set up a full React app in `apps/web` — that is E6's responsibility.

### Key Technical Decisions

1. **i18next over react-intl**: i18next is the industry standard for React i18n, has superior TypeScript support, namespace-based organisation, and mature ecosystem. react-intl (FormatJS) is more opinionated and less flexible for our namespace-per-module approach.

2. **Bundled locale files (no HTTP backend for MVP)**: Since MVP is English-only, locale files are imported directly and bundled. When multi-language support is added, switch to `i18next-http-backend` for lazy loading.

3. **Namespace-per-module strategy**: Namespaces (`common`, `validation`, `navigation`, `errors`) map to separate JSON files. Future business module namespaces (`finance`, `ar`, `sales`, etc.) will be added as those epics are implemented.

4. **User.locale field addition**: FR179 requires per-user language preference. The current Prisma schema lacks this field on the User model. This story adds it as a migration. The `CompanyProfile.defaultLanguage` field already exists (line 283 of schema.prisma).

### Fallback Chain Implementation

```
resolveLocale(userLocale?, companyLocale?) → string
  1. If userLocale is set AND is in SUPPORTED_LOCALES → use it
  2. If companyLocale is set AND is in SUPPORTED_LOCALES → use it
  3. Default → 'en'
```

The `I18nProvider` component calls `i18n.changeLanguage(resolvedLocale)` reactively when user/company locale props change.

### i18next Interpolation Format

i18next uses `{{ }}` double-brace syntax by default (not `{ }` single-brace):

```typescript
// Locale file: "required": "{{field}} is required"
// Usage:       t('validation.required', { field: 'Email' })
// Output:      "Email is required"
```

### Missing Key Handling

In development mode, missing keys trigger `console.warn` via i18next's `saveMissing` + `missingKeyHandler` config. In production, missing keys render the key string itself (e.g., `"finance.ledger.title"`) as a visible signal that a translation is missing.

### Remaining Issues for Human Review:

- **ISSUE #1: [HIGH]** `isDev` detection in `config.ts:71-73` is broken for Vite browser environments. The check uses `typeof process !== 'undefined' && process.env?.NODE_ENV !== 'production'`, but Vite does NOT polyfill `process` in the browser — it uses `import.meta.env.DEV` instead. This means `isDev` evaluates to `false` in the browser even during development, which silently disables AC #3's "logs a missing-key warning in development mode" requirement. The entire missing-key handler (`missingKeyHandler`) is set to `undefined` in the exact environment where developers most need it.
- **ISSUE #2: [HIGH]** No type-safe translation key registry. i18next v24+ supports a `resources.d.ts` / custom type module that provides compile-time checking for `t()` calls. Without it, `t('commmon:svae')` (typos) compiles silently and only fails at runtime by returning the raw key string. The story says "centralised translation key system" — centralisation without compile-time enforcement is half-built. This is especially dangerous because the ESLint rule only catches hardcoded strings in JSX, not misspelled translation keys.
- **ISSUE #3: [HIGH]** AC #4 example cannot actually be reproduced. The acceptance criterion specifies: `t('validation.required', { field: t('field.customerName') })` returning "Customer Name is required". But no `field` namespace or `field.customerName` key exists anywhere in the locale files. The test at `i18n-instance.test.ts:93-98` substitutes a hardcoded string `'Email'` instead of a nested `t()` call, sidestepping the AC's actual requirement of composable translation keys for field labels.
- **ISSUE #4: [MEDIUM]** `I18nProvider` does not include its own `<Suspense>` boundary despite `useSuspense: true` being set in the i18next config (`config.ts:110`). If any consuming code renders `<I18nProvider>` without a parent `<Suspense>`, the app crashes with an uncaught thrown promise. The test suite masks this by always wrapping in a `Wrapper` with `<Suspense>`. No JSDoc, comment, or runtime guard warns consumers. This is a pit-of-failure API design — the provider should either wrap its own `<Suspense>` with a sensible fallback, or throw a descriptive error if it detects no Suspense boundary.
- **ISSUE #5: [MEDIUM]** `server.ts` exports utilities but provides no server-side singleton with bundled resources. Backend code that needs `t()` for emails/PDFs (E4.2 AC #3) must call `createI18nInstance()` manually AND import JSON locale files themselves AND pass them as `resources`. Compare this with the client singleton (`i18n-instance.ts`) which bundles everything automatically. This asymmetry guarantees that the first E4.2 implementer will either: (a) import from `@nexa/i18n` main entry (pulling in React and crashing), or (b) write boilerplate that should be in the package.
- **ISSUE #6: [MEDIUM]** `FallbackConfig` type (`types.ts:33-37`) is exported in both `index.ts` and `server.ts` but never consumed anywhere — not by `resolveLocale()`, not by `I18nProvider`, not by tests. It's dead code that shipped on day one. `resolveLocale()` takes two loose `string | undefined` parameters instead of the structured config. Either the type should be deleted or `resolveLocale` should accept it.
- **ISSUE #7: [MEDIUM]** Flat JSON key structure conflicts with the documented patterns in `project-context.md`. The project context shows `t('invoice.status.posted')` — a dot-separated nested key. But all four locale files use flat top-level keys (`"save": "Save"`, not `"invoice": {"status": {"posted": "..."}}`). Since i18next's default `keySeparator` is `.`, a key like `t('invoice.status.posted')` would traverse nested objects. When business modules arrive (E14+), they'll face an undocumented decision: flat keys (inconsistent with spec) or nested keys (inconsistent with E4.1 precedent). The story should establish the convention explicitly.
- **ISSUE #8: [MEDIUM]** `buildInitOptions()` in `config.ts:79-115` uses a shallow spread (`...overrides`) that silently destroys nested configuration objects. If a consumer passes `{ interpolation: { prefix: '%%' } }`, the entire `interpolation` block is replaced — `escapeValue: false` (the React XSS protection) is silently dropped. A deep-merge utility or explicit override handling would be safer. This is a security-adjacent issue since `escapeValue: false` being lost could lead to double-escaping bugs, and a consumer accidentally setting it to `true` would break output.
- **ISSUE #9: [LOW]** `matchSupportedLocale()` does not handle BCP 47 case-insensitivity. Per RFC 5646, locale tags are case-insensitive (`en-GB` = `EN-gb`). The function compares with strict equality via `SUPPORTED_LOCALES.includes()`. The test at `config.test.ts:28` explicitly asserts `'EN'` returns `false`. While only `'en'` is supported in MVP, any future locale addition (e.g., `'fr-FR'`) will silently fail if the browser supplies `'FR-FR'`. A simple `.toLowerCase()` before comparison would fix this.
- **ISSUE #10: [LOW]** `TranslationParams` type (`types.ts:30`) defines `Record<string, string | number>` but the E4.2 story spec defines backend `TranslationMessage.params` as `Record<string, string>` (no `number`). When E4.2 is implemented, this type mismatch will either require backend messages to stringify numbers before sending, or the types will diverge between frontend and backend within the same `@nexa/i18n` package. Resolve the canonical type now before both stories ship.
- **ISSUE #11: [LOW]** `i18n-instance.ts` fires `i18n.use(initReactI18next).init(...)` as a module-level side effect on import. Any code that imports from the main `@nexa/i18n` entry point — even if it only needs types or `resolveLocale()` — triggers React initialization. The `package.json` `exports` field correctly provides `./server` as an alternative, but there's no `sideEffects` field or build-time guard to prevent accidental main-entry imports from Node.js code.
- **ISSUE #12: [LOW]** `errors.json` mixes two different key conventions: SCREAMING_SNAKE_CASE for error codes (`AUTH_INVALID_CREDENTIALS`, `NOT_FOUND`) and the rest of the locale files use camelCase (`save`, `required`, `minLength`). The locale-files test (`locale-files.test.ts:62-77`) explicitly allows both conventions, but there's no documentation explaining when to use which. This ambiguity will compound as business modules add keys.
