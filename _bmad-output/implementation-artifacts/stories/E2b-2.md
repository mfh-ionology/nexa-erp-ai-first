# Story E2b.2: Admin can create and configure access groups with permissions

Status: done

## Story

As an **administrator**,
I want to create custom access groups and set per-resource permissions (canAccess, canNew, canView, canEdit, canDelete),
So that I can define granular roles tailored to my company's needs.

## Acceptance Criteria

1. GIVEN an administrator is authenticated WHEN they POST /system/access-groups with code, name, description THEN a new access group is created for the current company AND the code is unique per company (409 if duplicate)
2. GIVEN an access group exists WHEN the admin GETs /system/access-groups THEN they receive a paginated list of access groups for the current company with id, code, name, description, isSystem, isActive, userCount
3. GIVEN an access group exists WHEN the admin GETs /system/access-groups/:id THEN they receive the full detail including permissions array and fieldOverrides array
4. GIVEN an access group exists WHEN the admin PATCHes /system/access-groups/:id with name and/or description THEN the metadata is updated (system groups can have name/description modified)
5. GIVEN an access group exists WHEN the admin PUTs /system/access-groups/:id/permissions with a permission array THEN all existing permissions are replaced with the provided set AND omitted resources have no permissions (all flags false)
6. GIVEN a new company is created WHEN the company creation completes THEN 12 pre-built access groups are seeded (FULL_ACCESS, FINANCE_MANAGER, FINANCE_CLERK, SALES_MANAGER, SALES_STAFF, PURCHASE_MANAGER, PURCHASE_CLERK, WAREHOUSE_STAFF, HR_MANAGER, HR_VIEWER, REPORT_VIEWER, READ_ONLY) AND all are marked isSystem: true AND the creating user is assigned the FULL_ACCESS group
7. GIVEN a system access group (isSystem: true) WHEN the admin attempts DELETE /system/access-groups/:id THEN they receive a 409 Conflict response
8. GIVEN a custom access group with no active users WHEN the admin DELETEs /system/access-groups/:id THEN the group is soft-deleted (isActive set to false)

## Dev Notes

### Critical Architecture: AccessGroup is Company-Scoped (has companyId)

Unlike the Resource table (which is global), AccessGroup and all related tables are **company-scoped**. Every query MUST include `companyId` in the WHERE clause. This follows the standard multi-company architecture from project-context.md §1.

**Tables and their scoping:**
| Table | Scoped? | Notes |
|-------|---------|-------|
| `Resource` | GLOBAL (no companyId) | Created in E2b-1, shared across all companies |
| `AccessGroup` | COMPANY-SCOPED | `@@unique([companyId, code])` |
| `AccessGroupPermission` | Via AccessGroup | FK to AccessGroup (inherits company scope) |
| `AccessGroupFieldOverride` | Via AccessGroup | FK to AccessGroup (inherits company scope) |
| `UserAccessGroup` | COMPANY-SCOPED | Has explicit `companyId` for cross-company assignment |

### Delete Error Code Mapping (DomainError → 409)

The `DomainError` class returns 422 by default. The AC requires 409 Conflict for system group deletion and groups with active users. The route handler must catch the `DomainError` and re-throw as `AppError` with status 409:

```typescript
try {
  await deleteAccessGroup(prisma, companyId, id, userId);
  return reply.status(204).send();
} catch (error) {
  if (error instanceof DomainError && ['SYSTEM_GROUP_PROTECTED', 'GROUP_HAS_USERS'].includes(error.code)) {
    throw new AppError(error.code, error.message, 409);
  }
  throw error;
}
```

```typescript
    const result = await createAccessGroup(prisma, ctx.companyId, request.body, ctx.userId);
    return sendSuccess(reply, result, undefined, 201);
  });
  // ... more routes
}

export const accessGroupRoutesPlugin = accessGroupRoutes;
```

### Previous Implementation Learnings (from E2b-1 and reverted E2b)

1. **In-memory permission cache with Map works for MVP** — Redis interface planned for scale, but Map is sufficient. E2b.4 will implement this.
2. **Fastify 5 requires getter/setter for reference-type request decorators** — if decorating `request` with objects, use `{ getter() { return val; }, setter(v) { val = v; } }` pattern.
3. **Zod `.default([])` makes arrays always truthy** — no need for null checks when using default empty arrays.
4. **Prisma `$transaction` accepts either a function or an array** — use function form for dependent operations (access group create + permission create).
5. **P2002 error code** — always catch in try/catch for unique constraint violations, not middleware.
6. **Resource table is GLOBAL** — do NOT accidentally add companyId to Resource queries. Only AccessGroup and UserAccessGroup have companyId.
7. **loadDefaultAccessGroups must accept TransactionClient** — because it's called inside `createCompanyProfile`'s `$transaction`. Use `Prisma.TransactionClient | PrismaClient` type.
8. **The reverted E2b implementation** (commits 51c78d9–a7b0c00, then reverted 5fb97e9–86d5d57) tried to do all of E2b in one shot. The current incremental approach (E2b-1 → E2b-2 → ...) is the correct pattern.
