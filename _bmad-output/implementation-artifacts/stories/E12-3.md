# Story E12-3: Default Templates

**Epic:** E12 — Document Templates & PDF
**Status:** done
**Dependencies:** E12-1 (Template Engine — Prisma schema, TemplateCompilerService, PdfGeneratorService, SampleDataGeneratorService), E12-2 (Template Management — CRUD service, document-template.service.ts extended with CRUD methods)
**Complexity:** Large (8 tasks)
**FRs:** FR79 (report templates), FR85 (document generation), FR191 (generate PDFs via Document Templates engine) | **NFRs:** NFR2 (CRUD <500ms), NFR3 (reports <5s), NFR41 (settings management)

---

## User Story

As a **system**,
I want built-in default templates for each DocumentType (invoice, credit note, PO, delivery note, statement, payslip, etc.), so that tenants can generate professional documents immediately without custom template creation.

---

## Acceptance Criteria

### AC1: All 14 DocumentTypes Have Default Templates
**GIVEN** a new tenant is provisioned (or the database is seeded)
**WHEN** the seed script runs
**THEN** default `DocumentTemplate` records exist for all 14 `DocumentType` enum values:
  - `SALES_INVOICE`, `CREDIT_NOTE`, `CASH_RECEIPT`, `PROFORMA_INVOICE`, `CUSTOMER_STATEMENT`
  - `SALES_ORDER`, `SALES_QUOTE`, `DELIVERY_NOTE`
  - `PURCHASE_ORDER`, `GOODS_RECEIPT_NOTE`, `SUPPLIER_REMITTANCE`
  - `PAYSLIP`, `P45`, `P60`
**AND** each template has `isDefault = true` and `isActive = true`
**AND** each template has a descriptive `name` (e.g., "Standard Invoice", "Standard Credit Note")
**AND** the unique constraint `[companyId, documentType, name]` is satisfied for each template

### AC2: Professional Invoice Template Layout
**GIVEN** the default `SALES_INVOICE` template
**WHEN** rendered with invoice data
**THEN** it includes:
  - Company logo (conditional on `showLogo`) with correct positioning (`logoPosition`)
  - Company details: legal name, address (multi-line), phone, email, website
  - Company VAT number (conditional on `showVatNumber`)
  - Company registration number (conditional on `showCompanyReg`)
  - Customer name and address block
  - Invoice number, date, due date, reference
  - Line items table: line number, item code, description, quantity, unit price, discount (if applicable), VAT rate, line total
  - VAT breakdown summary (grouped by rate)
  - Totals: subtotal, VAT amount, total, amount due
  - Bank details (conditional on `showBankDetails`): bank name, sort code, account number
  - Payment terms
  - Footer with page numbers
**AND** monetary values use `{{formatCurrency amount currency}}` helper
**AND** dates use `{{formatDate date}}` helper (UK format `DD/MM/YYYY`)
**AND** numbers use `{{formatNumber value decimals}}` helper
**AND** line items iterate via `{{#each lines}}...{{/each}}`

### AC3: Conditional Branding Sections
**GIVEN** each default template
**WHEN** the branding toggles are configured on the `DocumentTemplate` record
**THEN** all templates include conditional blocks:
  - `{{#if showLogo}}` — company logo section (positioned per `logoPosition`)
  - `{{#if showBankDetails}}` — bank account details section
  - `{{#if showVatNumber}}` — VAT registration number display
  - `{{#if showCompanyReg}}` — company registration number display
**AND** when a toggle is `false`, the corresponding section is omitted from the rendered PDF
**AND** the PDF layout does not break when sections are omitted (no blank gaps or layout shifts)

### AC4: Template Variety Across Document Types
**GIVEN** the 14 default templates
**WHEN** rendered with their respective sample data (from `SampleDataGeneratorService`)
**THEN** each template is appropriate for its document type:
  - **SALES_INVOICE**: Full invoice with line items, VAT, bank details, payment terms
  - **CREDIT_NOTE**: "CREDIT NOTE" header, reference to original invoice, line items showing credited amounts
  - **CASH_RECEIPT**: Compact receipt format, payment reference, amount received
  - **PROFORMA_INVOICE**: "PROFORMA INVOICE" header, "This is not a VAT invoice" notice, line items
  - **CUSTOMER_STATEMENT**: Date range header, opening balance, transaction list (date, reference, type, debit, credit, balance), closing balance, aging bands (Current, 30, 60, 90+ days)
  - **SALES_ORDER**: "SALES ORDER" header, delivery date, line items, terms & conditions area
  - **SALES_QUOTE**: "QUOTATION" header, validity date, line items, terms, "not a binding contract" notice
  - **DELIVERY_NOTE**: "DELIVERY NOTE" header, line items with quantities only (no pricing), delivery address, received-by signature area
  - **PURCHASE_ORDER**: "PURCHASE ORDER" header, supplier details, delivery address, line items, authorised-by area
  - **GOODS_RECEIPT_NOTE**: "GOODS RECEIVED NOTE" header, supplier reference, line items with received quantities, inspection notes area
  - **SUPPLIER_REMITTANCE**: "REMITTANCE ADVICE" header, payment date, invoice allocations table, total paid
  - **PAYSLIP**: Employee details, tax period, earnings table (description, amount), deductions table (PAYE, NI, pension), net pay, YTD totals
  - **P45**: HMRC P45 format — employee details, leaving date, tax code, total pay and tax to date, employer PAYE reference
  - **P60**: HMRC P60 format — tax year header, employee details, total pay, total tax deducted, NI contributions, employer details

### AC5: Seed Script Idempotency
**GIVEN** the seed script has already run and default templates exist
**WHEN** the seed script runs again
**THEN** existing templates are NOT duplicated
**AND** existing templates are updated if the seed data has changed (upsert behaviour)
**AND** if a tenant has customised a default template (changed `htmlTemplate`), the seed does NOT overwrite their customisation — it only upserts on the unique constraint `[companyId, documentType, name]`
**AND** the seed script logs the count of templates seeded/updated

### AC6: Templates Render Valid PDFs
**GIVEN** each of the 14 default templates
**WHEN** compiled by `TemplateCompilerService` with sample data from `SampleDataGeneratorService`
**AND** rendered to PDF by `PdfGeneratorService`
**THEN** each produces a valid PDF (buffer starts with `%PDF`)
**AND** the PDF is non-empty (size > 1KB for all types)
**AND** rendering completes within 5 seconds per document (NFR3)
**AND** no Handlebars compilation errors occur (no unresolved `{{undefined}}` in output)

### AC7: CSS Styling Consistency
**GIVEN** all 14 default templates
**WHEN** rendered as PDFs
**THEN** they share a consistent visual design:
  - Clean, professional typography (system fonts: Arial, Helvetica, sans-serif)
  - Consistent table styling for line items (bordered, alternating row backgrounds)
  - Proper page margins (A4 format, 20mm top/bottom, 15mm left/right)
  - Responsive layout that handles varying content lengths without page-break issues
  - CSS `page-break-inside: avoid` on line item rows and summary sections
  - Consistent colour scheme (headers, table headers, accent colours)
**AND** each template includes its CSS inline within a `<style>` block (for Puppeteer compatibility)

## Dev Notes

- **Template HTML location**: Architecture spec says `api/src/modules/system/templates/` for default HTML files. However, the seed runs from `packages/db/`. **Solution**: Define template HTML as exported string constants in TypeScript files within the `templates/` directory. The seed file will need to either (a) import these via a workspace path alias, or (b) duplicate/co-locate the template strings in the seed file. Option (a) is cleaner but requires build configuration; option (b) is simpler for MVP. **Recommended**: Place template constants in `apps/api/src/modules/system/templates/` and have the seed file import them via `../../../../apps/api/src/modules/system/templates/` relative path (workspaces allow this). If this causes build issues, fall back to defining templates inline in the seed file.
- **Handlebars variable names**: Templates MUST use the exact variable paths from `DocumentDataContext` interface (defined in `document-data-loader.service.ts`): `company.name`, `company.legalName`, `company.address`, `document.number`, `document.date`, `counterparty.name`, `lines[].lineNumber`, `totals.subtotal`, `metadata.currencyCode`, `branding.showLogo` (note: branding context is flattened into the top-level context, so use `{{#if showLogo}}` not `{{#if branding.showLogo}}`). Check `SampleDataGeneratorService` output to confirm exact paths.
- **Branding context injection**: In the template engine pipeline (E12-1), branding flags are injected as top-level variables in the data context by `DocumentDataLoaderService.loadContext()` (Task 5.6). Templates use `{{#if showLogo}}` directly, not `{{#if branding.showLogo}}`. Verify this by checking `document-data-loader.service.ts` before writing templates.
- **Payroll templates (P45, P60)**: These are HMRC-defined formats. For MVP, create reasonable approximations. Full HMRC compliance (exact field positions, barcode areas) can be a follow-up story. Use the stub data context from `SampleDataGeneratorService` which repurposes some fields (e.g., `totals.vatAmount` for tax deducted).
- **CSS inline vs separate**: Each template should include its CSS in a `<style>` block within the `<head>`. The `TemplateCompilerService.compile()` method also supports a separate `css` parameter that gets injected — use this for the shared base CSS, with template-specific styles in the HTML `<style>` block.
- **No external resources**: Templates should NOT reference external URLs for fonts, images, or stylesheets. All styling must be inline or in `<style>` blocks. Company logos are injected via `{{company.logoUrl}}` and rendered as `<img>` tags — Puppeteer handles these via the browser.
- **Page numbers**: Puppeteer's `headerTemplate` and `footerTemplate` support special CSS classes: `.pageNumber`, `.totalPages`. Use `<span class="pageNumber"></span>` in the footer HTML field of the DocumentTemplate record (not in the main htmlTemplate body).
- **Seed ordering**: The seed must run AFTER `seedDefaultCompany()` (needs `DEFAULT_COMPANY_ID`) and AFTER the Prisma migration that creates the `DocumentTemplate` table (E12-1 migration).
- **DocumentType import**: Use `import { DocumentType } from '../../generated/prisma/client'` in the seed file (matching existing seed patterns), NOT a hardcoded enum.
