## Successes

### 1. Rapid, Efficient Delivery
Entire epic completed in a single day — 2 stories, 13 story points, full backend + frontend implementation with tests. This is the most efficient delivery ratio of any epic in the project.

### 2. Clean Architecture — Dedicated Model + Existing Infrastructure
- Purpose-built `PrintPreference` model with typed enums and `[companyId, userId, documentType]` unique constraint — more robust than generic key-value storage
- Company-level defaults cleverly leverage existing `SystemSetting` model with key convention `print.default.{DOCUMENT_TYPE}`
- Clean separation: E13-1 (configuration) → `usePrintAction` hook (bridge) → E13-2 (execution)

### 3. Applied Past Lessons
- E13-1 explicitly addressed the dirty-state-after-save bug from E9-4 Issue #1 by clearing dirty state after successful save
- Dev notes document the anti-pattern and prevention, showing active learning from prior epics

### 4. No Backend Changes for E13-2
E13-2 was entirely frontend — all backend infrastructure from E12 (document generation) and E13-1 (print preferences) was sufficient. Clean dependency management.

### 5. Comprehensive Reference Documentation
Both story files include detailed "Existing Implementation" sections documenting exact file paths, export names, and patterns to follow. This is excellent for developer onboarding and cross-story continuity.

## Challenges

### 1. Unresolved HIGH Code Review Issues (6 total)
Both stories were marked "done" with unresolved HIGH issues. Key HIGH issues:
- **E13-1 #1:** Missing navigation i18n key — sidebar shows raw translation key
- **E13-1 #2-#3:** Test assertions reference wrong i18n keys — tests will fail when run
- **E13-2 #1:** Dead code — `useBatchGenerate` and `useBatchGenerateStatus` hooks never imported
- **E13-2 #2:** AC #3 not fully met — batch AUTO_DOWNLOAD doesn't use backend batch-generate or produce ZIP
- **E13-2 #3:** No guard against concurrent batch operations

### 2. Acceptance Criteria Gap (E13-2 AC #3)
The story specifies: "PDFs are generated for all selected documents and either batch-downloaded as a ZIP or printed sequentially." The implementation uses client-side sequential generation instead of the backend `POST /documents/batch-generate` endpoint. The batch hooks exist but are dead code.

### 3. Dead Code Accumulation
- `resolvePrintAction` utility (E13-1) — unused, only `lookupResolvedPreference` is active
- `useBatchGenerate` + `useBatchGenerateStatus` (E13-2) — created but never imported
- Three i18n translation keys (E13-2) — defined but never referenced in code

### 4. Persistent i18n Key Discipline Issues
i18n key mismatches appeared in BOTH stories:
- Navigation sidebar showing raw key instead of translated text
- Test assertions using wrong i18n keys
- Dead translation keys accumulating in JSON files
This pattern has recurred across E9, E10, E11, and now E13.

---

## Key Insights

1. **Small, focused epics execute extremely efficiently.** E13's 2-story structure allowed complete delivery in one day with minimal context switching.

2. **Code review catches real issues but lacks follow-through enforcement.** The CR process identified 6 HIGH issues, but there's no gate preventing "done" status with unresolved HIGHs.

3. **The CR4 fix pass works when applied.** E5d-6 proved that a dedicated fix iteration after code review can eliminate all HIGH issues. Not applying it to E13 was a regression in process discipline.

4. **Dead code signals plan-implementation mismatch.** When hooks/utilities are created but never used, it suggests the dev agent implemented what the story spec described rather than what the actual execution path required.

5. **i18n key management needs systematic tooling.** Manual cross-referencing of translation keys vs code references is error-prone. A lint rule or build-time check would eliminate this entire class of bug.
