# Story E5.5: Daily Briefing & Smart Suggestions

Status: done

## Story

As a **user**,
I want a personalised daily briefing based on my role and contextual smart suggestions when viewing records,
so that I start each day informed and always have relevant next actions available.

## Acceptance Criteria

1. **AC-1: Role-based briefing for Finance Manager** — GIVEN a user with role "Finance Manager" WHEN they request the daily briefing THEN it includes: pending approvals, overdue invoices, cash position, upcoming payment runs, and anomaly alerts

2. **AC-2: Role-based briefing for Business Owner** — GIVEN a user with role "Business Owner" WHEN they request the daily briefing THEN it includes: revenue vs prior period, overdue receivables, pending approvals across all modules, and AI-detected opportunities

3. **AC-3: Actionable briefing items** — GIVEN a briefing is generated WHEN each item is displayed THEN it includes actionable links (one-tap approve, chase, review) and period comparison data (delta/trend)

4. **AC-4: Contextual smart suggestions** — GIVEN a user is viewing a specific record (e.g., Customer Detail) WHEN AI suggestions are requested THEN contextual suggestions are returned (e.g., "Invoice this customer", "Show payment history", "Credit check")

5. **AC-5: Briefing caching and performance** — GIVEN the briefing generation runs WHEN the scheduled job executes THEN it completes within the AI response time target and caches the result for the day

6. **AC-6: Graceful degradation** — GIVEN the AI service is unavailable WHEN a user requests the briefing or suggestions THEN the system returns a graceful error response and the traditional UI remains fully functional (IMP-006)

### Implementation Patterns to Follow

1. **Route pattern**: Follow `prediction.routes.ts` — use `assertService()` guard, `buildContext()` helper, `createPermissionGuard()`
2. **Service pattern**: Follow `PredictionService` — constructor injection, `processDirect()` for inline prompts, `safeModelQuery()` for graceful model access
3. **Error handling**: Follow orchestrator pattern — never throw from public methods, return structured error responses, emit `ai.degraded` event
4. **Schema pattern**: Follow `prediction.schema.ts` — Zod schemas with `.describe()` for OpenAPI, shared between request validation and response serialization
5. **Test pattern**: Follow `prediction.service.test.ts` — mock `AiOrchestrator.processDirect()`, mock Prisma, test both success and degradation paths

### Key Architectural Decisions

- **No dedicated DB model for briefings** — generated on-demand, cached in Redis (Architecture §6.8)
- **AI Gateway routing** — briefing uses `routingTags: ['briefing']` which maps to Sonnet model (Architecture §6.1)
- **Suggestions are NOT AI-generated** — they are deterministic mappings based on page context, role, and time of day. Only the briefing content goes through the AI. This keeps suggestions fast and predictable.
- **BullMQ for scheduled generation** — pre-generates briefings at configurable time so users get instant cache hits on first access
- **IMP-006 enforcement** — all endpoints return 503 gracefully when AI is unavailable; no crash, no blocking of traditional UI

### Resource Codes for Permission Guards

- `ai.briefing` — for `GET /ai/briefing` (VIEWER minimum)
- `ai.suggestions` — for `POST /ai/suggestions` (STAFF minimum)

These must be added to `packages/db/default-data/company-defaults.json` resource registry and seeded.
