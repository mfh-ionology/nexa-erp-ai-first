# Story E6.4: ActionBar Component

Status: done

## Story

As a **user**,
I want a consistent action bar on every record screen with primary actions, persistent tools (Attachments, Links), and a grouped overflow menu, so that I always know where to find actions regardless of which module I am in.

## Acceptance Criteria

1. GIVEN a record screen (T2, T3, T5, T6, T7, T8) WHEN the action bar renders THEN it shows three zones: primary action zone (max 2 buttons, status-driven), persistent tools zone (Attachments with count badge, Links with count badge), and overflow menu button (⋯)
2. GIVEN the entity is in DRAFT status WHEN the action bar renders THEN the primary action is "Approve" (or "Save Draft") and the overflow Status Actions section shows only valid next transitions from the state machine
3. GIVEN the entity has 3 attachments and 2 record links WHEN the action bar renders THEN the Attachments button shows "(3)" badge and Links button shows "(2)" badge
4. GIVEN the overflow menu WHEN opened THEN actions are grouped into 5 sections with dividers: Document Actions, Status Actions, Record Actions, AI Actions, History — with empty sections hidden entirely
5. GIVEN an action that is invalid for the current entity status WHEN the overflow menu renders THEN the action is hidden entirely (not greyed out/disabled)
6. GIVEN a destructive status change action (Void, Cancel, Delete) WHEN clicked THEN a confirmation dialog appears showing the entity name and a consequence description before executing
7. GIVEN the user lacks the required access group permission for an action WHEN the action bar renders THEN that action is hidden (both primary zone and overflow menu respect permission checks independently of status checks)
8. GIVEN a SUPER_ADMIN user WHEN the action bar renders THEN all actions that are valid for the current status are shown (permission checks bypassed)
9. GIVEN the action bar WHEN navigated via keyboard THEN all buttons are focusable, overflow menu items are navigable with arrow keys, and keyboard shortcut hints are displayed
10. GIVEN the action bar WHEN rendered on phone (<768px) THEN primary actions collapse to a single primary button plus the overflow (⋯), persistent tools move into the overflow menu
11. GIVEN all action bar content WHEN inspected THEN every user-facing string uses `t()` from `@nexa/i18n` — no hardcoded English text

      /** Unique action key (e.g., 'approve', 'void', 'print') */
      key: string;
      /** i18n translation key for the action label */
      labelKey: string;
      /** Optional Lucide icon component */
      icon?: LucideIcon;
      /** Button variant */
      variant?: ActionVariant;
      /** Callback when action is triggered */
      onAction: () => void;
      /** Whether this action requires a confirmation dialog */
      requiresConfirmation?: boolean;
      /** i18n key for confirmation dialog title */
      confirmTitleKey?: string;
      /** i18n key for confirmation dialog description (receives {{ entityName }}) */
      confirmDescriptionKey?: string;
      /** Keyboard shortcut hint string (e.g., '⌘S', '⌘P') */
      shortcutHint?: string;
      /** Whether the action is currently loading/processing */
      isLoading?: boolean;
      /** Permission check: resource code (e.g., 'finance.invoices.detail') */
      permissionResource?: string;
      /** Permission check: required action flag */
      permissionAction?: 'canAccess' | 'canNew' | 'canView' | 'canEdit' | 'canDelete';
    }

    /** Primary action — appears in the primary zone (max 2) */
    export interface PrimaryAction extends ActionDefinition {
      /** Whether this is the main primary action (gets primary styling) */
      isPrimary?: boolean;
    }

    /** Overflow action — appears in a grouped section of the overflow menu */
    export interface OverflowAction extends ActionDefinition {
      /** Which overflow section this action belongs to */
      section: OverflowSection;
    }

    /** Status-to-actions mapping for an entity type */
    export interface EntityActionConfig {
      /** Entity type identifier (e.g., 'customerInvoice', 'salesOrder') */
      entityType: string;
      /** Map of status → available actions */
      statusActions: Record<string, StatusActionSet>;
      /** Actions available regardless of status (e.g., AI Explain, View Audit Log) */
      globalActions?: OverflowAction[];
    }

    /** Actions available for a specific entity status */
    export interface StatusActionSet {
      /** Primary actions for this status (max 2) */
      primary: PrimaryAction[];
      /** Overflow actions for this status — grouped by section */
      overflow: OverflowAction[];
    }

    /** Props for the ActionBar component */
    export interface ActionBarProps {
      /** Entity type identifier */
      entityType: string;
      /** Current entity status */
      status: string;
      /** Entity display name (shown in confirmation dialogs) */
      entityName: string;
      /** Number of attachments (displayed as badge count) */
      attachmentCount?: number;
      /** Number of record links (displayed as badge count) */
      linkCount?: number;
      /** Callback when Attachments button is clicked */
      onAttachmentsClick?: () => void;
      /** Callback when Links button is clicked */
      onLinksClick?: () => void;
      /** Override action config — allows callers to provide custom actions
       *  instead of relying on the default entity action config registry */
      actionConfig?: StatusActionSet;
      /** Additional global actions (AI, History) to merge with defaults */
      additionalActions?: OverflowAction[];
      /** Whether to show persistent tools (Attachments/Links) — default true */
      showPersistentTools?: boolean;
    }
    ```

## Dev Notes

### Architecture Summary

The ActionBar is the single most critical consistency element across all record screens. It provides a standardised three-zone layout for primary actions, persistent tools, and overflow actions on every T2, T3, T5, T6, T7, and T8 template. The component is **status-driven** (only valid state machine transitions appear) and **permission-driven** (only actions the user's access groups allow are visible).

The ActionBar integrates into existing templates via the `actionBarSlot` prop that was already provisioned in E6.3. Module pages will render `<ActionBar>` and pass it into the template's slot.

### Status-Driven Action Mapping

The action configuration system maps entity status → available actions. This follows the state machine transitions defined in the State Machine Reference:

| Entity | DRAFT → | APPROVED → | POSTED → | Terminal |
|--------|---------|------------|----------|----------|
| CustomerInvoice | Approve, Save Draft, Delete | Email, Post | Void, Create Credit Note | — |
| SalesOrder | Approve, Save Draft, Delete, Cancel | Create Dispatch, Cancel, Convert to Invoice | — | — |
| SalesQuote | Send, Save Draft, Delete | (SENT): Accept, Reject | — | — |
| PurchaseOrder | Approve, Save Draft, Delete | Send to Supplier, Cancel | — | — |
| JournalEntry | Post, Save Draft, Delete | — | Reverse | — |

### Permission Integration

Actions are filtered by two independent checks:

1. **Status check** — is this action valid for the current entity status? (from state machine)
2. **Permission check** — does the user's access group(s) grant the required permission?

Both checks must pass for an action to appear. The checks are independent:
- A POSTED invoice won't show "Approve" (status check fails) even for SUPER_ADMIN
- A DRAFT invoice won't show "Delete" for a user without `canDelete` permission (permission check fails) even though status allows it
- SUPER_ADMIN bypasses permission checks only (all status-valid actions are shown)

```

Empty sections are hidden entirely — no empty groups, no disabled items, no grey text.

### UX Action Bar Rules (from UX Design Spec)

1. **Status-driven and permission-driven visibility** — Both checks must pass
2. **Primary action changes with status** — DRAFT: "Approve"; APPROVED: "Email"; POSTED: no primary
3. **Maximum 2 primary actions** — Never more than 2 visible action buttons
4. **Persistent tools never hide** — Attachments and Links always visible (desktop), even with count 0
5. **Overflow sections hide when empty** — No empty groups, no disabled items
6. **Keyboard shortcut hints** — Menu items show hints (e.g., "Print ⌘P")

### Confirmation Dialogs for Destructive Actions

These actions ALWAYS require confirmation:
- **Void** — Reverses all GL postings, cannot be undone
- **Cancel** — Cancels the entire document, cannot be undone
- **Delete** — Permanently removes the draft record
- **Reverse** — Creates a contra-entry (journal reversal)
- **Reject** — Rejects an approval request

The dialog shows the entity name and a description of the consequences. The confirm button is styled as destructive (red).

### Cross-Cutting Patterns (MANDATORY)

- **i18n**: Every label, tooltip, ARIA label, dialog text uses `t()` from `@nexa/i18n`
- **Accessibility (NFR27, NFR28)**: `role="toolbar"`, `aria-label`, keyboard shortcuts, focus management in dialogs
- **TypeScript strict**: All types defined, no `any`, discriminated unions for action types
- **Permission model**: Access Group permissions from E2b — `useAuthStore` provides resolved permissions
