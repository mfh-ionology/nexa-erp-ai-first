# Story E2b.4: System enforces granular permissions on all routes

Status: done

## Story

As a **user**,
I want the system to enforce my access group permissions on every page and action,
So that I can only access what my groups allow, and administrators using SUPER_ADMIN bypass all checks.

## Acceptance Criteria

1. **AC1 — Most-permissive-wins resolution:** Given a user with multiple access groups where Group A has canView=true and Group B has canView=false for the same resource, when the system resolves permissions, then the effective canView is true (OR logic across all groups for all 5 action flags).

2. **AC2 — SUPER_ADMIN bypass:** Given a SUPER_ADMIN user, when they access any protected route, then they are always allowed (permission matrix bypassed entirely, no access group assignment required).

3. **AC3 — canAccess denied:** Given a user with no canAccess on a resource, when they request a route guarded by `createPermissionGuard` for that resource, then they receive a 403 Forbidden.

4. **AC4 — Action-level enforcement:** Given a user with canAccess=true but canNew=false, when they attempt a POST route guarded with action='new', then they receive a 403 Forbidden.

5. **AC5 — My-permissions endpoint:** Given any authenticated user, when they GET /system/my-permissions, then they receive their resolved permissions, fieldOverrides, accessGroups, role, isSuperAdmin, and enabledModules (derived: modules where any resource has canAccess=true).

6. **AC6 — Permission cache hit:** Given resolved permissions are requested, when the permission cache has a valid entry (within 60s TTL), then the cached result is returned without DB query.

7. **AC7 — Cache invalidation on group edit:** Given an access group's permissions are modified, when the change is saved, then the permission cache is invalidated for all affected users.

8. **AC8 — Cache invalidation on assignment change:** Given a user's access group assignments are changed, when the assignment is saved, then the permission cache is invalidated for that user+company.

9. **AC9 — Route migration:** Given all existing system routes (company-profile, users, company switch, resources, access-groups, user-access-groups, auth admin), when the migration is complete, then all routes use `createPermissionGuard` instead of the legacy `createRbacGuard`, and `createRbacGuard` is marked `@deprecated` but not deleted.

10. **AC10 — UserRole meaning narrowing:** Given the UserRole enum, when E2b-4 is complete, then SUPER_ADMIN provides platform-level permission bypass, ADMIN grants user and access group management authority, and MANAGER/STAFF/VIEWER are retained for backward compatibility but no longer drive page/action permissions.

11. **AC11 — Module access derivation (FR231):** Given a user with access groups, when enabledModules is computed, then a module is accessible if any resource within that module has canAccess=true in any of the user's assigned access groups.

## Dev Notes

### Architecture Overview

This story implements the **permission enforcement layer** that sits between JWT authentication (E2) and route handlers. It replaces the flat role-hierarchy guard (`createRbacGuard`) with a granular permission guard (`createPermissionGuard`) that checks per-resource, per-action permissions from the AccessGroup permission matrix built in E2b-1/E2b-2/E2b-3.

**Request lifecycle after E2b-4:**
```
Request → JWT verify (E2) → Company context (E2) → Permission guard (E2b-4) → Route handler
```

### Permission Resolution Algorithm

```
1. Check if user.role === SUPER_ADMIN → ALLOW (short-circuit, no DB/cache lookup)
2. Check cache for key `permissions:{userId}:{companyId}`
   - If cache hit and not expired → use cached permissions
   - If cache miss or expired → query DB:
     a. SELECT all UserAccessGroup WHERE userId = ? AND companyId = ?
     b. JOIN AccessGroup WHERE isActive = true
     c. SELECT all AccessGroupPermission WHERE accessGroupId IN (user's group IDs)
     d. Merge permissions per resource using OR logic:
        canAccess = OR(all groups), canNew = OR(all groups), etc.
     e. Cache result with 60s TTL
3. Check canAccess for target resourceCode → 403 if false
4. If action specified, check corresponding flag → 403 if false
5. Attach resolved permissions to request for downstream use
```

### In-Memory Cache Design (MVP)

Use a `Map<string, CacheEntry>` with lazy expiration:
- **Key format:** `permissions:{userId}:{companyId}`
- **TTL:** 60 seconds (configurable via env var for testing)
- **Invalidation:** Delete specific keys or prefix-match for bulk invalidation
- Redis interface planned for scale (E3+); keep the API surface compatible

### Key Implementation Decisions

1. **ADMIN role still required for access-group management routes:** Even with `createPermissionGuard`, the guard checks `canAccess`/`canEdit` etc. from the user's access groups. The `FULL_ACCESS` system group grants all flags on `system.access-groups.*`, but a custom group could too. The role-level ADMIN check is no longer needed since permission is now derived from access groups. However, BR-RBAC-008 states only ADMIN/SUPER_ADMIN users can manage access groups — implement this as the permission guard checking the access-group resources (which are only granted to ADMIN-level pre-built groups), NOT as a separate role check.

2. **Permission guard must NOT break unauthenticated routes:** Public routes (login, refresh, logout, health, docs) don't have `request.userId`. The guard runs as a preHandler on specific routes only, not globally. The JWT verify hook already skips public routes.

3. **Error responses:** Use existing `AuthError('FORBIDDEN', message, 403)` pattern from E2 for consistency.

4. **Fastify 5 request decorator pattern:** If adding `request.permissions`, use getter/setter pattern:
   ```typescript
   let permissionsVal: EffectivePermissions | null = null;
   fastify.decorateRequest('permissions', {
     getter() { return permissionsVal; },
     setter(v: EffectivePermissions | null) { permissionsVal = v; }
   });
   ```

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: Permission resolution queries UserAccessGroup WHERE companyId = request.companyId. Resource table is GLOBAL (no companyId) — one of the very few exceptions.
- **i18n**: Error messages use translation keys where applicable (e.g., 'error.forbidden').
- **Audit**: Permission-related events already defined in AppEventMap (E2b-2, E2b-3). Cache invalidation listeners subscribe to these events.
- **Attachments/Notes/Tasks**: Not applicable for this story.
