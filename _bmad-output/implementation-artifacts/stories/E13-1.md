# Story E13.1: Print Preferences

Status: done

## Story

As a **user**,
I want to configure my preferred print behaviour per document type (auto-download PDF, browser print dialog, or no action),
so that document handling matches my workflow without manual intervention after saving records.

## Acceptance Criteria

1. **GIVEN** a user preferences page **WHEN** the user opens print settings **THEN** a list of document types shows with a preference selector for each (Auto-Download, Browser Print, None)
2. **GIVEN** a company-level default **WHEN** set by an ADMIN **THEN** it applies to all users who have not set personal preferences
3. **GIVEN** a user has set "Auto-Download" for SALES_INVOICE **WHEN** they save an invoice **THEN** the PDF is automatically downloaded to their browser
4. **GIVEN** a user has set "Browser Print" for PURCHASE_ORDER **WHEN** they save a PO **THEN** the browser's native print dialog opens with the PDF
5. **GIVEN** a user with no personal preference for a document type **WHEN** the company default is "Auto-Download" **THEN** the resolved preference is "Auto-Download" (company default takes effect)
6. **GIVEN** a user with no personal preference and no company default **WHEN** the preference is resolved **THEN** the fallback is "NONE" (no automatic print action)

## Dev Notes

- **No existing system-settings CRUD routes**: The API contracts reference CRUD `/system/system-settings`, but these routes are not yet implemented. Rather than implementing the full generic CRUD, this story creates purpose-built print preference endpoints that are more ergonomic (typed document types, specific response shapes).
- **Company defaults via SystemSetting**: Company-level print defaults leverage the existing `SystemSetting` model with key convention `print.default.{DOCUMENT_TYPE}`. This avoids creating a redundant company-level preference model and reuses the established key-value pattern.
- **User preferences via dedicated model**: A new `PrintPreference` model with `[companyId, userId, documentType]` unique constraint provides type-safe, indexed storage. This is more robust than storing user preferences as JSON in a generic settings table.
- **Preference cascade**: Resolution order is user preference → company default → NONE. The service layer performs this cascade and annotates each preference with its `source` for UI display.
- **E13-2 dependency**: Task 8 creates the `usePrintAction` hook that E13-2 (Print Actions) will consume to trigger auto-download or browser print dialog after document save. This hook is the bridge between E13-1 (configuration) and E13-2 (execution).
- **Post-save dirty state**: Task 5.5 explicitly requires clearing dirty state after successful save — this addresses a known bug pattern from E9-4 (Issue #1) where dirty state was not reset after save.
- **Concept D prototype**: Check `_bmad-output/planning-artifacts/ux-prototypes/concept-d-purple-copilot.html` side-by-side for visual parity.
