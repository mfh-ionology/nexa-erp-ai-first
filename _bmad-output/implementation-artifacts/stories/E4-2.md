# Story E4.2: Backend i18n

Status: done

## Story

As a **developer**,
I want API error messages, validation messages, and system messages to use translation keys,
so that the backend is language-agnostic and ready for future locale support.

## Acceptance Criteria

1. GIVEN a validation error occurs on the API WHEN the error response is returned THEN the `message` field contains a translation key (e.g., `"validation.required"`) with parameters, not a hardcoded English string
2. GIVEN the API returns a structured error WHEN the frontend receives it THEN it can resolve the translation key via the `t()` helper to display the localised message
3. GIVEN a system-generated message (e.g., audit log description, notification text) WHEN it is persisted to the database THEN it stores a translation key and parameters, not rendered text
4. GIVEN a backend service needs to format a user-facing message WHEN it calls the message formatting utility THEN it produces a structured `{ key: string, params?: Record<string, string> }` object

## Dev Notes

### Architecture Context

This is a **Tier 1 infrastructure story** that connects the i18n translation system (created in E4-1) to the backend API error handling pipeline. After this story, all API errors, validation messages, and system messages use translation keys instead of hardcoded English strings.

**The API error response format changes from:**
```json
{
  "success": false,
  "error": {
    "code": "INVALID_CREDENTIALS",
    "message": "Invalid email or password"
  }
}
```

**To:**
```json
{
  "success": false,
  "error": {
    "code": "INVALID_CREDENTIALS",
    "message": "Invalid email or password",
    "messageKey": "errors:AUTH_INVALID_CREDENTIALS"
  }
}
```

The `message` field still contains the resolved English text (backward compat for API consumers). The `messageKey` field is new and lets the frontend resolve its own localised version. Both are populated from the same source of truth (`packages/i18n/locales/en/errors.json`).

### Key Technical Decisions

1. **Dual message approach (message + messageKey):** The `message` field is the resolved English string for immediate display and backward compat. The `messageKey` field lets the frontend resolve a localised version via `t()`. This avoids forcing all API consumers to implement i18n resolution.

2. **Server-side singleton (server-instance.ts):** The backend needs a pre-configured i18next instance with bundled English resources to resolve `tServer()` calls. This is the backend equivalent of `i18n-instance.ts` (which uses `initReactI18next`). Created separately to avoid React dependency. Uses `createI18nInstance()` from `config.ts` with the same locale JSON files.

3. **Zod error mapping in `@nexa/i18n` package (not in `apps/api`):** The Zod-to-translation-key mapping lives in the shared `@nexa/i18n` package because it may be reused by other services. The mapping produces `TranslationMessage` objects; resolution via `tServer()` happens at the call site.

4. **Optional `messageKey`/`messageParams` on AppError:** Adding new fields as optional preserves backward compatibility. All existing tests pass unchanged. New error throws should always provide `messageKey`.

5. **`TranslationMessage.params` uses `Record<string, string>` (strings only):** This differs from the frontend `TranslationParams` (`Record<string, string | number>`) intentionally. Backend messages are serialised to JSON for persistence (audit logs, notifications) and transport (API responses). Numbers must be formatted with locale-aware formatters on the client, not stringified on the server.

- `@nexa/i18n` package structure: `src/`, `locales/en/`, `__tests__/`
