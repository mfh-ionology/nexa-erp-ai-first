# Story E5.3: AI Action Framework

Status: done

## Story

As a **user**,
I want the AI to propose actions (create invoice, send email, update record) that I can review and confirm or reject before execution,
so that I maintain control over all AI-initiated changes.

## Acceptance Criteria

1. GIVEN the AI determines an action is needed (e.g., "create invoice for Acme") WHEN it formulates the action THEN it sends an `action_proposal` message with type, description, entity type, preview data, and confidence score
2. GIVEN the user receives an action proposal WHEN they click "Confirm" THEN the action executes through the standard API (same path as manual creation) and a `record_created` message is sent back
3. GIVEN the user receives an action proposal WHEN they click "Reject" THEN the action is cancelled, no data is modified, and the AI acknowledges the rejection
4. GIVEN a financial action (create invoice, post journal, process payment) WHEN the AI proposes it THEN user confirmation is ALWAYS required regardless of confidence score
5. GIVEN an action is executed via AI WHEN the audit trail records it THEN it includes `isAiAction: true`, `aiConfidence`, and the conversation ID

## Dev Notes

### Architecture Summary

E5-3 adds the **user confirmation layer** for AI-proposed actions. It implements the "Approve" step in the "Told, Shown, Approve, Done" paradigm. The key additions are:

1. **ActionPlanner** — Parses AI output to create structured action proposals with confidence scores and preview data
2. **GuardrailsService** — Evaluates every action against a hardcoded chain of safety rules, with financial actions ALWAYS requiring user confirmation (NFR16)
3. **ActionExecutor** — Executes confirmed actions through the standard service layer (same path as manual creation), with a registry pattern for future module handlers
4. **Audit integration** — Every AI-executed action logs `isAiAction: true`, `aiConfidence`, and `conversationId` in the immutable audit trail

**Flow:** AI Response → ResponseParser → ActionPlanner (create proposal + guardrails) → WebSocket `action_proposal` → User confirms/rejects → ActionExecutor (execute via standard API) → Business event emitted → Audit logged

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: All queries and operations must be scoped by `companyId`. Action proposals must validate the user can access the target entity within their company. The `ai.action.executed` event MUST include `companyId` for audit trail scoping.
- **i18n**: Error messages use translation keys via `messageKey` parameter (pattern established in `ai.errors.ts`). Action descriptions should use keys like `ai.action.createInvoice.description`. For MVP, hardcode English but include the key.
- **Audit**: AI action execution MUST emit `ai.action.executed` event, which the audit service subscribes to. Audit records include `isAiAction: true` and `aiConfidence`. Audit writes are best-effort (errors logged, not re-thrown) per NFR22.
- **Events**: Emit `ai.action.executed` event after successful action execution. The event payload must include `companyId`, `conversationId`, and `actionType` (to be added to the existing event type definition).

### AI Guardrails Compliance

- **IMP-005 (HARD):** AI NEVER auto-executes financial transactions. The `FINANCIAL_ACTION_TYPES` constant defines all financial action types. `GuardrailsService.evaluate()` ALWAYS sets `requiresApproval: true` for these types, regardless of confidence score or agent configuration. This is hardcoded and non-configurable.
- **BR-COM-013 (HARD):** AI chatbot actions require user confirmation before execution. The WebSocket handler sends `action_proposal` messages for user review. Only after receiving `action_confirm` does the `ActionExecutor` run.
- **IMP-006 (HARD):** AI degradation must not break traditional UI. If `ActionPlanner`, `GuardrailsService`, or `ActionExecutor` fail to initialize, they are set to `null`. Action proposals degrade gracefully to text-only responses.
- **NFR16 (IMMUTABLE):** Financial safety is the #6 rule in the guardrail chain. It is evaluated for EVERY action, not just when other rules pass. Even if an agent has `requiresApproval: false`, financial actions STILL require approval.

### Anti-Patterns to AVOID

1. **Do NOT bypass business rules for AI actions.** All action execution MUST go through the same service layer as manual creation. No direct Prisma writes for entity creation.
2. **Do NOT auto-execute ANY financial action.** Even with 99% confidence, financial actions (create invoice, post journal, process payment) ALWAYS require explicit user confirmation. This is NFR16 — immutable, non-configurable.
3. **Do NOT store action proposals in the database.** Use an in-memory Map keyed by proposal ID. Proposals are ephemeral — they live only for the duration of the conversation. If the server restarts, proposals are lost and the user must re-request.
4. **Do NOT modify the ResponseParser.** It already correctly produces `ActionProposal` objects from tool calls and structured output. The ActionPlanner enriches these with guardrail decisions — it doesn't replace the parser.
5. **Do NOT create HTTP endpoints for action confirmation in E5-3.** Action confirmation is exclusively via WebSocket (the existing stub). HTTP fallback for actions may be added in a future story if needed.
6. **Do NOT skip the guardrail chain for any action.** Every single action, regardless of type or confidence, must pass through `GuardrailsService.evaluate()` before being sent to the user.
7. **Do NOT modify the `ai.action.executed` event name or remove existing fields.** Only ADD new fields (`companyId`, `conversationId`, `actionType`).
8. **Do NOT implement actual business module handlers in E5-3.** The `ActionExecutor.registerHandler()` pattern is for future modules (Finance, AR, AP). E5-3 returns `ACTION_TYPE_NOT_IMPLEMENTED` for all action types.
