## User Story

As a **platform vendor**,
I want to curate and push best-practice knowledge articles, default configurations, and AI improvements to all tenants, so that every tenant benefits from collective intelligence — while each tenant admin retains full control over what enters their knowledge base.

## Acceptance Criteria

### AC1: Platform Knowledge Article CRUD (Vendor Admin)
**GIVEN** the `PlatformKnowledgeArticle` table (already created by E5d-3 migration)
**WHEN** a platform admin manages knowledge articles
**THEN** the following endpoints are available under `/admin/intelligence/knowledge`:
  - `GET /admin/intelligence/knowledge` — list platform knowledge articles with filters: `status` (DRAFT/PUBLISHED/ARCHIVED), `category` (BEST_PRACTICE/HELP/DEFAULT_CONFIG/SKILL_UPDATE), cursor pagination (permission: PLATFORM_ADMIN)
  - `GET /admin/intelligence/knowledge/:id` — get single article (permission: PLATFORM_ADMIN)
  - `POST /admin/intelligence/knowledge` — create new article in DRAFT status (permission: PLATFORM_ADMIN)
  - `PATCH /admin/intelligence/knowledge/:id` — update article content/metadata; if article is PUBLISHED, increment `version` on content changes (permission: PLATFORM_ADMIN)
  - `POST /admin/intelligence/knowledge/:id/publish` — transition from DRAFT → PUBLISHED, set `publishedAt` (permission: PLATFORM_ADMIN)
  - `POST /admin/intelligence/knowledge/:id/archive` — transition from PUBLISHED → ARCHIVED (permission: PLATFORM_ADMIN)
  - `DELETE /admin/intelligence/knowledge/:id` — hard-delete DRAFT articles only; PUBLISHED/ARCHIVED cannot be deleted (permission: PLATFORM_ADMIN)
**AND** all endpoints are protected by `requirePlatformRole(PlatformRole.PLATFORM_ADMIN)`
**AND** all admin write operations are audit-logged via the existing `PlatformAuditPlugin`
**AND** `targetIndustries` accepts an array of industry strings (empty array = all industries)
**AND** `targetPlanTiers` accepts an array of plan tier codes (empty array = all tiers)
**AND** responses use `sendSuccess(reply, data)` with `successEnvelope()` wrapper

### AC2: Platform Knowledge Response Tracking Schema
**GIVEN** the platform Prisma schema
**WHEN** migrations run
**THEN** a new `platform_knowledge_responses` table is created with:
  - `id` (UUID PK), `tenantId` (FK to Tenant), `articleId` (FK to PlatformKnowledgeArticle), `articleVersion` (Int — the version at time of response), `status` (ACCEPTED/REJECTED), `tenantArticleId` (String? — ID of the tenant-side copy if accepted), `respondedAt` (DateTime)
  - Unique constraint: `(tenantId, articleId)` — one response per tenant per article
  - Index: `(tenantId)` for efficient per-tenant lookups
  - Index: `(articleId)` for efficient per-article distribution tracking
**AND** the `PlatformKnowledgeArticle` model gains an inverse relation `responses PlatformKnowledgeResponse[]`
**AND** the `Tenant` model gains an inverse relation `knowledgeResponses PlatformKnowledgeResponse[]`
**AND** no `companyId` column exists on this table (project-context.md §8b: platform DB has no companyId)

### AC3: Internal Knowledge Distribution Endpoint (ERP-Facing)
**GIVEN** a tenant's ERP API needs to query available platform knowledge
**WHEN** the ERP calls the platform API via service token
**THEN** a new internal endpoint is available:
  - `GET /platform/tenants/:tenantId/knowledge/suggested` — returns PUBLISHED platform articles that:
    - Match the tenant's industry (or have empty `targetIndustries` = all)
    - Match the tenant's plan tier (or have empty `targetPlanTiers` = all)
    - Have NOT been accepted or rejected by this tenant at the current version (no `PlatformKnowledgeResponse` row with matching `articleVersion`)
    - If a previously-accepted article has a newer version, it re-appears as a suggestion
  - Response includes: `id`, `title`, `content`, `category`, `version`, `publishedAt`, `previousResponse` (null | { status, articleVersion } — if they responded to a previous version)
  - Authenticated via service token (same pattern as `/platform/tenants/:tenantId/entitlements`)
  - Supports cursor pagination with `limit` default 20
**AND** a `POST /platform/tenants/:tenantId/knowledge/:articleId/respond` endpoint allows:
  - Body: `{ status: 'ACCEPTED' | 'REJECTED', tenantArticleId?: string }`
  - Creates or upserts a `PlatformKnowledgeResponse` row
  - Authenticated via service token

### AC4: PlatformClient Extension for Knowledge
**GIVEN** the `@nexa/platform-client` package
**WHEN** the tenant ERP API needs to query platform knowledge
**THEN** the `PlatformClient` interface gains two new methods:
  - `getSuggestedKnowledge(tenantId: string, opts?: { cursor?: string; limit?: number }): Promise<SuggestedKnowledgeResult>`
  - `respondToKnowledge(tenantId: string, articleId: string, response: { status: 'ACCEPTED' | 'REJECTED'; tenantArticleId?: string }): Promise<void>`
**AND** the `PlatformClientService` implements these with HTTP calls to the platform API endpoints from AC3
**AND** suggested knowledge calls are NOT cached (always live — suggestions change when articles are published or tenant responds)
**AND** types are exported from `@nexa/platform-client`

### AC5: Tenant-Side Suggested Knowledge Endpoints
**GIVEN** the tenant API under `/ai/knowledge-articles`
**WHEN** a tenant admin queries suggested knowledge
**THEN** the following new endpoints are available:
  - `GET /ai/knowledge-articles/suggested` — proxies to `PlatformClient.getSuggestedKnowledge()` with the current tenant's ID, returns articles formatted for the UI with Accept/Reject/Edit&Accept actions (permission: ADMIN role)
  - `POST /ai/knowledge-articles/suggested/:platformArticleId/accept` — creates a copy in the tenant's `ai_knowledge_articles` with:
    - `source` = `PLATFORM_SUGGESTED`
    - `confidenceScore` = 0.9
    - `isConfirmed` = true
    - `sourceRef` = platform article ID
    - Content/title/category copied from the platform article
    - Then calls `PlatformClient.respondToKnowledge()` with status `ACCEPTED` and the new tenant article ID
    - Emits `ai.knowledge.articleCreated` event
    - Returns the created tenant article (permission: ADMIN role)
  - `POST /ai/knowledge-articles/suggested/:platformArticleId/reject` — calls `PlatformClient.respondToKnowledge()` with status `REJECTED` (permission: ADMIN role)
  - `POST /ai/knowledge-articles/suggested/:platformArticleId/accept-edited` — same as accept but with a request body containing modified `title`, `content`, and/or `category` that override the platform article's values (permission: ADMIN role)
**AND** if `PlatformClient` is unavailable (not configured), the suggested endpoints return empty results with a 200 (graceful degradation per IMP-006)
**AND** the accept flow triggers the chunking/embedding pipeline on the newly created tenant article (same as `KnowledgeArticleService.createArticle()`)
**AND** `companyId` scoping is enforced — tenant articles are created for the authenticated user's active company

### AC6: Platform Knowledge Article Versioning
**GIVEN** a platform knowledge article that has been PUBLISHED and accepted by some tenants
**WHEN** the vendor admin updates the content and re-publishes (content change on a PUBLISHED article)
**THEN** the article's `version` field is incremented automatically
**AND** existing `PlatformKnowledgeResponse` rows with the old `articleVersion` are NOT deleted — they remain as historical records
**AND** the article re-appears in the "Suggested Knowledge" list for tenants who accepted the previous version (because `articleVersion` in their response row doesn't match the current `version`)
**AND** the `previousResponse` field in the suggestion response indicates what the tenant did with the prior version: `{ status: 'ACCEPTED', articleVersion: N }`
**AND** tenants who rejected the previous version also see the new version (rejection only hides the specific version, not future versions)

### AC7: Skill Update Distribution
**GIVEN** skill effectiveness data from `ai_skill_effectiveness` showing a skill with `avgSuccessRate < 50%` across tenants
**WHEN** the vendor reviews and creates an improvement
**THEN** the vendor can create a platform knowledge article with category `SKILL_UPDATE` targeting that skill:
  - The article title follows the pattern: "Skill Update: {skillKey} — improved guidance"
  - The article content contains improved usage guidance, common corrections, and best practices for that skill
  - `targetIndustries` and `targetPlanTiers` can optionally narrow the audience
**AND** this article follows the standard publish → distribute flow (no special plumbing — it's just a knowledge article with category `SKILL_UPDATE`)
**AND** tenants see these in their Suggested Knowledge alongside other categories
**AND** the platform admin can link the article to a specific `PlatformAiInsight` of type `SKILL_IMPROVEMENT` by storing the insight ID in a notes/metadata field (informational, not enforced)

### AC8: Default Configuration Suggestions for New Tenant Onboarding
**GIVEN** a platform knowledge article with category `DEFAULT_CONFIG` targeting a specific industry
**WHEN** a new tenant is provisioned with that industry
**THEN** the standard suggested knowledge flow applies — the tenant admin sees DEFAULT_CONFIG articles in their Suggested Knowledge list after first login
**AND** no special onboarding code is required — the existing `GET /ai/knowledge-articles/suggested` endpoint handles industry filtering naturally
**AND** the vendor can pre-create DEFAULT_CONFIG articles for common industries (construction, retail, manufacturing, professional services, etc.) as published platform knowledge

### AC9: Distribution Analytics
**GIVEN** the `platform_knowledge_responses` table accumulates acceptance/rejection data
**WHEN** a platform admin queries article distribution status
**THEN** the `GET /admin/intelligence/knowledge/:id` response includes:
  - `distributionStats`: `{ totalEligibleTenants: number, accepted: number, rejected: number, pending: number }`
  - Calculated from `PlatformKnowledgeResponse` rows for the current `version`
  - `totalEligibleTenants` = count of active tenants matching industry/tier filters
**AND** the `GET /admin/intelligence/knowledge` list includes a `distributionSummary` per article: `{ accepted: number, rejected: number, pending: number }`

## Architecture Decision: Two-Database Communication

Platform knowledge distribution requires bi-directional communication between the Platform API and tenant ERP API:

```
Platform DB (platform-api)                    Tenant DB (api)
┌─────────────────────────┐                  ┌─────────────────────────┐
│ PlatformKnowledgeArticle│                  │ AiKnowledgeArticle      │
│   (DRAFT→PUBLISHED→    │  ──service-──→   │   (source: PLATFORM_    │
│    ARCHIVED)            │    token call     │    SUGGESTED)           │
│                         │                  │   (sourceRef: platform   │
│ PlatformKnowledgeResp.  │  ←──response──   │    article ID)          │
│   (ACCEPTED/REJECTED    │    recorded      │                         │
│    per tenant per ver.) │                  │ + chunking + embedding  │
└─────────────────────────┘                  └─────────────────────────┘

Flow:
1. Vendor creates + publishes platform article → PlatformKnowledgeArticle.status = PUBLISHED
2. Tenant admin opens Suggested Knowledge → tenant API calls PlatformClient.getSuggestedKnowledge()
3. PlatformClient → Platform API (service token) → queries eligible articles minus responded
4. Tenant admin accepts → tenant API:
   a. Creates local AiKnowledgeArticle copy (source=PLATFORM_SUGGESTED, confidence=0.9)
   b. Triggers chunking + embedding pipeline
   c. Calls PlatformClient.respondToKnowledge(ACCEPTED, tenantArticleId)
5. PlatformClient → Platform API → creates PlatformKnowledgeResponse row
6. Article no longer appears in tenant's suggestions (until vendor publishes new version)
```

## Dev Notes

- **File placement**: Platform routes in `apps/platform-api/src/routes/admin/` (admin-facing) and `apps/platform-api/src/routes/internal/` (ERP-facing). Tenant routes extend existing `apps/api/src/ai/knowledge-article.routes.ts`. Service in `apps/platform-api/src/services/`. Client extension in `packages/platform-client/src/`.
- **No companyId on platform tables**: `PlatformKnowledgeResponse` uses `tenantId`, NOT `companyId`. The tenant-side copy in `AiKnowledgeArticle` does use `companyId` (per multi-company pattern).
- **Auth patterns**:
  - Platform admin endpoints: `requirePlatformRole('PLATFORM_ADMIN')`
  - Internal ERP-facing endpoints: Service token guard (same as entitlements)
  - Tenant-side endpoints: `createRbacGuard({ minimumRole: UserRole.ADMIN })`
- **Audit logging**: Platform admin write operations automatically logged by `PlatformAuditPlugin`. Tenant-side article creation logged via `ai.knowledge.articleCreated` event.
- **Version increment**: Only on content changes to PUBLISHED articles. DRAFT articles can be freely edited without version bumps. Version starts at 1 (default from E5d-3 schema).
- **Response format**: Platform API uses `sendSuccess(reply, data)` with `successEnvelope()` wrapper. Tenant API uses same pattern from E5d-1.
- **Category mapping**: Platform articles use different categories than tenant articles:
  - Platform: `BEST_PRACTICE`, `HELP`, `DEFAULT_CONFIG`, `SKILL_UPDATE`
  - Tenant: `BUSINESS_PROCESS`, `TERMINOLOGY`, `INDUSTRY_RULES`, `CUSTOM_FIELDS`, `HISTORICAL_PATTERN`
  - When accepting, the tenant admin can override the category to match their local taxonomy (via `accept-edited` endpoint)
- **Graceful degradation**: If `PLATFORM_SERVICE_TOKEN` is not set, `fastify.platformClient` is null, and all suggested knowledge endpoints return empty results — no errors thrown for list, 503 for actions.
