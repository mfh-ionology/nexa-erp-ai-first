# Story E7.1: Database Schema & API Foundation

Status: done

## Story

As a **developer**,
I want the metadata-driven DataTable database schema and API endpoints created,
so that the frontend can build a fully automated list page experience from metadata-driven configuration.

## Acceptance Criteria

1. GIVEN the Prisma schema WHEN migrations run THEN the 6 tables (`data_views`, `data_view_fields`, `date_range_presets`, `user_column_preferences`, `saved_views`, `saved_view_conditions`) are created with all fields, indexes, foreign keys, and enums as specified in the Data Models document §3.1
2. GIVEN the database WHEN the seed script runs THEN 20 `DateRangePreset` records are seeded (CUSTOM RANGE, Today, Yesterday, Tomorrow, Last 3/7/30 days, Next 7/30 days, This/Last/Next Week/Month/Year, MTD, YTD) and at least one `DataView` with `DataViewField` rows is seeded (e.g., USERS) as a reference implementation
3. GIVEN the bundled init endpoint `GET /views/init?viewKey=USERS` WHEN called by an authenticated user THEN it returns the `dataView`, all `fields`, `datePresets`, the user's `savedViews` (with conditions), and `userColumnPreferences` in a single response within 100ms
4. GIVEN the saved view CRUD endpoints WHEN a STAFF user creates a view with `scope=PERSONAL` THEN only that user sees it; WHEN `scope=ROLE` THEN all users with the matching `roleId` see it; WHEN `scope=GLOBAL` THEN only ADMIN can create it and all users see it
5. GIVEN the filter-to-Prisma converter WHEN a saved view's conditions are applied THEN the backend correctly builds a Prisma `where` clause supporting all `FilterOperator` values (`EQUALS`, `NOT_EQUALS`, `CONTAINS`, `STARTS_WITH`, `ENDS_WITH`, `GT`, `GTE`, `LT`, `LTE`, `BETWEEN`, `IN`, `NOT_IN`, `IS_EMPTY`, `IS_NOT_EMPTY`) with AND/OR group bracketing
6. GIVEN the `POST /views/lov/batch` endpoint WHEN called with multiple LOV requests THEN it returns all LOV data in a single response, respecting `lovSearchMin` thresholds and `lovDependsOn` parent values
7. GIVEN the `PATCH /views/columns/:viewKey/:fieldId/width` endpoint WHEN called after a column drag-resize THEN the width is persisted to `UserColumnPreference` within 50ms

  - [x] 5.2 Implement operator mapping for all 14 `FilterOperator` values:
    | FilterOperator | Prisma Operator | Notes |
    |----------------|-----------------|-------|
    | `EQUALS` | `equals` | Type-aware: string, number, boolean, date |
    | `NOT_EQUALS` | `not: { equals }` | |
    | `CONTAINS` | `contains` (mode: insensitive) | Strings only |
    | `STARTS_WITH` | `startsWith` (mode: insensitive) | Strings only |
    | `ENDS_WITH` | `endsWith` (mode: insensitive) | Strings only |
    | `GT` | `gt` | Numbers and dates |
    | `GTE` | `gte` | Numbers and dates |
    | `LT` | `lt` | Numbers and dates |
    | `LTE` | `lte` | Numbers and dates |
    | `BETWEEN` | `gte` + `lte` combined | Uses `value` as JSON `[min, max]` or `valueList[0], valueList[1]` |
    | `IN` | `in` | Uses `valueList` array |
    | `NOT_IN` | `notIn` | Uses `valueList` array |
    | `IS_EMPTY` | `equals: null` OR `equals: ''` | Type-dependent |
    | `IS_NOT_EMPTY` | `not: null` AND `not: ''` | Type-dependent |
  - [x] 5.3 Implement group bracketing:
    - Conditions with the same `groupId` are combined with `groupLogic` (AND/OR)
    - Groups are combined with `outerLogic` (AND/OR)
    - Top-level `filterLogic` wraps the outermost combination
    ```typescript
    // Example: (status = POSTED AND amount > 1000) OR (dueDate < today)
    // Group 0: [status EQUALS POSTED, amount GT 1000], groupLogic: AND
    // Group 1: [dueDate LT today], outerLogic: OR
    // Result: { OR: [{ AND: [{status: 'POSTED'}, {amount: {gt: 1000}}] }, {dueDate: {lt: today}}] }
    ```

## Dev Notes

### Architecture Summary

E7.1 creates the **metadata-driven DataTable infrastructure** — a database-driven system where a single `data_view_fields` table defines every column, filter, sort option, and LOV dropdown for every T1 Entity List page. When future module epics are built (E14 Finance, E17 AR, etc.), developers simply seed `DataView` + `DataViewField` rows — zero custom filter/column UI code per entity.

The six tables form three layers:
1. **Metadata Layer:** `data_views` (page registry) + `data_view_fields` (column/filter automation engine) + `date_range_presets` (date filter presets)
2. **User Personalisation Layer:** `user_column_preferences` (per-user column config) + `saved_views` (named filter/sort/column configs)
3. **Filter Persistence Layer:** `saved_view_conditions` (individual filter conditions with group bracketing)

### Performance Strategy

- **Bundled init endpoint** (`GET /views/init?viewKey=X`): 1 API call instead of 4-5. Returns dataView + fields + datePresets + savedViews + columnPrefs.
- **Redis caching**: DataView metadata + fields + datePresets cached with 1hr TTL (key: `{tenantId}:views:meta:{viewKey}`). User-specific data (savedViews, columnPrefs) always fresh from DB.
- **Column width persistence**: `PATCH .../width` is a lightweight upsert — must complete within 50ms for smooth drag-resize UX.
- **Partial indexes**: `WHERE is_favourite = true` and `WHERE is_default = true` on `saved_views` — fast lookups for favourites and defaults without indexing all rows.

### Filter Builder Implementation Notes

The filter-builder converts `SavedViewCondition[]` into Prisma `where` clauses. Key considerations:

1. **Security**: Never allow raw SQL injection. All values are parsed by type and applied via Prisma's typed query builder.
2. **Group bracketing**: Conditions are grouped by `groupId`. Within a group, conditions are combined with `groupLogic`. Between groups, conditions are combined with `outerLogic`. The top-level `filterLogic` wraps everything.
3. **Date presets**: When `datePresetId` is set, the preset key is resolved to actual date values at query time (not stored as dates). This means "Today" always means today, not the day the view was saved.
4. **Nested fields**: For fields that resolve to related entities (e.g., `customer.name`), the filter-builder must produce nested Prisma where clauses: `{ customer: { name: { contains: 'Acme' } } }`. The `fieldKey` can use dot-notation for this.
