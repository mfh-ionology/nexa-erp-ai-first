# Story E5d-5: Knowledge Management UI (Tenant)

**Epic:** E5d — AI Knowledge Evolution & Cross-Tenant Intelligence
**Status:** done
**Dependencies:** E5d-1 (Knowledge Base Schema & RAG Pipeline — done), E5d-2 (Correction Loop & Training Examples — done), E5d-4 (Platform Knowledge Distribution — backlog, tenant-side suggested endpoints needed)
**Complexity:** X-Large (11 tasks)
**FRs:** FR4 (contextual AI), FR6 (human approval gate) | **NFRs:** NFR2 (CRUD <500ms), NFR27 (WCAG 2.1 AA), NFR28 (keyboard navigable), NFR29 (contrast 4.5:1)

---

## User Story

As a **tenant admin**,
I want to manage my company's AI knowledge base — upload documents, review corrections, manage training examples, and accept/reject platform suggestions — so that the AI accurately understands our business, gets smarter from corrections, and benefits from vendor-curated best practices.

---

## Acceptance Criteria

### AC1: Page Shell, Navigation, and Routing
**GIVEN** the Knowledge Management page at route `/ai/admin/knowledge`
**WHEN** a user with ADMIN role navigates to it
**THEN** the page renders with:
  - Breadcrumb: `AI Administration > Knowledge Management`
  - Page header with title "Knowledge Management" and action bar slot
  - Five tabs: `Knowledge Articles`, `Training Examples`, `Corrections`, `Suggested`, `Settings`
  - A persistent stats panel above the tabs (see AC9)
  - The page loads with the `Knowledge Articles` tab active by default
  - Tab selection is reflected in the URL hash (e.g., `/ai/admin/knowledge#corrections`) for deep-linking
**AND** the sidebar navigation shows a "Knowledge" item under the "AI Administration" section with a `BookOpen` icon
**AND** the route is lazy-loaded (React Router v7 `lazy()` pattern matching existing AI admin pages)
**AND** the route is protected by ADMIN role guard — non-admin users are redirected
**AND** the page background is `#f4f2ff` (Concept D) — not white or grey
**AND** the page uses `animate-fade-in-up` on load with staggered card delays

### AC2: Knowledge Articles Tab — List View
**GIVEN** the Knowledge Articles tab is active
**WHEN** rendered
**THEN** articles are displayed as cards grouped by category sections:
  - `Business Processes` (BUSINESS_PROCESS)
  - `Terminology` (TERMINOLOGY)
  - `Industry Rules` (INDUSTRY_RULES)
  - `Custom Fields` (CUSTOM_FIELDS)
  - `Historical Patterns` (HISTORICAL_PATTERN)
**AND** each article card shows:
  - `title` — primary text, Plus Jakarta Sans
  - Source badge with colour coding: `Admin` (purple), `AI Generated` (blue), `Platform` (green), `From Corrections` (amber)
  - Confidence score indicator: High (green, ≥0.9), Medium (amber, 0.7–0.89), Low (red, <0.7) — score value shown in JetBrains Mono
  - `isConfirmed` status: unconfirmed AI-generated articles show an amber "Needs Review" badge with prominent Confirm action
  - `chunkCount` in mono font (e.g., "4 chunks")
  - `usageCount` — "Used 12 times" with `lastUsedAt` relative timestamp
  - `isActive` toggle switch
  - Overflow menu: Edit, Confirm (if unconfirmed), Delete
**AND** the list supports cursor-based pagination with infinite scroll (load-more pattern)
**AND** the list supports filtering by: category (multi-select), source (multi-select), active/inactive status
**AND** the list has a search input that filters by title (client-side on loaded results)
**AND** empty state shows: dashed-border card with "No knowledge articles yet. Upload a document or create an article to help the AI understand your business."
**AND** loading state shows 3 skeleton cards per category section
**AND** cards use 12px radius, custom shadow, purple-tinted hover shadow (Concept D)

### AC3: Knowledge Articles Tab — Upload & Create Actions
**GIVEN** the Knowledge Articles tab action bar
**WHEN** rendered
**THEN** two primary actions are available:
  - `[Upload Document]` — opens upload dialog (primary button, `#7c3aed`)
  - `[Create Article]` — opens article creation dialog (secondary/outline button)
**AND** the Upload Document dialog contains:
  - A drag-drop zone with dashed border accepting `.txt`, `.md` files (PDF text extraction deferred to future enhancement per E5d-1 dev notes — accept `.pdf` with a note that text content must be pasted for now)
  - A file picker fallback button: "Drop files here or click to browse"
  - File type validation with error message for unsupported types
  - A text area for direct paste of content (alternative to file upload)
  - Category selector dropdown (required)
  - Title field (required, 1–500 chars)
  - Submit triggers `POST /ai/knowledge-articles` with content, title, and category
  - Processing status indicator: after submit, show "Processing… chunking & embedding" with animated spinner until completion
  - Success toast: "Knowledge article created and indexed"
**AND** the Create Article dialog contains:
  - Title field (required, 1–500 chars)
  - Category selector (required)
  - Content area with markdown support (textarea, 1–100,000 chars)
  - Submit triggers `POST /ai/knowledge-articles`
  - Success toast and list refresh
**AND** dialogs use `animate-step-in` animation and 12px radius (Concept D)

### AC4: Knowledge Articles Tab — Edit, Confirm, Delete
**GIVEN** a knowledge article in the list
**WHEN** the admin clicks Edit from the overflow menu
**THEN** an edit dialog opens pre-filled with the article's current `title`, `content`, and `category`
**AND** the `source` field is displayed read-only (immutable per E5d-1 AC9)
**AND** if `content` is changed, the dialog shows a note: "Content changes will re-chunk and re-embed this article"
**AND** submit triggers `PATCH /ai/knowledge-articles/:id` with optimistic UI update
**AND** if the article is unconfirmed (`isConfirmed = false`), the Confirm action:
  - Sends `PATCH /ai/knowledge-articles/:id` with `{ isConfirmed: true }`
  - Backend auto-upgrades `confidenceScore` to 0.8 (per E5d-2 AC9)
  - Optimistic update: badge changes to "Confirmed" immediately
  - Success toast: "Article confirmed — confidence upgraded to 0.8"
**AND** the Delete action:
  - Shows `AlertDialog` confirmation: "This will deactivate the article. It will no longer be used for AI responses."
  - On confirm, sends `DELETE /ai/knowledge-articles/:id` (soft-delete: `isActive = false`)
  - Optimistic removal from the list
  - Undo toast with 5-second window (re-activate via `PATCH /ai/knowledge-articles/:id` with `{ isActive: true }`)

### AC5: Training Examples Tab
**GIVEN** the Training Examples tab is active
**WHEN** rendered
**THEN** training examples are displayed as cards showing:
  - `inputText` — label "When user asks:", truncated to 2 lines, in normal font
  - Arrow indicator (→)
  - `outputText` — label "AI should answer:", truncated to 2 lines, in normal font
  - `skillKey` badge (if set): mono font, purple bg
  - `category` badge
  - Source badge: `Admin Curated` (purple) or `From Corrections` (amber)
  - `isActive` toggle switch
  - `[Test]` button per row — disabled with tooltip "Coming soon" for MVP (no test endpoint exists yet)
  - Overflow menu: Edit, Delete
**AND** the action bar has a `[+ Add Example]` primary button that opens a dialog with:
  - Input text area (required, label "When user asks…", 1–10,000 chars)
  - Output text area (required, label "AI should respond…", 1–10,000 chars)
  - Skill key input (optional, with helper text: "Optionally scope to a specific skill")
  - Category selector (required)
  - Submit triggers `POST /ai/training-examples`
**AND** the list supports filtering by: `category` (multi-select), `skillKey` (text input), `isActive` status
**AND** the list uses cursor-based pagination with load-more
**AND** Edit dialog pre-fills all fields except `source` (read-only, immutable)
**AND** Delete shows confirmation dialog, triggers soft-delete (`DELETE /ai/training-examples/:id`)
**AND** all mutations use optimistic updates matching the E5c pattern

### AC6: Corrections Tab
**GIVEN** the Corrections tab is active
**WHEN** rendered
**THEN** corrections are displayed in two parts:
  1. **Stats row** at the top showing (from `GET /ai/corrections/stats`):
     - Total corrections (all time)
     - Corrections last 30 days with trend indicator (↑ increasing / ↓ decreasing)
     - Breakdown by type: horizontal segmented bar chart (each correctionType a segment)
     - Auto-resolved count with percentage
  2. **Corrections list** grouped by `correctionType` (collapsible accordion sections):
     - `TERMINOLOGY`, `PROCESS`, `DATA`, `PREFERENCE`, `OTHER`
     - Each correction shows:
       - `originalResponse` — italic, muted text, truncated to 2 lines
       - `correctedResponse` — bold text, truncated to 2 lines
       - `skillKey` badge (if present)
       - `wasAutoResolved` badge: green "Auto-resolved" if true, grey "Pending" if false
       - `createdAt` relative timestamp
       - `[Create Article]` button — opens pre-filled knowledge article dialog
         - Title pre-filled: "From correction: {first 80 chars of correctedResponse}"
         - Content pre-filled: correctedResponse
         - Category auto-mapped from correctionType (TERMINOLOGY→TERMINOLOGY, PROCESS→BUSINESS_PROCESS, DATA→CUSTOM_FIELDS, PREFERENCE→BUSINESS_PROCESS, OTHER→BUSINESS_PROCESS)
         - Submit triggers `POST /ai/corrections/:correctionId/create-article`
         - Success toast: "Knowledge article created from correction"
**AND** the list supports filtering by: `correctionType`, `skillKey`, `wasAutoResolved`, date range (`from`, `to`)
**AND** the list uses cursor-based pagination
**AND** corrections are read-only — no edit or delete actions (immutable log per E5d-2 AC1)
**AND** the stats panel data is fetched separately from the list (parallel queries)

### AC7: Suggested Knowledge Tab
**GIVEN** the Suggested tab is active
**WHEN** rendered and `platformClient` is available (configured)
**THEN** platform-suggested knowledge articles are displayed as cards showing:
  - `title` from the platform article
  - Category badge (platform taxonomy): `Best Practice` (blue), `Help` (teal), `Default Config` (purple), `Skill Update` (amber)
  - Content preview (first ~200 chars, truncated) with "Read more" toggle
  - `version` indicator (e.g., "v2")
  - `publishedAt` date
  - `previousResponse` context: "Previously accepted v1" / "Previously rejected v1" (amber info banner if present)
  - Three action buttons per card:
    - `[Accept]` (primary, purple) — calls `POST /ai/knowledge-articles/suggested/:platformArticleId/accept`
      - Creates tenant copy with source=PLATFORM_SUGGESTED, confidence=0.9
      - Success toast: "Knowledge article accepted and added to your knowledge base"
      - Card transitions out of the list with slide-out animation
    - `[Reject]` (destructive/outline) — calls `POST /ai/knowledge-articles/suggested/:platformArticleId/reject`
      - Confirmation: "Reject this suggestion? You can still see new versions if the vendor updates it."
      - Card transitions out
    - `[Edit & Accept]` (secondary) — opens editable dialog pre-filled with platform article content
      - Admin can modify `title`, `content`, and remap `category` to tenant taxonomy (BUSINESS_PROCESS, TERMINOLOGY, etc.)
      - Submit calls `POST /ai/knowledge-articles/suggested/:platformArticleId/accept-edited`
      - Success toast: "Knowledge article accepted with your edits"
**AND** if `platformClient` is not configured (null), the tab shows a friendly info state: "Platform suggestions are not available. Contact your administrator to enable platform integration."
**AND** if the suggested list is empty (no pending suggestions), show: "You're all caught up — no new suggestions from the vendor."
**AND** the list uses cursor-based pagination with load-more

### AC8: Settings Tab
**GIVEN** the Settings tab is active
**WHEN** rendered
**THEN** a settings form is displayed in T7 Settings layout (label left, control right on desktop; stacked on mobile):
  - **Enable AI Knowledge Base** — Switch toggle
    - Description: "When enabled, the AI uses your company's knowledge articles to provide more accurate answers."
    - Controls whether RAG retrieval is active for this company
  - **Share Anonymised Patterns** — Switch toggle
    - Description: "Allows anonymised usage patterns to be shared with the vendor to improve AI for all tenants. No actual data leaves your system — only statistical counts and categories."
    - Maps to `TenantFeatureFlag` key `share_anonymised_ai_patterns`
  - **Knowledge Categories** — Multi-select checkboxes
    - Which categories are included in RAG retrieval
    - Options: Business Processes, Terminology, Industry Rules, Custom Fields, Historical Patterns
  - **Auto-generated Article Retention** — Number input (days, default 90, range 30–365)
    - Description: "How long unconfirmed AI-generated articles are kept before automatic cleanup."
  - **RAG Token Budget** — Slider with number display (default 1000, range 500–2000, step 100)
    - Description: "Maximum tokens allocated for knowledge context in AI responses. Higher values provide more context but may slow responses."
**AND** the form has a `[Save Settings]` primary action button in the action bar
**AND** settings are loaded from the API on tab mount and saved on submit
**AND** unsaved changes show a confirmation prompt on tab/page navigation
**AND** the form respects the T7 responsive pattern: two-column on desktop, single-column stacked on tablet/phone

### AC9: Stats Panel (Persistent Above Tabs)
**GIVEN** the Knowledge Management page is loaded
**WHEN** rendered
**THEN** a stats panel is always visible above the tab bar showing 4 KPI cards:
  1. **Total Articles** — count of active knowledge articles, with a small breakdown by source (Admin/AI/Platform/Corrections) as a segmented mini-bar
  2. **RAG Retrieval Rate** — percentage of AI queries that used knowledge articles (from learning signals), displayed as a percentage with a coloured ring indicator
  3. **Correction Trend** — sparkline or mini-chart showing correction count over the last 30 days with direction indicator (↑/↓)
  4. **Pending Reviews** — count of articles with `isConfirmed = false` (amber highlight if > 0, links to filtered Knowledge Articles tab)
**AND** stats are fetched from:
  - `GET /ai/knowledge-articles` with `isActive=true` count for total articles
  - `GET /ai/corrections/stats` for correction trend
  - Count of unconfirmed articles for pending reviews
**AND** stats cards use Concept D styling: 12px radius, custom shadow, purple-tinted hover shadow
**AND** stats cards have `animate-fade-in-up` with staggered `animationDelay` (0ms, 50ms, 100ms, 150ms)
**AND** the "Pending Reviews" card is clickable — navigates to the Knowledge Articles tab filtered by `isConfirmed = false`

### AC10: Concept D Visual Design Compliance
**GIVEN** the Knowledge Management page and all its components
**WHEN** rendered
**THEN** the following Concept D design tokens are applied:
  - Background: `#f4f2ff` (primary-50)
  - Cards: `rounded-xl` (12px), `shadow-[0_1px_3px_rgba(0,0,0,0.06)]`, hover: `shadow-[0_4px_12px_rgba(124,58,237,0.10)]`
  - Primary button: `bg-[#7c3aed] hover:bg-[#5b21b6] rounded-lg` (8px)
  - Typography: headings in Plus Jakarta Sans (`font-serif`), body in Inter (`font-sans`), scores/codes/counts in JetBrains Mono (`font-mono`)
  - Active/selected elements: `bg-[#f5f3ff] text-[#7c3aed] border border-[#7c3aed]/20`
  - Destructive actions: `bg-[#dc2626] hover:bg-[#b91c1c]`
  - Animations: `animate-fade-in-up` on page/card load, `animate-step-in` on modals/dialogs
  - `prefers-reduced-motion` respected — all animations disabled when user preference is set
**AND** all colour combinations meet WCAG 2.1 AA contrast requirements (4.5:1 for normal text, 3:1 for large text)
**AND** all interactive elements are keyboard-navigable (Tab, Enter, Escape)
**AND** status badges use both colour AND text labels (not colour-only — NFR29)

### AC11: Responsive Design
**GIVEN** the Knowledge Management page
**WHEN** viewed on different screen sizes
**THEN** the layout adapts:
  - **Desktop (≥1024px):** Full layout — stats panel horizontal row, tabs as horizontal tab bar, article cards in grid
  - **Tablet (768–1023px):** Stats panel wraps to 2×2 grid, tabs as horizontal tab bar (may scroll), article cards stack vertically
  - **Mobile (<768px):** Stats panel stacks vertically, tabs render as a dropdown selector, article cards full-width, Settings tab uses single-column layout
**AND** the upload drag-drop zone remains functional on all sizes
**AND** all dialogs are full-screen on mobile (sheet pattern from bottom)

## Dev Notes

- **File placement**: All new files go in `apps/web/src/features/ai-admin/knowledge/` and `apps/web/src/features/ai-admin/api/` — matching the existing AI admin feature structure
- **i18n**: All files should have `/* eslint-disable i18next/no-literal-string */` at the top (matching all existing AI admin pages). Translation keys are used where feasible but literal strings are acceptable for MVP
- **Upload is text content**: For MVP, the upload dialog extracts text from `.txt`/`.md` files via `FileReader.readAsText()`. The API receives the text content in the `content` field — no binary file upload endpoint exists. PDF text extraction is a future enhancement per E5d-1 dev notes
- **Test button**: The "Test" button on training examples is scaffolded as disabled for MVP. No `/ai/training-examples/:id/test` endpoint exists. The button will be enabled when a test endpoint is added in a future story
- **Settings persistence**: If a dedicated knowledge settings API endpoint does not exist, use the existing tenant feature flags API for boolean toggles (e.g., `share_anonymised_ai_patterns`) and localStorage for numeric settings (retention, RAG budget). Document this as a follow-up to add a proper settings endpoint
- **Suggested tab dependency on E5d-4**: The suggested tab requires the tenant-side endpoints from E5d-4 Tasks 5–6 to be implemented. If E5d-4 is not complete when this story starts, the Suggested tab should be scaffolded with a "Coming Soon" state and wired up once E5d-4 is done
- **No new backend work**: This story is frontend-only. All API endpoints are created by E5d-1, E5d-2, and E5d-4. No Prisma schema changes, no new routes, no backend services
- **Animation pattern**: Match `ai-config-dashboard.tsx` and `agent-list-page.tsx` exactly — `animate-fade-in-up` on cards, `animate-step-in` on dialogs, staggered delays on stats cards
- **Permission pattern**: The page-level ADMIN guard protects the route. Individual mutation hooks don't need extra permission checks — the API enforces ADMIN role on POST/PATCH/DELETE. For read-only users (if route guard is loosened in future), hide action buttons conditionally based on user role from `useAuthStore`
