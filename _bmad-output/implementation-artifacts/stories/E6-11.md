# Story E6-11: Permission-Driven Route Guards & Field Visibility

**Epic:** E6 — Web Frontend Shell + Mobile Scaffold
**Status:** done
**Dependencies:** E6.1 (React App Bootstrap), E6.2 (Navigation Shell — sidebar, header, company switcher), E6.3 (Screen Template System — T1-T8 templates), E6.4 (ActionBar — status-driven action visibility), E2b (Granular RBAC backend — `GET /system/my-permissions` API, permission resolution, field filter)
**Complexity:** Large (10 tasks)

---

## User Story

As a user, I want the frontend to enforce my access group permissions — hiding unauthorized sidebar items, blocking unauthorized routes, and respecting field-level visibility — so that I only see and interact with what my permissions allow.

---

## Acceptance Criteria

### AC1: Route-Level Permission Guards via `beforeLoad`
**GIVEN** a user who lacks `canAccess=true` for any resource in the "finance" module
**WHEN** they navigate directly to `/finance/journals` via URL
**THEN** they are redirected to the `/403` Access Denied page
**AND** the redirect happens in TanStack Router's `beforeLoad` hook (before the route component renders)
**AND** the `ModuleGuard` component wrapper is removed from individual page components in favour of the `beforeLoad` approach

### AC2: SUPER_ADMIN Bypasses All Route Guards
**GIVEN** a user with `isSuperAdmin=true` in their resolved permissions
**WHEN** they navigate to any route in the application
**THEN** the route renders normally regardless of their access group permissions
**AND** no `/403` redirect ever occurs for SUPER_ADMIN users

### AC3: Permission-Driven Sidebar — Module-Level Filtering
**GIVEN** a user with `enabledModules: ['system', 'sales']` in their resolved permissions
**WHEN** the sidebar renders
**THEN** only the System and Sales module groups are visible
**AND** all other module groups (Finance, AR, AP, Purchasing, Inventory, CRM, HR, Manufacturing, Reporting) are hidden entirely
**AND** SUPER_ADMIN users see all 11 module groups (existing behaviour preserved)

### AC4: Permission-Driven Sidebar — Item-Level Filtering
**GIVEN** a user with `canAccess=true` for `system.users.list` but `canAccess=false` for `system.access-groups.list`
**WHEN** the System module group is expanded in the sidebar
**THEN** the "Users" nav item is visible
**AND** the "Access Groups" nav item is hidden
**AND** if ALL items within a module group are hidden, the entire module group header is also hidden (even if the module is in `enabledModules`)

### AC5: Button Visibility — `[+ New]` and CRUD Actions
**GIVEN** a user with `canNew=false` for a resource
**WHEN** a T1 Entity List page renders for that resource
**THEN** the `[+ New]` button is hidden (not disabled, fully removed from DOM)
**AND** given `canEdit=false`, the Edit action in overflow menus is hidden
**AND** given `canDelete=false`, the Delete action and batch delete toolbar button are hidden
**AND** both status-driven AND permission-driven checks must pass for any action to appear (per UX Design Spec Action Bar Rule #1)

### AC6: Field-Level Visibility — HIDDEN Fields
**GIVEN** a user whose resolved `fieldOverrides` for resource `sales.orders.detail` includes `{ costPrice: 'HIDDEN' }`
**WHEN** a record detail page or form renders for that resource
**THEN** the `costPrice` field is not rendered at all (removed from DOM, not just hidden via CSS)
**AND** the field's label, input, and container are all absent

### AC7: Field-Level Visibility — READ_ONLY Fields
**GIVEN** a user whose resolved `fieldOverrides` for resource `sales.orders.detail` includes `{ discountPercent: 'READ_ONLY' }`
**WHEN** a record detail form renders for that resource
**THEN** the `discountPercent` field is rendered as a disabled input (visible but non-editable)
**AND** the field has `aria-readonly="true"` for screen readers

### AC8: `_fieldMeta` Integration from API Responses
**GIVEN** the backend's `filterFieldsByPermission` hook returns a `_fieldMeta` object on API responses with READ_ONLY markers
**WHEN** the frontend processes the API response
**THEN** the `_fieldMeta` markers are applied to form fields matching the field paths
**AND** `_fieldMeta` supplements the Zustand-cached `fieldOverrides` (API response is authoritative over local cache for READ_ONLY markers)

### AC9: SUPER_ADMIN Field Visibility Bypass
**GIVEN** a user with `isSuperAdmin=true`
**WHEN** any form or detail page renders
**THEN** all fields are visible and editable regardless of `fieldOverrides` configuration
**AND** no fields are hidden or rendered as read-only

### AC10: My Permissions Page
**GIVEN** an authenticated user navigates to `/system/my-permissions`
**WHEN** the page renders
**THEN** it displays:
  - Current user name and role (e.g., "ADMIN")
  - Whether the user is SUPER_ADMIN (boolean badge)
  - Current company name
  - List of assigned access groups with code and name
  - Collapsible permission matrix grouped by module (resource code → canAccess, canNew, canView, canEdit, canDelete)
  - List of enabled modules
  - Field overrides grouped by resource code
**AND** the page is read-only (informational display only, no edit capability)
**AND** a sidebar nav item for "My Permissions" is added under the System module

### AC11: Permission Refresh on Company Switch
**GIVEN** a user switches company via the company switcher in the sidebar
**WHEN** the company context changes (Zustand `activeCompanyId` updates)
**THEN** `GET /system/my-permissions` is called again for the new company
**AND** the sidebar, route guards, and field visibility all update to reflect the new company's permission set
**AND** if the user was on a route they no longer have access to in the new company, they are redirected to `/403`

### AC12: Access Denied Page Enhancement
**GIVEN** a user is redirected to `/403` by a route guard
**WHEN** the Access Denied page renders
**THEN** it displays a clear 403 heading and explanation message
**AND** a "Go Home" link navigates to the authenticated home page
**AND** a "Contact your administrator" hint is shown
**AND** the page is fully accessible (WCAG 2.1 AA, keyboard-navigable, screen reader compatible)

### AC13: Permission Hook API
**GIVEN** a developer building a page component
**WHEN** they need to check a specific permission
**THEN** a `usePermission(resourceCode)` hook is available that returns `{ canAccess, canNew, canView, canEdit, canDelete, isSuperAdmin }`
**AND** a `useFieldVisibility(resourceCode)` hook is available that returns a map of fieldPath → `'VISIBLE' | 'READ_ONLY' | 'HIDDEN'`
**AND** both hooks read from the Zustand auth store (no additional API calls)
**AND** both hooks return fully permissive values for SUPER_ADMIN users

### AC14: Accessibility (NFR27, NFR28)
**GIVEN** all permission-gated UI elements
**WHEN** rendered or hidden
**THEN** hidden elements are removed from the DOM (not just `display: none`) so screen readers do not announce them
**AND** the Access Denied page has proper heading hierarchy and ARIA landmarks
**AND** all interactive elements on the My Permissions page are keyboard-navigable with visible focus indicators
**AND** colour contrast meets WCAG 2.1 AA for all permission-related indicators (badges, read-only field styling)

  ```typescript
  import { redirect } from '@tanstack/react-router';
  import { useAuthStore } from '@/stores/auth-store';

  /**
   * Creates a TanStack Router beforeLoad function that checks module-level access.
   * Redirects to /403 if the user lacks canAccess for any resource in the module.
   * SUPER_ADMIN always passes.
   */
  export function createModuleBeforeLoad(moduleKey: string) {
    return () => {
      const { permissions } = useAuthStore.getState();
      if (!permissions) {
        throw redirect({ to: '/login' });
      }
      if (permissions.isSuperAdmin) return;
      if (!permissions.enabledModules.includes(moduleKey)) {
        throw redirect({ to: '/403' });
      }
    };
  }
  ```

    ```tsx
    export const Route = createFileRoute('/_authenticated/system/')({
      beforeLoad: createModuleBeforeLoad('system'),

      return <section>...</section>;
    ```

### Task 5: Field-Level Visibility in Forms
**AC:** #6, #7, #8, #9
**Files:**

- `apps/web/src/lib/form-utils.ts` (modify)

**Subtasks:**
- [x] 5.1 Create `useFieldVisibility(resourceCode: string)` as a standalone hook file (re-export from `use-permissions.ts` if preferred):
  ```typescript
  /**
   * Returns field visibility map for a resource, merging Zustand-cached
   * fieldOverrides with _fieldMeta from API responses.
   */
  export function useFieldVisibility(
    resourceCode: string,
    fieldMeta?: Record<string, string>,
  ): Record<string, 'VISIBLE' | 'READ_ONLY' | 'HIDDEN'> {
    const permissions = useAuthStore((s) => s.permissions);

    return useMemo(() => {
      if (permissions?.isSuperAdmin) return {};

      const overrides = permissions?.fieldOverrides[resourceCode] ?? {};
      if (!fieldMeta) return overrides;

      // Merge: API _fieldMeta is authoritative for READ_ONLY markers
      const merged = { ...overrides };
      for (const [field, vis] of Object.entries(fieldMeta)) {
        if (!merged[field] || merged[field] === 'VISIBLE') {
          merged[field] = vis as 'VISIBLE' | 'READ_ONLY' | 'HIDDEN';
        }
      }
      return merged;
    }, [permissions, resourceCode, fieldMeta]);
  }
  ```
- [x] 5.2 Create `<PermissionField>` wrapper component in `apps/web/src/components/permission-field.tsx`:
  ```typescript
  interface PermissionFieldProps {
    fieldPath: string;
    visibility: Record<string, 'VISIBLE' | 'READ_ONLY' | 'HIDDEN'>;
    children: ReactNode; // the form field
  }
  ```
  - If visibility is `HIDDEN`: render `null` (field removed from DOM)
  - If visibility is `READ_ONLY`: clone children with `disabled={true}` and `aria-readonly="true"`
  - If visibility is `VISIBLE` or not in map: render children unchanged
- [x] 5.3 Create a `getFieldProps(fieldPath: string, visibility: FieldVisibilityMap)` utility:
  ```typescript
  export function getFieldProps(
    fieldPath: string,
    visibility: Record<string, 'VISIBLE' | 'READ_ONLY' | 'HIDDEN'>,
  ): { hidden: boolean; disabled: boolean; ariaReadOnly?: boolean } {
    const vis = visibility[fieldPath] ?? 'VISIBLE';
    return {
      hidden: vis === 'HIDDEN',
      disabled: vis === 'READ_ONLY',
      ariaReadOnly: vis === 'READ_ONLY' ? true : undefined,
    };
  }
  ```
  - Provides a simpler API for forms that don't want to use `<PermissionField>` wrapper

**Implementation notes:**
- The `_fieldMeta` object is returned by the backend's `filterFieldsByPermission` onSend hook. It's present on individual record API responses (GET detail endpoints). The frontend should extract it and pass it to `useFieldVisibility`.
- For list endpoints, `_fieldMeta` applies uniformly to all returned records.
- HIDDEN fields are already stripped from the API response by the backend. The frontend additionally removes the form field from the DOM for fields that the user shouldn't see (defense in depth).
- Forms built with React Hook Form + Zod should use `getFieldProps` to conditionally render/disable fields.

## Dev Notes

1. **Frontend-Only Story**: This story requires NO backend changes. All permission resolution is done by the E2b backend. The frontend consumes `GET /system/my-permissions` and renders accordingly.

2. **Permissions Map Key Format**: The `permissions.modules` map in the Zustand store is keyed by **resource code** (e.g., `system.users.list`), NOT by module key (e.g., `system`). Module-level access is determined by the `enabledModules` array. Don't confuse the two.

3. **Defense in Depth**: The backend already enforces permissions on every API endpoint via `createPermissionGuard()`. The frontend permission enforcement is a UX convenience — preventing users from seeing UI they can't use. It is NOT a security boundary. If a user bypasses the frontend, the backend still blocks unauthorized actions.

4. **SUPER_ADMIN Is Always Permissive**: Every hook, guard, and component must check `isSuperAdmin` and short-circuit to "allow everything". This is a critical invariant — SUPER_ADMIN users must never see a 403 page or hidden fields.

5. **Company Switch = Full Permission Refresh**: When the user switches company, the entire permission set changes. The sidebar, route guards, and field visibility must all update. There's no partial refresh — it's a full `GET /system/my-permissions` call.

6. **`ModuleGuard` Deprecation**: The existing `ModuleGuard` component uses `useEffect` + `useNavigate` which causes a flash of content before redirect. The `beforeLoad` approach prevents this by blocking the route before the component renders. Mark `ModuleGuard` as deprecated but don't delete it yet.

7. **Resource Codes Must Match Backend**: The `resourceCode` values on `NavigationItem` entries and in `usePermission` calls must exactly match the dot-notation codes in the backend's Resource registry table. If a code doesn't match, the permission check will fail-closed (no access). Verify codes against `packages/db/default-data/company-defaults.json`.

8. **Field Visibility Is Additive**: Both the Zustand-cached `fieldOverrides` and the API response's `_fieldMeta` contribute to field visibility. The API response is authoritative (it reflects the server's current permission state), while the Zustand cache may be up to 60 seconds stale.

9. **My Permissions Page Access**: The My Permissions page at `/system/my-permissions` should be accessible to ALL authenticated users, not just those with system module access. Every user should be able to see what permissions they have. The route guard should check authentication only, not system module access.

10. **Test Strategy**: Focus tests on permission resolution edge cases: SUPER_ADMIN bypass, null permissions (loading state), empty permissions (no access), partial permissions (some modules enabled), and multi-group union scenarios. The backend handles the complex resolution; the frontend just needs to display the resolved result correctly.
