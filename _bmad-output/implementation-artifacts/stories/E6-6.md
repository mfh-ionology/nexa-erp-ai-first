# Story E6.6: Mobile Scaffold

Status: done

## Story

As a **mobile user**,
I want a React Native app with Expo, authentication (email/password + biometric unlock), a tab bar navigation shell, push notification setup, and shared API client integration,
so that I can access AI chat, briefings, and approvals on my phone.

## Acceptance Criteria

1. GIVEN the `apps/mobile` package WHEN it is built THEN it produces an Expo (React Native) application with TypeScript strict mode, Expo Router for file-based routing, and shared workspace packages (`@nexa/api-client`, `@nexa/shared`, `@nexa/i18n`)
2. GIVEN the mobile app WHEN the user opens it unauthenticated THEN they see a login screen with email/password fields, MFA challenge support, and an optional biometric authentication toggle (Face ID / fingerprint) using `expo-local-authentication`
3. GIVEN the mobile app WHEN the user authenticates successfully THEN a bottom tab bar shows four tabs: Chat (primary), Briefing, Approvals, More — with the Chat tab selected by default
4. GIVEN the shared API client in `packages/api-client` WHEN the mobile app makes API calls THEN it uses the same typed client as the web app, with JWT Bearer auth, `X-Company-Id` header, `Accept-Language` header, response envelope parsing, and 401 token refresh
5. GIVEN push notifications WHEN configured with Expo Push THEN the app can register for push tokens on login, store the push token on the server, and receive and display approval requests, briefing alerts, and stock alerts with deep linking to the relevant screen
6. GIVEN any user-facing string in the mobile app WHEN it is displayed THEN it uses the `@nexa/i18n` translation key system (`t()` function) — no hardcoded strings
7. GIVEN the mobile app WHEN the user's session expires or they tap Sign Out THEN tokens are cleared from secure storage, biometric unlock is disabled, and the user is returned to the login screen

### Token Storage Security (Mobile)

Mobile token storage differs from web:
- **Access token**: stored in memory (Zustand store) — same as web
- **Refresh token**: stored in `expo-secure-store` (encrypted OS keychain) — more secure than web's `sessionStorage`
- **Biometric flag**: stored in `expo-secure-store`
- Session restoration: on app launch, read refresh token from `SecureStore` → call `POST /auth/refresh` → restore session without re-login
- Biometric flow: authenticate locally via Face ID/fingerprint → then use stored refresh token (biometric is a local gate, not a server-side auth method)

### Mobile Strategy (from Project Context §8)

- Mobile is "AI-first" — the conversational interface IS the primary mobile experience
- Chat tab is the default/primary tab (not Briefing)
- Web screens drive design; mobile adapts
- Tab screens are placeholders in this story — fully wired in subsequent epics (E9 Notifications, E5 AI integration)
- Push notifications are infrastructure — actual notification content comes from E9

### Platform Compatibility Notes

- `packages/api-client` MUST be platform-agnostic — use `fetch` API only (available in both web and React Native)
- No `window`, `localStorage`, `document`, `sessionStorage` in shared packages
- Platform-specific storage abstracted behind interfaces (web: `sessionStorage`, mobile: `expo-secure-store`)
- Metro bundler config needed to resolve monorepo workspace packages

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: API client sends `X-Company-Id` header on every request from auth store's `activeCompanyId`
- **i18n**: Every user-facing string uses `t()` from `@nexa/i18n` — no hardcoded English strings
- **Secure storage**: Tokens in encrypted OS keychain (`expo-secure-store`), never in AsyncStorage
- **Error Handling**: API errors surfaced as alerts with translated messages, session errors trigger logout flow
