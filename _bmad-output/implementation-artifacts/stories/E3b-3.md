# Story E3b.3: AI Gateway Service + Provider Adapters

Status: done

## Story

As a **developer**,
I want a single AI Gateway service through which all LLM calls are routed,
so that every AI interaction is quota-checked, proxied, and usage-recorded with zero loss.

## Acceptance Criteria

1. GIVEN any ERP module WHEN it needs to call an LLM THEN it must call `aiGateway.complete()` (no direct LLM API calls — all provider SDKs encapsulated in adapters) per BR-PLT-007
2. GIVEN an AI call request WHEN the AI Gateway receives it THEN it first calls `POST /platform/tenants/:id/ai/check` with estimated tokens and feature key, and only proceeds if `allowed: true`
3. GIVEN the quota check returns `allowed: false` (hard limit reached) WHEN `plan.aiHardLimit = true` THEN the gateway returns an `AI_QUOTA_EXCEEDED` error without calling the LLM
4. GIVEN a successful AI call WHEN the LLM response is received THEN the gateway calls `POST /platform/tenants/:id/ai/record` with usage data (fire-and-forget with retry queue) and returns the response to the calling module
5. GIVEN the Platform API is unreachable WHEN the gateway performs a quota check THEN it serves from cached quota data and queues the usage record for later sync per BR-PLT-020
6. GIVEN the AI Gateway adds overhead WHEN measured end-to-end THEN the quota check + usage recording adds no more than 100ms latency per NFR47
7. GIVEN the AI Gateway WHEN it receives a completion request THEN it resolves the provider adapter from the AiModel registry, resolves credentials (vendor key or tenant BYOK), calls the provider adapter, and normalises the response to a unified `LLMResponse` format
8. GIVEN a primary model fails (rate limit, 5xx, timeout >10s) WHEN a `fallbackModelId` is configured on the AiModel THEN the gateway retries with the fallback model and records `fallbackUsed: true` in the usage record

## Dev Notes

### Architecture Context

This story creates the **AI Gateway package** in `packages/ai-gateway/` and the **Platform API AI endpoints** in `apps/platform-api/src/routes/platform/ai.routes.ts`. The AI Gateway is the **single enforcement point** for all AI usage in Nexa ERP — no ERP module may call any LLM provider API directly (BR-PLT-007).

**AI Gateway Flow:**
```
ERP Module → aiGateway.complete({ tenantId, userId, featureKey, messages, tools })
  → 1. Resolve model (by name or routing tags)
  → 2. Estimate tokens
  → 3. POST /platform/tenants/:id/ai/check { estimatedTokens, featureKey }
       → { allowed: true/false, remainingTokens, quotaPct, warning? }
  → 4. If allowed → resolve credentials (BYOK or vendor key)
  → 5. Call provider adapter (Anthropic/OpenAI)
  → 6. On provider error → attempt fallback if configured
  → 7. POST /platform/tenants/:id/ai/record { usage data } (fire-and-forget)
  → 8. Return normalised LLMResponse to caller
```

### Key Business Rules

| Code | Rule | Enforcement |
|------|------|-------------|
| BR-PLT-007 | All AI calls via gateway — no direct LLM API calls | HARD — `packages/ai-gateway` is the only dependency with LLM SDK credentials |

| BR-PLT-011 | Unusual usage spikes flagged (3x rolling 7-day average) | INFORMATIONAL — background job (out of scope for this story) |
| BR-PLT-020 | ERP degrades gracefully if Platform API unreachable | HARD — circuit breaker serves stale cache |

### Performance Target

**NFR47:** Quota check + usage recording adds **no more than 100ms** end-to-end latency. Measured via p95 in performance benchmarks.

### AI Quota State Machine (Runtime, not persisted)

| Threshold | State | Behaviour |
|-----------|-------|-----------|
| 0–49% | NORMAL | AI calls proceed normally |
| 50% | ALERT_50 | Platform admin notified (internal) |
| 80% (softLimitPct) | SOFT_LIMIT | `tenant.quota_warning` event. ERP shows warning banner. |
| 100% (hardLimitPct) | HARD_LIMIT | If `aiHardLimit = true`: AI calls blocked. If false: overage logged. |
| >100% + spike | ANOMALY | Admin alert for investigation (BR-PLT-011, out of scope) |

### Credential Resolution (BYOK)

Resolution order: (1) tenant BYOK key from `TenantProviderCredential` if configured and active, (2) vendor platform key from environment. When BYOK is used, usage is recorded for audit but **not billed against the tenant's token quota** (`isByok: true` → skip quota increment).

### Fallback Chains

Each model has an optional `fallbackModelId`. On provider error (429, 5xx, timeout >10s), the gateway retries with the fallback model. Maximum 1 fallback attempt. Cross-provider fallback is supported.
