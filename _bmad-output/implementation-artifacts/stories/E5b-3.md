**Epic:** E5b — AI Co-Pilot Intelligence — Memory, Skills & Dynamic Context
**Status:** done
**Dependencies:** E5b-1 (Memory Storage & Retrieval API — done), E5b-2 (Skills Registry, Tool Framework & Dynamic Context — done)

## User Story

As a **user**,
I want the AI to learn my preferences and patterns over time without me having to tell it, and I want it to preserve important context even when conversations are long,
so that the AI becomes increasingly helpful and personalised without losing valuable information during extended sessions.

## Acceptance Criteria

### AC1: Implicit Memory from Repeated Actions (Pattern Detection)
**GIVEN** repeated user actions (e.g., always opening "Overdue Invoices" view, always using Net 30 terms)
**WHEN** the pattern threshold is met (3+ occurrences within a 30-day window)
**THEN** the AI creates an IMPLICIT memory with category matching the pattern type (PREFERENCE for UI patterns, WORKFLOW for process patterns)
**AND** the memory importance is set to 0.5 (default implicit weight)
**AND** a semantic dedup check is performed before creation (cosine similarity ≥ 0.85 against existing memories → merge instead of create)
**AND** the `ai.memory.created` event is emitted with `source: 'IMPLICIT'`

### AC2: Explicit Memory from User Instructions
**GIVEN** an explicit instruction from the user (e.g., "Remember that I prefer Net 30 for new customers", "Always use FIFO costing")
**WHEN** the AI parses the message and detects a "remember" or instruction intent
**THEN** an EXPLICIT memory is created with category INSTRUCTION and high importance (1.5)
**AND** the memory `content` captures the instruction in a normalised form (e.g., "User prefers Net 30 payment terms for new customers")
**AND** memory settings are checked — if memory is disabled or INSTRUCTION category is disabled, the memory is NOT created and the user is informed

### AC3: Memory Citation in AI Responses
**GIVEN** the AI makes a suggestion or takes an action based on a stored memory
**WHEN** presenting the response to the user
**THEN** the AI cites the memory source (e.g., "Based on your preference for Net 30 terms..." or "I noticed you frequently view overdue invoices, so...")
**AND** the cited memory's `lastAccessedAt` is updated (triggering importance recalculation via temporal decay)

### AC4: Explicit Overrides Implicit (Conflict Resolution)
**GIVEN** an implicit memory conflicts with an explicit instruction (e.g., implicit: "user prefers LIFO" vs explicit: "use FIFO costing")
**WHEN** both memories are present during context assembly
**THEN** the explicit instruction takes precedence (higher importance weight: explicit 1.5x vs implicit 1.0x)
**AND** the conflicting implicit memory is marked for review (metadata flag `{ conflictsWith: explicitMemoryId }`)

### AC5: User Correction Updates Memory
**GIVEN** a user corrects the AI (e.g., "No, I actually prefer FIFO now" or "That's wrong, use Net 60")
**WHEN** the correction is processed
**THEN** the old memory is updated with the corrected content (NOT deleted and recreated — preserves the memory ID and creation date)
**AND** the memory source is upgraded to EXPLICIT if it was previously IMPLICIT
**AND** the importance score is boosted to 1.5 (explicit weight)
**AND** the `ai.memory.updated` event is emitted

### AC6: Pre-Compaction Memory Flush
**GIVEN** a conversation approaching the context window limit (>80% of maxInputTokens consumed)
**WHEN** pre-compaction is triggered (before the orchestrator trims old messages)
**THEN** the system scans the most recent messages (those about to be trimmed) for:
  - Novel facts and decisions
  - User instructions and preferences
  - Entity context (projects, customers being discussed)
**AND** for each extracted fact, creates a new IMPLICIT memory if it doesn't already exist
**AND** performs semantic similarity check (cosine ≥ 0.85) against existing memories to avoid duplicates
**AND** if a near-duplicate exists, merges by updating the existing memory's content if the new content adds information
**AND** updates importance scores of any existing memories that were referenced in the conversation
**AND** only then allows context compaction (message trimming) to proceed

### AC7: Semantic Deduplication Check
**GIVEN** the system attempts to create a new memory (either from pattern detection, explicit instruction, or pre-compaction flush)
**WHEN** a semantic similarity check is performed against existing user memories
**THEN** if an existing memory has cosine similarity ≥ 0.85 with the new memory content:
  - Do NOT create a duplicate
  - Instead, merge: update the existing memory's content to incorporate any new information
  - Update the existing memory's `lastAccessedAt` and recalculate importance
**AND** if no near-duplicate exists (all similarities < 0.85), create the new memory normally
**AND** the similarity check uses the VectorSearchService from E5b-4 if available, falling back to simple keyword overlap if pgvector is not yet installed

## Dev Notes

### Architecture Patterns

- **Service Pattern:** Each service follows the existing E5b pattern — constructor DI with `PrismaClient`, `Logger`, `EventBus`, no Fastify dependency
- **Graceful Degradation:** All new services are optional plugins in `apps/api/src/ai/index.ts`. If any fail to initialise, the orchestrator continues without implicit learning. Use the setter pattern: `orchestrator.setPatternDetection(service)`
- **Event Emission:** Use the typed event bus from E3. All new events must be added to `event-bus.types.ts`
- **Memory CRUD:** Reuse `MemoryService` from E5b-1 for all memory operations. Do NOT create parallel memory access patterns
- **Token Budgets:** Citation instructions are within the existing 2000-token memory budget — no new budget allocation needed

### Key Implementation Decisions

1. **Pattern Detection is in-memory first:** Actions are buffered in memory and periodically analysed. No new DB table for action tracking — use a simple Map<userId, action[]> with a 30-day rolling window. If the server restarts, the buffer resets (acceptable for pattern detection — patterns will be re-detected over time)

2. **Pre-compaction is triggered by the orchestrator, not by a cron:** The orchestrator already knows when it's about to trim messages. The pre-compaction service is called synchronously before trimming. This ensures no data loss between cron runs

3. **Semantic dedup uses keyword fallback:** E5b-4 (Hybrid Search & Memory Intelligence) adds pgvector. Until then, the keyword-based Jaccard similarity is the fallback. The SemanticDedupService should be designed to switch seamlessly when VectorSearchService becomes available

4. **Memory corrections update in-place:** Do NOT delete and recreate memories on correction. This preserves the memory ID, creation date, and any references to it. The `updatedAt` timestamp reflects the correction

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: All memory queries must include `WHERE companyId = ?` — memories are per-user-per-company, never cross-company
- **i18n**: Error messages returned to the user must use error codes (e.g., `MEMORY_DISABLED`, `CATEGORY_DISABLED`). The frontend handles translation
- **Audit**: All memory creation, update, and deletion must emit typed events via the event bus
- **userId isolation**: A user can only access/modify their own memories. Never expose another user's memories
