# Story E5c.3: AI Model & Prompt Admin UI

Status: done

## Story

As an **admin**,
I want to view and manage AI models, prompt templates, and prompt versions through a web interface,
so that I can configure the AI infrastructure without database access.

## Acceptance Criteria

1. **AC-1: AI Configuration Dashboard** — GIVEN the AI Configuration Dashboard WHEN an admin navigates to it THEN they see cards for: Active Models (count + cost this month), Active Agents (count), Active Skills (count by module), Automations (active/paused), and a daily token usage chart (line chart, 30-day rolling). All cards follow Concept D styling (12px radius, purple-tinted hover shadow). Typography uses Plus Jakarta Sans for headings, Inter for body, JetBrains Mono for counts/costs.

2. **AC-2: Model Registry list** — GIVEN the Model Registry page WHEN rendered THEN it shows a T1 Entity List of all AI models with columns: name (mono font), provider, model ID (mono font), max tokens (mono font), cost/M input (mono font), cost/M output (mono font), routing tags (as purple badges), active status (StatusBadge), default flag (purple "Default" badge). Supports search, column sorting, and cursor-based pagination.

3. **AC-3: Model add/edit form** — GIVEN the Model Registry WHEN an admin clicks "Add Model" or clicks a model row THEN a detail form opens with: name (required, unique), provider (dropdown: anthropic, openai, google, or custom text), model ID (required), display name, max input tokens, max output tokens, cost per million input tokens (decimal), cost per million output tokens (decimal), routing tags (multi-select tag input: reasoning, standard, cheap, fast, vision, structured_output, briefing), capabilities (JSON editor), fallback model (dropdown of other models), config (JSON editor, optional), active toggle, default toggle. **Business rules**: (a) Setting `isDefault=true` automatically unsets the previous default model. (b) Cannot deactivate a model that `isDefault=true` — must assign new default first. (c) Cannot delete a model referenced by any AiAgent — show error with agent count. (d) Fallback model dropdown must not allow circular chains (A→B→A).

4. **AC-4: Prompt Templates list** — GIVEN the Prompt Templates page WHEN rendered THEN it shows a T1 Entity List of all prompts with columns: name (mono font), category (badge), active version (mono font), variable count, active status (StatusBadge), last updated. Supports grouping by category, search, and cursor-based pagination.

5. **AC-5: Prompt Template editor** — GIVEN a prompt template WHEN an admin clicks to edit THEN a T4-style editor opens with: (a) syntax-highlighted system prompt editor with variable autocomplete triggered on `{{`, showing all available variables grouped by source type (System, DB Fields, Page Fields, Custom), (b) syntax-highlighted user template editor with same autocomplete, (c) parameters JSON schema editor with syntax highlighting, (d) version history sidebar showing all AiPromptVersion records with version number, change reason, created by, and created date — clicking a version shows the diff between that version and the current active version, (e) "Test Prompt" button that renders the prompt with sample variables via the PromptRenderer and displays the fully resolved output.

6. **AC-6: Prompt versioning on save** — GIVEN the prompt editor WHEN an admin saves changes THEN a new AiPromptVersion is created (append-only), the prompt's `activeVersion` is incremented, and a `changeReason` is required (modal or inline field). The version record stores snapshots of `systemPrompt`, `userTemplate`, and `parameters`.

7. **AC-7: Version restore** — GIVEN the prompt editor's version history WHEN an admin clicks a previous version THEN they see the diff between that version and the current active version, with an option to "Restore This Version" which creates a new version copying the old content and auto-populates changeReason as "Restored from version N".

  - `listModels(companyId: string, params)` → paginated list with cursor, sortable by name/provider/createdAt. Include `_agentCount` for each model (count of AiAgent records referencing this model)

  - `createModel(data)` → create new AiModel. If `isDefault=true`, unset `isDefault` on all other models in a transaction
  - `updateModel(id: string, data)` → update model. Business rules:

    - If setting `fallbackModelId` → validate no circular chain (traverse fallback pointers, max depth 10)
  - `deleteModel(id: string)` → check if any AiAgent references this model. If yes → reject with 422 "This model is referenced by N agents. Remove agent references first." If no → hard delete (models are infrastructure config, not business data)

- [x] 2.2 Implement circular fallback detection:
  - `detectFallbackCycle(modelId: string, fallbackModelId: string)` → boolean
  - Starting from `fallbackModelId`, traverse the fallback chain. If any node equals `modelId` → cycle detected
  - Max depth 10, reject if exceeded

### Task 3: Create AiPrompt admin service with versioning (AC: #4, #5, #6, #7) ✅

- [x] 3.1 Create `apps/api/src/ai/admin/admin-prompt.service.ts`:
  - `listPrompts(params)` → paginated list with cursor. Include `_variableCount` (count of AiPromptVariable records), `_latestVersionCreatedAt` for "last updated". Support filtering by category, isActive, search (name/description)
  - `getPrompt(id: string)` → single prompt with current version content, variable list, and version count
  - `createPrompt(userId: string, data)` → create AiPrompt + initial AiPromptVersion (version 1). Set `activeVersion=1`, `createdBy=userId`. First version's `changeReason` = "Initial version"
  - `updatePrompt(id: string, userId: string, data)` → update AiPrompt fields AND create a new AiPromptVersion if `systemPrompt`, `userTemplate`, or `parameters` changed:
    1. Load current prompt
    2. Determine next version number (`activeVersion + 1`)
    3. Create AiPromptVersion with snapshots of new content + `changeReason` (required)
    4. Update AiPrompt: set `activeVersion` to new version, update changed fields
    5. All in a transaction
    6. If only non-content fields changed (description, category, isActive) → update without creating version
  - `deletePrompt(id: string)` → check if any AiAgent references this prompt. If yes → reject with 422. If no → hard delete (cascades to versions and variables)

- [x] 3.2 Implement version management methods:
  - `listVersions(promptId: string)` → all versions ordered by version DESC. Return: version, changeReason, createdBy, createdAt, and a preview snippet (first 100 chars of systemPrompt)
  - `getVersion(promptId: string, version: number)` → full version content (systemPrompt, userTemplate, parameters)
  - `restoreVersion(promptId: string, version: number, userId: string)` → create a NEW version copying the old version's systemPrompt, userTemplate, and parameters. Set changeReason to `"Restored from version ${version}"`. Update prompt's activeVersion. Return the new version

- [x] 3.3 Implement prompt test rendering:
  - `testRender(promptId: string, context: TestRenderInput)` → use existing `PromptRenderer.render()` with provided sample context. Return `{ systemPrompt: string, userTemplate: string, resolvedVariables: Record<string, string>, unresolvedCount: number }`

### Key Design Decisions

1. **No Prisma migration needed**: All AI infrastructure models (AiModel, AiPrompt, AiPromptVersion, AiPromptVariable) already exist from E5 and E5c-1. This story is purely service layer + API + frontend.

2. **Separate admin route prefix**: All admin endpoints use `/ai/admin/*` prefix to clearly distinguish from the existing `/ai/*` user-facing endpoints. This also simplifies RBAC — the entire `/admin` prefix requires ADMIN role.

3. **Prompt versioning on update**: Every update that changes content (systemPrompt, userTemplate, parameters) creates a new version. Metadata-only updates (description, category, isActive) do NOT create versions. This prevents version noise.

4. **Model delete is hard delete**: Unlike business entities, AI models are infrastructure configuration. Hard delete is appropriate since we validate no agents reference the model first. Soft-delete would leave ghost records in the registry.

5. **Prompt delete cascades**: Deleting an AiPrompt cascades to AiPromptVersion and AiPromptVariable (defined in Prisma schema). We validate no AiAgent references the prompt first.

6. **Variable autocomplete is read-only**: The prompt editor's `{{` autocomplete fetches from the existing `GET /ai/variables` endpoint. Variable CRUD is done through the separate variable management UI (already built in E5c-2). The editor just references existing variables.

7. **Simple diff instead of rich diff library**: For version comparison, use a line-by-line text comparison instead of importing a full diff library. Prompts are typically short enough that line-by-line diff is readable. This avoids a new dependency.

8. **No CodeMirror/Monaco in first pass**: Use styled `<textarea>` with mono font instead of a full code editor. The variable autocomplete popover provides the key "smart editor" feature. CodeMirror/Monaco can be added later as an enhancement if needed — they add significant bundle size.
