# Story E10-1: SMTP Outbound Service

**Epic:** E10 — Email Integration
**Status:** done
**Dependencies:** E9 (Notifications — done, provides EmailSender service, BullMQ patterns, Nodemailer)
**Complexity:** X-Large (10 tasks)
**FRs:** FR187 (email send with per-company SMTP) | **NFRs:** NFR31 (retry with exponential backoff)

---

## User Story

As a **system**,
I want a reliable email queue with SMTP sending, retry logic, and delivery status tracking,
so that business emails (invoices, notifications, PO confirmations) are delivered reliably using per-company SMTP settings.

---

## Acceptance Criteria

### AC1: Email Creation Service
**GIVEN** a service call to create an outbound email
**WHEN** `EmailService.createEmail()` is called with subject, body, recipients, and optional source entity reference
**THEN** an `EmailMessage` record is created with:
  - `direction` = `OUTBOUND`
  - `status` = `DRAFT`
  - `messageNumber` auto-generated via NumberSeries "EM-00001" (pattern: `EM-{seq}`)
  - `companyId` set to the caller's active company
  - `sourceEntityType` and `sourceEntityId` populated if sending from a document (e.g., `CustomerInvoice`, `PurchaseOrder`)
**AND** `EmailRecipient` rows are created for each recipient:
  - `recipientType` = `FROM` (company email), `TO`, `CC`, `BCC`
  - `emailAddress` always populated (even for internal users)
  - Validated per BR-COM-001: internal userId must exist; external addresses must pass RFC 5322 format check
**AND** duplicate recipients are rejected per BR-COM-002: unique constraint on `(emailMessageId, emailAddress, recipientType)`
**AND** the `companyId` on the EmailMessage is scoped to the caller's company

### AC2: Email Queue System with BullMQ
**GIVEN** an email is queued for sending
**WHEN** `EmailService.queueEmail(emailMessageId)` is called
**THEN** the `EmailMessage.status` transitions from `DRAFT` to `QUEUED`
**AND** an `EmailQueue` record is created with:
  - `emailMessageId` (unique 1:1 with EmailMessage)
  - `status` = `PENDING`
  - `priority` = 0 (normal) or 1 (high priority for time-sensitive emails)
  - `maxAttempts` = 3
  - `attempts` = 0
**AND** a BullMQ job is added to the `email-send` queue with `{ emailQueueId }` payload
**AND** the email cannot transition back from QUEUED to DRAFT per BR-COM-003

### AC3: SMTP Sending via Per-Company Configuration
**GIVEN** the `email-send` BullMQ worker processes a job
**WHEN** it picks up a PENDING EmailQueue entry
**THEN** it loads the SMTP configuration for the email's company:
  - Per-company SMTP config stored in `SystemSettings` with key `smtp.*` (host, port, secure, user, pass, fromName, fromEmail)
  - Falls back to global env-based SMTP config (from E9's `email-config.ts`) if no per-company config exists
**AND** it creates a Nodemailer transporter with the resolved SMTP settings
**AND** it supports TLS/STARTTLS (`secure: true` for port 465, STARTTLS for port 587)
**AND** it sends the email via the transporter with:
  - `from`: per-company sender alias or company email
  - `to/cc/bcc`: resolved from `EmailRecipient` rows
  - `subject`: from `EmailMessage.subject`
  - `html/text`: from `EmailMessage.bodyHtml` / `EmailMessage.bodyText`
  - Attachments: loaded from `Attachment` records linked to the EmailMessage (via entity type/id per §2.20)
**AND** Company A's emails use Company A's SMTP settings, never a shared server

### AC4: Retry with Exponential Backoff
**GIVEN** an email send fails
**WHEN** the SMTP server returns an error (connection timeout, auth failure, 4xx/5xx response)
**THEN** the system retries with exponential backoff delays: 30s, 120s, 300s (NFR31: max 3 retries)
**AND** the `EmailQueue.status` is set to `RETRYING`
**AND** `EmailQueue.attempts` is incremented
**AND** `EmailQueue.lastError` stores the error message/SMTP response
**AND** `EmailQueue.nextRetryAt` is set to the next retry timestamp
**AND** the BullMQ job uses a custom backoff strategy matching the [30s, 120s, 300s] schedule

### AC5: Delivery Status Tracking
**GIVEN** an email is sent successfully
**WHEN** the SMTP server accepts the email
**THEN** the following updates are applied:
  - `EmailMessage.status` = `SENT`
  - `EmailMessage.sentAt` = current timestamp
  - `EmailQueue.status` = `SENT`
  - `EmailQueue.deliveredAt` = current timestamp
  - `EmailQueue.smtpResponse` stores the SMTP server's acceptance response
**AND** the `email.sent` event is emitted per event catalog §14 with `{ emailMessageId, recipientEmail, subject, documentType? }`

### AC6: Permanent Failure Handling
**GIVEN** all 3 retry attempts are exhausted
**WHEN** the email still fails to send
**THEN** the following updates are applied:
  - `EmailMessage.status` = `FAILED`
  - `EmailQueue.status` = `FAILED`
  - `EmailQueue.lastError` contains the final error message
**AND** a `notification.created` event is raised to alert the user who created the email (via E9 notification system)
**AND** failed emails can be manually retried via `POST /email/messages/:id/send` (resets the queue)

### AC7: Email Signature Handling
**GIVEN** an outbound email is being composed
**WHEN** the email body is prepared for sending
**THEN** the user's `EmailSignature` (HTML) is appended to the body once (BR-COM-009: prevent double-append)
**AND** the system tracks whether the signature has been appended via an internal flag to prevent duplicate insertion on re-queue

### AC8: CRUD API for Email Messages
**GIVEN** the REST API endpoints under `/email/messages`
**WHEN** authenticated users call them
**THEN** the following endpoints are available:
  - `GET /email/messages` — list outbound emails for the current company with pagination, filtering by status/direction (permission: STAFF+)
  - `GET /email/messages/:id` — get single email with recipients (permission: STAFF+)
  - `POST /email/messages` — create email message with recipients (permission: STAFF+)
  - `POST /email/messages/:id/send` — queue email for sending, or re-queue a FAILED email (permission: STAFF+)
  - `PATCH /email/messages/:id/read` — mark as read/unread for current user (permission: STAFF+)
  - `DELETE /email/messages/:id` — soft-delete/archive email (permission: STAFF+)
**AND** all endpoints enforce `companyId` scoping
**AND** cross-company access attempts return 404 (not 403)

### AC9: Number Series Registration
**GIVEN** the number series system from E0
**WHEN** email messages are created
**THEN** a NumberSeries entry exists for `EmailMessage` with prefix `EM-`, starting from `00001`
**AND** the `messageNumber` is auto-generated on creation and is unique per company

### AC10: Prisma Schema Migration
**GIVEN** the email-related models are not yet in the Prisma schema
**WHEN** the migration runs
**THEN** the following tables are created:
  - `email_messages` with all fields from Architecture §2.29
  - `email_recipients` with all fields including unique constraint on `(emailMessageId, emailAddress, recipientType)`
  - `email_queue` with all fields including indexes on `(status, priority)` and `(nextRetryAt)`
  - `email_signatures` with per-user HTML/text signatures
**AND** the enums `EmailMessageStatus`, `EmailDirection`, `EmailRecipientType`, `EmailRecipientStatus`, `EmailQueueStatus` are created
**AND** the migration uses `prisma migrate dev` (never `db push`)
**AND** `PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION=yes` is set
