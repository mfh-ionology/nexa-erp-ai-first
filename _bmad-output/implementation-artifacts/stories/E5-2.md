# Story E5.2: AI Chat Session Management

Status: done

## Story

As a **user**,
I want to have multi-turn conversations with the AI assistant via WebSocket, with persistent chat history and session management,
so that I can have contextual, ongoing interactions through the Co-Pilot drawer.

## Acceptance Criteria

1. GIVEN a user opens the Co-Pilot drawer WHEN a WebSocket connection is established THEN the connection authenticates via JWT and associates with the user's tenant and company context
2. GIVEN a user sends a message WHEN the AI processes it THEN the response streams back token-by-token with a typing indicator until complete
3. GIVEN an active conversation WHEN the user sends a follow-up message THEN the AI has full context of the previous messages in the session (multi-turn)
4. GIVEN a user creates a new chat session WHEN they click "+ New Chat" THEN a new `AiConversation` record is created and the AI starts fresh while retaining user/tenant awareness
5. GIVEN a user returns to the application WHEN they open the Co-Pilot drawer THEN their previous conversations are listed with auto-generated titles, most recent first
6. GIVEN an HTTP fallback is needed (WebSocket unavailable) WHEN the user sends a message via POST THEN the response is returned as a complete message (non-streaming)

## Dev Notes

### Architecture Summary

E5.2 adds the real-time communication layer on top of E5.1's AI service layer. The key addition is a Socket.io WebSocket handler that:
1. Authenticates clients via JWT (same auth as REST endpoints)
2. Streams AI responses token-by-token from the orchestrator to the client
3. Manages chat session CRUD (create, list, get, end)
4. Auto-generates conversation titles from the first user message
5. Falls back to HTTP POST `/ai/chat/message` (already implemented in E5.1) when WebSocket is unavailable

**Flow:** Co-Pilot Drawer → WebSocket connect (JWT auth) → Send message → AiOrchestrator.processStream() → Stream chunks back → Persist messages → stream_end

### Socket.io + Fastify Integration Notes

Socket.io attaches to the raw Node.js HTTP server, not to Fastify's request pipeline:
```typescript
import { Server as SocketServer } from 'socket.io';

const io = new SocketServer(fastify.server, {
  cors: { origin: corsOrigin, credentials: true },
  path: '/api/v1/ai/chat',        // Socket.io path (not Fastify route)
  transports: ['websocket', 'polling'],
});

// Auth middleware runs on Socket.io namespace, NOT Fastify preHandler
io.use((socket, next) => {
  const token = socket.handshake.auth.token || socket.handshake.query.token;
  // verify JWT...
  next();
});
```

**Important:** Socket.io auth is separate from Fastify's JWT hooks. Reuse the JWT verification logic but implement it as a Socket.io middleware, not a Fastify hook.

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: All `AiConversation` queries must include `WHERE companyId = ?`. Session listing, session get, and message queries must be scoped by companyId + userId. This is critical for multi-tenant isolation.
- **i18n**: WebSocket error messages should use translation key pattern (`messageKey` field). For MVP, hardcode English strings but include the key for future i18n.
- **Audit**: Chat sessions themselves don't require audit log entries (conversations are themselves an audit trail). AI-initiated actions (E5.3) will need audit entries.
- **Events**: No new events for E5.2. The `ai.degraded` event from E5.1 already covers AI failure scenarios.
- **Permission Guard**: REST endpoints use `createPermissionGuard('ai.chat', 'view')`. WebSocket auth uses JWT verification directly (Socket.io middleware bypasses Fastify route guards). Check that the user has the `ai.chat` canAccess permission in the WebSocket auth middleware using `PermissionService` directly.

### AI Guardrails Compliance

- **BR-COM-013 (HARD):** `action_confirm` / `action_reject` message types are defined in the WebSocket contract but actual action execution is E5.3's scope. In E5.2, stub these handlers to log the intent and return an acknowledgement. E5.3 will implement the full action framework.
- **IMP-005 (HARD):** AI never auto-executes. The WebSocket handler passes AI responses as-is — any `action_proposal` in the response is sent to the client for user review, never auto-executed.
- **IMP-006 (HARD):** If AI Gateway fails during streaming, the WebSocket handler sends an `error` server message — it never crashes or disconnects the socket.

### Anti-Patterns to AVOID

1. **Do NOT** create a separate HTTP server for WebSocket — Socket.io MUST attach to the existing Fastify HTTP server via `fastify.server`
2. **Do NOT** implement action execution in E5.2 — `action_confirm` and `action_reject` are stubbed; full implementation is E5.3
3. **Do NOT** bypass the orchestrator for AI calls — all AI requests must go through `AiOrchestrator` which handles gateway routing, prompt resolution, and message persistence
4. **Do NOT** store chat messages in Redis — they must be persisted in PostgreSQL via the `AiMessage` table for audit trail and history retrieval
5. **Do NOT** use Fastify's routing for WebSocket endpoints — Socket.io has its own event-based routing that runs outside Fastify's request pipeline
