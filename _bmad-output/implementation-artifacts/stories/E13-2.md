# Story E13.2: Print Actions

Status: done

## Story

As a **user**,
I want documents to auto-download as PDF on save or trigger the browser print dialog based on my preferences, with support for batch printing,
so that printing is seamless and I don't have to manually generate and print documents after every save.

## Acceptance Criteria

1. **GIVEN** a user saves a document (invoice, PO, etc.) **WHEN** their preference is "Auto-Download" **THEN** the system generates the PDF via E12 and triggers a browser file download
2. **GIVEN** a user saves a document **WHEN** their preference is "Browser Print" **THEN** the system generates the PDF, opens it in a hidden iframe, and calls `window.print()` to trigger the native print dialog
3. **GIVEN** a batch of documents selected on a list page **WHEN** the user clicks "Print Selected" **THEN** PDFs are generated for all selected documents and either batch-downloaded as a ZIP or printed sequentially
4. **GIVEN** the print action **WHEN** the PDF is being generated **THEN** a loading indicator is shown and the user can continue working (non-blocking)

### API Client Pattern for Binary Responses

The shared `apiGet`/`apiPost` always JSON-parse. For PDF binary responses, use **raw `fetch`** with auth headers (same pattern as `usePreviewTemplate` in `apps/web/src/features/settings/document-templates/api.ts:432-487`):

```typescript
const { accessToken, activeCompanyId } = useAuthStore.getState();
const headers: Record<string, string> = {
  Accept: 'application/pdf',
  'Content-Type': 'application/json',
};
if (accessToken) headers['Authorization'] = `Bearer ${accessToken}`;
if (activeCompanyId) headers['X-Company-Id'] = activeCompanyId;

const baseUrl = (import.meta.env.VITE_API_BASE_URL as string | undefined) ?? '';
const response = await fetch(`${baseUrl}/api/v1/system/documents/generate`, {
  method: 'POST',
  headers,
  body: JSON.stringify({ documentType, recordId, outputFormat: 'attachment' }),
});
const blob = await response.blob();
```

## Dev Notes

### Architecture Patterns

- **No backend changes needed** — E13-2 is entirely frontend. All backend infrastructure exists from E12 (document generation) and E13-1 (print preferences). The only backend endpoints consumed are `POST /system/documents/generate`, `POST /system/documents/batch-generate`, and `GET /system/documents/batch-generate/:batchJobId/status`.
- **Binary response pattern** — `POST /documents/generate` returns raw PDF binary, not JSON. You MUST use raw `fetch` (not `apiGet`/`apiPost`) with manual auth header injection. Follow the exact pattern from `usePreviewTemplate()` in `apps/web/src/features/settings/document-templates/api.ts:432-487`.
- **Blob URL lifecycle** — Always `URL.revokeObjectURL()` after use. Use `useRef` to track blob URLs and clean up in `useEffect` cleanup (see `PreviewPanel` component for reference pattern).
- **iframe print pattern** — Create a hidden `<iframe>` with `style="display:none"`, set `src` to blob URL, wait for `onload` event, then call `iframe.contentWindow.print()`. Remove iframe after print dialog closes (listen for `afterprint` event or use `setTimeout` fallback).

### Critical Anti-Patterns to Avoid

- **DO NOT** use `apiGet`/`apiPost` for PDF binary responses — they JSON-parse and will corrupt the blob
- **DO NOT** block the save operation on print action — print is fire-and-forget, errors show toasts only
- **DO NOT** try to open multiple print dialogs simultaneously — browsers only support one at a time. For batch BROWSER_PRINT, generate and print sequentially
- **DO NOT** store blob URLs in React state without cleanup — memory leak
- **DO NOT** create new API endpoints — all backend infrastructure exists
