# Story E3b.5: Plan & Billing Management

Status: done

## Story

As a **platform administrator**,
I want to manage subscription plans and track billing status per tenant,
so that I can enforce plan limits and handle payment escalation.

## Acceptance Criteria

1. GIVEN PLATFORM_ADMIN role WHEN I call `POST /admin/plans` with plan data THEN a new Plan is created with code, displayName, maxUsers, maxCompanies, monthlyAiTokenAllowance, aiHardLimit, enabledModules, apiRateLimit
2. GIVEN PLATFORM_ADMIN role WHEN I call `POST /admin/tenants/:id/assign-plan` with a new planId THEN the tenant's plan is changed, a `tenant.plan_changed` webhook is pushed to the ERP, and a PlatformAuditLog entry is created
3. GIVEN a tenant's billing status WHEN `GET /admin/tenants/:id/billing` is called THEN it returns stripeCustomerId, subscriptionStatus, currentPeriodEnd, gracePeriodDays, lastPaymentAt, dunningLevel, and current enforcementAction
4. GIVEN billing enforcement escalation WHEN a tenant's payment becomes overdue THEN the system progresses through NONE→WARNING→READ_ONLY→SUSPENDED based on configurable dunning thresholds per BR-PLT-004
5. GIVEN a tenant in READ_ONLY enforcement WHEN the ERP checks entitlements THEN write operations are blocked and a billing notice is shown (BR-PLT-005)
6. GIVEN a plan change WHEN it takes effect THEN the new module entitlements apply immediately after the webhook invalidates the ERP cache (BR-PLT-006)

### Billing Enforcement State Machine (BR-PLT-004)

```
NONE ──→ WARNING ──→ READ_ONLY ──→ SUSPENDED
  ↑        ↓  ↑          ↓  ↑           │
  └────────┘  └──────────┘  └───────────┘
  (payment)   (admin/pay)    (full payment
                              + admin)
```

| From | Allowed Targets | Trigger | Side Effects |
|------|-----------------|---------|--------------|
| NONE | WARNING | Payment overdue, dunningLevel >= 1 | Webhook: `billing.enforcement_changed`, billingStatus → GRACE |
| WARNING | NONE, READ_ONLY | Payment received / Grace period expired | Webhook: `billing.enforcement_changed` |
| READ_ONLY | NONE, WARNING, SUSPENDED | Full payment / Admin override / Further non-payment | Webhook: `billing.enforcement_changed`. If → SUSPENDED, also: Tenant.status → SUSPENDED + `tenant.suspended` webhook |
| SUSPENDED | NONE | Full payment + admin reactivation | Webhook: `billing.enforcement_changed` + Tenant.status → ACTIVE + `tenant.reactivated` webhook |

**Critical enforcement rule (BR-PLT-005):** READ_ONLY blocks all ERP writes. The ERP checks `enforcementAction` from cached entitlements before every write operation. This is enforced by the Platform Client SDK (E3b-4) — NOT by this story's routes. This story ensures the enforcement state is correctly set and propagated via webhook.

**Plan change immediate effect (BR-PLT-006):** When a plan is assigned, the `tenant.plan_changed` webhook busts the ERP's entitlement cache. New limits/modules apply on the next ERP page load. No server restart needed.

### Events Published by This Story

| Event | Trigger | Payload |
|-------|---------|---------|
| `tenant.plan_changed` | POST /admin/tenants/:id/assign-plan | `{ tenantId, oldPlanCode, newPlanCode, changedBy, enabledModules }` |
| `billing.enforcement_changed` | PATCH /admin/tenants/:id/billing/enforcement | `{ tenantId, oldAction, newAction, reason }` |
| `tenant.suspended` | Enforcement → SUSPENDED | `{ tenantId, reason, suspendedBy, enforcementAction: 'SUSPENDED' }` |
| `tenant.reactivated` | Enforcement SUSPENDED → NONE | `{ tenantId, reactivatedBy }` |

All events are delivered to ERP via `WebhookService.push()` (fire-and-forget, 3 retries).

### Previous Story Issues to Avoid

**From E3b-1 Code Review:**
- [HIGH #2] Audit log NOT wrapped in try/catch → violates BR-PLT-017. **Always** wrap `platformAudit.log()` in try/catch.
- [MEDIUM #7] PATCH accepts empty body → no-op audit entries. **Always** validate body has at least one field.
- [MEDIUM #8] updateData typed as `Record<string, unknown>` → bypasses type safety. Use proper Zod-inferred types.

**From E3b-2 Code Review:**
- [HIGH #2] Webhook service sends `Authorization: Bearer` with empty string if `PLATFORM_SERVICE_TOKEN` unset. **Always** check env var exists before constructing WebhookService.
- [MEDIUM #3] suspendTenant hardcodes enforcementAction in webhook but doesn't update `TenantBilling.enforcementAction`. **This story MUST keep enforcement state consistent** — when enforcement escalates to SUSPENDED, both `TenantBilling.enforcementAction` AND `Tenant.status` must update.
- [MEDIUM #4] PATCH uses `findUnique` instead of `SELECT FOR UPDATE` → no race condition protection. Use Prisma `$transaction` for all enforcement state changes.

**From E3b-4 Code Review:**
- [HIGH #1] `invalidateCache` does NOT clear stale fallback. When this story pushes `billing.enforcement_changed` webhooks, the ERP side invalidates cache — but the stale cache issue in E3b-4 means enforcement changes may not take immediate effect. **This is an E3b-4 issue, not this story's scope**, but document it for awareness.
