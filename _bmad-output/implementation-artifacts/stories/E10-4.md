# Story E10.4: Email Template Management Frontend

Status: done

## Story

As an **administrator**,
I want a frontend page to browse, create, edit, and preview email templates with live Handlebars rendering,
so that I can configure professional email formatting per document type without developer assistance.

## Acceptance Criteria

### AC1: Email Template List Page (T1 Entity List)
**GIVEN** an ADMIN user navigates to `/system/email-templates`
**WHEN** the page loads
**THEN** a table displays all email templates with columns: Code, Name, Document Type, Language, Status (Active/Inactive), Last Updated
**AND** the table supports cursor-based pagination, search by code/name, and filtering by documentType and isActive
**AND** a `[+ New Template]` primary action button is visible
**AND** clicking a row navigates to `/system/email-templates/:id` (edit mode)

### AC2: Email Template Create/Edit Form (T7 Settings — Split-Pane)
**GIVEN** an ADMIN user navigates to `/system/email-templates/new` or `/system/email-templates/:id`
**WHEN** the page loads
**THEN** a split-pane layout shows a form on the left and a live preview on the right
**AND** the form contains fields: Code (read-only on edit), Name, Description, Document Type (Select), Language (Select), Subject Template, Body HTML Template (code editor), Body Text Template (optional), Attach PDF (toggle), Auto Send (toggle)
**AND** dirty tracking enables/disables the Save button
**AND** unsaved changes trigger a navigation-away warning

### AC3: Live Preview Pane
**GIVEN** the user edits the Subject Template or Body HTML Template
**WHEN** the content changes
**THEN** the preview pane re-renders the template with sample data (debounced 500ms) via `POST /email/templates/:id/preview`
**AND** Handlebars syntax errors are shown as an error message in the preview pane (not a crash)
**AND** the available variables panel lists all valid variables for the selected document type

### AC4: Variable Reference Panel
**GIVEN** the user selects a Document Type
**WHEN** the available variables update
**THEN** a panel shows all valid Handlebars variables for that document type (e.g., `{{invoiceNumber}}`, `{{customerName}}`)
**AND** each variable has a copy-to-clipboard button
**AND** clicking a variable inserts it at the cursor position in the active editor field

### AC5: Template CRUD Operations
**GIVEN** the user fills out the form and clicks `[Save]`
**WHEN** the API call succeeds
**THEN** a success toast appears and the form resets to clean state
**AND** creating a new template calls `POST /email/templates`, editing calls `PATCH /email/templates/:id`
**AND** soft-deleting a template (from overflow menu) calls `DELETE /email/templates/:id` and redirects to list
**AND** validation errors from the API (duplicate code, invalid Handlebars syntax) are shown as form-level error banners

### AC6: Responsive Design
**GIVEN** any viewport width
**WHEN** the page renders
**THEN** desktop (≥1024px) shows the split-pane layout (form left, preview right)
**AND** tablet (768–1023px) stacks preview below the form
**AND** mobile (<768px) shows form-only with a `[Preview]` button that opens a full-screen dialog

## Dev Notes

### Critical Implementation Context

- **Backend is 100% complete** — All 6 API endpoints exist at `/email/templates` (E10-2), fully tested with 90/90 pass rate
- **No companyId on EmailTemplate** — Templates are system-wide config entities, NOT per-company. API calls don't need companyId scoping for templates
- **ADMIN role required** — All template endpoints use `createRbacGuard({ minimumRole: UserRole.ADMIN })`. Frontend must check role before rendering edit actions
- **7 seeded default templates** exist — INVOICE_SEND, STATEMENT_SEND, QUOTE_SEND, ORDER_CONFIRM, PO_SEND, CREDIT_NOTE_SEND, PAYSLIP_SEND. The list page will show these on first load
- **Handlebars validation happens server-side** — The `POST` and `PATCH` endpoints validate template syntax. Frontend should show API errors, NOT duplicate Handlebars parsing
