# Story E2b.3: Admin can assign access groups to users

Status: done

## Story

As an **administrator**,
I want to assign one or more access groups to each user per company,
So that users get the exact permissions their role requires.

## Acceptance Criteria

1. GIVEN an administrator is authenticated WHEN they GET /system/users/:id/access-groups THEN they receive the list of access groups assigned to that user for the current company AND each group includes id, code, name, description, isSystem, assignedBy, and assignedAt
2. GIVEN an administrator WHEN they PUT /system/users/:id/access-groups with an array of accessGroupIds THEN all current group assignments are replaced with the provided set AND the assignedBy field records who made the assignment
3. GIVEN an administrator tries to assign zero access groups WHEN they PUT /system/users/:id/access-groups with an empty array THEN they receive a 422 Business Rule Violation (at least one group required)
4. GIVEN an administrator provides an accessGroupId from a different company WHEN they PUT /system/users/:id/access-groups THEN they receive a 400 Validation Error

### Critical Architecture: UserAccessGroup is Company-Scoped (has companyId)

The UserAccessGroup table has explicit `companyId` because a user can be assigned different access groups in different companies. Every query MUST include `companyId` in the WHERE clause.

**Table scoping context (from E2b-1/E2b-2):**
| Table | Scoped? | Notes |
|-------|---------|-------|
| `Resource` | GLOBAL (no companyId) | Created in E2b-1, shared across all companies |
| `AccessGroup` | COMPANY-SCOPED | `@@unique([companyId, code])` |
| `AccessGroupPermission` | Via AccessGroup | FK to AccessGroup (inherits company scope) |
| `AccessGroupFieldOverride` | Via AccessGroup | FK to AccessGroup (inherits company scope) |
| `UserAccessGroup` | COMPANY-SCOPED | Has explicit `companyId`, `@@unique([userId, accessGroupId, companyId])` |

### User Existence Validation Pattern

The target user must be validated as belonging to the current company. Check against `UserCompanyRole` (not just User table) because a user might exist in the platform but not be associated with the current company.

```typescript
// Check user exists in THIS company
const userInCompany = await prisma.userCompanyRole.findFirst({
  where: { userId: targetUserId, companyId },
  select: { id: true },
});
if (!userInCompany) {
  throw new NotFoundError('USER_NOT_FOUND', 'User not found in this company');
}
```

### 422 vs 400 for Empty Array Validation (AC #3)

The AC requires **422** for "at least one group required", not 400 (which is what Zod validation returns by default). Two approaches:

**Option A (Recommended):** Don't use `.min(1)` on the Zod schema. Instead, allow empty arrays through schema validation and check in the service/route:
```typescript
if (accessGroupIds.length === 0) {
  throw new DomainError('MIN_GROUPS_REQUIRED', 'At least one access group is required');
}
// DomainError maps to 422 via the error handler
```

**Option B:** Use `.min(1)` in Zod and override the error handler to return 422 for this specific validation. This is more complex and less clean.

### Replace-All Assignment Pattern

The PUT endpoint uses a **replace-all** pattern (same as `setAccessGroupPermissions` in E2b-2):
1. Delete all existing UserAccessGroup records for (userId, companyId)
2. Create new records for each provided accessGroupId
3. Re-query to return the full response

This must happen inside a `$transaction` to maintain consistency.
