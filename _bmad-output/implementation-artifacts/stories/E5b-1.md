# Story E5b.1: Memory Storage & Retrieval API

Status: done

## Story

As a **developer**,
I want a memory persistence layer so that the AI can store and retrieve user-specific memories, conversation summaries, and preferences,
so that the Co-Pilot remembers decisions, preferences, and context across sessions — making Nexa genuinely AI-first.

## Acceptance Criteria

1. **AC-1: Prisma schema migration** — GIVEN the Prisma schema WHEN `prisma migrate dev` runs THEN the 3 tables (`ai_memories`, `ai_conversation_summaries`, `ai_memory_settings`) are created with proper indexes, foreign keys to `User` and `CompanyProfile`, and all columns as specified in the epic

2. **AC-2: Create memory** — GIVEN the `POST /ai/memories` endpoint WHEN called with `{ content, category, source }` by an authenticated user THEN a memory is stored with category (PREFERENCE | WORKFLOW | ENTITY_CONTEXT | DECISION | INSTRUCTION), content text, source (EXPLICIT | IMPLICIT), importance score (default: 1.0 for EXPLICIT, 0.5 for IMPLICIT), and the memory is scoped to the user + current company

3. **AC-3: List memories** — GIVEN the `GET /ai/memories` endpoint WHEN called by an authenticated user with optional `?category=PREFERENCE&search=invoice` THEN only that user's memories for the current company are returned, ordered by importance (descending) then recency, with pagination support

4. **AC-4: Update memory** — GIVEN the `PATCH /ai/memories/:id` endpoint WHEN called with updated content THEN the memory content is updated, `updatedAt` is refreshed, and only the owning user can modify their own memories

5. **AC-5: Delete memory** — GIVEN the `DELETE /ai/memories/:id` endpoint WHEN called THEN the memory is permanently deleted (hard delete, not soft) and only the owning user can delete their own memories

6. **AC-6: Forget all** — GIVEN the `POST /ai/memories/forget-all` endpoint WHEN called by an authenticated user THEN all of that user's memories for the current company are permanently deleted and a success count is returned

7. **AC-7: Conversation summary service** — GIVEN the conversation summary service WHEN an AI session ends (conversation status changes to "ended" or "completed") THEN the key decisions, actions taken, and context from the conversation are compressed into a summary and stored in `ai_conversation_summaries` linked to the conversation

8. **AC-8: Memory injection service** — GIVEN the memory injection service WHEN a new AI session starts THEN relevant memories and the 3-5 most recent conversation summaries are assembled into a `<user_context>` block within a token budget of ~2000 tokens, ordered by importance with temporal decay applied

9. **AC-9: Memory settings** — GIVEN memory settings WHEN a user disables memory (`isEnabled: false`) THEN no new memories are created and existing memories are NOT injected into AI sessions (but are NOT deleted)

10. **AC-10: Importance scoring** — GIVEN the importance scoring algorithm WHEN a memory is accessed or referenced in an AI session THEN its `lastAccessedAt` is updated and the effective importance score is calculated as: `effectiveScore = baseScore * (explicitWeight) * temporalDecay(daysSinceAccess, halfLife=30)` where `temporalDecay = 0.5^(days/halfLife)` and explicit memories get 1.5x weight over implicit

11. **AC-11: Memory pruning** — GIVEN the memory pruning service WHEN the scheduled job runs THEN memories exceeding the per-user limit (default: 500 per company) that have low effective importance (< 0.1) and haven't been accessed in 90+ days are deleted

12. **AC-12: Events emitted** — GIVEN a memory is created or deleted WHEN the operation completes THEN the corresponding typed event (`AiMemoryCreated`, `AiMemoryDeleted`, `AiConversationSummarised`) is emitted via the event bus with the standard payload including `companyId`, `userId`, and entity metadata

### Task 4: Implement memory injection service (AC: #8, #9) ✅

- [x] 4.1 Create `apps/api/src/ai/memory-injection.service.ts`:
  - `assembleUserContext(userId, companyId, conversationContext?)` → returns `<user_context>` block string
  - **Steps:**
    1. Check `AiMemorySettings.isEnabled` — if false, return empty context
    2. Fetch active memories for user+company, filtered by `enabledCategories`
    3. Apply temporal decay to importance scores
    4. Sort by effective importance DESC
    5. Fetch 3-5 most recent conversation summaries
    6. Assemble into formatted `<user_context>` block
    7. Truncate to ~2000 token budget (estimate: ~4 chars per token)
  - Format:
    ```
    <user_context>
    ## Your Memories About This User
    - [INSTRUCTION] Always use Net 30 payment terms for new customers
    - [PREFERENCE] Prefers overdue invoices sorted by amount descending

    ## Recent Conversation Summaries
    - 2026-02-24: Discussed quarterly AR review. Approved write-off of 3 invoices.
    - 2026-02-20: Set up new customer Acme Ltd with special pricing.
    </user_context>
    ```

- [x] 4.2 Integrate memory injection into the orchestrator:
  - Modify `apps/api/src/ai/orchestrator.ts` to call `memoryInjectionService.assembleUserContext()` before each AI call
  - Inject the `<user_context>` block as part of the system prompt (prepend to existing system prompt content)
  - **Do NOT break existing orchestrator behaviour** — this is additive only
  - If memory injection fails, log warning and proceed without memory context (graceful degradation per IMP-006)

### Key Architectural Constraints

1. **AI Gateway is mandatory** — the conversation summary service MUST use `packages/ai-gateway` for LLM calls, never call Claude/OpenAI directly (BR-PLT-007)
2. **Graceful degradation** — if memory injection fails, the AI session must still work (IMP-006). Wrap injection in try/catch, log warning, proceed without memory
3. **No pgvector yet** — this story does NOT implement vector search. That comes in E5b-4 (Hybrid Search). Memory retrieval here uses standard SQL queries with importance ordering
4. **BullMQ for scheduled jobs** — use existing BullMQ connection (same as briefing scheduler), not raw `setInterval` or `cron`
5. **Hard delete for memories** — per epic spec, `DELETE` and `forget-all` are permanent (not soft delete). This is a privacy feature
6. **Token budget estimation** — use ~4 chars per token as rough estimate for token budgeting. The AI Gateway has a `tokenEstimator` utility
