# Story E3.2: Audit Trail Service

Status: done

## Story

As an **administrator**,
I want an immutable audit log recording every business entity mutation,
So that I can trace all changes for compliance and investigation purposes.

## Acceptance Criteria

1. GIVEN a business event is emitted WHEN the audit subscriber receives it THEN an append-only AuditLog record is created with entityType, entityId, action (CREATE/UPDATE/DELETE/APPROVE/POST), beforeData (JSONB), afterData (JSONB), userId, isAiAction, aiConfidence, timestamp, and correlationId
2. GIVEN the audit_log table WHEN any UPDATE or DELETE SQL is attempted THEN the database rejects it via PostgreSQL rules (no_update_audit, no_delete_audit)
3. GIVEN the audit query API WHEN GET /system/audit-log is called with filters (entityType, entityId, action, userId, dateRange) THEN matching records are returned with cursor pagination
4. GIVEN the audit query API WHEN GET /system/audit-log/:entityType/:entityId is called THEN the full change history for that specific entity is returned in chronological order
5. GIVEN an AI-initiated action WHEN it is audit logged THEN isAiAction is true and aiConfidence contains the confidence score from the AI orchestration layer
6. GIVEN the retention policy WHEN audit records are queried THEN records older than 6 years are accessible (no automatic deletion per NFR40)

## Dev Notes

### Critical: Append-Only Design (IMP-003)

The AuditLog table is fundamentally different from all other tables in the schema:
- **NO `updatedAt`** — records are never modified after creation
- **NO `updatedBy`** — no one can edit an audit record
- **PostgreSQL RULES** enforce immutability at the database level, not just the application level
- `CREATE RULE no_update_audit AS ON UPDATE TO audit_logs DO INSTEAD NOTHING;` silently discards any UPDATE
- `CREATE RULE no_delete_audit AS ON DELETE TO audit_logs DO INSTEAD NOTHING;` silently discards any DELETE
- This means Prisma's `prisma.auditLog.update()` and `prisma.auditLog.delete()` will "succeed" (no error) but have no effect — tests must verify the record is unchanged, not just that no error is thrown

### Architecture Decision: Event-Driven Audit

Per Architecture §2.6 and Project Context §11 Rule 4: the audit service subscribes to business events via the EventBus rather than being called explicitly by route handlers. This provides:
- **Decoupling** — business logic doesn't need to know about audit
- **Consistency** — every event automatically gets an audit record
- **Extensibility** — new modules just emit events, audit service picks them up

The AuditService uses a mapping registry (`AUDIT_EVENT_MAPPINGS`) that defines how to extract audit fields from each event payload. Future modules register their own mappings when they add new event types.

### Event Payload companyId Strategy

Several existing events (user.login, user.mfa.*) were designed without `companyId` in their payloads. Since AuditLog requires `companyId` (multi-company scoping, Architecture §2.3), Task 3 adds `companyId` to these payloads:
- Login handler: the user record fetched during authentication includes `companyId`
- MFA handlers: `request.companyId` is set by company-context middleware (post-auth)
- This is a backward-compatible change — existing event subscribers ignore extra fields

### correlationId: Optional for Now

The `correlationId` is available on `request.correlationId` (set by correlation-id middleware) but is NOT automatically included in event payloads. Since event handlers receive only the payload (not the request), correlationId is:
- **Optional** on the AuditLog model
- NOT included in event emissions in this story
- **Future enhancement:** use Node.js `AsyncLocalStorage` to thread correlationId through async context, or add correlationId to a base event payload type

### Polymorphic Entity Pattern (BR-SYS-013, BR-SYS-014)

The `entityType` and `entityId` fields form a polymorphic reference:
- `entityType` is a string (not an FK) — it references the conceptual entity type (e.g., "User", "AccessGroup", "CompanyProfile")
- `entityId` is a string (not an FK) — it references the entity's PK in the target table
- There is NO foreign key constraint on `entityId` — validation is at the application layer (BR-SYS-013)
- The application maintains a registry of valid entity type strings (BR-SYS-014)

### Cursor Pagination Pattern

Follow the existing cursor pagination pattern from resources.routes.ts:
- Use `cursor` query param (UUID of the last record's `id`)
- Use `limit` for page size (default 50, max 200)
- Response `meta` includes: `{ cursor: string | null, hasMore: boolean, total: number }`
- Prisma cursor-based pagination: `cursor: { id: cursorValue }, skip: 1` (skip the cursor record itself)

### Retention Policy (NFR40)

Per NFR40, audit records must be retained for at least 6 years. This story does NOT implement automatic deletion — records are kept indefinitely. The test (10.12) verifies that old records are still queryable. Future stories may add archival/partitioning for performance at scale.

### AI Action Tracking (AC #5)

The `isAiAction` and `aiConfidence` fields support the AI-first architecture. Currently:
- No AI events are emitted (the AI orchestration layer is a future module)
- `isAiAction` defaults to `false` for all current audit records
- The `ai.action.executed` event type exists in BusinessEvents as a placeholder
- When the AI module is implemented, it will emit events with `isAiAction: true` and `aiConfidence: 0.0–1.0`
- The audit mapping structure already supports this — the `ai.action.executed` mapping will be added in the AI module's epic

### Cross-Cutting Patterns (MANDATORY)

- **companyId**: AuditLog is company-scoped. All queries MUST filter by `request.companyId`. Event payloads include `companyId` for the audit service.
- **i18n**: Not applicable — audit log is an internal data store; no user-facing display strings in this story.
- **Audit**: This IS the audit system — self-referential. The AuditLog table does NOT audit itself (no infinite recursion).
- **Events**: The audit service SUBSCRIBES to events. It does NOT emit its own events (no audit.created event).
