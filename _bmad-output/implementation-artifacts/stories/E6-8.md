# Story E6.8: Access Group Management Pages

Status: done

## Story

As an **administrator**,
I want to list, create, view, edit, and deactivate access groups with an interactive permission matrix, so that I can define and manage granular roles for my company.

## Acceptance Criteria

1. GIVEN an authenticated ADMIN user WHEN they navigate to `/system/access-groups` THEN a T1 Entity List page renders showing all active access groups for the current company via `GET /system/access-groups`, displaying columns: Code, Name, System badge (for `isSystem: true`), User Count, and Created date
2. GIVEN the access group list WHEN it renders THEN a search input is available and typing text filters access groups by partial match on `code` or `name` fields, with a "Load More" cursor-based pagination button when `hasMore` is true
3. GIVEN the access group list WHEN a `[+ New]` button is clicked THEN the user navigates to `/system/access-groups/new` which renders a create form with fields: Code (required, UPPER_SNAKE_CASE validation), Name (required), Description (optional)
4. GIVEN the create form WHEN submitted with valid data THEN `POST /system/access-groups` is called and on 201 success the user is redirected to the detail page `/system/access-groups/:id` with a success toast
5. GIVEN the create form WHEN submitted and the API returns a `409 CONFLICT` (duplicate code) THEN the form displays an inline error on the Code field: "An access group with this code already exists"
6. GIVEN a row in the access group list WHEN clicked THEN the user navigates to `/system/access-groups/:id` which renders a T7 Settings detail page showing the group's metadata (code, name, description) and a tabbed layout with Permissions and Field Overrides tabs
7. GIVEN the access group detail page WHEN it loads THEN `GET /system/access-groups/:id` is called and the response populates: editable Name and Description fields, a read-only Code field, and the permissions array populates the permission matrix
8. GIVEN a system group (`isSystem: true`) WHEN the detail page loads THEN a banner is displayed: "This is a system access group. Its code cannot be changed and it cannot be deactivated." — the Code field is read-only (always), and the Deactivate option in the overflow menu is disabled
9. GIVEN the permission matrix on the Permissions tab WHEN it renders THEN it displays a checkbox grid with columns: Resource Name, canAccess, canNew, canView, canEdit, canDelete — grouped by module (Finance, Sales, Purchasing, etc.) with collapsible module sections
10. GIVEN the permission matrix WHEN the administrator checks/unchecks checkboxes and clicks "Save Permissions" THEN `PUT /system/access-groups/:id/permissions` is called with the full permissions array (replace-all semantics) and a success toast confirms the save
11. GIVEN the permission matrix WHEN the API returns a `400 VALIDATION_ERROR` (invalid resource codes) THEN a toast error is displayed listing the invalid resource codes
12. GIVEN the detail page metadata section WHEN the administrator edits the Name and/or Description fields and clicks "Save" THEN `PATCH /system/access-groups/:id` is called with the updated fields and a success toast confirms the update
13. GIVEN the detail page overflow menu WHEN clicked THEN it shows a "Deactivate" option (for non-system groups) which, when selected, shows a confirmation dialog: "Deactivate [group name]? Users assigned to this group will lose its permissions."
14. GIVEN the deactivate confirmation dialog WHEN confirmed THEN `DELETE /system/access-groups/:id` is called (soft-delete, sets `isActive: false`) and the user is redirected to the access group list with a success toast
15. GIVEN a deactivation attempt WHEN the API returns a `409 CONFLICT` (active users assigned) THEN a toast error is displayed: "Cannot deactivate: users are still assigned to this group. Reassign them first."
16. GIVEN a non-ADMIN user WHEN they navigate directly to `/system/access-groups` or any sub-route THEN they receive a 403 Access Denied page since the endpoint requires ADMIN role
17. GIVEN all visible text on any page in this story WHEN inspected THEN every user-facing string uses `t()` from `@nexa/i18n` — including column headers, form labels, validation errors, breadcrumbs, page titles, toasts, and empty state messages
18. GIVEN the access group list WHEN viewed on phone (<768px) THEN access groups render as mobile cards with code, name, system badge, and user count

  - [x] 1.1 Create `apps/web/src/features/admin/access-groups/api/types.ts`:
    - Define TypeScript interfaces matching the API contracts:
      ```typescript
      interface AccessGroup {
        id: string;
        code: string;
        name: string;
        description: string | null;
        isSystem: boolean;
        isActive: boolean;
        userCount: number;
        createdAt: string;
        updatedAt: string;
      }

      interface AccessGroupDetail extends AccessGroup {
        companyId: string;
        permissions: AccessGroupPermission[];
        fieldOverrides: AccessGroupFieldOverride[];
        createdBy: string;
        updatedBy: string;
      }

      interface AccessGroupPermission {
        resourceCode: string;
        resourceName: string;
        resourceModule: string;
        resourceType: 'PAGE' | 'REPORT' | 'SETTING' | 'MAINTENANCE';
        canAccess: boolean;
        canNew: boolean;
        canView: boolean;
        canEdit: boolean;
        canDelete: boolean;
      }

      interface AccessGroupFieldOverride {
        resourceCode: string;
        resourceName: string;
        fieldPath: string;
        visibility: 'VISIBLE' | 'READ_ONLY' | 'HIDDEN';
      }

      interface CreateAccessGroupRequest {
        code: string;
        name: string;
        description?: string;
      }

      interface UpdateAccessGroupRequest {
        name?: string;
        description?: string;
      }

      interface SetPermissionsRequest {
        permissions: Array<{
          resourceCode: string;
          canAccess: boolean;
          canNew: boolean;
          canView: boolean;
          canEdit: boolean;
          canDelete: boolean;
        }>;
      }
      ```

  - [x] 1.2 Create `apps/web/src/features/admin/access-groups/api/use-access-groups.ts`:
    - `useAccessGroups(params)` — `useInfiniteQuery` for list endpoint `GET /system/access-groups`:
      - Query key: `['access-groups', params]`
      - Params: `{ cursor?, limit?, search?, isActive?, isSystem? }`
      - Supports cursor-based pagination via `getNextPageParam`
    - `useAccessGroup(id)` — `useQuery` for detail endpoint `GET /system/access-groups/:id`:
      - Query key: `['access-groups', id]`
      - Returns `AccessGroupDetail` with permissions and field overrides
      - Enabled only when `id` is truthy

  - [x] 1.3 Create `apps/web/src/features/admin/access-groups/api/use-access-group-mutations.ts`:
    - `useCreateAccessGroup()` — `useMutation` calling `POST /system/access-groups`:
      - On success: invalidate `['access-groups']` query, return created group
      - Error handling: map 409 → `{ field: 'code', message: t('accessGroups.error.duplicateCode') }`
    - `useUpdateAccessGroup(id)` — `useMutation` calling `PATCH /system/access-groups/:id`:
      - On success: invalidate `['access-groups', id]` and `['access-groups']` queries
    - `useSetPermissions(id)` — `useMutation` calling `PUT /system/access-groups/:id/permissions`:
      - On success: invalidate `['access-groups', id]` query
      - Error handling: map 400 → display invalid resource codes in toast
    - `useDeactivateAccessGroup()` — `useMutation` calling `DELETE /system/access-groups/:id`:
      - On success: invalidate `['access-groups']` query
      - Error handling: map 409 → display "users still assigned" message

  **CRITICAL RULES:**
  - Use the shared API client pattern from E6.1 (`@/lib/api-client` or `@nexa/api-client`)
  - Query key factory: `['access-groups', ...]` for consistent cache invalidation
  - TypeScript strict: no `any`, all response types explicitly typed
  - All error messages use `t()` from `@nexa/i18n`

  - Module groups are collapsible to manage the potentially large list (60+ resources)

## Dev Notes

### Architecture Summary

The Access Group Management pages provide the CRUD frontend for the granular RBAC system. This is the primary admin interface for defining custom roles — each access group is a named collection of per-resource permissions (canAccess, canNew, canView, canEdit, canDelete) that can be assigned to users per company. The backend (E2b) already provides all API endpoints; this story builds the frontend.

The story spans three pages:
1. **List page** (`/system/access-groups`) — T1 Entity List showing all groups with search and pagination
2. **Create page** (`/system/access-groups/new`) — Simple form for creating a new group
3. **Detail page** (`/system/access-groups/:id`) — T7 Settings pattern with metadata editing, permission matrix, and deactivation

### RBAC Business Rules (from Business Rules Compendium)

| Rule | Description | Relevance |
|------|-------------|-----------|
| BR-RBAC-001 | Most-permissive-wins across all user access groups | Informs users that permissions are additive |
| BR-RBAC-002 | SUPER_ADMIN bypasses permission matrix entirely | SUPER_ADMIN sees this page without access group check |
| BR-RBAC-003 | System access groups cannot be deleted, only modified or cloned | Deactivate disabled for `isSystem: true` groups |
| BR-RBAC-004 | Company creator auto-assigned FULL_ACCESS group | System group exists from company creation |
| BR-RBAC-008 | Non-ADMIN cannot manage access groups | ADMIN role required on all routes |
