# Story E5b-5: Memory & Skills Management UI

**Epic:** E5b — AI Co-Pilot Intelligence — Memory, Skills & Dynamic Context
**Status:** done
**Dependencies:** E5b-1 (Memory Storage & Retrieval API — done), E5b-2 (Skills Registry & Tool Framework — done), E5b-4 (Hybrid Search & Memory Intelligence — done), E6 (Frontend Shell — done)
**Complexity:** X-Large (12 tasks)
**FRs:** FR7 (conversation history / persistent memory) | **NFRs:** NFR27 (WCAG 2.1 AA), NFR2 (CRUD <500ms)

---

## User Story

As a **user**,
I want to view, edit, and delete what the AI remembers about me, and I want to see what skills the AI has available,
so that I maintain full control and transparency over my AI experience — knowing what it knows about me and what it can do.

---

## Acceptance Criteria

### AC1: Memory Management Page — Route & Layout
**GIVEN** the app navigation
**WHEN** the user clicks "AI > My Memory" in the sidebar
**THEN** they are navigated to `/ai/memory`
**AND** the page renders using the T7 Settings template with Concept D styling
**AND** the page has two logical sections: (1) Memory Settings panel at the top, (2) Memory list below with search/filter
**AND** the sidebar shows a top-level "AI" section with items: Morning Briefing, My Memory, Skills
**AND** the page title is translated via i18n key `ai.memory.title` ("My Memory")

### AC2: Memory Settings Panel
**GIVEN** the Memory Management page
**WHEN** it renders
**THEN** the settings panel shows:
  - An "Enable AI Memory" toggle switch (maps to `ai_memory_settings.isEnabled`)
  - Category checkboxes: Preferences, Workflows, Decisions, Instructions, Entity Context (maps to `enabledCategories[]`)
  - A retention period selector: 30 / 60 / 90 / 180 / 365 / Never (maps to `retentionDays`)
  - A "Forget Everything" destructive action button (red variant)
**AND** settings are loaded from `GET /ai/memories/settings` on mount
**AND** toggling any setting calls `PATCH /ai/memories/settings` with optimistic UI update
**AND** when "Enable AI Memory" is toggled off, the category toggles and retention selector are visually disabled but the memory list remains visible (memories are preserved, just not used)
**AND** all labels use i18n translation keys under the `ai.memory.settings.*` namespace

### AC3: Memory List with Category Grouping
**GIVEN** the Memory Management page with memories loaded
**WHEN** the page renders
**THEN** memories are fetched from `GET /ai/memories` and displayed in a scrollable list
**AND** memories are grouped by category: Preferences, Workflows, Decisions, Instructions, Entity Context
**AND** each category group is collapsible/expandable (accordion pattern) with a count badge showing the number of memories
**AND** each memory card shows: content text, category badge (coloured), source badge (Explicit = purple, Learned = grey), creation date (relative), last used date (relative)
**AND** empty groups are hidden
**AND** the list supports infinite scroll or "Load More" pagination (cursor-based, using the cursor from the API response)

### AC4: Memory Search & Filter
**GIVEN** the Memory Management page
**WHEN** the user types in the search input
**THEN** the memory list is filtered client-side (debounced 300ms) to show only memories whose content matches the search text (case-insensitive)
**AND** a category filter (multi-select pill chips) allows filtering by one or more categories
**AND** both filters combine: search text AND selected categories
**AND** when no results match, an empty state is shown with "No memories match your search" message

### AC5: Edit Memory
**GIVEN** a memory card
**WHEN** the user clicks the Edit button (pencil icon)
**THEN** an edit dialog opens with the memory content in a textarea
**AND** the dialog title shows "Edit Memory"
**AND** the user can modify the content text
**AND** clicking Save calls `PATCH /ai/memories/:id` with the updated content
**AND** the UI updates optimistically (memory content changes immediately, reverts on error)
**AND** on success, a toast notification confirms "Memory updated"
**AND** on error, a toast notification shows the error and the content reverts

### AC6: Delete Single Memory
**GIVEN** a memory card
**WHEN** the user clicks the Delete button (trash icon)
**THEN** a confirmation dialog appears: "Delete this memory? This action cannot be undone."
**AND** on confirmation, calls `DELETE /ai/memories/:id`
**AND** the memory card is removed with a fade-out animation
**AND** on success, a toast notification confirms "Memory deleted"
**AND** the category group count badge decrements, and if the group is now empty it collapses/hides

### AC7: Forget Everything
**GIVEN** the "Forget Everything" button in the settings panel
**WHEN** clicked
**THEN** a destructive confirmation dialog appears requiring the user to type "FORGET" to confirm
**AND** the dialog title is "Forget Everything" with a warning icon
**AND** the dialog body explains: "This will permanently delete ALL of your AI memories. This action cannot be undone."
**AND** the confirm button is disabled until the user types "FORGET" exactly
**AND** on confirmation, calls `POST /ai/memories/forget-all`
**AND** on success, the memory list clears with an animation and shows an empty state
**AND** a toast notification confirms "All memories have been deleted"
**AND** the event `ai.memory.allDeleted` fires on the backend

### AC8: Memory Stats Panel
**GIVEN** the Memory Management page
**WHEN** it renders
**THEN** a stats summary is shown (above or beside the settings panel) displaying:
  - Total memory count
  - Breakdown by source: Explicit vs Learned (count + percentage)
  - Breakdown by category (count per category)
**AND** stats are derived from the loaded memory data (client-side computation, not a separate API call)

### AC9: AI Skills Browser Page — Route & Layout
**GIVEN** the app navigation
**WHEN** the user clicks "AI > Skills" in the sidebar
**THEN** they are navigated to `/ai/skills`
**AND** the page renders using the T7 Settings template with Concept D styling
**AND** skills are fetched from `GET /ai/skills` on mount
**AND** the page title is translated via i18n key `ai.skills.title` ("AI Skills")

### AC10: Skills List with Module Grouping
**GIVEN** the AI Skills page
**WHEN** skills are loaded
**THEN** skills are displayed in an accordion layout grouped by module (moduleKey)
**AND** each module group header shows: module display name, skill count, expand/collapse toggle
**AND** each skill card within a module group shows:
  - Skill name and description
  - Trigger phrases as green tag pills
  - Negative triggers as red tag pills (if any)
  - Orchestration pattern badge (SEQUENTIAL, PARALLEL, ITERATIVE, CONTEXT_AWARE, DOMAIN_INTELLIGENCE)
  - Active/inactive status indicator (green dot = active, grey dot = inactive)
**AND** modules with no skills are hidden
**AND** the first module group is expanded by default, others collapsed

### AC11: Skills Search & Module Filter
**GIVEN** the AI Skills page
**WHEN** the user types in the search input
**THEN** the skills list is filtered client-side (debounced 300ms) to show only skills whose name, description, or trigger phrases match the search text
**AND** a module filter dropdown allows filtering by a specific module
**AND** when no results match, an empty state is shown
**AND** the search includes both positive trigger phrases and negative triggers for matching

### AC12: Admin Skill Editing (ADMIN Role Only)
**GIVEN** an ADMIN user on the AI Skills page
**WHEN** they click on a skill card
**THEN** a slide-out sheet (or modal) opens showing the full skill details:
  - Name, display name, description (read-only for system fields)
  - Trigger phrases (editable — add/remove tags via tag input)
  - Negative triggers (editable — add/remove tags)
  - Active/inactive toggle
  - Priority number (editable)
  - Orchestration pattern (read-only)
  - Skill content / instructions (read-only, displayed in monospace)
  - Required tools list (read-only)
  - Examples (read-only, if present)
**AND** saving changes calls `PUT /ai/skill-overrides/:skillId` to create/update a tenant-specific override (NOT modifying the base skill)
**AND** the skill card in the list updates to reflect the override
**AND** a "Reset to Default" button removes the tenant override via `DELETE /ai/skill-overrides/:skillId`
**AND** STAFF users see the same slide-out sheet but all fields are read-only (no save button, no toggles)

### AC13: Concept D Visual Compliance
**GIVEN** both the Memory Management and AI Skills pages
**WHEN** rendered
**THEN** they follow Concept D design system:
  - Purple primary colour (#7c3aed) on action buttons, toggles, active states
  - Card components use 12px border-radius, custom shadow, purple-tinted hover shadow
  - Category badges use semantic colours per category type
  - Typography: Plus Jakarta Sans for headings, Inter for body text
  - Background: `#f4f2ff` (light purple) page background
  - Animations: `fadeInUp` on card entry with staggered delays
  - Skeleton loading states during data fetch (no spinners)
  - Focus rings on all interactive elements (NFR27 WCAG 2.1 AA)
  - `prefers-reduced-motion` respected for all animations

### AC14: Responsive & Accessible
**GIVEN** the Memory Management and AI Skills pages
**WHEN** viewed on different devices
**THEN** phone (375px): single column, settings sections stack vertically, accordion for categories
**AND** tablet (768px): single column form layout
**AND** desktop (1024px+): two-column settings panel (label left, input right), full memory/skill cards
**AND** all interactive elements are keyboard navigable (Tab, Enter, Escape)
**AND** all buttons and toggles have accessible labels (aria-label)
**AND** all colour-coded badges have text labels alongside colour (not colour-only information)
**AND** screen readers can navigate the full page structure (proper heading hierarchy, landmark roles)

## Dev Notes

### Architecture Patterns

- **Feature Module Pattern:** Follow the existing feature module structure in `apps/web/src/features/`. Each feature has its own directory with `components/`, `types.ts`, `api.ts`, and page components
- **TanStack Query:** Use `useQuery` for reads and `useMutation` for writes. Follow existing patterns in the codebase for query keys, invalidation, and optimistic updates
- **TanStack Router:** File-based routing under `apps/web/src/routes/_authenticated/`. Use `createFileRoute` and lazy component imports
- **Concept D Design System:** Reference the v0 components in `apps/web/src/components/v0-reference/epic-E5-E5b/` for visual guidance. Adapt to use existing design tokens and component library — do NOT copy v0 code directly
- **i18n:** ALL user-facing strings must use translation keys via `useTranslation()`. No hardcoded English strings
- **Error Handling:** Mutations show toast notifications on success/error using the existing toast system
- **Optimistic Updates:** For settings toggles and memory edits, update the UI immediately and revert on error (using TanStack Query's `onMutate`/`onError` pattern)

### Key Implementation Decisions

1. **Client-Side Filtering:** Memory and skills lists are expected to be small enough (<500 memories, <100 skills) for client-side search/filter. No server-side search needed — fetch all on mount, filter in-browser
2. **Settings Lazy Upsert:** The `GET /ai/memories/settings` endpoint auto-creates default settings if none exist (lazy upsert pattern implemented in E5b-1). The UI doesn't need to handle a "no settings" state
3. **Override Pattern for Skills:** Admin edits create `ai_skill_overrides` rows (per-tenant), NOT modifying the base `ai_skills` rows. This preserves system defaults and allows "Reset to Default"
4. **No New API Endpoints:** All required endpoints already exist from E5b-1 (memory CRUD + settings) and E5b-2 (skills CRUD + overrides). This story is frontend-only
5. **v0 Reference — Adapt, Don't Copy:** The v0 reference files use Next.js App Router patterns. Adapt to TanStack Router, use existing i18n hooks, existing auth context, and existing API client patterns

### Known AC Deviations

1. **AC12 — Negative triggers are read-only instead of editable for ADMIN.** The backend `upsertOverrideBodySchema` (in `skill-overrides.schema.ts`) only accepts `isActive`, `triggerPhrasesOverride`, and `priorityOverride`. There is no `negativeTriggersOverride` field in the schema, service, or Prisma model. Making the UI editable without backend support would silently discard edits on save. A follow-up task is needed to add `negativeTriggersOverride` to the backend before the UI can be made editable. See `skill-detail-sheet.tsx` TODO comment.

### Cross-Cutting Patterns (MANDATORY)

- **companyId:** API calls are auto-scoped by the backend via JWT auth context. Frontend doesn't need to pass companyId explicitly
- **i18n:** All strings via `useTranslation('ai')`. Category names, patterns, module names — all translated
- **RBAC:** Admin-only features (skill toggle, override save) check `user.role` from auth context. Conditionally render edit controls
- **Responsive:** T7 template handles responsive layout. Memory cards and skill cards must be usable on phone (375px)
- **Accessibility (NFR27):** WCAG 2.1 AA — keyboard navigation, aria-labels, focus management in dialogs, colour + text for status indicators
