# Story E7.2: Column Customization & Views UI

Status: done

## Story

As a **user**,
I want to manage columns (reorder, hide/show, pin, resize) and manage saved views (create, edit, delete, set default, star) from a Views & Columns modal,
so that I can customise how I browse data.

## Acceptance Criteria

1. GIVEN any T1 Entity List page WHEN the page mounts THEN it calls `GET /views/init` and auto-generates columns from `data_view_fields` metadata, applying user column preferences if they exist, or the user's default saved view if set
2. GIVEN the [Views & Columns] button WHEN clicked THEN a modal opens with two tabs: Views and Columns
3. GIVEN the Views Tab WHEN rendered THEN saved views are grouped into My Views (PERSONAL), Team Views (ROLE), and Global Views (GLOBAL) sections, each showing star toggle, edit/delete actions on hover, and a "Set as Default" action
4. GIVEN the Views Tab WHEN "Save as New View" is clicked THEN a form captures name, group name, and scope, and saves the current filter + sort + column configuration as a new SavedView
5. GIVEN a named saved view is loaded WHEN "Save Current View" is clicked THEN the view's configuration is updated in-place (overwrite)
6. GIVEN the Columns Tab WHEN rendered THEN all available columns appear as a checkbox list with drag handles for reorder and pin toggle (Left/Right/None)
7. GIVEN a column header border on the table WHEN the user drags it THEN the column resizes in real-time and the new width is persisted via `PATCH /views/columns/:viewKey/:fieldId/width` with debounce
8. GIVEN a column is pinned Left WHEN the table scrolls horizontally THEN the pinned column stays fixed with a shadow indicator

    ```typescript
    function useViewMutations(viewKey: string) {
      const queryClient = useQueryClient();

      // Invalidate view init cache after any mutation
      const invalidateViews = () =>
        queryClient.invalidateQueries({ queryKey: queryKeys.views.init(viewKey) });

      const createView = useMutation({
        mutationFn: (data: CreateSavedViewRequest) => createSavedView(data),
        onSuccess: invalidateViews,
      });

      const updateView = useMutation({
        mutationFn: ({ id, data }: { id: string; data: UpdateSavedViewRequest }) =>
          updateSavedView(id, data),
        onSuccess: invalidateViews,
      });

      const replaceView = useMutation({
        mutationFn: ({ id, data }: { id: string; data: CreateSavedViewRequest }) =>
          replaceSavedView(id, data),
        onSuccess: invalidateViews,
      });

      const removeView = useMutation({
        mutationFn: (id: string) => deleteSavedView(id),
        onSuccess: invalidateViews,
      });

      const toggleFav = useMutation({
        mutationFn: (id: string) => toggleFavourite(id),
        // Optimistic update: toggle isFavourite in cache
        onMutate: async (id) => {
          await queryClient.cancelQueries({ queryKey: queryKeys.views.init(viewKey) });
          const prev = queryClient.getQueryData<ViewInitResponse>(queryKeys.views.init(viewKey));
          if (prev) {
            queryClient.setQueryData(queryKeys.views.init(viewKey), {
              ...prev,
              savedViews: prev.savedViews.map(v =>
                v.id === id ? { ...v, isFavourite: !v.isFavourite } : v
              ),
            });
          }
          return { prev };
        },
        onError: (_err, _id, context) => {
          if (context?.prev) {
            queryClient.setQueryData(queryKeys.views.init(viewKey), context.prev);
          }
        },
        onSettled: invalidateViews,
      });

      const setDef = useMutation({
        mutationFn: (id: string) => setDefault(id),
        // Optimistic update: set isDefault, unset others
        onMutate: async (id) => {
          await queryClient.cancelQueries({ queryKey: queryKeys.views.init(viewKey) });
          const prev = queryClient.getQueryData<ViewInitResponse>(queryKeys.views.init(viewKey));
          if (prev) {
            queryClient.setQueryData(queryKeys.views.init(viewKey), {
              ...prev,
              savedViews: prev.savedViews.map(v => ({
                ...v,
                isDefault: v.id === id,
              })),
            });
          }
          return { prev };
        },
        onError: (_err, _id, context) => {
          if (context?.prev) {
            queryClient.setQueryData(queryKeys.views.init(viewKey), context.prev);
          }
        },
        onSettled: invalidateViews,
      });

      return { createView, updateView, replaceView, removeView, toggleFav, setDef };
    }
    ```

    ```typescript
    function useColumnMutations(viewKey: string) {
      const queryClient = useQueryClient();

      const bulkUpdate = useMutation({
        mutationFn: (prefs: UserColumnPreferenceDto[]) =>
          bulkUpdateColumns(viewKey, prefs),
        onSuccess: () =>
          queryClient.invalidateQueries({ queryKey: queryKeys.views.init(viewKey) }),
      });

      // Debounced width update — called during/after column border drag-resize
      const widthUpdateRef = useRef<ReturnType<typeof setTimeout>>();
      const updateWidth = useMutation({
        mutationFn: ({ fieldId, width }: { fieldId: string; width: number }) =>
          updateColumnWidth(viewKey, fieldId, width),
      });

      const debouncedUpdateWidth = useCallback(
        (fieldId: string, width: number) => {
          if (widthUpdateRef.current) clearTimeout(widthUpdateRef.current);
          widthUpdateRef.current = setTimeout(() => {
            updateWidth.mutate({ fieldId, width });
          }, 300); // 300ms debounce
        },
        [updateWidth],
      );

      return { bulkUpdate, debouncedUpdateWidth };
    }
    ```

## Dev Notes

### Architecture Summary

E7.2 builds the **frontend layer** of the metadata-driven DataTable system. It consumes the backend infrastructure created in E7.1 (6 tables, bundled init endpoint, saved view CRUD, column preference APIs) and delivers the React component library that powers all T1 Entity List pages.

The key architectural insight is that **zero custom column/filter UI code is needed per entity type**. The `useViewState(viewKey)` hook fetches metadata from the backend and auto-generates TanStack Table column definitions, filter configurations, and sort options. When future module epics add new entity lists (Invoices, Customers, Items, etc.), they simply pass `viewKey="INVOICES"` to the template — all column management, view saving, and filtering works automatically.

### Performance Strategy

- **Bundled init**: Single `GET /views/init` call on page mount instead of 4-5 separate calls. Response ~5-15KB, <50ms from Redis cache.
- **TanStack Query cache**: `staleTime: 30s` prevents redundant refetches during modal open/close cycles.
- **Optimistic updates**: `toggleFavourite` and `setDefault` update the cache immediately, then reconcile with server response.
- **Debounced width persistence**: Column drag-resize fires `PATCH .../width` with 300ms debounce — prevents flooding the API during continuous drag.
- **Memoised column definitions**: `useMemo` on `tanstackColumns` prevents TanStack Table from re-rendering on every state change.
- **Fire-and-forget mutations**: Width updates and favourite toggles don't block UI — errors are silently retried or toasted.

### Cross-Cutting Patterns (MANDATORY)

- **companyId**: Handled by the API client automatically (injects `x-company-id` header from auth store).
- **i18n**: ALL user-facing text must use `t()` translation keys. No hardcoded English strings. Key namespace: `views.*`.
- **RBAC**: Use `usePermissions()` to check if user can create GLOBAL views (requires ADMIN). Edit/delete limited to view owner or ADMIN.
- **Mobile adaptation**: Use `useBreakpoint()` to simplify the modal on mobile (single column layout, no pin toggle, simplified view list).
- **Accessibility (WCAG 2.1 AA)**: Keyboard navigation for all interactive elements, focus trapping in modal, aria-labels on icon buttons, screen reader announcements for state changes (toast notifications).
- **Concept D compliance**: All components must use theme tokens from `globals.css`. No hardcoded colours, shadows, or border-radius values. Purple accent (#7c3aed), 12px card radius, Plus Jakarta Sans headings, Inter body text.
