# Story E2b.6: Admin can export and import permission configurations

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an **administrator**,
I want to export my company's access group configuration as a JSON file and import it into other companies,
So that I can standardize permissions across multiple companies or share best-practice configurations.

## Acceptance Criteria

1. **AC1 — Export returns complete JSON:** Given an authenticated ADMIN user, when they GET /system/company-profile/export-defaults, then they receive a JSON object containing: `version` (schema version string), `description`, `exportedAt` (ISO datetime), `exportedFrom` (company name), `resources` (all active resources), `accessGroups` (company's groups with permissions and field overrides), `vatCodes`, `paymentTerms`, `numberSeries`, and `currencies` for the current company.

2. **AC2 — Import upserts access groups:** Given an ADMIN user POSTing a valid defaults JSON to /system/company-profile/import-defaults, when an access group code already exists in the target company, then its name and description are updated but its `isSystem` flag is preserved. New codes are created.

3. **AC3 — Import upserts resources:** Given the import JSON contains resources, when the import runs, then resources are upserted by code (global table — create or update). Existing resources not in the import file are NOT deleted.

4. **AC4 — Import replaces permissions and field overrides:** Given an imported access group, when it includes `permissions` and `fieldOverrides` arrays, then permissions are replaced (delete-all + create-many) per group, and field overrides are replaced per group.

5. **AC5 — Import summary returned:** Given a successful import, when the operation completes, then the response contains `status: 'APPLIED'`, a `summary` with counts of created/updated records for each entity type, and a `warnings` array for non-fatal issues.

6. **AC6 — Dry run mode:** Given `dryRun: true` in the import request, when the endpoint processes the data, then it validates and counts what would change but does NOT persist. The response has `status: 'DRY_RUN'`.

7. **AC7 — Cache invalidation on import:** Given a successful non-dry-run import that modifies access groups or permissions, when the import completes, then the `company.defaultData.imported` event is emitted, triggering permission cache invalidation for ALL users in the company.

8. **AC8 — Validation errors:** Given invalid JSON structure or missing required fields, when the import endpoint processes the data, then it returns 400 VALIDATION_ERROR. Given an unsupported version, it returns 400 UNSUPPORTED_VERSION.

9. **AC9 — ADMIN role required:** Given a non-ADMIN user (MANAGER, STAFF, VIEWER), when they attempt export or import, then they receive 403 Forbidden.

10. **AC10 — SUPER_ADMIN bypass:** Given a SUPER_ADMIN user, when they attempt export or import, then the permission guard is bypassed and the operation succeeds.

## Dev Notes

### Architecture Overview

This story implements the **export/import permission configuration** endpoints — the final story in the E2b epic. These endpoints allow administrators to export their company's access group configuration as JSON and import it into other companies, enabling permission standardization across multi-company setups.

**Request lifecycle:**
```
Request → JWT verify (E2) → Company context (E2) → Permission guard (E2b-4, ADMIN required) → Route handler (export/import) → Response
```

### Key Implementation: Inline JSON Import (NOT Multipart)

The API contract specifies both multipart file upload and inline JSON. **For MVP, implement inline JSON only** (`request.body.data`). Multipart file upload can be added later. This avoids the complexity of `@fastify/multipart` registration and file handling.

```typescript
// Import request body — inline JSON only for E2b-6
interface ImportDefaultsRequestBody {
  data: ExportDefaultsResponse;  // The full export JSON
  dryRun?: boolean;              // Default: false
}
```

### Dry Run Implementation Strategy

Use Prisma's `$transaction` with intentional rollback for dry run:

```typescript
if (dryRun) {
  // Execute entire import inside a transaction, then throw to rollback
  try {
    await prisma.$transaction(async (tx) => {
      const result = await executeImport(tx, companyId, data, userId);
      // Store result before rollback
      throw { __dryRunResult: result };
    });
  } catch (e: unknown) {
    if (e && typeof e === 'object' && '__dryRunResult' in e) {
      return { status: 'DRY_RUN' as const, ...(e as { __dryRunResult: ImportSummary }).__dryRunResult };
    }
    throw e; // Real error
  }
} else {
  const result = await prisma.$transaction(async (tx) => {
    return executeImport(tx, companyId, data, userId);
  });
  appEvents.emit('company.defaultData.imported', { companyId, importedBy: userId, version: data.version });
  return { status: 'APPLIED' as const, ...result };
}
```

### Existing Default Data Loader Pattern (packages/db)

The `packages/db/src/services/default-data-loader.service.ts` already implements upsert logic for resources and access groups from the static `company-defaults.json` file. **Do NOT reuse this directly** — it reads from the filesystem and uses a cached module-level object. The import service needs to accept arbitrary JSON input. However, the **upsert patterns** (find existing → upsert → delete/recreate permissions) should be replicated.

Key patterns from `default-data-loader.service.ts`:
- Resource upsert: `prisma.resource.upsert({ where: { code }, create: {...}, update: {...} })`
- Access group upsert: `prisma.accessGroup.upsert({ where: { companyId_code: { companyId, code } }, ... })`
- Permission replace-all: `deleteMany({ where: { accessGroupId } })` → `createMany({ data: [...] })`
- `isSystem` flag: on create uses the imported value, on update does NOT change it (preserves existing)

### Resource Table: Global (No companyId)

**CRITICAL:** Resources are NOT company-scoped. The Resource table is global — all companies share the same resource registry. When importing resources, upsert by `code` without companyId scoping.

### Access Groups: Company-Scoped

Access groups ARE company-scoped. When importing, upsert by `{ companyId, code }` using the current company's ID. Never import access groups across company boundaries.

### Permission & Field Override Validation

When importing permissions/field overrides, validate that the `resourceCode` references exist in the Resource table. If a resourceCode doesn't exist:
- Add a warning: `"Skipped unknown resource code '{code}' in access group '{groupCode}' permissions"`
- Skip that permission/field override entry
- Do NOT fail the entire import

### Event System Integration

Add the new event type to the typed event emitter:

```typescript
// In event-emitter.ts
export interface CompanyDefaultDataImportedEvent {
  companyId: string;

  version: string;

// In AppEventMap
'company.defaultData.imported': CompanyDefaultDataImportedEvent;
```

Register cache invalidation in `permission-cache-listeners.ts`:

```typescript
appEvents.on('company.defaultData.imported', (payload) => {
  // Invalidate all cached permissions for the company
  // permissionService.invalidateAll() clears the entire cache
  // This is correct because import may affect any user's permissions
  permissionService.invalidateAll();
});
```

**Note:** `invalidateAll()` clears the entire in-memory Map cache. For MVP this is acceptable since permission cache is per-process. When Redis is added (E3+), this should be scoped to `permissions:*:{companyId}`.

### Reference Data Models (Prisma)

For export/import of vatCodes, paymentTerms, numberSeries, currencies — check which fields are available:

**VatCode:** `{ companyId, code, name, rate (Decimal), type (VatType enum), isDefault }`
- Upsert key: unique constraint on `(companyId, code)` — check schema. If no compound unique, use findFirst + create/update.

**PaymentTerms:** `{ companyId, code, name, dueDays, isDefault }`
- Upsert key: needs `(companyId, code)` compound unique

**NumberSeries:** `{ companyId, entityType, prefix, padding, nextValue, isActive }`
- Upsert key: `(companyId, entityType)` — do NOT import `nextValue` (preserve existing sequence position)

**Currency:** `{ code, name, symbol, minorUnit }` — GLOBAL table (no companyId)
- Upsert key: `code`

**IMPORTANT:** Verify the actual Prisma schema has the compound unique indexes needed for upserts before implementing. If a compound unique doesn't exist, use `findFirst` + conditional `create`/`update` pattern instead of `upsert`.

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: Export queries access groups, vatCodes, paymentTerms, numberSeries with `WHERE companyId = ?`. Resources and currencies are global (no companyId). Import creates all company-scoped entities with `companyId = request.companyId`.
- **i18n**: Error messages use translation-key-friendly codes (e.g., `VALIDATION_ERROR`, `UNSUPPORTED_VERSION`). No user-facing prose strings.
- **Audit**: Emit `company.defaultData.imported` event on successful non-dry-run import. This satisfies audit trail requirements for configuration changes.
- **Attachments/Notes/Tasks**: Not applicable for export/import configuration.
