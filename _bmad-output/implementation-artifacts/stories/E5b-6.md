# Story E5b-6: E7 Skill Pack Validation (Proof-of-Concept)

**Epic:** E5b — AI Co-Pilot Intelligence — Memory, Skills & Dynamic Context
**Status:** done
**Dependencies:** E5b-2 (Skills Registry, Tool Framework & Dynamic Context — done), E7 (Saved Views, Filters & Columns — done)
**Complexity:** Large (10 tasks)

---

## User Story

As a product team, we want to validate the skills architecture end-to-end by seeding and testing the E7 (Saved Views) skill pack, proving that the AI can correctly route user intents like "show me overdue invoices" through L0→L1→L2 to the right action.

---

## Acceptance Criteria

### AC1: E7 Skill Pack Registration
**GIVEN** the E7 skill pack seed
**WHEN** it runs
**THEN** the following 5 skills are registered: `open_entity_list`, `search_views`, `apply_filter`, `list_saved_views`, `create_saved_view`
**AND** each skill has moduleKey=`views`, packKey=`views-core`
**AND** each skill has properly defined triggerPhrases, negativeTriggers, requiredTools, parameters, examples, orchestrationPattern, and priority

### AC2: E7 Module Knowledge Registration
**GIVEN** the E7 module knowledge seed
**WHEN** it runs
**THEN** knowledge entries are registered for moduleKey=`views` with types: OVERVIEW, ENTITIES, WORKFLOWS
**AND** the OVERVIEW entry describes the Saved Views module capabilities
**AND** the ENTITIES entry describes DataView, SavedView, DataViewField, and SavedViewCondition
**AND** the WORKFLOWS entry describes view creation, filter application, and column customisation flows

### AC3: E7 Entity Trigger Registration
**GIVEN** the E7 entity trigger seed
**WHEN** it runs
**THEN** entity triggers are registered for trigger words: "view", "saved view"
**AND** each trigger maps to the correct entity type, search endpoint, and display field

### AC4: L0 Meta-Router Classifies to Views Module
**GIVEN** the meta-router (L0)
**WHEN** the user says "show me all invoices"
**THEN** L0 classifies this as module=`views` with confidence > 0
**AND** the `views` module appears in the available modules summary

### AC5: L1 Pack Loading and Skill Selection
**GIVEN** the views module pack (L1)
**WHEN** loaded for a user message "show me all invoices"
**THEN** all 5 skills are loaded (no tenant overrides assumed)
**AND** trigger phrases for each skill are matched: "show me" + "invoices" → `open_entity_list` selected with high confidence

### AC6: L2 Skill Activation
**GIVEN** `open_entity_list` skill is selected (L2)
**WHEN** activated
**THEN** the full skill instructions are loaded into context
**AND** the required tools (`open_entity_list`) are resolved from the ToolRegistry
**AND** ENTITIES knowledge for the views module is injected
**AND** total token count is within budget (~600 tokens for L2 + knowledge)

### AC7: Search Views Routing
**GIVEN** the user says "show me the Overdue for more than 20 days View"
**WHEN** processed through L0→L1→L2
**THEN** L1 selects the `search_views` skill
**AND** L2 activates with the `search_views` tool resolved
**AND** the skill's parameters define a `query` string input

### AC8: Apply Filter Routing
**GIVEN** the user says "filter invoices by this month"
**WHEN** processed through L0→L1→L2
**THEN** L1 selects the `apply_filter` skill
**AND** L2 activates with the `apply_filter` tool resolved
**AND** the skill's parameters define `viewKey`, `conditions` inputs

### AC9: Create Saved View Routing
**GIVEN** the user says "create a new view called Big Invoices for amounts over 10000"
**WHEN** processed through L0→L1→L2
**THEN** L1 selects the `create_saved_view` skill
**AND** L2 activates with the `create_saved_view` tool resolved
**AND** the skill's parameters define `name`, `viewKey`, `conditions` inputs

### AC10: Negative Trigger Exclusion
**GIVEN** a negative trigger scenario
**WHEN** the user says "create an invoice"
**THEN** the views skill pack does NOT activate any skill (negative trigger: "create an invoice" ≠ "create a view")
**AND** L1 selectSkill returns null for the views module pack

### AC11: Tool Registration for Views Module
**GIVEN** the views module tool definitions
**WHEN** the module registers its tools
**THEN** 5 tool definitions are registered in the ToolRegistry: `open_entity_list` (query), `search_views` (query), `apply_filter` (action), `list_saved_views` (query), `create_saved_view` (action)
**AND** each tool has a complete inputSchema with properly typed properties
**AND** each tool has moduleKey=`views`

### AC12: Query Tool Handler Registration
**GIVEN** the views query tool handlers
**WHEN** the module registers query handlers
**THEN** `open_entity_list`, `search_views`, and `list_saved_views` have query handlers registered in the QueryExecutor
**AND** each handler enforces companyId scoping when querying DataView/SavedView tables
**AND** results are returned as structured JSON with proper rowCount

### AC13: Seed Idempotency
**GIVEN** the E7 seed scripts (skill pack, module knowledge, entity triggers)
**WHEN** run multiple times
**THEN** no duplicate entries are created
**AND** existing entries are updated (upsert pattern for skills and triggers, delete-and-recreate for knowledge)

### AC14: Dynamic Context Assembly with Views Skill
**GIVEN** the dynamic context assembler in INTERACTIVE mode
**WHEN** the user says "show me overdue invoices"
**THEN** the assembled context includes: user memories + views skill chain (L0→L1→L2) + views module knowledge + user permissions
**AND** the tools array contains only the resolved tools for the activated skill (not all registered tools)
**AND** total context is within the ~5000 token budget

  - Input: `{ viewKey?: string }`
  - Query: List all `SavedView` records visible to the user, optionally filtered by viewKey, scoped by companyId
  - Return: Array of `{ id, name, viewKey, viewName, scope, isDefault, isFavourite }`, ordered by name
  - Enforce companyId scoping

- Query handlers receive `{ companyId, userId, input }` and must scope all Prisma queries by `companyId`
- The saved view visibility logic must match E7's scope-based filtering: PERSONAL views visible only to creator, ROLE views visible to users with matching role, GLOBAL views visible to all
- Result data should be concise — the AI doesn't need full SavedViewCondition details, just summaries
- These are READ-only handlers — `apply_filter` and `create_saved_view` use the ActionExecutor pattern from E5 and are NOT implemented in this story (they require the full guardrails + user confirmation flow)
