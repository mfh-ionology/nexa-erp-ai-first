# Story E12-2: Template Management

**Epic:** E12 — Document Templates & PDF
**Status:** done
**Dependencies:** E12-1 (Template Engine — Prisma schema, DocumentTemplateService, TemplateCompilerService, PdfGeneratorService), E6 (Frontend Shell — layout, routing, Concept D), E2b (RBAC — permission guards)
**Complexity:** X-Large (11 tasks)
**FRs:** FR79 (report templates), FR191 (generate PDFs via Document Templates engine), FR192 (admin print preferences per document type) | **NFRs:** NFR2 (CRUD <500ms), NFR41 (settings management)

---

## User Story

As an **administrator**,
I want to create, edit, and version document templates with active/draft management and preview capability, so that I can customise the look of business documents (invoices, POs, statements, etc.) for my company.

---

## Acceptance Criteria

### AC1: Template CRUD Operations
**GIVEN** an authenticated user with ADMIN role
**WHEN** they create a document template via `POST /api/v1/document-templates`
**THEN** they can specify:
  - `documentType` (one of 14 DocumentType enum values)
  - `name` (string, 1–200 chars)
  - `htmlTemplate` (Handlebars HTML body)
  - `pageSize` (A4, A5, Letter — default A4)
  - `orientation` (portrait, landscape — default portrait)
  - `marginTop`, `marginBottom`, `marginLeft`, `marginRight` (decimal mm values)
  - `showLogo`, `logoPosition`, `showBankDetails`, `showVatNumber`, `showCompanyReg` (branding toggles)
  - `description` (optional), `headerHtml`, `footerHtml`, `cssStyles` (optional overrides)
  - `isDefault` (boolean — whether this is the default template for the document type)
**AND** the template is scoped to the authenticated user's `companyId`
**AND** a unique constraint on `[companyId, documentType, name]` prevents duplicates (returns 409 on conflict)
**AND** the response returns the created template with its `id`
**AND** if `isDefault = true`, any existing default template for the same `documentType` in the same company has its `isDefault` set to `false` (at most one default per type per company)
**AND** the operation completes within 500ms (NFR2)

### AC2: Template Listing and Retrieval
**GIVEN** an authenticated user with ADMIN role
**WHEN** they call `GET /api/v1/document-templates`
**THEN** the response returns all templates for the user's company with:
  - Filtering by `documentType` (optional)
  - Filtering by `isActive` (optional, default true)
  - Search by `name` (optional, case-insensitive partial match)
  - Cursor-based pagination (cursor + limit, default 50, max 100)
  - Results include version count per template
**AND** templates are ordered by `documentType` ASC, then `name` ASC
**AND** `GET /api/v1/document-templates/:id` returns a single template with all its versions included
**AND** both endpoints enforce `companyId` scoping (only templates belonging to the user's company are returned)
**AND** accessing a template from a different company returns 404 (not 403)

### AC3: Template Update and Soft-Delete
**GIVEN** an authenticated user with ADMIN role
**WHEN** they call `PATCH /api/v1/document-templates/:id` with updated fields
**THEN** only the provided fields are updated (partial update)
**AND** the `updatedAt` timestamp is refreshed
**AND** if `isDefault` is changed to `true`, the previous default for that documentType is unset
**AND** `companyId` and `documentType` cannot be changed after creation
**WHEN** they call `DELETE /api/v1/document-templates/:id`
**THEN** the template is soft-deleted (`isActive = false`) rather than hard-deleted
**AND** soft-deleted templates are excluded from list results by default (unless `isActive=false` filter is explicitly used)
**AND** soft-deleted templates cannot be used for document generation (the version selection algorithm in E12-1 already filters by `isActive`)

### AC4: Version Management
**GIVEN** a `DocumentTemplate` record
**WHEN** an ADMIN user creates a new version via `POST /api/v1/document-templates/:id/versions`
**THEN** a `DocumentTemplateVersion` record is created with:
  - Selection criteria: `languageCode`, `branchCode`, `numberSeriesId`, `accessGroup`, `customerGroupId` (all optional)
  - Override fields: `htmlOverride`, `cssOverride`, `headerOverride`, `footerOverride` (all optional)
  - Email settings: `emailSubject`, `emailBody`, `replyToEmail`, `ccEmails` (all optional)
  - `priority` (integer, default 0)
  - `isActive` (boolean, default true)
**AND** the previous versions are retained (versions are never deleted on create — they accumulate)
**AND** the new version can be set as active or kept as draft (`isActive = false`)
**WHEN** an ADMIN user calls `PATCH /api/v1/document-templates/:id/versions/:versionId`
**THEN** the version's fields are updated (partial update)
**WHEN** an ADMIN user calls `DELETE /api/v1/document-templates/:id/versions/:versionId`
**THEN** the version is hard-deleted (versions are not soft-deleted — they are removed)
**AND** all version operations verify the version belongs to the specified template (returns 404 otherwise)
**AND** all version operations verify the template belongs to the user's company (companyId scoping)

### AC5: Template Preview
**GIVEN** a `DocumentTemplate` record
**WHEN** an ADMIN user calls `POST /api/v1/document-templates/:id/preview`
**THEN** the system:
  1. Loads the template from the database
  2. Generates sample data appropriate for the template's `documentType` (mock company, customer/supplier, line items, totals)
  3. Compiles the Handlebars template with the sample data (using E12-1's TemplateCompilerService)
  4. Renders the compiled HTML to PDF (using E12-1's PdfGeneratorService)
  5. Returns the PDF as `Content-Type: application/pdf`, `Content-Disposition: inline`
**AND** sample data includes realistic UK-formatted values (GBP currency, UK date format, sample VAT rates 20%/5%/0%)
**AND** the optional request body allows specifying a `versionId` to preview a specific version's overrides
**AND** if no `versionId` is specified, the base template is previewed
**AND** preview completes within 5 seconds (NFR3)
**AND** the endpoint returns 404 if the template doesn't exist or belongs to a different company

### AC6: Template Management UI — Template List
**GIVEN** an ADMIN user navigates to `/settings/document-templates`
**THEN** the page uses the T7 Settings template layout (Concept D design system)
**AND** displays a list of templates grouped by document type (collapsible sections)
**AND** each template shows: name, status badge (active/inactive), default badge (if isDefault), version count
**AND** the page includes:
  - A document type filter dropdown
  - A search input for filtering by name
  - An "Add Template" primary action button
  - Overflow menu per template with: Edit, Preview, Clone, Deactivate/Activate, Set as Default
**AND** the page follows Concept D visual fidelity: 12px card radius, purple-tinted shadows, Plus Jakarta Sans headings, `#f4f2ff` background

### AC7: Template Management UI — Editor Form
**GIVEN** an ADMIN user clicks "Add Template" or "Edit" on an existing template
**THEN** a form is displayed (slide-over panel or dedicated section) with:
  - Document type selector (dropdown, disabled on edit)
  - Template name input
  - Description textarea (optional)
  - Page settings: page size dropdown, orientation dropdown, margin inputs (top/bottom/left/right in mm)
  - Branding toggles: show logo (with logo position dropdown), show bank details, show VAT number, show company registration — all as switches
  - HTML template editor with monospace font (JetBrains Mono), minimum height 400px
  - CSS styles editor with monospace font (optional, collapsible)
  - Header HTML editor (optional, collapsible)
  - Footer HTML editor (optional, collapsible)
  - isDefault checkbox
  - Save / Cancel buttons
**AND** the form validates: name required (1–200 chars), htmlTemplate required, documentType required on create
**AND** on save, the form calls the appropriate API endpoint (POST for create, PATCH for update)
**AND** on successful save, the template list is refreshed and a success toast is shown
**AND** on validation error, field-level error messages are displayed

### AC8: Template Management UI — Version Management
**GIVEN** an ADMIN user is viewing/editing a template
**THEN** a "Versions" section is displayed showing all versions for the template
**AND** each version shows: priority, selection criteria summary (e.g., "Language: fr, Branch: MAIN"), status badge (active/draft), created date
**AND** the section includes:
  - "Create Version" button
  - Per-version actions: Edit, Clone, Activate/Deactivate, Delete (with confirmation dialog)
**AND** the version editor form includes:
  - Selection criteria: languageCode, branchCode, numberSeriesId, accessGroup, customerGroupId inputs
  - Override fields: HTML override editor, CSS override editor, header/footer override editors (all collapsible)
  - Email settings: emailSubject, emailBody editor, replyToEmail, ccEmails
  - Priority number input
  - isActive toggle
  - Save / Cancel buttons
**AND** creating/editing/deleting a version refreshes the version list inline

### AC9: Template Management UI — Preview Panel
**GIVEN** an ADMIN user clicks "Preview" (either from the template list overflow menu or within the editor)
**THEN** the system calls `POST /api/v1/document-templates/:id/preview` (optionally with `versionId`)
**AND** displays a loading spinner while the PDF is being generated
**AND** on success, renders the PDF inline in an `<iframe>` or `<object>` element within the page
**AND** includes controls: Download, Open in New Tab, Print (via `window.print`)
**AND** if the PDF generation fails, an error message is shown (not a crash)

## Dev Notes

- **Extending existing service**: E12-1 created `DocumentTemplateService` with version selection only. E12-2 adds CRUD + version management methods to the same service class. Do NOT create a separate service — extend the existing one.
- **Prisma schema already exists**: E12-1 created the `DocumentTemplate` and `DocumentTemplateVersion` models with all fields, indexes, and constraints. No schema migration needed for E12-2.
- **Import DocumentType from @nexa/db**: E12-1's code review flagged hardcoded enum values (Issue #3). E12-2's schemas should use `z.nativeEnum(DocumentType)` with `import { DocumentType } from '@nexa/db'` to maintain a single source of truth.
- **Route prefix**: Template management routes go under `/system/document-templates` (via the system module plugin). Document generation routes are already under `/system/documents` (from E12-1). These are different prefixes — do not confuse them.
- **HTML editor for MVP**: Use a styled `<textarea>` with monospace font. Adding a full code editor (CodeMirror/Monaco) would significantly increase bundle size and complexity. This can be a follow-up enhancement.
- **Preview uses existing Puppeteer instance**: The preview endpoint reuses the `pdfGeneratorService` instance that was initialised in E12-1's system module wiring. No additional Puppeteer setup needed.
- **companyId scoping pattern**: Follow existing patterns from `access-groups.service.ts` — all queries include `companyId`, cross-company access returns null (not a 403), caller converts null to 404.
- **Error pattern**: Use `AppError` for HTTP-level errors, `DomainError` for business rule violations (matching existing `core/errors/` patterns).
- **Frontend file location**: Create feature folder at `apps/web/src/features/settings/document-templates/` following the feature-based folder structure used by existing features (e.g., `features/ai-admin/knowledge/`).
- **Blob URL cleanup**: When displaying PDF preview via `URL.createObjectURL()`, ensure cleanup via `URL.revokeObjectURL()` in a `useEffect` return function to prevent memory leaks.
