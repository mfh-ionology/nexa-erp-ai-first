# Story E3.3: Event Persistence & Dead Letter

Status: done

## Story

As a **developer**,
I want failed event handlers to be retried with exponential backoff and persisted to a dead-letter queue,
So that transient failures do not cause permanent data loss.

## Acceptance Criteria

1. GIVEN an event handler that fails on first attempt WHEN the retry mechanism triggers THEN the handler is retried up to 3 times with exponential backoff (1s, 2s, 4s)
2. GIVEN an event handler that fails all 3 retries WHEN the final retry fails THEN the event is persisted to a dead-letter queue (BullMQ) with the full event payload and error details
3. GIVEN the dead-letter queue WHEN an administrator queries it THEN they can see failed events with event name, payload, error message, retry count, and timestamp
4. GIVEN a dead-letter item WHEN an administrator triggers a manual re-process THEN the event is re-emitted through the event bus for all subscribers
5. GIVEN event handlers WHEN they are designed THEN they must be idempotent: processing the same event twice produces no duplicate side effects, using sourceId/correlationId for deduplication

## Dev Notes

### Architecture Decision: BullMQ for Dead-Letter Queue

Per Architecture section 4.2, BullMQ is the chosen technology for async job processing. The DLQ leverages BullMQ's Redis-backed job queue for persistence. This is consistent with the architecture's plan to use BullMQ for "long-running side effects (email sending, PDF generation, bank feed sync)."

**Infrastructure available:**
- Redis `7-alpine` is already in `docker-compose.yml` on port 6379
- `REDIS_URL=redis://localhost:6379` is already in `.env.example`
- No `ioredis` or `bullmq` packages are currently installed

### Critical Design: EventBus Backward Compatibility

The retry mechanism is **opt-in** via `setRetryExecutor()`. When no executor is set, `emit()` behaves exactly as before — single attempt with catch + log. This ensures:
- All existing tests pass without modification
- The EventBus singleton remains functional before the Fastify plugin initializes
- Tests that don't need retry can use the EventBus directly without mocking BullMQ/Redis

### Retry Mechanism Design

```
Handler called → Fails? → Wait backoffBaseMs * 2^(attempt-1) → Retry
                   ↓ (after maxRetries)
              Add to DLQ (BullMQ job)
```

- **Backoff schedule**: 1s, 2s, 4s (for default config: base=1000ms, max=3 retries)
- **Total max delay**: 7s per handler per event (1+2+4)
- **Non-blocking**: Retry loop runs inside the existing `queueMicrotask` async wrapper — does NOT block `emit()` return
- **Error isolation**: Each handler has its own independent retry loop — one handler's retries don't affect others

### Important: Handlers That Catch Their Own Errors

Some existing handlers already catch errors defensively (e.g., AuditService catches DB errors and logs them without re-throwing). These handlers will NEVER trigger retry because they don't throw. This is correct behavior — the retry mechanism only activates when a handler throws an uncaught error.

Handlers affected:
- `AuditService.registerEventSubscriptions()` handlers — catch errors internally
- `permission-cache-listeners.ts` handlers — fire-and-forget async

### BullMQ DLQ Implementation Detail

BullMQ jobs are used as DLQ storage. Each dead-letter entry is a BullMQ job in the `"event-dead-letter"` queue:

```typescript
// Job data structure
{
  eventName: string;          // e.g., "invoice.posted"
  payload: unknown;           // The original event payload (serialized JSON)
  error: string;              // Error message from final retry
  stack?: string;             // Error stack trace
  retryCount: number;         // Number of retry attempts made (always = maxRetries)
  originalTimestamp: string;  // ISO timestamp when event was originally emitted
  reprocessed: boolean;       // false initially, true after manual reprocess
  reprocessedAt?: string;     // ISO timestamp when reprocessed
}
```

The BullMQ queue is configured with:
- `defaultJobOptions.removeOnComplete: false` — keep completed (reprocessed) jobs for audit
- `defaultJobOptions.removeOnFail: false` — keep failed jobs
- `defaultJobOptions.attempts: 1` — no BullMQ-level retry (retry is handled by EventBus)

### DLQ List Endpoint — BullMQ Pagination

BullMQ's `Queue.getJobs()` returns jobs by status. For the DLQ list API:
- Use `Queue.getJobs(['waiting', 'completed', 'failed'])` to get all entries
- Apply in-memory filtering (by eventName, reprocessed status) — DLQ volume is expected to be low
- Cursor pagination uses BullMQ job IDs
- If DLQ volume becomes large, consider adding a Prisma model in future (post-MVP)

### Reprocess Endpoint — Type Safety Consideration

When re-emitting an event from the DLQ via `eventBus.emit(eventName, payload)`, the event name is a `string` from storage, not a `keyof BusinessEvents`. To maintain type safety:
- Cast the event name: `eventBus.emit(eventName as keyof BusinessEvents, payload as BusinessEvents[keyof BusinessEvents])`
- This is an acceptable cast since the DLQ only stores events that were previously type-checked at original emission time
- Add a runtime check that `eventName` is a known event (optional validation)

### Idempotency — Current Scope and Future Enhancement

AC #5 requires idempotent handlers. This story:
- Creates an `IdempotencyGuard` utility for handler use
- Documents the pattern for future module developers
- Does NOT retroactively add correlationId to all event payloads (this would modify all existing event emissions)

**Future enhancement (not in this story):** Use Node.js `AsyncLocalStorage` to thread `request.correlationId` through async context and automatically include it in event payloads. This was identified in E3-2 dev notes.

### Known Issue from E3-1 Code Review: drain() Infinite Loop Risk

E3-1 code review identified that `drain()` could loop indefinitely if cascading events create new pending promises. The retry mechanism adds longer-lived promises (up to 7s per handler). During `drain()`:
- Retry promises are tracked in `this.pending`
- `drain()` will wait for all retries to complete before resolving
- This is correct behavior for graceful shutdown — all retries should finish
- The infinite loop risk from cascading events is pre-existing and unrelated to retry

### Graceful Degradation: Redis Unavailable

If Redis is not available when the dead-letter plugin initializes:
- Log a warning: `[DeadLetterPlugin] Redis connection failed — dead-letter queue disabled`
- Set `deadLetterService` to null on the EventBus
- The EventBus continues to function with retry-only (no persistence of failed events)
- Failed events after max retries are logged to the application log (existing behavior)

This ensures the API server starts and functions even without Redis, which is important for:
- Local development when Redis isn't running
- Tests that don't need DLQ functionality
- Graceful handling of Redis outages in production

### Permission Guard: system.dead-letter-queue.list

The DLQ routes use `createPermissionGuard('system.dead-letter-queue.list', 'view')` for GET and `'manage'` for POST reprocess. This resource code must exist in the resource registry (Task 9.5). Grant to:
- **FULL_ACCESS** — canAccess: true, canView: true, canCreate: true (for reprocess)

### Cross-Cutting Patterns (MANDATORY)

- **companyId**: DLQ entries are NOT company-scoped. Events from all companies flow through the same event bus and DLQ. This is infrastructure-level, not tenant-level. The DLQ admin routes require ADMIN permission.
- **i18n**: Not applicable — DLQ is an internal infrastructure view; no user-facing display strings in this story.
- **Audit**: The DLQ itself does NOT emit audit events. When an event is reprocessed, the re-emitted event will trigger its normal audit subscribers.
- **Events**: The retry mechanism wraps event handler execution. It does NOT create new event types.

- EventBus uses `queueMicrotask` for async — retry delays must live inside the existing async wrapper
- Handler deduplication uses `Set` — retry does not affect dedup
- `drain()` waits on `this.pending` Set — retry promises must be tracked there

**E3-2 Learnings:**
