# Story E10-3: Document-to-Email

**Epic:** E10 — Email Integration
**Status:** done
**Dependencies:** E10-1 (SMTP Outbound Service — done), E10-2 (Email Template Management — done), E8 (Attachments + RecordLinks — done)
**Complexity:** X-Large (10 tasks)
**FRs:** FR188 (send emails from within business records with PDF attachment) | **NFRs:** NFR2 (CRUD < 500ms p95)

---

## User Story

As a **user**,
I want to send invoices, purchase orders, and statements as PDF attachments via email directly from the record screen,
so that I can communicate with customers and suppliers without leaving the ERP.

---

## Acceptance Criteria

### AC1: Email Composition Dialog from Action Bar
**GIVEN** a posted invoice (or any document in a sendable state)
**WHEN** the user clicks "Email" in the action bar overflow → Document Actions section
**THEN** an email composition dialog opens with:
  - `From`: pre-filled with the company's email alias (from SystemSettings `smtp.fromEmail` or company profile)
  - `To`: pre-filled with the customer/supplier email (resolved from Customer.email → Contact.email with role "Accounts")
  - `Subject`: pre-filled from the document-type-specific email template (Handlebars rendered with record data)
  - `Body`: pre-filled from the same template (Handlebars rendered with record data)
  - `Attachment`: auto-generated PDF of the document shown as a preview card
**AND** the dialog title includes the document identifier (e.g., "Send Invoice INV-00234 via Email")
**AND** the user can edit any field before sending

### AC2: CC/BCC Recipients
**GIVEN** the email composition dialog is open
**WHEN** the user clicks `[+ Cc]` or `[+ Bcc]` toggle links
**THEN** the CC/BCC fields expand with multi-email chip inputs
**AND** the user can add multiple recipients using email addresses
**AND** all recipients (TO, CC, BCC) are validated per BR-COM-001 (RFC 5322 format check)
**AND** duplicate recipients across TO/CC/BCC are warned about per BR-COM-002

### AC3: Template Resolution and Rendering
**GIVEN** a document type (e.g., `CustomerInvoice`)
**WHEN** the email template is resolved for the composition dialog
**THEN** the system uses `resolveTemplate(documentType, languageCode?)` from E10-2 with the BR-COM-010 fallback hierarchy:
  1. Custom template matching documentType + languageCode
  2. System default template matching documentType (default "en")
  3. Error if no template exists
**AND** the template's Handlebars variables are populated from the source record data (invoice number, customer name, total amount, due date, company name, etc.)
**AND** the user can switch templates via the TemplateSelector dropdown and "Reset to Template" to repopulate fields

### AC4: RecordLink Between Email and Source Document
**GIVEN** an email is sent successfully from the composition dialog
**WHEN** the send is confirmed
**THEN** a `RecordLink` is created:
  - `sourceEntityType`: the document type (e.g., "CustomerInvoice")
  - `sourceEntityId`: the source document UUID
  - `targetEntityType`: "EmailMessage"
  - `targetEntityId`: the created EmailMessage UUID
  - `linkType`: `RELATES_TO`
  - `isSystemGenerated`: true
**AND** an Activity record is created (type: EMAIL, subject: "Invoice emailed to {recipient}", entityType: source document type)

### AC5: Batch Statement Email
**GIVEN** the batch statement generation endpoint
**WHEN** the user triggers `POST /ar/reports/statements/batch` with a date range and optional customer filters
**THEN** the system:
  - Identifies all customers with a non-zero balance for the period
  - For each customer: generates a statement PDF, resolves the STATEMENT_SEND email template, creates an EmailMessage with the statement attached, and queues for sending
  - Uses a BullMQ job for batch processing to avoid blocking the API
  - Returns a batch job ID for progress tracking
**AND** each batch email creates a RecordLink between the statement and the EmailMessage

### AC6: Document Email API Endpoint
**GIVEN** a `POST /documents/email` request
**WHEN** called with `{ documentType, recordId, recipientOverrides?, cc?, bcc? }`
**THEN** the system:
  - Validates the source document exists and is in a sendable state
  - Resolves the email template for the document type (BR-COM-010)
  - Renders the template with record data (Handlebars merge)
  - Generates or retrieves the PDF for the document
  - Creates an EmailMessage (direction: OUTBOUND, status: DRAFT, sourceEntityType, sourceEntityId)
  - Creates EmailRecipient rows (FROM: company email, TO: resolved or override, CC, BCC)
  - Attaches the PDF via the Attachment model (entityType: "EmailMessage", entityId) — BR-COM-015
  - Sets `hasAttachments = true` on the EmailMessage (fixing E10-1 CR issue #6)
  - Appends email signature once (BR-COM-009)
  - Queues the email for sending (DRAFT → QUEUED)
  - Creates RecordLink (AC4)
**AND** permission level: STAFF+
**AND** responds with the created EmailMessage ID and queue status

## Dev Notes

- **Path conventions**: Backend files go in `apps/api/src/modules/communications/email/`. Frontend files go in `apps/web/src/features/email/`.
- **E12 dependency**: The PDF generation depends on E12 (Document Templates & PDF). If E12 is not ready, create a placeholder that generates a simple "Document attached" PDF or skip the PDF attachment step with a TODO marker. The dialog should still function with a "PDF will be generated when available" placeholder.
- **E10-1 CR issue #6**: The `hasAttachments` field is never set to `true` in E10-1. This story MUST explicitly set it when attaching the PDF. Do this via a direct Prisma update after creating the EmailMessage.
- **E10-1 CR issue #5**: The FROM address should NOT be freely editable. Populate it from per-company SMTP config (`smtp.fromEmail`), not from user input. The dialog's From field is a dropdown of configured company aliases, not a free-text input.
- **E10-1 CR issue #2**: `messageNumber` has a global `@unique` instead of per-company unique. This is a known issue — do not rely on messageNumber uniqueness across companies. The E10-3 code should not create additional workarounds; this should be fixed separately.
- **RecordLink**: Use the existing `createRecordLink()` from `apps/api/src/modules/cross-cutting/record-link.service.ts`. It validates both source and target entities exist and enforces company scoping via the entity-registry.
- **Action bar placement**: The "Email" action goes in the overflow menu's "Document Actions" section, not as a primary action. However, for APPROVED invoices, consider promoting it to a more visible position.
- **Error pattern**: Use existing `AppError` from `core/errors/index.js`
- **Responsive**: The dialog follows the Concept D design system. Desktop: 600px centered modal. Tablet: 90% viewport width. Mobile: full-screen bottom sheet.
