# Story E6.7: Resource Registry Page

Status: done

## Story

As an **administrator**,
I want a read-only Resource Registry page listing all controllable system resources, so that I can see what pages, reports, settings, and maintenances are available for permission configuration.

## Acceptance Criteria

1. GIVEN an authenticated ADMIN user WHEN they navigate to `/system/resources` THEN a T1 Entity List page renders showing all resources from the `Resource` table via `GET /system/resources`
2. GIVEN the Resource Registry page WHEN it loads THEN the data table displays columns: Code (`code`), Name (`name`), Module (`module`), Type (`type`), Sort Order (`sortOrder`) — sorted by module then sortOrder by default
3. GIVEN the page WHEN it renders THEN a Module filter dropdown is available listing all distinct module values (system, finance, sales, purchasing, inventory, crm, hr, manufacturing, reporting, etc.) and selecting a module filters the table to resources in that module only
4. GIVEN the page WHEN it renders THEN a Type filter dropdown is available with the `ResourceType` enum values (PAGE, REPORT, SETTING, MAINTENANCE) and selecting a type filters the table to resources of that type only
5. GIVEN the page WHEN it renders THEN a search input is available and typing text filters resources by partial match on `code`, `name`, or `description` fields
6. GIVEN the page WHEN rendered THEN the `[+ New]` button is NOT shown, no row actions (edit/delete) are available, and no batch actions are available — the page is entirely read-only since resources are system-managed
7. GIVEN the table WHEN a row is clicked THEN nothing happens (no detail navigation) — resources are view-only at the list level
8. GIVEN a non-ADMIN user (MANAGER, STAFF, VIEWER) WHEN they navigate directly to `/system/resources` THEN they receive a 403 Access Denied page (per the route-level permission guard) since the endpoint requires ADMIN role
9. GIVEN the page WHEN the API returns data THEN the response follows the standard envelope (`{ success: true, data: [...], meta: { cursor, hasMore } }`) and the page supports cursor-based pagination with a "Load More" button
10. GIVEN all visible text on the page WHEN inspected THEN every user-facing string uses `t()` from `@nexa/i18n` — including column headers, filter labels, breadcrumbs, page title, and empty state messages
11. GIVEN the page WHEN viewed on phone (<768px) THEN resources render as mobile cards (per T1 responsive behaviour) with code as card title and module/type as card details
12. GIVEN the Module and Type filters WHEN both are applied simultaneously THEN the filters compose with AND logic (e.g., module="finance" AND type="PAGE" shows only finance page resources)

## Dev Notes

### Architecture Summary

The Resource Registry is a read-only T1 Entity List page at `/system/resources` that displays all `Resource` table entries. Resources are system-managed entities (seeded from `company-defaults.json` on company creation) that define the controllable pages, reports, settings, and maintenance screens in the ERP. They are the foundation of the granular RBAC system — each resource row represents something that can be granted or denied via Access Group permissions.

This page is intentionally simple: it's a read-only list with filtering and search. No CRUD operations, no detail page, no batch actions. Admins use this page to understand what resources are available when configuring Access Group permissions (E6.8).

### Data Model Reference

**Resource table** (`packages/db/prisma/schema.prisma:556`):

```prisma
model Resource {
  id          String       @id @default(uuid())
  code        String       @unique       // e.g., "sales.orders.list"
  name        String                     // e.g., "Sales Orders"
  module      String                     // e.g., "sales"
  type        ResourceType               // PAGE, REPORT, SETTING, MAINTENANCE
  parentCode  String?                    // FK → Resource.code (detail → list)
  sortOrder   Int          @default(0)   // Display order
  icon        String?                    // Icon key
  description String?                    // Help text
  isActive    Boolean      @default(true)

  @@index([module, sortOrder])
  @@index([type])
  @@map("resources")
}
```

**ResourceType enum:** `PAGE`, `REPORT`, `SETTING`, `MAINTENANCE`

### API Endpoint Reference

| Method | Path | Description | Min Role |
|--------|------|-------------|----------|
| `GET` | `/system/resources` | List all resources (for admin UI) | ADMIN |

**Query parameters:** Standard list params (`cursor`, `limit`, `sort`, `order`, `search`) plus:
- `module` (string) — filter by module name
- `type` (string) — filter by ResourceType enum value

**Response:**

```json
      "isActive": true
    }
  ],

    "cursor": "uuid-of-last-item",
    "hasMore": true
  }
}
```

### RBAC Business Rules (from Business Rules Compendium)

| Rule | Description | Relevance |
|------|-------------|-----------|
| BR-RBAC-002 | SUPER_ADMIN bypasses the permission matrix entirely | SUPER_ADMIN sees this page without access group check |
| BR-RBAC-006 | Module access derived from access group permissions | System module sidebar item visible only if user has canAccess on any system resource |
| BR-RBAC-008 | Non-ADMIN users cannot manage access groups or assign them | Resource Registry requires ADMIN role |

### Read-Only Enforcement

This page is read-only by design. Resources are system-managed:
- **No [+ New] button** — `canCreate={false}`
- **No row click** — `onRowClick` is undefined
- **No batch actions** — `batchActions=[]`
- **No overflow actions** — `overflowActions=[]`
- **No edit/delete** — no row action column
- Resources are seeded from `company-defaults.json` — not user-created

### Filter Composition

Filters compose with AND logic at the API level:
- `GET /system/resources?module=finance&type=PAGE` → returns only finance module pages
- `GET /system/resources?search=invoice&module=sales` → returns sales resources matching "invoice"
- All three filters (module, type, search) can be active simultaneously

### Cross-Cutting Patterns (MANDATORY)

- **i18n**: Every label, column header, filter option, breadcrumb, empty state uses `t()` from `@nexa/i18n`
- **Accessibility (NFR27, NFR28)**: semantic HTML, `aria-label` on main, accessible names on all interactive elements
- **TypeScript strict**: all types defined, no `any`, `Resource` interface matches Prisma schema
- **Permission model**: ADMIN role required — enforced at route level via permission guard
