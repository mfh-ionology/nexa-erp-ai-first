# Story E4.3: Number/Date/Currency Formatting

Status: done

## Story

As a **user**,
I want numbers, dates, and currency values to be formatted according to my locale settings,
so that financial data is displayed in a familiar format.

## Acceptance Criteria

1. GIVEN a user with locale `en-GB` WHEN a monetary value of 1234.56 GBP is displayed THEN it renders as "£1,234.56"
2. GIVEN a user with locale `en-GB` WHEN a date of 2026-02-17 is displayed THEN it renders as "17/02/2026" (DD/MM/YYYY)
3. GIVEN a currency with `minorUnit = 0` (e.g., JPY) WHEN an amount is displayed THEN it shows no decimal places
4. GIVEN the `Intl` API WHEN formatting numbers THEN it uses the user's locale for thousands separator and decimal point
5. GIVEN a formatting utility WHEN called from both web and mobile apps THEN it produces consistent output (shared package)

  - [x] 2.1 Create `packages/shared/src/formatters/format-currency.ts`:
    ```typescript
    /**
     * Format a monetary value according to locale and currency rules.
     *
     * @param amount  - Numeric value or Decimal string (API returns monetary
     *                  fields as strings to preserve Decimal(19,4) precision).
     * @param currencyCode - ISO 4217 currency code (e.g., 'GBP', 'JPY', 'BHD').
     * @param locale  - BCP 47 locale string (e.g., 'en-GB').
     * @param options - Optional overrides (minorUnit, useSymbol).
     * @returns Formatted currency string (e.g., "£1,234.56").
     */
    export function formatCurrency(
      amount: string | number,
      currencyOrCode: string | CurrencyInfo,
      locale: string,
      options?: FormatCurrencyOptions,
    ): string
    ```
    Implementation details:
    - Parse `amount` to `number` via `Number(amount)`. If `NaN`, return empty string or throw.
    - Second parameter accepts `string` (ISO 4217 code) or `CurrencyInfo` object (from Task 1.1). When `CurrencyInfo` is passed, `minorUnit` is used automatically unless overridden via `options.minorUnit`.
    - Use `Intl.NumberFormat(locale, { style: 'currency', currency: currencyCode })`.
    - If `options.minorUnit` is provided (or derived from `CurrencyInfo`), set `minimumFractionDigits` and `maximumFractionDigits` to override Intl's default. This is critical for currencies like JPY (0 decimals) and BHD (3 decimals) — see AC #3.
    - If `options.useSymbol` is `false`, use `{ currencyDisplay: 'code' }` instead of the default `'symbol'`.
    - **Never use floating-point arithmetic on the amount** — parsing to `number` is safe for display formatting since `Number()` preserves enough precision for visual rendering of Decimal(19,4) values. Actual calculations must use fixed-point (NFR38).

## Dev Notes

### Architecture Context

This is the **third and final Tier 1 i18n infrastructure story**. It creates locale-aware formatting utilities for numbers, dates, currencies, and percentages. The formatters are split across two packages:

1. **`packages/shared/src/formatters/`** — Pure utility functions with no React dependency. These are usable by both web (`apps/web`) and mobile (`apps/mobile`, created in E6) apps, satisfying AC #5. They accept `locale` as an explicit parameter.

2. **`packages/i18n/src/format-hooks.ts`** — React hooks that wrap the shared formatters with automatic locale resolution from the i18n context (created in E4-1). These provide a convenient API for React components.

**Critical constraint: `apps/web` is still a stub** (same as E4-1). The hooks are tested independently using `@testing-library/react` + `renderHook()`. Integration into the web app is E6's responsibility.

### Key Technical Decisions

1. **Pure functions in `@nexa/shared`, hooks in `@nexa/i18n`:** The formatters are pure utility functions that accept explicit `locale` + options parameters — no React dependency, no side effects. The React hooks in `@nexa/i18n` are thin wrappers that bind the current locale from the i18n context. This separation ensures mobile (React Native) and backend code can use the same formatters.

2. **`amount` accepts `string | number`:** The API returns monetary values as string Decimal(19,4) representations (NFR38, API Contracts §1). The `formatCurrency()` function accepts both `string` (from API) and `number` (from in-memory calculations) and parses to `number` for `Intl.NumberFormat`. This parsing is safe for display formatting — no arithmetic is performed on the value.

3. **`minorUnit` override via options, not automatic lookup:** The `formatCurrency()` function accepts an optional `minorUnit` in its options parameter. It does NOT fetch currency metadata from the database or API. Consumers must supply `minorUnit` from the Currency model (fetched separately, cached in React Query when the web app exists in E6+). This keeps the formatter a pure function with no I/O.

4. **`Intl` API exclusively — no dependencies:** All formatting uses the built-in JavaScript `Intl` API (`Intl.NumberFormat`, `Intl.DateTimeFormat`). No external formatting libraries (date-fns, luxon, numeral.js). This aligns with the Architecture specification and keeps the bundle lean.

5. **Date presets instead of format strings:** Instead of accepting arbitrary format patterns (like `'DD/MM/YYYY'`), `formatDate()` accepts preset names (`'short'`, `'medium'`, `'long'`) that map to `Intl.DateTimeFormat` options. This ensures locale-correct formatting — hardcoded patterns like `DD/MM/YYYY` would break for non-UK locales when i18n expands.

6. **UTC timezone by default for date formatting:** `formatDate()` and `formatDateTime()` use `timeZone: 'UTC'` to prevent date-only ISO strings (e.g., `'2026-02-17'`) from shifting to the previous day in negative-offset timezones. The JavaScript `Date` constructor parses date-only strings as UTC midnight, so formatting in local time could display the wrong date. This is the correct default for an ERP where dates come from the API as ISO 8601 strings.

7. **`formatCurrency()` accepts `string | CurrencyInfo` for currency parameter:** Task 1.1 defines `CurrencyInfo` with the explicit note "Consumers fetch currency data from the API and pass this type to `formatCurrency()`." The implementation accepts both a plain ISO 4217 string and a `CurrencyInfo` object, using `CurrencyInfo.minorUnit` automatically when provided. This avoids requiring consumers to destructure `CurrencyInfo` and pass `minorUnit` via options separately.

### Monetary Value Handling (NFR38 / IMP-002)

The Architecture mandates `Decimal(19,4)` for all monetary values with NO floating-point arithmetic:

- **Database:** Stores as `Decimal` (Prisma type → PostgreSQL `NUMERIC(19,4)`)
- **API transport:** Returns as `string` (e.g., `"1234.5600"`) to preserve precision
- **Frontend display:** `formatCurrency()` parses string to `number` for `Intl.NumberFormat` — safe because this is display-only formatting, not arithmetic
- **Frontend calculations:** Must use a fixed-point library (e.g., `decimal.js`) — NOT the formatters

### Currency.minorUnit Integration

The Prisma Currency model (`packages/db/prisma/schema.prisma:30-47`):
```prisma
model Currency {
  code       String  @id @db.VarChar(3)    // "GBP", "JPY", "BHD"
  symbol     String                         // "£", "¥", ".د.ب"
  minorUnit  Int     @default(2)            // 2 for GBP/USD, 0 for JPY, 3 for BHD
  roundTotal Int     @default(2)
  roundVat   Int     @default(2)
  roundLine  Int     @default(2)
  ...
}
```

The `minorUnit` field controls decimal places in `Intl.NumberFormat`:
- **GBP/USD (minorUnit=2):** `£1,234.56` — standard 2 decimal places
- **JPY (minorUnit=0):** `¥1,234` — no decimal places (AC #3)
- **BHD (minorUnit=3):** `BD1,234.567` — 3 decimal places

When `options.minorUnit` is passed to `formatCurrency()`, it overrides `Intl.NumberFormat`'s built-in defaults:
```typescript
const intlOptions: Intl.NumberFormatOptions = {
  style: 'currency',
  currency: currencyCode,
};
if (options?.minorUnit !== undefined) {
  intlOptions.minimumFractionDigits = options.minorUnit;
  intlOptions.maximumFractionDigits = options.minorUnit;
}
```
