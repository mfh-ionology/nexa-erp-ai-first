# Story E5c.6: Automation Monitoring & Run History

Status: done

## Story

As an **admin**,
I want to view automation execution history with per-step details, monitor active runs, and handle failures,
so that I can ensure automations are running correctly and troubleshoot issues.

## Acceptance Criteria

1. **AC-1: Automation Runs list page** — GIVEN the Automation Runs page WHEN rendered THEN it shows a T1 Entity List of all runs across all automations with columns: automation name (bold, link to automation detail), trigger type badge (same colour mapping as E5c-5: purple "Scheduled", blue "Event", amber "Chain", grey "Manual"), triggered by (e.g. "scheduler", "manual:user@example.com", "chain:automationName"), started at (formatted datetime), duration (computed: completedAt - startedAt, displayed as "Xs" or "Xm Ys" or "—" if still running), status (StatusBadge: green "Completed" / red "Failed" / amber "Running" / grey "Pending" / muted "Cancelled"), total tokens (mono font, formatted with comma separators), total cost (mono font, formatted as £X.XXXX), step count (from stepRuns array length on detail — or show "—" on list). Supports column sorting on started at, duration, status, and total tokens.

2. **AC-2: Run history filters** — GIVEN the Automation Runs list page WHEN rendered THEN it includes filter controls: status multi-select dropdown (PENDING, RUNNING, COMPLETED, FAILED, CANCELLED — mapped to StatusBadge colours), date range picker (from/to date inputs), and an optional automation name filter (dropdown of all automations for scoping). Filters are applied as query params to `GET /ai/automations/runs`. Results are cursor-based paginated at 50 per page and load within 500ms (NFR2). Clearing filters resets to showing all runs.

3. **AC-3: Run detail page** — GIVEN a specific run WHEN an admin clicks to view THEN a T2 Detail page opens showing:
   - **Run summary header:** automation name (link back), run ID (mono), status (large StatusBadge), triggered by, started at, completed at, total duration
   - **Metrics cards row:** total tokens (input + output breakdown), total cost (£X.XXXX), total steps (completed/total), total turns across all steps
   - **Step-by-step timeline:** vertical timeline with status indicator per step (green circle "Completed", red circle "Failed", amber spinner "Running", grey circle "Pending", strikethrough "Skipped"). Each step shows: step number, agent name (with display name), status badge, duration (latencyMs formatted), token count (inputTokens + outputTokens)
   - **Step expandable details:** clicking a step expands to show: goal text, input data (collapsible JSON viewer with syntax highlighting), output data (collapsible JSON viewer), model used, turns count, detailed token breakdown (input tokens / output tokens), latency in milliseconds

4. **AC-4: Failed step detail view** — GIVEN a failed run step WHEN expanded THEN the admin sees: error message (displayed in a red-bordered alert box), the step's input data (so the admin can see what was fed to the agent), the step status FAILED badge, and action buttons: "Retry from This Step" (primary action) and "View Automation" (secondary, navigates to automation builder). If the run has subsequent steps that were SKIPPED, they are shown greyed out with "Skipped (previous step failed)" label.

5. **AC-5: Retry from failed step** — GIVEN a failed run WHEN the admin clicks "Retry from This Step" on the run detail page THEN a confirmation dialog appears explaining: "This will create a new run starting from step {N} ({step goal preview}). Previous step outputs will be preserved." On confirm, `POST /ai/automations/runs/:runId/retry` is called. On success (202): toast "Retry started" with link to the new run, and the page navigates to the new run detail. On error: toast with error message. The new run appears in the run list linked to the original via `retryOfRunId`.

6. **AC-6: Automation Health dashboard section** — GIVEN the AI Configuration Dashboard WHEN rendered THEN it includes an "Automation Health" section (below the existing summary cards) showing:
   - **Automations by status:** mini donut/pie chart showing active vs paused vs inactive count with legend
   - **Failed runs (last 24h):** alert count badge — if > 0, displayed in red with count; if 0, green "All healthy"
   - **Upcoming scheduled runs:** list of next 6 scheduled runs (automation name, next run time as relative "in 2 hours") — from `AiAutomationSchedule.nextRunAt` where `isPaused = false`
   - **Token spend trend:** small line/area chart showing daily automation token usage for the last 7 days
   Clicking the section header or any item navigates to the Automation Runs page with appropriate filters pre-applied.

7. **AC-7: Circuit breaker warning** — GIVEN an automation that has been auto-paused by the circuit breaker (3 consecutive failures) WHEN the AI Configuration Dashboard renders THEN it shows a warning banner at the top of the Automation Health section: amber alert with icon, text: "{automation name} has been paused after 3 consecutive failures. [View Runs] [Resume]". The "View Runs" link navigates to the run list filtered to that automation's failed runs. The "Resume" action calls `PATCH /ai/automations/:id` with `{ isActive: true }` and unpauses the schedule, with a confirmation dialog: "Are you sure you want to resume {name}? It will run on its next scheduled time."

8. **AC-8: Run detail from automation list** — GIVEN the Automation list page (E5c-5) WHEN an admin clicks on an automation's "last run status" badge THEN it navigates directly to that run's detail page. The automation list page also gains a "View Runs" action in the per-row dropdown menu that navigates to the run list filtered to that automation.

9. **AC-9: Visual design fidelity** — GIVEN all monitoring pages WHEN rendered THEN they match Concept D visual design: 12px card radius, purple-tinted hover shadows, `#7c3aed` primary buttons, `#f4f2ff` background, Plus Jakarta Sans headings, Inter body text, JetBrains Mono for code/tokens/costs, StatusBadge with coloured dot + label, fadeInUp animation on page load. Step timeline uses purple connector lines between steps. Failed steps use red border accent. Charts use purple gradient fills.

## Technical Notes

### Backend Endpoints (All Exist — No Backend Work Required)

All API endpoints were implemented in E5c-1 and E5c-2. Frontend types and hooks for runs were scaffolded in E5c-5. This story is **frontend-only** — building the UI pages that consume these existing endpoints.

| Method | Endpoint | Purpose | Notes |
|--------|----------|---------|-------|
| GET | `/ai/automations/runs` | List all runs across automations | Query: cursor, limit (default 50), status, dateFrom, dateTo |
| GET | `/ai/automations/:id/runs` | List runs for specific automation | Query: cursor, limit, status, dateFrom, dateTo |
| GET | `/ai/automations/runs/:runId` | Run detail with step runs | Returns full stepRuns array |
| POST | `/ai/automations/runs/:runId/retry` | Retry from failed step | Returns 202 with `{ message, automationId, originalRunId }` |
| GET | `/ai/automations` | List automations (for filters, health) | Used for automation name filter dropdown |
| PATCH | `/ai/automations/:id` | Update automation (resume paused) | Used by circuit breaker Resume action |
