# Story E3b.2: Tenant Management API

Status: done

## Story

As a **platform administrator**,
I want to manage tenant lifecycle (create, view, suspend, reactivate, archive) and configure per-tenant module overrides and feature flags,
so that I can onboard new customers, control their access, and customise their feature set.

## Acceptance Criteria

1. GIVEN PLATFORM_ADMIN role WHEN `POST /admin/tenants` is called with valid tenant data THEN a new Tenant record is created in PROVISIONING status with the assigned plan, database connection metadata, and a PlatformAuditLog entry
2. GIVEN a tenant in PROVISIONING status WHEN provisioning completes (manual trigger via `POST /admin/tenants/:id/activate`) THEN the status transitions to ACTIVE and a `tenant.created` event is emitted
3. GIVEN a tenant in ACTIVE status WHEN `POST /admin/tenants/:id/suspend` is called with a reason THEN the status transitions to SUSPENDED, a `tenant.suspended` webhook is pushed to the ERP, and a PlatformAuditLog entry is created
4. GIVEN a tenant in SUSPENDED status WHEN `POST /admin/tenants/:id/reactivate` is called THEN the status transitions to ACTIVE, a `tenant.reactivated` webhook is pushed, and a PlatformAuditLog entry is created
5. GIVEN a tenant in SUSPENDED status WHEN `POST /admin/tenants/:id/archive` is called THEN the status transitions to ARCHIVED (irrecoverable from UI per BR-PLT-003), a `tenant.archived` webhook is pushed, and a PlatformAuditLog entry is created
6. GIVEN invalid state transitions (e.g., ACTIVE to ARCHIVED directly, or ARCHIVED to any state) WHEN attempted THEN a 422 INVALID_STATE_TRANSITION error is returned per BR-PLT-001
7. GIVEN PLATFORM_ADMIN role WHEN `PUT /admin/tenants/:id/modules` is called with module overrides THEN the TenantModuleOverride records are upserted, a `tenant.modules_changed` webhook is pushed per changed module, and a PlatformAuditLog entry is created
8. GIVEN PLATFORM_ADMIN role WHEN `PUT /admin/tenants/:id/feature-flags` is called with feature flag updates THEN the TenantFeatureFlag records are upserted and a PlatformAuditLog entry is created
9. GIVEN PLATFORM_VIEWER or PLATFORM_ADMIN role WHEN `GET /admin/tenants` is called THEN a paginated list of tenants is returned with status, plan, region, last activity, and active module count — filterable by status, planId, and search term
10. GIVEN PLATFORM_VIEWER or PLATFORM_ADMIN role WHEN `GET /admin/tenants/:id` is called THEN the full tenant detail is returned including plan, module overrides, feature flags, and billing summary

## Dev Notes

### Architecture Context

This story builds on the **Platform API Fastify server** created in E3b.1. It adds tenant management capabilities to the Platform API, including:

1. **CRUD operations** — Create, read, update tenants
2. **Lifecycle management** — State machine transitions (PROVISIONING -> ACTIVE -> SUSPENDED -> ARCHIVED)
3. **Module overrides** — Per-tenant module on/off overrides
4. **Feature flags** — Per-tenant feature toggles
5. **Webhook push** — Notify ERP tenants of lifecycle and configuration changes

### Current State of `apps/platform-api/`

E3b.1 (DONE) established the following infrastructure that this story builds upon:
- `app.ts` — Fastify app builder with full plugin chain
- `core/auth/` — Platform JWT auth, service token auth, role guard
- `core/audit/` — PlatformAuditService (append-only), audit-on-response hook
- `core/errors/` — AppError hierarchy (AuthError, ValidationError, NotFoundError, DomainError)
- `core/schemas/` — successEnvelope() Zod wrapper
- `core/utils/` — sendSuccess() helper
- `routes/admin/users.routes.ts` — Example CRUD route pattern to follow
- `routes/platform/entitlements.routes.ts` — Entitlement endpoints (reads tenant data)

### Key Patterns to Follow (from E3b.1)

1. **Route plugin pattern:** Use `fp()` wrapper with named dependencies
2. **Schema pattern:** Separate `*.schema.ts` file with Zod schemas + inferred types
3. **Response pattern:** Use `sendSuccess(reply, data, meta?, statusCode)` with `successEnvelope(schema)` in route schema
4. **Role guard pattern:** `requirePlatformRole('PLATFORM_ADMIN')` as preHandler
5. **Audit pattern:** For routes where you need the created entity's ID, use explicit `fastify.platformAudit.log()` wrapped in try/catch. For simple updates, use `config.audit` on the route.
6. **Error pattern:** Throw `AppError`, `NotFoundError`, `DomainError` for business errors. 422 for invalid state transitions, 409 for conflicts, 404 for not found.
7. **Prisma pattern:** Use `getPlatformPrisma()` lazy singleton. Always use `select` to pick only needed fields — never return full model (prevents leaking passwordHash, mfaSecret, etc.).

### Tenant Lifecycle State Machine (State Machine Reference §20.1)

```
  PROVISIONING ──► ACTIVE ◄──── Reactivate ──── SUSPENDED ──── Archive ──► ARCHIVED
                     │  ▲                            ▲
                     │  │                            │
                     │  └── Payment/Override ──┐     │
                     │                         │     │
                     └── Billing ──► READ_ONLY ─┴─── Escalate
```

Valid transitions (this story implements):
| From | To | Trigger | This Story? |
|------|----|---------|-------------|
| PROVISIONING | ACTIVE | activate endpoint | Yes |
| ACTIVE | SUSPENDED | suspend endpoint (manual) | Yes |
| ACTIVE | READ_ONLY | billing enforcement | No (E3b.5) |
| READ_ONLY | ACTIVE | payment/override | No (E3b.5) |
| READ_ONLY | SUSPENDED | billing escalation | Yes (service supports it) |
| SUSPENDED | ACTIVE | reactivate endpoint | Yes |
| SUSPENDED | ARCHIVED | archive endpoint | Yes |

### Business Rules Enforced

| Rule | Description | Implementation |
|------|-------------|----------------|
| BR-PLT-001 | Strict state machine transitions | `validateTransition()` in tenant-lifecycle.service.ts |
| BR-PLT-002 | Suspension takes effect within 30 seconds | Webhook pushed immediately on suspend (NFR51) |
| BR-PLT-003 | Archived tenants irrecoverable from UI | No outbound transitions from ARCHIVED state |
| BR-PLT-016 | Platform audit log immutable | PlatformAuditService is append-only |
| BR-PLT-017 | Every state-changing action logged | All lifecycle + CRUD operations create audit entries |

### Events Emitted (Event Catalog §19)

| Event | Trigger | Webhook to ERP? |
|-------|---------|-----------------|
| `tenant.created` | PROVISIONING -> ACTIVE | No (ERP not provisioned yet at this point) |
| `tenant.suspended` | ACTIVE/READ_ONLY -> SUSPENDED | Yes — bust entitlement cache |
| `tenant.reactivated` | SUSPENDED -> ACTIVE | Yes — bust entitlement cache |
| `tenant.archived` | SUSPENDED -> ARCHIVED | Yes — block all access |
| `tenant.modules_changed` | Module override upserted | Yes — bust module cache |

- **Webhook service** uses `PLATFORM_SERVICE_TOKEN` for authentication — same token validated by the ERP's service-token.guard.
- **Code uniqueness** enforced at both validation and DB constraint level.
