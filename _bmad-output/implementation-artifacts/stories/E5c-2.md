# Story E5c.2: Prompt Variable Binding System

Status: done

## Story

As an **admin**,
I want to define variables in prompt templates that are automatically resolved from database fields, page state, or computed expressions,
so that prompts are dynamic and context-rich without hardcoding.

## Acceptance Criteria

1. **AC-1: DB_FIELD prompt rendering** — GIVEN a prompt template with `{{customer.name}}` WHEN the prompt is rendered for an AI session with a customer entity context (`entityId` + `entityType: 'customer'`) THEN the variable is replaced with the actual customer name from the database, scoped by `companyId`

2. **AC-2: DB_QUERY resolution** — GIVEN a variable with source `DB_QUERY` and query `SELECT COUNT(*) FROM customer_invoices WHERE status = 'OVERDUE' AND company_id = :companyId` WHEN resolved THEN the count is injected into the prompt. Query results exceeding 2000 characters are truncated with `... [truncated]`

3. **AC-3: PAGE_FIELD resolution** — GIVEN a variable with source `PAGE_FIELD` and field `currentPage.selectedFilters` WHEN resolved in a chat session THEN the current page's filter state is injected from the session's page context (enables "what am I looking at?" queries)

4. **AC-4: SYSTEM variables** — GIVEN a variable with source `SYSTEM` WHEN resolved THEN ALL system variables are available: `{{today}}` (ISO date), `{{currentUser.name}}`, `{{currentUser.role}}`, `{{currentUser.id}}`, `{{company.name}}`, `{{company.baseCurrency}}`, `{{company.defaultCurrency}}`, `{{company.id}}`

5. **AC-5: EXPRESSION evaluation** — GIVEN a variable with source `EXPRESSION` and value `{{today - 30 days}}` WHEN resolved THEN the expression is evaluated and the result injected. Supports: date arithmetic (days/weeks/months), basic math (+, -, *, /), and string operations (concat, uppercase, lowercase, trim)

6. **AC-6: Variable registry API** — GIVEN the variable registry endpoints WHEN called by an authenticated admin THEN variables can be created, listed (grouped by sourceType), updated, deleted, and test-resolved. `GET /ai/variables` returns all variables grouped by source type with descriptions for autocomplete

7. **AC-7: Graceful fallback** — GIVEN a prompt with unresolvable variables WHEN rendered in INTERACTIVE mode THEN the system logs a warning, replaces with `defaultValue` (if set) or safe fallback `[unknown: varName]`, and does NOT crash the AI session

8. **AC-8: Prompt rendering service** — GIVEN a prompt ID and execution context (companyId, userId, entityId, pageContext) WHEN the prompt rendering service is called THEN it loads the prompt's active version template, resolves all bound variables, and returns the fully rendered prompt text

9. **AC-9: DB_FIELD model allowlist** — GIVEN a DB_FIELD variable WHEN the handler resolves it THEN only models from an explicit allowlist are queryable (Customer, Supplier, Item, Invoice, SalesOrder, PurchaseOrder, etc.). Attempts to access non-allowlisted models (User, AiModel, platform tables) are rejected with a logged warning

10. **AC-10: Test resolution endpoint** — GIVEN the `POST /ai/variables/:id/test` endpoint WHEN called with sample context data THEN the variable is resolved against the provided context and the result returned (for admin testing/preview without needing a full AI session)

### Key Design Decisions

1. **Extend, don't rewrite**: The `VariableResolver` from E5c-1 is solid. This story adds a new handler (PAGE_FIELD), fixes gaps (SystemHandler, model allowlist), and wraps it in a higher-level `PromptRenderer` service. No breaking changes to existing interface.

2. **PromptRenderer is the public API**: Other services (orchestrator, automation executor) should use `PromptRenderer.render()`, not `VariableResolver` directly. `PromptRenderer` handles loading the prompt, finding variables, and calling the resolver — a cohesive unit.

3. **PAGE_FIELD is chat-only**: The `PageFieldHandler` resolves from the chat session's frontend state. In automation/autonomous mode, `pageContext` will be undefined and PAGE_FIELD variables will fall back to defaults. This is by design — automations don't have a "current page".

4. **Model allowlist for DB_FIELD**: This addresses code review Issue #7 from E5c-1. The allowlist prevents arbitrary model access via crafted `sourceConfig.table` values. New models can be added to the allowlist as modules are built.

5. **DB_QUERY result truncation**: Long query results (e.g., listing all overdue invoices) could blow up the token budget. Truncation at 2000 chars protects against this. The truncation happens at the resolver level, before the value enters the prompt template.

6. **No new Prisma migration**: The `AiPromptVariable` model was created in E5c-1. This story extends the service and API layer without schema changes.
