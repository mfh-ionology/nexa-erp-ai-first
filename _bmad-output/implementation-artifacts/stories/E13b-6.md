# Story E13b.6: Platform Audit Log Viewer

Status: done

## Story

As a **platform administrator**,
I want to search and filter the immutable platform audit log by action, target, user, and date range, with detail view and CSV export, so that I can investigate incidents and demonstrate compliance.

## Acceptance Criteria

1. GIVEN the Audit Log page WHEN it loads THEN a T1 Entity List shows audit records with columns: timestamp, admin user, action, target type, target name, IP address
2. GIVEN the audit log WHEN the admin filters by action type (e.g., "tenant.suspend") THEN only matching records are shown
3. GIVEN the audit log WHEN the admin filters by date range THEN only records within that range are shown
4. GIVEN an audit log entry WHEN clicked THEN a detail view shows the full action details JSON, before/after state (if applicable), user agent, and IP address
5. GIVEN the audit log WHEN the admin clicks "Export CSV" THEN a CSV file downloads with the filtered records

### Data Model Reference

```prisma
model PlatformAuditLog {
  id             String   @id @default(uuid())
  platformUserId String   @map("platform_user_id")
  action         String   @map("action") @db.VarChar(100)
  targetType     String?  @map("target_type") @db.VarChar(50)
  targetId       String?  @map("target_id")
  details        Json?    @map("details") @db.JsonB
  ipAddress      String   @map("ip_address") @db.VarChar(45)
  userAgent      String?  @map("user_agent") @db.VarChar(500)
  timestamp      DateTime @default(now()) @map("timestamp")
  createdAt      DateTime @default(now()) @map("created_at")

  platformUser   PlatformUser @relation(fields: [platformUserId], references: [id])

  @@index([platformUserId, timestamp])
  @@index([targetType, targetId])
  @@map("platform_audit_log")
}
```

### Remaining Issues for Human Review:

- - **ISSUE #1: [HIGH]** Cursor-based pagination lacks a secondary sort key. `audit-log.routes.ts:161` orders by `{ timestamp: 'desc' }` but uses `id` as cursor. When multiple audit entries share the same timestamp (e.g., burst of near-simultaneous actions), the ordering among them is non-deterministic. Between page fetches, records within the same timestamp group can shift position, causing entries to be duplicated or skipped at page boundaries. Fix: add `orderBy: [{ timestamp: 'desc' }, { id: 'desc' }]` to guarantee stable ordering.
- - **ISSUE #2: [MEDIUM]** CSV export is faux-streaming. `audit-log.routes.ts:204` loads all 10,000 rows into the `entries` array via `findMany`, then iterates them into a `PassThrough` stream. The entire result set sits in memory before any bytes are written. For 10k rows with JSONB `details` payloads, this can be a significant memory spike. A true streaming approach would use chunked cursor-based fetches or raw SQL cursors.
- - **ISSUE #3: [MEDIUM]** Unsafe type double-casting throughout route handlers. `serializeListItem` (line 64), `serializeDetail` (line 85), and `formatCsvRow` (line 112) all accept `Record<string, unknown>` and immediately cast to a hardcoded shape with `as`. Prisma's `findMany`/`findUnique` already return fully-typed results with the `include` clause. The `as Record<string, unknown>` at the call sites (`line 178`, `line 235`, `line 273`) discards that static typing. If the Prisma schema or `include` clause changes, these casts will silently produce runtime errors.
