# Story E6-10: Field Override Configuration

**Epic:** E6 — Web Frontend Shell + Mobile Scaffold
**Status:** done
**Dependencies:** E6.1 (React App Bootstrap), E6.8 (Access Group Management Pages — detail page with tabs), E2b.5 (Field override backend API — `PUT /system/access-groups/:id/field-overrides`)
**Complexity:** Medium (7 tasks)

---

## User Story

As an administrator, I want to configure field-level visibility (VISIBLE, READ_ONLY, HIDDEN) per access group per resource, so that sensitive fields like cost prices can be hidden from specific user groups.

---

## Acceptance Criteria

### AC1: Field Overrides Tab Integration
**GIVEN** an authenticated administrator navigates to `/system/access-groups/:id`
**WHEN** they click the "Field Overrides" tab
**THEN** the existing placeholder is replaced with the field overrides configuration panel
**AND** it displays the resource selector dropdown and any existing field overrides for this access group

### AC2: Resource Selector Dropdown
**GIVEN** the Field Overrides panel is active
**WHEN** the administrator clicks the resource selector dropdown
**THEN** a searchable combobox lists all active resources from `GET /system/resources`
**AND** resources are grouped by module with module headers as group labels
**AND** the administrator can search/filter resources by name or code
**AND** selecting a resource shows its field overrides in the table below

### AC3: Resource Selector Empty State
**GIVEN** no resource has been selected yet
**WHEN** the Field Overrides panel first renders
**THEN** an empty state message is displayed: "Select a resource to configure field visibility overrides"
**AND** no field override table is shown
**AND** the Save button is hidden or disabled

### AC4: Per-Field Visibility Table
**GIVEN** a resource has been selected
**WHEN** the resource has existing field overrides for this access group
**THEN** a table displays with columns: Field Path, Visibility, Actions (remove)
**AND** each override for the selected resource is shown as a row
**AND** the Visibility column contains a `<Select>` dropdown with options: Visible, Read Only, Hidden
**AND** changing a visibility dropdown updates the local state immediately (not persisted until Save)

### AC5: Add New Field Override Row
**GIVEN** a resource is selected
**WHEN** the administrator clicks the "Add Field Override" button
**THEN** a new row appears with an empty field path input and visibility defaulting to VISIBLE
**AND** the field path input is auto-focused on the new row
**AND** entering a field path that already exists for this resource shows an inline validation error: "This field already has an override for this resource"

### AC6: Remove Field Override Row
**GIVEN** an existing field override row in the table
**WHEN** the administrator clicks the remove (x) button on that row
**THEN** the row is removed from the local state immediately
**AND** the change is not persisted until the Save button is clicked
**AND** the remove button has `aria-label="Remove override for {fieldPath}"`

### AC7: Save Field Overrides (Replace-All)
**GIVEN** the administrator has made changes to field overrides (added, modified, or removed)
**WHEN** they click the "Save Field Overrides" button
**THEN** the system calls `PUT /system/access-groups/:id/field-overrides` with ALL overrides for ALL resources (not just the currently selected resource)
**AND** a success toast confirms "Field overrides saved successfully"
**AND** the access group detail query cache is invalidated and refetched
**AND** the Save button returns to a disabled state (no longer dirty)

### AC8: Persisted Overrides Reload Correctly
**GIVEN** field overrides were previously saved for this access group
**WHEN** the administrator navigates to the Field Overrides tab
**THEN** the existing overrides from the `GET /system/access-groups/:id` response are displayed correctly
**AND** selecting a resource that has saved overrides shows those overrides in the table
**AND** selecting a resource with no overrides shows an empty table with the "Add Field Override" button

### AC9: Validation Error Handling
**GIVEN** the save request returns 400 VALIDATION_ERROR (one or more invalid resourceCode values)
**WHEN** the administrator clicks Save
**THEN** an error toast displays the invalid resource codes from the server response
**AND** the local state is preserved (no data loss)
**AND** the administrator can correct the issue and retry

### AC10: Dirty State Tracking
**GIVEN** the administrator has unsaved changes to field overrides
**WHEN** they switch between the Permissions tab and Field Overrides tab
**THEN** the unsaved field override changes are preserved (not lost)
**AND** the Save Field Overrides button is only enabled when there are unsaved changes
**AND** switching resources within the Field Overrides tab also preserves changes for previously edited resources

### AC11: Accessibility
**GIVEN** the field overrides panel
**WHEN** using keyboard navigation
**THEN** all interactive elements (resource selector, field path inputs, visibility dropdowns, add/remove buttons, save button) are keyboard-navigable with visible focus indicators (NFR28)
**AND** the resource selector combobox follows WAI-ARIA combobox pattern with `aria-expanded`, `aria-activedescendant`
**AND** visibility dropdowns follow WAI-ARIA listbox pattern
**AND** success/error toasts use `role="status"` for screen reader announcement
**AND** colour contrast meets WCAG 2.1 AA (NFR27)

## Dev Notes

1. **Replace-All Semantics**: The `PUT /system/access-groups/:id/field-overrides` endpoint uses replace-all semantics. The frontend must send ALL overrides across ALL resources, not just the ones for the currently selected resource. The local state manages the complete working set.

2. **Reuse Existing Hooks**: The resource selector MUST reuse `useResources()` from `features/admin/resources/api/use-resources.ts`. Do not create a duplicate query for fetching resources.

3. **Tab Preservation**: The Field Overrides panel MUST preserve its state when the user switches between the Permissions and Field Overrides tabs. Use `forceMount` on `<TabsContent>` or equivalent approach to prevent unmounting.

4. **No Field Registry**: The system does not have a predefined registry of fields per resource. Field paths are free-text strings (e.g., "costPrice", "purchasePrice"). The admin must know the field names. In future, a field registry could be added — but for MVP, free-text input is sufficient.

5. **Local ID for React Keys**: When adding/removing override rows, use stable local IDs (e.g., `crypto.randomUUID()`) rather than array indices to prevent React key collisions and stale state during rapid add/remove operations.

6. **Existing Placeholder**: The current `access-group-detail-page.tsx` (line 344-351) has a placeholder `<Card>` for the Field Overrides tab. This story replaces that placeholder entirely.

7. **Backend Already Complete**: The backend API for field overrides was built and tested in E2b.5 (55/55 tests passed). This story is frontend-only — no backend changes needed.

8. **Permission Cache**: The backend automatically invalidates the Redis permission cache when field overrides change. The frontend only needs to invalidate its own TanStack Query cache (`queryKeys.system.accessGroup(id)`).
