**Epic:** E5d — AI Knowledge Evolution & Cross-Tenant Intelligence
**Status:** done
**Dependencies:** E5d-1 (Knowledge Base Schema & RAG Pipeline — done), E5b-3 (Pattern Detection — done), E5 (AI Orchestration — done)

**FRs:** FR4 (contextual AI) | **NFRs:** NFR2 (CRUD <500ms)

## User Story

As a **system**,
I want to capture every user correction to AI responses and use them to improve future AI behaviour, so that the AI gets measurably smarter over time for each tenant — corrections feed back into knowledge articles and training examples, and per-skill learning signals expose where the AI underperforms.

## Acceptance Criteria

### AC1: Correction Capture Service
**GIVEN** a user corrects an AI response (edits a suggestion, provides explicit negative feedback, or overrides an AI action)
**WHEN** the correction is processed via the `CorrectionCaptureService`
**THEN** an immutable `AiCorrectionLog` record is created in the tenant DB with:
  - `companyId` — current company (multi-company scoped per project-context.md §1)
  - `userId` — the correcting user
  - `conversationId` — the chat session (if from chat)
  - `messageId` — the specific AI message corrected (if from chat)
  - `skillKey` — the skill used to generate the original response (if any)
  - `originalResponse` — the AI's original text
  - `correctedResponse` — what the user changed it to
  - `correctionType` — auto-categorised as one of: `TERMINOLOGY`, `PROCESS`, `DATA`, `PREFERENCE`, `OTHER`
  - `wasAutoResolved` — defaults to `false` (set to `true` when a knowledge article is later auto-generated)
**AND** the event `ai.correction.logged` is emitted per event catalog §17
**AND** the correction is immutable — no UPDATE or DELETE allowed on `AiCorrectionLog` (append-only)
**AND** the operation completes within 200ms (fire-and-forget side effect from chat, never blocking the user)
**AND** `companyId` scoping is enforced — corrections are only created for the authenticated user's active company

### AC2: Correction Auto-Categorisation
**GIVEN** a correction is captured
**WHEN** the `CorrectionCaptureService.categorise()` method runs
**THEN** it assigns a `correctionType` based on keyword heuristics:
  - `TERMINOLOGY` — correction involves renaming, abbreviation changes, company-specific terms (keywords: "called", "means", "we say", "abbreviation", "code")
  - `PROCESS` — correction involves workflow steps, approval chains, procedures (keywords: "step", "workflow", "approval", "process", "should do", "instead")
  - `DATA` — correction involves wrong data values, amounts, account codes (keywords: "amount", "code", "account", "number", "value", "rate")
  - `PREFERENCE` — correction involves format, sorting, default values (keywords: "prefer", "always", "default", "format", "sort", "order")
  - `OTHER` — no strong keyword match
**AND** categorisation is deterministic and fast (string matching, no LLM call)
**AND** the categorisation algorithm is exposed as a pure function for unit testing

### AC3: Pattern Detection — Auto-Generate Knowledge Articles
**GIVEN** N corrections on the same topic (threshold: 3, configurable)
**WHEN** the pattern detection logic runs (triggered after each correction via event handler)
**THEN** the system:
  1. Groups corrections by `(companyId, correctionType, topic)` where topic is derived from the first 100 characters of correctedResponse normalised to lowercase
  2. Counts corrections matching that topic within the last 90 days
  3. If count >= threshold, auto-generates a draft `AiKnowledgeArticle` with:
     - `source` = `CORRECTION_DERIVED`
     - `confidenceScore` = 0.5 (unconfirmed)
     - `isConfirmed` = false
     - `category` = maps from correctionType (TERMINOLOGY→TERMINOLOGY, PROCESS→BUSINESS_PROCESS, DATA→CUSTOM_FIELDS, PREFERENCE→BUSINESS_PROCESS, OTHER→BUSINESS_PROCESS)
     - `title` = "Auto-learned: {summary of correction topic}"
     - `content` = synthesised from the most recent corrections on that topic
  4. Marks all source corrections as `wasAutoResolved = true`
  5. Emits `ai.correction.autoArticleGenerated` event
**AND** the auto-generated article goes through the E5d-1 chunking and embedding pipeline
**AND** duplicate detection: if a similar correction-derived article already exists (same topic within the company), do NOT create a new one — instead increment the existing article's confidence by 0.1 (max 0.8)
**AND** the pattern detection is async — runs in an event handler, never blocking the correction capture

### AC4: Training Example CRUD
**GIVEN** REST API endpoints under `/ai/training-examples`
**WHEN** authenticated users call them
**THEN** the following endpoints are available:
  - `GET /ai/training-examples` — list examples for current company, filtered by `category`, `skillKey`, `isActive`, ordered by category + createdAt DESC (permission: `ai.chat` view)
  - `GET /ai/training-examples/:id` — get single example (permission: `ai.chat` view)
  - `POST /ai/training-examples` — create training example (permission: ADMIN role)
  - `PATCH /ai/training-examples/:id` — update example input/output/category (permission: ADMIN role)
  - `DELETE /ai/training-examples/:id` — soft-delete (set `isActive = false`) (permission: ADMIN role)
**AND** all endpoints enforce `companyId` scoping — only the current company's examples are accessible
**AND** cross-tenant access attempts return 404 (not 403, to avoid information leakage — matching E5d-1 pattern)
**AND** `source` is set on create and immutable thereafter
**AND** valid sources: `ADMIN_CURATED`, `CORRECTION_DERIVED`
**AND** valid categories: `BUSINESS_PROCESS`, `TERMINOLOGY`, `INDUSTRY_RULES`, `CUSTOM_FIELDS`, `HISTORICAL_PATTERN` (same as knowledge articles)

### AC5: Training Example Few-Shot Injection
**GIVEN** a user's AI query during a chat session
**WHEN** the dynamic context is assembled (step 2.5 in `DynamicContextService`)
**THEN** alongside the RAG knowledge injection, relevant training examples are retrieved:
  1. Query `ai_training_examples` for the current company where `isActive = true`
  2. If `skillKey` matches the current skill being used, prioritise those examples
  3. Also retrieve any category-matching examples (match the conversation topic)
  4. Select up to 3 training examples, sorted by relevance (skill match > category match)
  5. Inject as few-shot examples in the `<tenant_knowledge>` block:
     ```
     ## Training Examples
     - Q: "What VAT for EU purchase?" → A: "Use reverse charge — VAT code 3"
     - Q: "How do we handle returns?" → A: "Process via credit note, not void"
     ```
**AND** training examples share the ~1000 token knowledge budget with RAG chunks
**AND** training examples are injected AFTER RAG chunks (RAG chunks get priority for token budget)
**AND** if no training examples match, nothing extra is injected (graceful absence)

### AC6: Learning Signals Aggregation Service
**GIVEN** the learning signals aggregation job
**WHEN** it runs (daily, triggered by cron or scheduled event)
**THEN** it calculates per-skill, per-company metrics for the current day:
  - `totalQueries` — total AI interactions involving this skill
  - `successCount` — interactions with no subsequent correction
  - `correctionCount` — interactions followed by a correction (within 10 minutes)
  - `avgConfidence` — average confidence score from AI responses for this skill
**AND** results are upserted into `ai_learning_signals` using the unique constraint `(companyId, skillKey, signalDate)`
**AND** the aggregation processes one day at a time (the previous day's data)
**AND** if no data exists for a skill on a given day, no signal row is created (sparse storage)
**AND** the aggregation can be triggered manually via an admin API endpoint: `POST /ai/admin/learning-signals/aggregate`

### AC7: High Correction Rate Alert
**GIVEN** a skill with a correction rate >30% over the last 7 days
**WHEN** the learning signals aggregation detects this
**THEN** the system:
  1. Logs a warning: "Skill {skillKey} has {rate}% correction rate over last 7 days for company {companyId}"
  2. Emits `ai.learning.signalAggregated` event with the correction rate included
  3. (Future: E5d-5 will surface this in the Knowledge Management UI Settings tab)
**AND** the alert is per-company, per-skill — not a global threshold
**AND** skills with fewer than 10 total queries in the 7-day window are excluded (insufficient sample size)

### AC8: Correction Review API
**GIVEN** REST API endpoints for reviewing corrections
**WHEN** an admin accesses them
**THEN** the following endpoints are available:
  - `GET /ai/corrections` — list corrections for current company, grouped by `correctionType`, with pagination and filters (permission: ADMIN role)
    - Filters: `correctionType`, `skillKey`, `wasAutoResolved`, date range (`from`, `to`)
    - Response includes: correction records + aggregate stats (total count, count by type, count by skill)
  - `GET /ai/corrections/stats` — aggregated stats: total corrections, corrections by type, corrections by skill, auto-resolved count, correction trend (last 30 days) (permission: ADMIN role)
  - `POST /ai/corrections/:correctionId/create-article` — manually create a knowledge article from a specific correction (permission: ADMIN role)
    - Pre-fills article title and content from the correction's `correctedResponse`
    - Sets `source` = `CORRECTION_DERIVED`, `sourceRef` = correctionId
    - Returns the created article
**AND** all endpoints enforce `companyId` scoping
**AND** `GET /ai/corrections` supports cursor-based pagination (matching E5d-1 pattern)

### AC9: Admin Review of Auto-Generated Knowledge
**GIVEN** an auto-generated knowledge article (from AC3) with `source = CORRECTION_DERIVED` and `isConfirmed = false`
**WHEN** an admin reviews it via the knowledge article CRUD API (E5d-1)
**THEN** they can:
  - **Confirm**: `PATCH /ai/knowledge-articles/:id` with `{ isConfirmed: true }` → confidence score is upgraded to 0.8
  - **Edit and Confirm**: `PATCH /ai/knowledge-articles/:id` with updated content + `{ isConfirmed: true }` → re-chunked, re-embedded, confidence 0.8
  - **Reject**: `DELETE /ai/knowledge-articles/:id` → soft-deleted (set `isActive = false`)
**AND** no new endpoints are needed — this reuses E5d-1's existing knowledge article CRUD
**AND** the E5d-1 `PATCH` endpoint is updated to handle the `isConfirmed` field change: if transitioning from `false → true`, automatically update `confidenceScore` to 0.8
