## Epic Summary

Epic E11 delivered the cross-cutting task management system for Nexa ERP — enabling users to create, assign, and track tasks from any business record. The epic covers a backend CRUD service with polymorphic entity linking and status state machine, a full-featured frontend with "My Tasks" page, embedded task panels on record detail pages, and dashboard integration, plus notifications for task assignment, completion, and overdue detection via BullMQ cron job.

| E11.3 | Task Notifications — event-driven notifications, BullMQ overdue cron job, target resolver updates | done | 7 | 0 recorded |

**FRs Delivered:** FR181 (task creation from business records), FR182 (task assignment + notifications), FR183 (centralised task list)

## Successes

### 1. Clean Backend Execution (E11.1 + E11.3)
Both backend stories executed cleanly with zero recorded code review issues. E11.1 followed the established note/attachment cross-cutting pattern precisely, and E11.3 cleanly composed E9's notification infrastructure without reimplementing any delivery logic. This demonstrates mature backend architectural patterns.

### 2. Excellent Dependency Reuse
E11.3 is a showcase for compositional architecture:
- Reused E9's `createNotificationsFromEvent()` pipeline (no new notification delivery code)
- Reused E9's `NOTIFICATION_TRIGGERING_EVENTS` generic loop (just added 3 event names)
- Reused E9's target resolver with surgical additions (2 new strategies, 2 actor exclusions)
- Reused E9's BullMQ dispatch pattern for the overdue cron job
- Total new code for E11.3 notifications = minimal vs. the infrastructure it leverages

### 3. Compositional Story Architecture Applied
E10's retrospective recommended compositional story architecture (foundation -> management -> composition). E11 followed this:
- E11.1: Foundation (backend CRUD, events, state machine)
- E11.2: UI layer (consuming E11.1 APIs)
- E11.3: Integration layer (notifications consuming E11.1 events + E9 infrastructure)
Each story has clear inputs/outputs and minimal coupling.

### 4. Continued Quality Improvement Trend
Total issues dropped from 22 (E10) to 15 (E11) — a 32% reduction. HIGH issues dropped from 5 to 3. Issues per story: 5.0 (best ever, down from 7.3 in E10).

### 5. Rich v0-to-Production Workflow
E11.2 demonstrates the v0 design reference workflow at scale — 5 v0 reference files mapped to 18 production files with clear adaptation rules (Next.js -> TanStack Router, mock data -> React Query, hardcoded strings -> i18n). The v0 files served as effective design specifications.

### 6. Comprehensive Feature Coverage
The task feature is end-to-end complete:
- Backend: CRUD, status state machine, assignee management, polymorphic entity linking, event emission
- Frontend: My Tasks page (T1 template), Task Panel (embedded in record pages), Create Dialog, Detail Sheet, Dashboard "Tasks Today" card, sidebar navigation, batch actions
- Notifications: task.assigned, task.status_changed (COMPLETED only), task.overdue (daily BullMQ cron with deduplication)
- i18n: 40+ translation keys

## Challenges

### 1. HIGH Issues Still Not Blocking Story Completion (5th Consecutive Epic)
Despite E8, E5c, E9, and E10 retrospectives all establishing that HIGH code review issues must be fixed before marking stories "done" — and E10's explicit "TOOLING MANDATE" — E11 continued the pattern: 3 HIGH issues in E11.2, story marked "done." This is now the **5th consecutive retrospective** flagging this violation.

### 2. Task Count Guideline Still Violated (5th Consecutive Epic)
- E11.1: 8 tasks (exceeds <=5 guideline)
- E11.2: 10 tasks (exceeds <=5 guideline)
- E11.3: 7 tasks (exceeds <=5 guideline)

All 3 stories exceeded the <=5 task guideline. The tooling mandate from E10 was not implemented.

### 3. Frontend Stories Accumulate More Issues Than Backend
Across E11, all 15 code review issues are in E11.2 (frontend). E11.1 (backend) and E11.3 (backend + infrastructure) had zero issues. This pattern has been consistent across multiple epics — frontend stories are more complex and harder to get right, particularly around responsive design, state management, and visual fidelity.

### 4. Specific E11.2 HIGH Issues
- **Responsive breakpoint deviation**: Assignees column hidden at `lg` instead of spec-required `md`
- **`isOverdue()` logic inconsistent**: Different implementations between `task-helpers.ts` and `bottom-cards.tsx`
- **Utility duplication**: `bottom-cards.tsx` duplicates `isOverdue()` and `isDueTodayOrOverdue()` instead of importing from `task-helpers.ts`

### 5. Reassign Batch Action is a No-Op
E11.2 MEDIUM issue #10: the "Reassign..." batch action button renders but does nothing. This is a functional gap in the batch actions feature (AC #7).
