# Story E5b-4: Hybrid Search & Memory Intelligence

**Epic:** E5b — AI Co-Pilot Intelligence — Memory, Skills & Dynamic Context
**Status:** done
**Dependencies:** E5b-1 (Memory Storage & Retrieval API — done), E5b-3 (Implicit Memory Learning & Pre-Compaction Flush — done)
**Complexity:** X-Large (11 tasks)
**FRs:** FR4 (contextual AI) | **NFRs:** NFR2 (CRUD <500ms), NFR1 (AI responses <3s)

---

## User Story

As a **developer**,
I want the memory system to use hybrid search (keyword + semantic vector) with temporal decay and diversity re-ranking so that memory retrieval is accurate, contextually relevant, and recency-aware,
so that the AI Co-Pilot recalls the most useful memories for each conversation — not just the most recently stored or highest-importance ones.

## Acceptance Criteria

### AC1: pgvector Extension & Embedding Column Migration
**GIVEN** the migration runs
**WHEN** pgvector is needed
**THEN** the migration includes `CREATE EXTENSION IF NOT EXISTS vector` (one-time, idempotent)
**AND** the `ai_memories` table gains a `embedding vector(1536)` column for storing OpenAI-compatible embeddings
**AND** an HNSW index is created on the embedding column for approximate nearest-neighbour search: `CREATE INDEX idx_ai_memories_embedding ON ai_memories USING hnsw (embedding vector_cosine_ops)`
**AND** the migration is created with `prisma migrate dev --create-only` first, raw SQL appended, then applied (per Prisma Migration Rules)
**AND** the pgvector extension is shared infrastructure — reused by E5d (Knowledge RAG) and any future vector search

### AC2: Full-Text Search Column Migration
**GIVEN** the migration runs
**WHEN** BM25 keyword search is needed
**THEN** the `ai_memories` table gains a `search_vector tsvector` column populated from `content` via a trigger
**AND** a GIN index is created: `CREATE INDEX idx_ai_memories_search_vector ON ai_memories USING gin (search_vector)`
**AND** a PostgreSQL trigger is created to auto-update `search_vector` on INSERT or UPDATE of `content`:
```sql
CREATE OR REPLACE FUNCTION ai_memories_search_vector_update() RETURNS trigger AS $$
BEGIN
  NEW.search_vector := to_tsvector('english', NEW.content);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER ai_memories_search_vector_trigger
  BEFORE INSERT OR UPDATE OF content ON ai_memories
  FOR EACH ROW EXECUTE FUNCTION ai_memories_search_vector_update();
```
**AND** existing rows are backfilled: `UPDATE ai_memories SET search_vector = to_tsvector('english', content)`

### AC3: Embedding Generation on Memory Create/Update
**GIVEN** a memory is created via `MemoryService.createMemory()` or updated via `MemoryService.updateMemory()`
**WHEN** the content changes
**THEN** an embedding is generated via the EmbeddingService (which calls the AI Gateway's configured embedding model)
**AND** the embedding is stored in the `embedding` column of the `ai_memories` row
**AND** embedding generation is **fire-and-forget** — if it fails, the memory is still created/updated (the embedding column remains NULL) and an error is logged
**AND** existing memories without embeddings are backfilled via a one-time migration script or a background job (Task 5)
**AND** embedding dimension is 1536 (OpenAI text-embedding-3-small compatible; configurable via EmbeddingService)

### AC4: Hybrid Search — BM25 + Semantic Fusion
**GIVEN** a memory retrieval query (e.g., user sends a message and context needs to be assembled)
**WHEN** the VectorSearchService performs a hybrid search
**THEN** it executes **both**:
  1. **BM25 keyword search** via PostgreSQL full-text search (`ts_rank_cd` with `search_vector @@ plainto_tsquery('english', query)`)
  2. **pgvector cosine similarity search** (`1 - (embedding <=> query_embedding)` with similarity ≥ 0.5 threshold)
**AND** results are fused using **weighted Reciprocal Rank Fusion (RRF)**:
  - `RRF_score = w_keyword / (k + rank_keyword) + w_semantic / (k + rank_semantic)` where `k=60` (standard RRF constant)
  - Default weights: `w_keyword = 0.3`, `w_semantic = 0.7`
**AND** the fused results are returned ordered by RRF score DESC
**AND** if the query embedding cannot be generated (AI Gateway down), falls back to BM25-only search
**AND** if neither search produces results, returns empty array

### AC5: MMR Re-Ranking for Diversity
**GIVEN** a set of candidate memories from hybrid search
**WHEN** ranking for injection into the `<user_context>` block
**THEN** MMR (Maximal Marginal Relevance) re-ranking is applied:
  - `MMR(d) = λ * Sim(d, query) - (1 - λ) * max(Sim(d, selected))` where λ=0.7
  - 70% weight on relevance to the query, 30% penalty for similarity to already-selected memories
**AND** this prevents the top-N memories from all being about the same topic (e.g., 5 variations of "user prefers Net 30")
**AND** MMR uses cosine similarity between memory embeddings for the diversity term
**AND** if embeddings are unavailable for some memories, those memories are appended after MMR-ranked ones (graceful fallback)

### AC6: Temporal Decay in Retrieval
**GIVEN** a memory with `lastAccessedAt` = 30 days ago
**WHEN** temporal decay is applied during scoring
**THEN** its effective importance is halved: `effectiveScore = baseScore * sourceWeight * 0.5^(daysSinceAccess / halfLife)`
**AND** `halfLife` defaults to 30 days, configurable per-user in `ai_memory_settings` (existing column: `retentionDays` informs pruning, but a new `decayHalfLifeDays` column with default 30 is added)
**AND** temporal decay is applied **after** hybrid search scoring and **before** MMR re-ranking
**AND** the existing MemoryInjectionService temporal decay calculation is replaced by the VectorSearchService's unified scoring pipeline

### AC7: Enhanced Memory Pruning
**GIVEN** a memory that hasn't been accessed in 90+ days
**AND** its effective importance score (with temporal decay) is below threshold (< 0.1)
**WHEN** the pruning job runs
**THEN** the memory is **archived** (set a `metadata.archived: true` flag and clear the embedding to save storage) if the user's retention setting allows archiving
**OR** the memory is **deleted** if it exceeds the user's `retentionDays` setting
**AND** the pruning job now uses the VectorSearchService's `calculateEffectiveImportance()` for consistent scoring across the system

### AC8: SemanticDedupService Upgrade
**GIVEN** the SemanticDedupService from E5b-3 (currently using keyword-based Jaccard fallback)
**WHEN** E5b-4's VectorSearchService is available
**THEN** the `tryVectorSearch()` stub in `SemanticDedupService` is implemented to call `VectorSearchService.findSimilar()`:
  - Generate embedding for the new content via EmbeddingService
  - Search existing user memories with cosine similarity ≥ 0.85 threshold
  - Return the most similar existing memory as a `DedupResult`
**AND** the keyword-based fallback is preserved for cases where embedding generation fails
**AND** the transition is seamless — existing dedup behavior is unchanged when VectorSearchService is unavailable

### AC9: MemoryInjectionService Upgrade to Hybrid Retrieval
**GIVEN** the MemoryInjectionService `assembleUserContext()` method (currently fetches memories via Prisma `findMany` + importance sort)
**WHEN** VectorSearchService is available
**THEN** memory retrieval is replaced with hybrid search:
  1. Extract a query from the current conversation context (last 2-3 user messages → concatenated text)
  2. Call `VectorSearchService.hybridSearch(query, userId, companyId, ...)` to get RRF-fused results
  3. Apply temporal decay to RRF scores
  4. Apply MMR re-ranking for diversity
  5. Select top memories within the 2000-token budget
**AND** conflict resolution logic (EXPLICIT > IMPLICIT, newer > older) from E5b-3 is preserved and applied after MMR
**AND** when VectorSearchService is unavailable, falls back to the existing Prisma-based fetch (graceful degradation per IMP-006)

### AC10: Performance — Memory Retrieval < 100ms
**GIVEN** a user with up to 10,000 memories stored
**WHEN** the hybrid search is executed (BM25 + pgvector + RRF + MMR)
**THEN** the total retrieval time is < 100ms (p95)
**AND** this is validated via a performance benchmark test with synthetic data (10K memory rows with embeddings)
**AND** the HNSW index parameters are tuned (`m=16`, `ef_construction=64`) for this target

- [x] 3.1 Create `apps/api/src/ai/vector-search.service.ts` with:

- [x] 3.2 Implement semantic similarity search:
  - `similaritySearch(queryEmbedding: number[], table: string, filters: Record<string, unknown>, limit: number, minSimilarity?: number): Promise<SimilarityResult[]>`
  - Executes raw SQL: `SELECT id, content, 1 - (embedding <=> $1::vector) AS similarity FROM {table} WHERE {filters} AND embedding IS NOT NULL AND 1 - (embedding <=> $1::vector) >= {minSimilarity} ORDER BY embedding <=> $1::vector LIMIT {limit}`
  - Default `minSimilarity`: 0.5
  - Returns `SimilarityResult[]`: `{ id: string, content: string, similarity: number, metadata?: unknown }`
  - Uses parameterised queries to prevent SQL injection

- [x] 3.3 Implement BM25 keyword search:
  - `keywordSearch(query: string, table: string, filters: Record<string, unknown>, limit: number): Promise<KeywordResult[]>`
  - Executes raw SQL: `SELECT id, content, ts_rank_cd(search_vector, plainto_tsquery('english', $1)) AS rank FROM {table} WHERE {filters} AND search_vector @@ plainto_tsquery('english', $1) ORDER BY rank DESC LIMIT {limit}`
  - Returns `KeywordResult[]`: `{ id: string, content: string, rank: number }`

- [x] 3.4 Implement Reciprocal Rank Fusion (RRF):
  - `hybridSearch(query: string, queryEmbedding: number[], table: string, filters: Record<string, unknown>, opts?: HybridSearchOpts): Promise<HybridResult[]>`
  - `HybridSearchOpts`: `{ limit?: number, keywordWeight?: number, semanticWeight?: number, rrf_k?: number }`
  - Defaults: `limit=50`, `keywordWeight=0.3`, `semanticWeight=0.7`, `rrf_k=60`
  - Algorithm:
    1. Execute `keywordSearch()` and `similaritySearch()` in parallel
    2. Assign rank position to each result in each list (1-indexed)
    3. For each unique result ID: `score = w_kw / (k + rank_kw) + w_sem / (k + rank_sem)`
    4. If a result appears in only one list, its missing rank uses `limit + 1` as penalty
    5. Sort by fused score DESC, return top `limit`
  - Returns `HybridResult[]`: `{ id: string, content: string, rrf_score: number, keyword_rank?: number, semantic_rank?: number, metadata?: unknown }`

- [x] 3.5 Implement MMR re-ranking:
  - `mmrRerank(candidates: HybridResult[], queryEmbedding: number[], candidateEmbeddings: Map<string, number[]>, limit: number, lambda?: number): HybridResult[]`
  - Default `lambda`: 0.7 (70% relevance, 30% diversity)
  - Algorithm:
    1. Start with empty `selected` set
    2. For each remaining candidate: `MMR(d) = λ * cosine_sim(d.embedding, queryEmbedding) - (1 - λ) * max_over_selected(cosine_sim(d.embedding, s.embedding))`
    3. Select the candidate with the highest MMR score, add to `selected`, remove from candidates
    4. Repeat until `selected.length === limit` or no candidates remain
  - If a candidate has no embedding (NULL), it is appended after MMR-ranked results (graceful fallback)
  - Uses the `cosineSimilarity()` utility for in-memory vector math (no DB round-trip for re-ranking)

- [x] 3.6 Implement utility functions:
  - `cosineSimilarity(a: number[], b: number[]): number` — dot product / (norm_a * norm_b)
  - `calculateEffectiveImportance(baseScore: number, source: string, lastAccessedAt: Date, halfLifeDays: number): number` — unified temporal decay formula used across all services
  - `findSimilar(content: string, table: string, filters: Record<string, unknown>, threshold?: number): Promise<SimilarityResult | null>` — convenience method: generate embedding + search + return best match above threshold (used by SemanticDedupService)

    ```typescript
    private async tryVectorSearch(
      userId: string,
      companyId: string,
      newContent: string,
    ): Promise<DedupResult | null> {
      if (!this.vectorSearchService) return null;
      try {
        const result = await this.vectorSearchService.findSimilar(
          newContent,
          'ai_memories',
          { user_id: userId, company_id: companyId },
          VECTOR_SIMILARITY_THRESHOLD, // 0.85
        );
        if (!result) return null;
        // Fetch full memory record for merge
        const existing = await this.db.aiMemory.findUnique({ where: { id: result.id } });
        if (!existing) return null;
        return {
          isDuplicate: true,
          existingMemory: this.toMemoryRecord(existing),
          similarity: result.similarity,
        };
      } catch (err) {
        this.logger.warn({ err }, 'Vector search failed in dedup, falling back to keyword');
        return null; // Fallback to keyword
      }
    }
    ```

## Dev Notes

### Architecture Patterns

- **Shared Infrastructure:** VectorSearchService and EmbeddingService are designed as shared services — not memory-specific. E5d (Knowledge RAG) will reuse them for chunk embedding and retrieval. Design APIs generically (accept `table` parameter, not hardcoded to `ai_memories`)
- **Service Pattern:** Constructor DI with `PrismaClient`, `Logger`, and peer service dependencies. No Fastify dependency. Follow E5b-1/E5b-2/E5b-3 conventions
- **Graceful Degradation (IMP-006):** Every new service is optional. If pgvector is not installed, if the embedding model is not configured, or if any service fails to initialise, the system falls back to keyword-based search (E5b-3 behavior). Never throws — returns empty/null on failure
- **Fire-and-Forget Embeddings:** Embedding generation on memory create/update must NOT block the memory operation. The embedding column can be NULL — hybrid search handles this gracefully (BM25-only for un-embedded memories)
- **Raw SQL for pgvector:** Prisma ORM does not support pgvector types natively. All vector operations (`INSERT embedding`, `SELECT ... <=> ...`, `HNSW index`) use `db.$executeRaw` and `db.$queryRaw`. The Prisma schema uses `Unsupported("vector(1536)")` for type documentation only

### Key Implementation Decisions

1. **pgvector over custom ML:** We use PostgreSQL's pgvector extension rather than external vector databases (Pinecone, Weaviate) because: (a) it lives in the same DB instance — no network hop, (b) it supports our database-per-tenant architecture, (c) it's sufficient for our scale (≤10K memories per user), (d) it's shared by E5b and E5d

2. **RRF over linear combination:** Reciprocal Rank Fusion is score-distribution-agnostic — it works with ranks, not raw scores. This avoids normalization issues between BM25 ranks and cosine similarity scores

3. **MMR for diversity:** Without MMR, a user with 20 "Net 30" memories would get all of them in the top-N. MMR ensures the AI sees a diverse set of preferences, not just variations of the most common one

4. **Embedding dimension 1536:** Matches `text-embedding-3-small` (OpenAI). If we switch to a different provider, the dimension is read from the model's config, not hardcoded. A migration to change the column dimension would be needed

5. **search_vector trigger vs application-level:** We use a PostgreSQL trigger to maintain `search_vector` rather than updating it in application code. This ensures consistency even for raw SQL updates and batch operations

6. **Backfill is separate from migration:** The embedding backfill runs as a background job after the server starts, not in the migration. Migrations should be fast and deterministic. Embedding generation involves external API calls (AI Gateway) which can fail or be slow

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: All vector searches and BM25 queries must include `WHERE company_id = ?`. The VectorSearchService `filters` parameter enforces this
- **i18n**: Error messages use error codes (e.g., `EMBEDDING_GENERATION_FAILED`, `VECTOR_SEARCH_UNAVAILABLE`). The frontend handles translation
- **Audit**: Embedding backfill progress and vector search performance metrics logged via structured logging. No new events needed (existing `ai.memory.created` / `ai.memory.updated` events cover CRUD)
- **userId isolation**: Vector searches are always scoped to `user_id = ?`. Cross-user memory search is architecturally impossible
