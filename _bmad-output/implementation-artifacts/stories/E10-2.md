# Story E10-2: Email Template Management

**Epic:** E10 — Email Integration
**Status:** done
**Dependencies:** E10-1 (SMTP Outbound Service — done, provides EmailMessage model with `emailTemplateId` FK)
**Complexity:** Large (8 tasks)
**FRs:** FR189 (email templates with merge fields per document type) | **NFRs:** NFR41 (TypeScript strict mode), NFR2 (CRUD < 500ms p95)

---

## User Story

As an **administrator**,
I want to create and manage email templates with variable substitution and preview capability,
so that business emails have consistent, professional formatting across all document types.

---

## Acceptance Criteria

### AC1: Email Template CRUD
**GIVEN** an ADMIN user
**WHEN** they create an email template via `POST /email/templates`
**THEN** the template is created with:
  - `code` (unique machine identifier, e.g., `INVOICE_SEND`)
  - `name` (human-readable, max 200 chars)
  - `description` (optional text)
  - `documentType` (required, one of the 7 supported document types)
  - `subjectTemplate` (Handlebars syntax, max 500 chars)
  - `bodyHtmlTemplate` (HTML body with Handlebars merge fields)
  - `bodyTextTemplate` (optional plain text fallback)
  - `languageCode` (default `"en"`, max 5 chars)
  - `attachPdf` (default `true`)
  - `autoSend` (default `false`)
  - `isActive` (default `true`)
  - `openingTextCode` and `closingTextCode` (optional StandardText references)
  - `createdBy` and `updatedBy` audit fields
**AND** the template `code` is unique across all templates
**AND** `companyId` is NOT on EmailTemplate — templates are system-wide configuration entities
**AND** only ADMIN users can create, update, or delete templates

### AC2: Handlebars Template Variable Validation
**GIVEN** a template body containing Handlebars variables (e.g., `{{customer.name}}`, `{{invoice.number}}`)
**WHEN** the template is saved (create or update)
**THEN** the system:
  - Compiles the `subjectTemplate` and `bodyHtmlTemplate` using Handlebars
  - Validates that all referenced variables are in the known variable schema for the specified `documentType`
  - Rejects save with a descriptive error if unknown variables are used
  - Allows Handlebars helpers (`{{#if}}`, `{{#each}}`, `{{formatCurrency}}`, `{{formatDate}}`)

### AC3: Template Preview
**GIVEN** a template exists
**WHEN** preview is requested via `POST /email/templates/:id/preview`
**THEN** the system:
  - Loads the template's `subjectTemplate` and `bodyHtmlTemplate`
  - Renders them with sample data appropriate to the template's `documentType`
  - Returns the rendered `subject` and `bodyHtml` for display in the template editor
  - Sample data includes realistic values (e.g., invoice number "INV-00042", customer "Acme Ltd")

### AC4: System Default Template Fallback
**GIVEN** a document type (e.g., `CustomerInvoice`)
**WHEN** no custom template exists for that document type
**THEN** the system falls back to the seeded system default template (BR-COM-010)
**AND** the fallback hierarchy is:
  1. Custom template matching `documentType` + `languageCode`
  2. System default template matching `documentType` (any language)
  3. Error if no template exists at all

### AC5: Template Resolution Service
**GIVEN** the document-to-email pipeline needs a template (consumed by E10-3)
**WHEN** `resolveTemplate(documentType, languageCode?)` is called
**THEN** it returns the best-matching active template per the AC4 hierarchy
**AND** a compiled Handlebars render function is returned alongside the template metadata
**AND** the service caches compiled templates to avoid re-compilation on every email send

### AC6: Template CRUD Endpoints
**GIVEN** the REST API endpoints under `/email/templates`
**WHEN** authenticated ADMIN users call them
**THEN** the following endpoints are available:
  - `GET /email/templates` — list templates with filtering by `documentType`, `isActive`, `languageCode` (permission: ADMIN)
  - `GET /email/templates/:id` — get single template (permission: ADMIN)
  - `POST /email/templates` — create template (permission: ADMIN)
  - `PATCH /email/templates/:id` — update template (permission: ADMIN)
  - `DELETE /email/templates/:id` — soft-delete by setting `isActive = false` (permission: ADMIN)
  - `POST /email/templates/:id/preview` — render template with sample data (permission: ADMIN)
**AND** pagination uses cursor-based pagination matching existing patterns
**AND** all responses use `sendSuccess()` / `successEnvelope()` wrappers

### AC7: Seed Default Templates
**GIVEN** the system is freshly deployed
**WHEN** the seed script runs
**THEN** 7 system default templates are seeded:

| Code | Document Type | Subject Template |
|------|--------------|-----------------|
| `INVOICE_SEND` | `CustomerInvoice` | `Invoice {{invoiceNumber}} from {{companyName}}` |
| `STATEMENT_SEND` | `CustomerStatement` | `Account Statement from {{companyName}}` |
| `QUOTE_SEND` | `SalesQuote` | `Quotation {{quoteNumber}} from {{companyName}}` |
| `ORDER_CONFIRM` | `SalesOrder` | `Order Confirmation {{orderNumber}} from {{companyName}}` |
| `PO_SEND` | `PurchaseOrder` | `Purchase Order {{poNumber}} from {{companyName}}` |
| `CREDIT_NOTE_SEND` | `CreditNote` | `Credit Note {{creditNoteNumber}} from {{companyName}}` |
| `PAYSLIP_SEND` | `Payslip` | `Your Payslip for {{payPeriod}} — {{companyName}}` |

**AND** each default template has a professional HTML body layout with company branding placeholders
**AND** default templates have `isActive = true`

## Dev Notes

- **Path conventions**: New files go in `apps/api/src/modules/communications/email/` alongside existing E10-1 email files
- **No companyId on EmailTemplate**: Templates are system-wide configuration, not per-company. This matches the Architecture §2.29 schema which has no `companyId` on the model
- **emailTemplateId FK**: The `EmailMessage` model already has `emailTemplateId String?` but no relation yet because `EmailTemplate` doesn't exist. Task 1 adds the model and the relation
- **Handlebars dependency**: Install `handlebars` and `@types/handlebars` via pnpm. Handlebars is a well-established templating library with AST introspection for variable validation
- **Error pattern**: Use existing `AppError` from `core/errors/index.js`
- **Migration**: Use `prisma migrate dev` only, never `db push`. Set `PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION=yes`
- **Preview is read-only**: The preview endpoint doesn't modify any data — it compiles and renders on the fly with sample data
- **Cache invalidation**: When a template is updated, invalidate the compiled template cache entry so the next render uses the updated content
- **E10-3 integration point**: The `resolveTemplate()` method and `compileTemplate()` from the engine service will be consumed by E10-3's document-to-email pipeline
