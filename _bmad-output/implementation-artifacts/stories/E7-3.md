# Story E7.3: Filter Builder & Favourites

Status: done

## Story

As a **user**,
I want to build filters using Simple mode (field list with multi-select) or Advanced mode (AND/OR with group bracketing), use date presets, manage sort rules, and access my favourite views from the app header,
so that I can find specific records efficiently from any page.

## Acceptance Criteria

1. GIVEN the [Filter & Sort] button WHEN clicked THEN a wider modal (~640px) opens with two tabs: Filters and Sort
2. GIVEN the Filters Tab in Simple mode WHEN rendered THEN it shows a vertical list of all filterable fields (where `filterable=true`), each with a searchable multi-select dropdown populated from the field's LOV configuration
3. GIVEN a date field in Simple mode WHEN the user clicks it THEN a dropdown of date presets appears (from `date_range_presets` table: Today, This Week, Last 30 Days, YTD, Custom, etc.). Selecting "Custom" shows from/to date pickers
4. GIVEN the Filters Tab WHEN toggled to Advanced mode THEN the user can add condition rows with field/operator/value selection, AND/OR toggle between conditions, and group bracketing with `( )` for complex logic
5. GIVEN active filters WHEN applied THEN the list immediately refreshes showing only matching records, and the [Filter & Sort] button shows a badge with the active filter count
6. GIVEN the Sort Tab WHEN rendered THEN the user can add priority-numbered sort rules with field selection, direction toggle (ASC/DESC), and drag reorder
7. GIVEN the star icon in the app header WHEN clicked THEN a dropdown shows all favourite views from ALL data_views, grouped by `groupName` (e.g., "Invoices", "Sales", "CRM"). Clicking a favourite navigates to the entity list with the view applied
8. GIVEN a filter configuration WHEN included in a saved view (via E7.2) THEN the filters persist and re-apply when the view is loaded
9. GIVEN the LOV batch endpoint WHEN the filter modal opens THEN all VIEW_SPECIFIC LOVs are fetched in a single `POST /views/lov/batch` call. For LOVs with >50 items, search is server-side with debounce

  - [x] 1.1 Add LOV API functions to `apps/web/src/features/views/api.ts`:
    ```typescript
    // Batch LOV fetch for filter modal
    export async function batchFetchLov(
      items: Array<{
        fieldId: string;
        lovScope: string;
        search?: string;
        parentValue?: string;
        limit?: number;
      }>
    ): Promise<Record<string, Array<{ value: string; label: string }>>> {
      const res = await apiPost<{ results: Record<string, Array<{ value: string; label: string }>> }>(
        '/views/lov/batch',
        { items }
      );
      return res.results;
    }

    // Single LOV fetch with server-side search
    export async function fetchLov(
      lovScope: string,
      search?: string,
      limit?: number
    ): Promise<Array<{ value: string; label: string }>> {
      const params = new URLSearchParams();
      if (search) params.set('search', search);
      if (limit) params.set('limit', String(limit));
      return apiGet<Array<{ value: string; label: string }>>(
        `/views/lov/${lovScope}?${params.toString()}`
      );
    }
    ```

### Key architectural decisions
- **Filter state is transient** — editing filters in the modal does NOT persist until "Apply" is clicked. This allows cancel/discard of in-progress edits
- **LOVs are lazy-loaded** — batch LOV fetch only happens when the filter modal opens, not on page mount (performance)
- **Server-side search threshold** — fields with `lovSearchMin > 0` require that many characters before triggering server search (prevents fetching 10,000 customer names on modal open)
- **Simple vs Advanced mode** — Simple mode is the default for 90% of users. Advanced mode is opt-in. Both produce the same `SavedViewCondition[]` output
- **Favourites navigation** — requires a viewKey-to-route mapping. This should be configurable (e.g., a small registry or derived from the router config) rather than hardcoded

### Operator-to-field-type mapping
| Field Type | Available Operators |
|-----------|-------------------|
| STRING | EQUALS, NOT_EQUALS, CONTAINS, STARTS_WITH, ENDS_WITH, IN, NOT_IN, IS_EMPTY, IS_NOT_EMPTY |
| NUMBER, CURRENCY | EQUALS, NOT_EQUALS, GT, GTE, LT, LTE, BETWEEN, IN, IS_EMPTY, IS_NOT_EMPTY |
| DATE | EQUALS, NOT_EQUALS, GT, GTE, LT, LTE, BETWEEN, IS_EMPTY, IS_NOT_EMPTY |
| BOOLEAN | EQUALS |
| ENUM | EQUALS, NOT_EQUALS, IN, NOT_IN |

### Performance requirements
- Filter modal open → LOV batch fetch < 200ms (NFR2: CRUD < 500ms)
- Apply filters → list refresh immediate (optimistic UI, no loading state for filter application)
- Server-side LOV search debounce: 300ms
- Favourites dropdown: cached via TanStack Query, invalidated when any view's favourite status changes
