# Story E13b.2: Tenant Management Dashboard

Status: done

## Story

As a **platform administrator**,
I want to view all tenants in a list with status indicators, drill into tenant details, and perform lifecycle actions (activate, suspend, archive),
so that I can manage the tenant fleet.

## Acceptance Criteria

1. GIVEN the Tenants page WHEN it loads THEN a T1 Entity List shows all tenants with columns: name, code, plan, status (colour-coded badge), billing status, last activity, user count
2. GIVEN a tenant row WHEN clicked THEN a T2 Record Detail page shows tabbed detail: Overview, Modules & Flags, Users, AI Usage, Billing, Diagnostics, Audit
3. GIVEN an ACTIVE tenant WHEN the admin clicks "Suspend" THEN a confirmation dialog requires a reason, and on confirm the tenant is suspended and the ERP webhook fires within 30 seconds
4. GIVEN a SUSPENDED tenant WHEN the admin clicks "Reactivate" THEN the tenant returns to ACTIVE status and the ERP entitlement cache is busted
5. GIVEN the Modules & Flags tab WHEN the admin toggles a module override or feature flag THEN the change takes effect immediately via webhook

## Dev Notes

### Architecture Context

This story delivers the **Tenant Management Dashboard** — the primary admin interface for managing the tenant fleet. It replaces the placeholder `/tenants` page from E13b.1 with a full T1 Entity List and T2 Record Detail with 7 tabs.

**Backend is already implemented** in E3b (Platform API). This story is **frontend-only** — it builds React pages that consume existing `apps/platform-api` endpoints.

### Data Flow

```
Platform Admin UI  →  apiGet/apiPost/apiPatch  →  Platform API (port 5101)
     ↑                                                    ↓
React Query cache  ←  JSON response  ←  Prisma (Platform DB)
                                                          ↓
                                              WebhookService → ERP tenant
                                                          ↓
                                              Entitlement cache bust
```

### Prisma Models (Platform DB — for type reference)

```prisma
model Tenant {
  id              String    @id @default(uuid())
  code            String    @unique @db.VarChar(50)
  displayName     String
  legalName       String?
  status          TenantStatus @default(PROVISIONING)
  planId          String
  billingStatus   BillingStatus @default(CURRENT)
  region          String    @default("uk-south") @db.VarChar(30)
  dbHost          String
  dbName          String
  dbPort          Int       @default(5432)
  sandboxEnabled  Boolean   @default(false)
  lastActivityAt  DateTime?
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt
  plan            Plan      @relation(...)
  moduleOverrides TenantModuleOverride[]
  featureFlags    TenantFeatureFlag[]
  billing         TenantBilling?
}

enum TenantStatus { PROVISIONING, ACTIVE, SUSPENDED, READ_ONLY, ARCHIVED }
enum BillingStatus { CURRENT, GRACE, OVERDUE, BLOCKED }
```

### State Machine (Tenant Lifecycle — BR-PLT-001)

```
PROVISIONING → ACTIVE       (system: provisioning complete)
ACTIVE → SUSPENDED          (admin: manual suspend, or billing enforcement)
ACTIVE → READ_ONLY          (system: billing enforcement grace expired)
SUSPENDED → ACTIVE          (admin: manual reactivate)
SUSPENDED → ARCHIVED        (admin: manual archive — IRREVERSIBLE, BR-PLT-003)
READ_ONLY → ACTIVE          (system: billing settled)
```

Action bar buttons must respect this state machine:
- ACTIVE: show Suspend
- SUSPENDED: show Reactivate, Archive (in overflow)
- READ_ONLY: informational only (billing enforcement — system manages)
- ARCHIVED: no actions (terminal state)
- PROVISIONING: no actions (system manages)

- Sidebar: slate-900 background, purple accent for active item

- Cards: standard shadcn/ui cards with slate borders
- Buttons: purple primary (#7c3aed), red destructive (#dc2626)
- NOT Concept D's `#f4f2ff` background or custom shadows — platform admin has its own simpler, utilitarian visual style
