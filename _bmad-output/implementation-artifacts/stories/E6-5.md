# Story E6.5: Co-Pilot Dock

Status: done

## Story

As a **user**,
I want a Cmd+K header input for quick search/AI commands and a collapsible 380px right-side drawer for multi-turn AI conversations, so that I can interact with the AI assistant from any screen.

## Acceptance Criteria

1. GIVEN any screen WHEN the user presses Cmd+K (Mac) or Ctrl+K (Windows) THEN the unified search input in the header bar gains focus and opens a command palette dropdown
2. GIVEN the user types in the command palette WHEN the input matches entity patterns (INV-, PO-, SO-, etc.) THEN an autocomplete dropdown shows matching entities for direct navigation (top section), matching pages (middle section), and AI prompt suggestions (bottom section)
3. GIVEN the user types a natural language command WHEN they press Enter THEN the Co-Pilot drawer opens (if closed) and the AI response begins streaming in the drawer conversation area
4. GIVEN the Co-Pilot drawer WHEN opened on desktop (>=1024px) THEN it is 380px wide, slides in from the right with 200ms ease-out animation, the main content area resizes to accommodate it, and it shows: chat history selector, conversation area, quick prompts, and input area
5. GIVEN the Co-Pilot drawer WHEN on phone (<768px) THEN it renders as a full-screen overlay with a close button, and when minimised it shrinks to a floating pill at the bottom-right showing the last AI message preview
6. GIVEN the chat button in the header WHEN clicked THEN it toggles the Co-Pilot drawer open/closed, the button is no longer disabled, and the tooltip says "Open Co-Pilot" / "Close Co-Pilot" based on state
7. GIVEN the conversation area WHEN an AI response streams THEN text appears incrementally with a typing indicator (pulsing dots), AI messages are left-aligned with grey background, user messages are right-aligned with purple background
8. GIVEN an AI message WHEN it contains inline action buttons or record links THEN the buttons are clickable and record links navigate to the record on click (drawer stays open)
9. GIVEN the chat history selector WHEN the dropdown opens THEN it lists previous conversations with auto-generated titles (most recent first) and a "+ New Chat" button to start a fresh conversation
10. GIVEN the user is on a specific page WHEN quick prompts render THEN they are role-based and context-aware (e.g., on Invoice List: "Show Overdue", "Create Invoice"; on Dashboard: "Daily Briefing", "Revenue Summary")
11. GIVEN the Co-Pilot input area WHEN the user types and presses Enter THEN the message is sent; Shift+Enter inserts a new line; a file drop zone overlay appears when files are dragged onto the input
12. GIVEN all Co-Pilot UI content WHEN inspected THEN every user-facing string uses `t()` from `@nexa/i18n` вҖ” no hardcoded English text
13. GIVEN the Co-Pilot drawer WHEN navigated via keyboard THEN all interactive elements are focusable, Tab order follows visual reading order, and Escape closes the drawer
14. GIVEN any AI action proposal WHEN displayed in the conversation THEN a confirmation step is shown before any create/modify/delete operation executes (per BR-COM-013)

## Dev Notes

### Architecture Summary

The Co-Pilot Dock is the primary AI interaction surface in Nexa ERP. It consists of two connected elements: (1) a Cmd+K unified search/command palette in the header bar, and (2) a collapsible right-side drawer for multi-turn AI conversations. Together they implement the "Dual Interface Pattern" from Architecture В§5.4 вҖ” every operation is accessible via both AI natural language and traditional UI.

This story builds the **frontend shell** for all AI interactions. The actual AI WebSocket integration, response streaming from the backend, and action execution are NOT wired in this story вҖ” those depend on the AI backend (E5) being fully integrated. Instead, this story uses placeholder responses and stubs to validate the UI components independently.

### WebSocket Contract (for future integration вҖ” NOT wired in this story)

The AI chat uses WebSocket at `WS /ai/chat` (API Contracts В§2.6). Types from the contract:

**Client вҶ’ Server:**
```typescript
interface AiChatClientMessage {
  type: 'message' | 'action_confirm' | 'action_reject';
  sessionId: string;
  content?: string;
  actionId?: string;
}
```

**Server вҶ’ Client:**
```typescript
interface AiChatServerMessage {
  type: 'text' | 'action_proposal' | 'record_created' | 'error' | 'stream_chunk' | 'stream_end';
  sessionId: string;
  messageId: string;
  content?: string;
  action?: { id, type, description, entityType, previewData, confidence };
  record?: { entityType, entityId, displayRef };
  error?: { code, message };
}
```

The `ChatMessage` type in the copilot store is designed to align with these server messages. When the WebSocket integration is built, the mapping will be:
- `stream_chunk` вҶ’ `updateStreamingMessage()`
- `stream_end` вҶ’ `completeStreamingMessage()`
- `action_proposal` вҶ’ message with `actionProposal` field
- `record_created` вҶ’ message with `recordLinks` field

### Co-Pilot Behaviour Rules (from UX Design Spec)

1. **Context follows the user** вҖ” Co-Pilot maintains conversation but gains awareness of new page context
2. **Actions execute in main content area** вҖ” NOT inside the drawer. Drawer shows summary + link
3. **Drawer doesn't block workflows** вҖ” User can work in main content while drawer is open
4. **Smart collapse** вҖ” Phone uses full-screen overlay; minimise to floating pill
5. **Cross-session persistence** вҖ” Chat history persists across sessions (MVP: in Zustand; future: server-side)

### Business Rule: AI Actions Require Confirmation (BR-COM-013)

Per BR-COM-013: "AI chatbot actions require user confirmation. AI Engine must surface proposed action before executing create/modify/delete operations."

This is enforced in the `CopilotChat` component via the `ActionProposal` card. The AI never auto-executes; it proposes an action, the user reviews the preview data and confidence score, then explicitly confirms or rejects.
