# Story E2b.5: Admin can control field-level visibility per access group

Status: done

## Story

As an **administrator**,
I want the system to enforce field-level visibility rules from access group overrides on every API response,
So that sensitive fields (e.g., cost prices, salary data) are hidden or read-only based on the user's access groups.

## Acceptance Criteria

1. **AC1 — HIDDEN fields stripped:** Given an access group with a field override setting `fieldPath="costPrice"` to `HIDDEN` on resource `sales.orders.detail`, when a user in that group requests a sales order detail, then the `costPrice` field is completely absent from the `data` object in the response.

2. **AC2 — READ_ONLY fields annotated:** Given an access group with a field override setting `fieldPath="totalExVat"` to `READ_ONLY` on a resource, when a user in that group requests that resource, then the `totalExVat` field is present in `data` AND a `_fieldMeta` property is added to the response containing `{ "totalExVat": "readOnly" }`.

3. **AC3 — Most-permissive-wins for field visibility:** Given a user assigned to Group A (costPrice=HIDDEN) and Group B (costPrice=READ_ONLY), when the system resolves field visibility, then the effective visibility is READ_ONLY (VISIBLE > READ_ONLY > HIDDEN).

4. **AC4 — Default VISIBLE:** Given no `AccessGroupFieldOverride` row exists for a field, when the response hook runs, then that field is treated as VISIBLE — present in `data` and absent from `_fieldMeta`.

5. **AC5 — SUPER_ADMIN bypass:** Given a SUPER_ADMIN user, when they receive any API response, then the `filterFieldsByPermission` hook is skipped entirely — no fields stripped, no `_fieldMeta` added.

6. **AC6 — List response filtering:** Given a list endpoint returning an array of items, when field overrides apply, then HIDDEN fields are stripped from every item in the array, and `_fieldMeta` is added once at the envelope level.

7. **AC7 — Hook registration on routes:** Given a route with `onSend: filterFieldsByPermission('resource.code')`, when the response is sent, then the hook intercepts the JSON payload, applies field visibility rules, and returns the modified payload.

8. **AC8 — Cached field overrides:** Given the `PermissionService` has already resolved and cached the user's effective permissions (including `fieldOverrides`), when the `filterFieldsByPermission` hook runs, then it reads field overrides from `request.permissions.fieldOverrides[resourceCode]` without additional DB queries.

## Dev Notes

### Architecture Overview

This story implements the **field-level visibility response hook** — the final layer of the granular RBAC system. It sits as an `onSend` hook after route handlers, inspecting and modifying response payloads based on `AccessGroupFieldOverride` settings.

**Request lifecycle after E2b-5:**
```
Request → JWT verify (E2) → Company context (E2) → Permission guard (E2b-4, attaches request.permissions) → Route handler → filterFieldsByPermission onSend hook (E2b-5) → Client
```

### Key Implementation: Fastify `onSend` Hook

The `onSend` hook receives the serialized payload as a string. The hook must:
1. Parse the JSON string
2. Modify the parsed object (strip/annotate fields)
3. Return the re-serialized JSON string

**Fastify onSend signature:**
```typescript
type OnSendHook = (
  request: FastifyRequest,
  reply: FastifyReply,
  payload: string | Buffer | null,
) => Promise<string | Buffer | null> | string | Buffer | null;
```

**CRITICAL:** The `onSend` hook runs AFTER the serializer compiler. At this point, the payload is already a JSON string (or Buffer). Parse with `JSON.parse()`, modify, and return with `JSON.stringify()`.

### `filterFieldsByPermission` Implementation Pattern

```typescript
// apps/api/src/core/rbac/field-filter.hook.ts
import type { FastifyRequest, FastifyReply } from 'fastify';
import { UserRole } from '@nexa/db';
import type { FieldOverrides } from './permission.types.js';

export function filterFieldsByPermission(resourceCode: string) {
  return async (
    request: FastifyRequest,
    _reply: FastifyReply,
    payload: string | Buffer | null,
  ): Promise<string | Buffer | null> => {
    // SUPER_ADMIN bypass (BR-RBAC-002)
    if (request.userRole === UserRole.SUPER_ADMIN) {
      return payload;
    }

    // No permissions resolved (e.g., unauthenticated routes) — pass through
    const overrides = request.permissions?.fieldOverrides?.[resourceCode];
    if (!overrides || Object.keys(overrides).length === 0) {
      return payload;
    }

    // Only process string payloads (JSON)
    if (typeof payload !== 'string') {
      return payload;
    }

    let body: Record<string, unknown>;
    try {
      body = JSON.parse(payload) as Record<string, unknown>;
    } catch {
      return payload; // Non-JSON, pass through
    }

    // Only filter success responses with data
    if (body.success !== true || !body.data) {
      return payload;
    }

    const fieldMeta: Record<string, string> = {};

    // Apply field overrides
    if (Array.isArray(body.data)) {
      // List response — filter each item
      for (const item of body.data) {
        applyFieldOverrides(item as Record<string, unknown>, overrides, fieldMeta);
      }
    } else {
      // Single object response
      applyFieldOverrides(body.data as Record<string, unknown>, overrides, fieldMeta);
    }

    // Add _fieldMeta only if there are READ_ONLY annotations
    if (Object.keys(fieldMeta).length > 0) {
      body._fieldMeta = fieldMeta;
    }

    return JSON.stringify(body);
  };
}

function applyFieldOverrides(
  data: Record<string, unknown>,
  overrides: FieldOverrides,
  fieldMeta: Record<string, string>,
): void {
  for (const [fieldPath, visibility] of Object.entries(overrides)) {
    if (visibility === 'HIDDEN') {
      delete data[fieldPath];
    } else if (visibility === 'READ_ONLY') {
      if (fieldPath in data) {
        fieldMeta[fieldPath] = 'readOnly';
      }
    }
    // VISIBLE — no action needed (default)
  }
}
```

### Permission Service Already Implemented (E2b-4)

The `PermissionService` already provides everything the hook needs:

- **`getEffectivePermissions()`** — Resolves and caches permissions + `fieldOverrides` for user+company
- **`fieldOverrides`** structure: `Record<string, FieldOverrides>` where key is `resourceCode`, value is `Record<string, FieldVisibility>`
- **Most-permissive-wins merge** already implemented in `permission.service.ts` (lines 149-163) using `VISIBILITY_LEVEL: { VISIBLE: 3, READ_ONLY: 2, HIDDEN: 1 }`
- **`getFieldVisibility()`** — Standalone method for per-resource field overrides (can be used as alternative to request.permissions)

**CRITICAL: The hook should read from `request.permissions.fieldOverrides[resourceCode]` which was already attached by `createPermissionGuard` in the preHandler (E2b-4). Do NOT call `getFieldVisibility()` from the hook — it would bypass the cache and make an additional DB query.**

### How `request.permissions` Gets Set

In `permission.guard.ts` (E2b-4), the guard attaches permissions to the request:
```typescript
const effective = await permissionService.getEffectivePermissions(
  prisma, request.userId, request.companyId, request.userRole, request.enabledModules,
);
request.permissions = effective;
```

The `effective.fieldOverrides` is already resolved with most-permissive-wins merge. The onSend hook just reads this.

### Fastify 5 onSend Hook Registration

Register the hook on individual routes, NOT globally:
```typescript
fastify.get('/company-profile', {
  preHandler: createPermissionGuard('system.company-profile.detail', 'view'),
  onSend: filterFieldsByPermission('system.company-profile.detail'),
  schema: { ... },
}, handler);
```

**Why per-route and not global?** Different routes serve different resources. The resourceCode must match the route's resource to look up the correct field overrides.

### AccessGroupFieldOverride Prisma Model

```prisma
model AccessGroupFieldOverride {
  id            String          @id @default(uuid())
  accessGroupId String          @map("access_group_id")
  resourceCode  String          @map("resource_code")
  fieldPath     String          @map("field_path") @db.VarChar(255)
  visibility    FieldVisibility @map("visibility")

  accessGroup AccessGroup @relation(fields: [accessGroupId], references: [id], onDelete: Cascade)
  resource    Resource    @relation(fields: [resourceCode], references: [code])

  @@unique([accessGroupId, resourceCode, fieldPath])
  @@map("access_group_field_overrides")
}
```
