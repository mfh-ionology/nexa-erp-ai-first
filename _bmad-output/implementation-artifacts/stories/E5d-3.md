# Story E5d-3: Cross-Tenant Intelligence Pipeline (Platform)

**Epic:** E5d — AI Knowledge Evolution & Cross-Tenant Intelligence
**Status:** done
**Dependencies:** E5d-2 (Correction Loop & Training Examples — done), E5d-1 (Knowledge Base Schema & RAG Pipeline — done), E3b (Platform API — done)
**Complexity:** X-Large (9 tasks)
**FRs:** FR6 (AI-powered analytics) | **NFRs:** NFR50 (data privacy/durability), NFR9 (cross-tenant isolation)

---

## User Story

As a **platform vendor**,
I want anonymised aggregate intelligence from all tenants so that I can detect feature gaps, identify common workflows, improve skill effectiveness, and optimise default configurations — all without ever exposing one tenant's data to another or violating GDPR.

---

## Acceptance Criteria

### AC1: Platform Intelligence Schema Migration
**GIVEN** the platform Prisma schema (`apps/platform-api/prisma/schema.prisma`)
**WHEN** `prisma migrate dev` runs
**THEN** 5 new tables are created in the platform database:
  - `tenant_ai_patterns` — anonymised daily usage patterns per tenant (query categories, skill usage, view patterns, automation usage)
  - `tenant_ai_corrections` — anonymised correction patterns aggregated across tenants (no PII, grouped by correctionType + skillKey)
  - `platform_knowledge_base` — vendor-curated knowledge articles for distribution to tenants (used by E5d-4)
  - `ai_skill_effectiveness` — cross-tenant skill performance metrics (success rate, correction rate, confidence, trend)
  - `platform_ai_insights` — vendor-generated insights from aggregate data (feature gaps, workflow opportunities, default candidates)
**AND** the `Tenant` model gains an `industry` field (`String?`, optional) for industry-based grouping
**AND** the `Tenant` model gains an inverse relation `aiPatterns TenantAiPattern[]`
**AND** unique constraints are created: `(tenantId, patternDate)` on patterns, `(skillKey, measureDate)` on skill effectiveness
**AND** indexes are created per the epic's proposed schema: `(patternDate)`, `(industry)` on patterns; `(patternDate, correctionType)`, `(skillKey)` on corrections; `(status, category)` on knowledge base; `(measureDate)` on effectiveness; `(insightType, status)`, `(severity, status)` on insights
**AND** no `companyId` column exists on any platform table (project-context.md §8b: "Platform database has no companyId")
**AND** the migration uses `prisma migrate dev --create-only` first, then applied (per Prisma Migration Rules)

### AC2: Tenant DB Read Connector
**GIVEN** the platform aggregation service needs to read from tenant databases
**WHEN** it processes a tenant's data
**THEN** it creates a temporary `pg` client connection to the tenant's database using:
  - `dbHost`, `dbName`, `dbPort` from the `Tenant` record in the platform DB
  - `TENANT_DB_SERVICE_USER` and `TENANT_DB_SERVICE_PASSWORD` from platform environment variables (shared read-only service account)
**AND** connections are opened per-tenant, used for read-only queries, then closed immediately (no persistent pool per tenant)
**AND** if a tenant DB connection fails (unreachable, auth failure), the tenant is skipped with a warning log and the aggregation continues with remaining tenants (graceful degradation per IMP-006)
**AND** a connection timeout of 5 seconds is enforced — unresponsive tenant DBs do not block the aggregation of other tenants

### AC3: Anonymisation Service — PII Stripping
**GIVEN** raw data read from a tenant's database (learning signals, correction logs, usage data)
**WHEN** the anonymisation service processes it
**THEN** the output contains ONLY:
  - Statistical counts (query counts, correction counts, skill usage counts)
  - Percentage values (success rates, correction rates)
  - Category labels (correctionType, skill keys, module names, knowledge categories)
  - Boolean patterns (which saved views exist, which automations are configured)
**AND** the output contains NONE of:
  - Company names, customer/supplier names, or any entity names
  - Invoice amounts, account balances, or any monetary values
  - User names, email addresses, or any user identifiers
  - Free-text from knowledge articles, correction responses, or chat messages
  - IP addresses, specific dates of individual transactions, or any PII
**AND** the anonymisation is implemented as a pure function with no side effects (fully unit-testable)
**AND** an automated PII detection test suite validates that anonymised output passes a keyword/pattern scan for PII indicators (names, emails, amounts, etc.)
**AND** `commonCorrection` in `TenantAiCorrection` contains only generic category-level summaries (e.g., "Tenants correct EU VAT code suggestions"), never verbatim correction text

### AC4: Daily Aggregation Job — Usage Patterns & Corrections
**GIVEN** a tenant with `shareAnonymisedPatterns` feature flag enabled (via `TenantFeatureFlag` with key `share_anonymised_ai_patterns`, default: enabled)
**WHEN** the daily aggregation job runs (triggered by admin endpoint or cron)
**THEN** anonymised patterns are extracted from the tenant's DB and stored in `tenant_ai_patterns`:
  - `queryCategories` — distribution of AI query types by module (e.g., `{ "ar": 45, "finance": 30 }`)
  - `skillUsage` — count of each skill's invocations (e.g., `{ "create_invoice": 12, "apply_filter": 89 }`)
  - `viewPatterns` — which saved view types exist (boolean flags, no view names)
  - `automationUsage` — count of automation runs by type (no automation names)
  - `industry` — copied from `Tenant.industry` (may be null)
  - `planTier` — derived from `Tenant.plan.code`
**AND** anonymised correction patterns are aggregated into `tenant_ai_corrections`:
  - Grouped by `(patternDate, correctionType, skillKey)`
  - `occurrenceCount` — total corrections of this type from this tenant
  - `tenantCount` — incremented atomically (how many tenants have this correction pattern)
  - `commonCorrection` — a generic summary, never verbatim text
**AND** existing patterns for the same `(tenantId, patternDate)` are upserted (idempotent re-runs)
**AND** the aggregation processes the previous day's data (T-1)
**AND** the aggregation can be triggered via admin endpoint: `POST /admin/intelligence/aggregate` (permission: PLATFORM_ADMIN)
**AND** the response includes: `{ processedTenants: number, skippedTenants: number, patternsCreated: number, correctionsCreated: number }`

### AC5: Skill Effectiveness Aggregation
**GIVEN** `ai_learning_signals` data from multiple tenants
**WHEN** the skill effectiveness aggregation runs (as part of the daily job)
**THEN** cross-tenant skill metrics are calculated and stored in `ai_skill_effectiveness`:
  - `skillKey` — the skill being measured
  - `measureDate` — the date of the measurement
  - `tenantCount` — how many tenants use this skill
  - `totalQueries` — sum of all queries across tenants for this skill
  - `avgSuccessRate` — weighted average success rate: `sum(successCount) / sum(totalQueries)`
  - `avgCorrectionRate` — weighted average: `sum(correctionCount) / sum(totalQueries)`
  - `avgConfidence` — weighted average confidence across tenants
  - `trend` — calculated by comparing current 7-day average to previous 7-day average: `IMPROVING` (>5% better), `DECLINING` (>5% worse), `STABLE` (within 5%)
**AND** existing records for the same `(skillKey, measureDate)` are upserted (idempotent)
**AND** skills with fewer than 3 tenants reporting are still recorded but flagged with a low `tenantCount`
**AND** data is sourced from the `ai_learning_signals` table in each tenant's DB (created by E5d-2's learning signals aggregation)

### AC6: Tenant Opt-Out — Zero Data When Disabled
**GIVEN** a tenant with `shareAnonymisedPatterns` feature flag set to `false` (via `TenantFeatureFlag` with key `share_anonymised_ai_patterns`)
**WHEN** the daily aggregation job runs
**THEN** that tenant's data is completely excluded — zero rows created in `tenant_ai_patterns`, zero contribution to `tenant_ai_corrections`, zero contribution to `ai_skill_effectiveness`
**AND** existing pattern rows for that tenant are NOT retroactively deleted (data already aggregated remains)
**AND** if the feature flag row does not exist for a tenant, the default behaviour is opt-IN (sharing enabled)
**AND** the aggregation logs: "Tenant {tenantCode} opted out of anonymised pattern sharing — skipping"
**AND** the opt-out is verified by an integration test: create patterns, disable flag, run aggregation, assert zero new rows

### AC7: Weekly Insights Generation
**GIVEN** accumulated data in `tenant_ai_patterns`, `tenant_ai_corrections`, and `ai_skill_effectiveness`
**WHEN** the weekly insights generation service runs (triggered by admin endpoint or cron)
**THEN** it generates `PlatformAiInsight` records of the following types:
  - **FEATURE_GAP** — high-failure queries with no matching skill, detected by: AI usage records with `featureKey` values that have no corresponding skill definition, and high correction rates (>40%) across multiple tenants (>=3). Severity based on tenant count: >=10 HIGH, >=5 MEDIUM, else LOW
  - **WORKFLOW_OPPORTUNITY** — repeated manual patterns detected across tenants, identified by: common automation configurations that >30% of tenants have created manually (e.g., same cron schedule + skill combination). Severity: >=50% of tenants HIGH, >=30% MEDIUM
  - **DEFAULT_CANDIDATE** — configurations that >60% of tenants create manually, identified by: common `viewPatterns` or `automationUsage` patterns across tenants. Severity: >=80% HIGH, >=60% MEDIUM
  - **SKILL_IMPROVEMENT** — skills with `avgCorrectionRate > 0.30` and `tenantCount >= 5` from `ai_skill_effectiveness`. Severity: correction rate >=50% HIGH, >=30% MEDIUM
**AND** each insight includes an `evidence` JSON field with supporting data (counts, percentages, tenant counts, skill keys)
**AND** insights are created with `status = 'NEW'` — they do not overwrite existing reviewed/actioned insights
**AND** duplicate detection: if an insight with the same `insightType` and similar `title` already exists and is not `DISMISSED`, do NOT create a new one — instead update the `evidence` field with fresh data
**AND** the service can be triggered via admin endpoint: `POST /admin/intelligence/generate-insights` (permission: PLATFORM_ADMIN)
**AND** the response includes: `{ insightsGenerated: number, byType: { featureGap: N, workflowOpportunity: N, defaultCandidate: N, skillImprovement: N } }`

### AC8: Platform Admin Pattern Query API
**GIVEN** REST API endpoints under `/admin/intelligence`
**WHEN** platform admins call them
**THEN** the following endpoints are available:
  - `GET /admin/intelligence/patterns` — list aggregated patterns with filters: `industry`, `planTier`, `dateFrom`, `dateTo`, `tenantId` + cursor pagination (permission: PLATFORM_ADMIN)
  - `GET /admin/intelligence/corrections` — list aggregated correction patterns with filters: `correctionType`, `skillKey`, `dateFrom`, `dateTo` + cursor pagination (permission: PLATFORM_ADMIN)
  - `GET /admin/intelligence/skill-effectiveness` — list skill effectiveness data with filters: `skillKey`, `dateFrom`, `dateTo`, `trend` + cursor pagination (permission: PLATFORM_ADMIN)
  - `GET /admin/intelligence/insights` — list insights with filters: `insightType`, `severity`, `status` + cursor pagination (permission: PLATFORM_ADMIN)
  - `PATCH /admin/intelligence/insights/:id` — update insight status (`REVIEWED`, `ACTIONED`, `DISMISSED`) with optional `reviewedById` (permission: PLATFORM_ADMIN)
  - `GET /admin/intelligence/summary` — dashboard summary: total tenants contributing, total patterns, total corrections, overall AI success rate, top 5 skills by usage, top 5 insights by severity (permission: PLATFORM_ADMIN)
  - `POST /admin/intelligence/aggregate` — trigger daily aggregation manually (permission: PLATFORM_ADMIN)
  - `POST /admin/intelligence/generate-insights` — trigger weekly insights generation manually (permission: PLATFORM_ADMIN)
**AND** all endpoints are protected by `requirePlatformRole(PlatformRole.PLATFORM_ADMIN)`
**AND** all admin write operations are audit-logged via the existing `PlatformAuditPlugin`
**AND** responses use `sendSuccess(reply, data)` with `successEnvelope()` wrapper (matching platform-api conventions)

### AC9: Seed Data & Default Feature Flags
**GIVEN** the platform migration is applied
**WHEN** the seed script runs (or a new tenant is provisioned)
**THEN** each active tenant receives a default `TenantFeatureFlag` row:
  - `featureKey` = `share_anonymised_ai_patterns`
  - `enabled` = `true` (opt-in by default)
  - `changedBy` = `system`
**AND** no default `tenant_ai_patterns` or `tenant_ai_corrections` rows are seeded (data comes from aggregation)
**AND** no default `platform_ai_insights` are seeded (insights are generated from real data)
**AND** the seed script verifies that all 5 new tables exist and are queryable

## Dev Notes

- **File placement**: New services go in `apps/platform-api/src/services/`, new routes in `apps/platform-api/src/routes/admin/`, tests in `apps/platform-api/src/__tests__/`
- **No companyId**: Platform DB tables use `tenantId` (referencing `tenants.id`), NOT `companyId`. This is a fundamental architectural difference from the tenant-side ERP DB
- **Auth pattern**: All new endpoints use `requirePlatformRole(PlatformRole.PLATFORM_ADMIN)`, NOT service token guard (these are admin-facing, not ERP-calling)
- **Audit logging**: Existing `PlatformAuditPlugin` auto-logs all POST/PATCH/DELETE requests — no manual audit calls needed
- **Response format**: Use `sendSuccess(reply, data)` from `core/utils/response.ts` with `successEnvelope()` wrapper
- **Error pattern**: Use `NotFoundError`, `DomainError` from `core/errors/app-error.ts`
- **PlatformKnowledgeArticle**: The schema is created in this story but CRUD endpoints are deferred to E5d-4 (Platform Knowledge Distribution). Only the table creation is in scope
- **Cron automation**: For MVP, aggregation and insights generation are triggered manually via admin endpoints. Automated cron scheduling (via node-cron or BullMQ) is a future enhancement — the services are designed to be called by any scheduler
- **`tenant_ai_corrections` is an aggregate table**: Unlike `tenant_ai_patterns` (which has a `tenantId` FK), corrections are cross-tenant aggregates. The `tenantCount` field tracks how many tenants exhibit a pattern, but individual tenant identity is not stored on correction rows. This is by design for privacy
- **Connection to tenant DBs**: Uses raw `pg` client, NOT Prisma. Prisma requires schema generation per database which is not feasible for dynamic multi-tenant connections. Raw SQL queries target the known E5d-1/E5d-2 table schemas
