# Story E3b.1: Platform API Server

Status: done

## Story

As a **platform developer**,
I want a separate Fastify instance for the Platform API with its own authentication system,
so that platform admin operations and ERP runtime entitlement checks are served independently from tenant ERP requests.

## Acceptance Criteria

1. GIVEN the Platform API server in `apps/platform-api` WHEN it starts THEN it connects to the Platform database (not any tenant ERP database) and serves on a separate port
2. GIVEN a PlatformUser with PLATFORM_ADMIN role and MFA enabled WHEN they POST `/admin/auth/login` with correct credentials and TOTP code THEN a platform-level JWT is issued with `platformUserId` and `platformRole` claims
3. GIVEN a PLATFORM_ADMIN account WHEN MFA is not enabled THEN login is blocked per BR-PLT-018 (mandatory MFA for PLATFORM_ADMIN)
4. GIVEN the Platform API WHEN `GET /admin/monitoring/health` is called THEN it returns platform health status including database connectivity, Redis availability, and uptime
5. GIVEN an internal service token WHEN ERP calls `GET /platform/tenants/:id/entitlements` THEN the request is authenticated and the endpoint responds within 50ms (NFR46)
6. GIVEN the Platform API WHEN any state-changing admin action is performed THEN a PlatformAuditLog record is created with actor, action, target, details, IP, and timestamp (BR-PLT-017)

## Dev Notes

### Architecture Context

This story creates the **Platform API Fastify server** in `apps/platform-api/`. The Platform API is a completely **separate application** from the ERP API (`apps/api/`). It:

1. Has its own database (`PLATFORM_DATABASE_URL` → Platform PostgreSQL, NOT tenant DB)
2. Has its own auth system (Platform JWT, not ERP JWT)
3. Serves on a separate port (default 3001)
4. Has NO concept of `companyId` or company-context middleware
5. Has two auth pathways:
   - **Admin auth** (`/admin/*`) — PlatformUser JWT with `platformUserId` and `platformRole` claims
   - **Service token auth** (`/platform/*`) — Bearer token for ERP runtime calls, validated against `PLATFORM_SERVICE_TOKEN` env var

### Current State of `apps/platform-api/`

The package exists as a **Prisma data-access layer only** — created in E1-6. It contains:
- `prisma/schema.prisma` — all Platform DB models (Tenant, Plan, PlatformUser, PlatformAuditLog, etc.)
- `prisma/seed.ts` — seeds 3 plans, 1 dev tenant, 1 platform admin user
- `src/client.ts` — `getPlatformPrisma()` lazy singleton (PROTECTED FILE — do not overwrite)
- `src/index.ts` — barrel exports (PROTECTED FILE — do not overwrite, only append new exports)
- `generated/platform-prisma/` — generated Prisma client
- `prisma.config.ts` — Prisma 7 config (reads `.env` from monorepo root)

**CRITICAL: Do NOT overwrite `src/client.ts` or `src/index.ts`.** Add new files alongside them. Update `src/index.ts` only to add new exports.

### PrismaClient Pattern Difference

- **Platform** uses lazy singleton: `getPlatformPrisma()` (function call)
- **ERP** uses eager singleton: `export const prisma = new PrismaClient(...)`
- The Platform pattern was chosen because the app may not always need DB access. Use `getPlatformPrisma()` in all Platform API service code.

### Key Differences from ERP API

| Aspect | ERP API (`apps/api/`) | Platform API (`apps/platform-api/`) |
|--------|----------------------|-------------------------------------|
| Database | Tenant PostgreSQL (`DATABASE_URL`) | Central Platform PostgreSQL (`PLATFORM_DATABASE_URL`) |
| Auth | User JWT (`userId`, `tenantId`, `role`, `enabledModules`) | Platform JWT (`platformUserId`, `platformRole`) OR Service Token |
| Company context | `X-Company-ID` header, `companyId` on every query | None — Platform is cross-tenant |
| Request decorators | `request.userId`, `request.tenantId`, `request.companyId`, `request.userRole` | `request.platformUserId`, `request.platformRole` |
| Audit | ERP AuditLog (per-tenant, with companyId) | PlatformAuditLog (central, no companyId, append-only) |
| Port | 3000 | 3001 |
| RBAC | Access Groups with resource-level permissions | Two roles only: PLATFORM_ADMIN, PLATFORM_VIEWER |

### Cross-Cutting Patterns (MANDATORY)

- **companyId**: NOT applicable to Platform API. Platform DB has no companyId. All queries are cross-tenant.
- **i18n**: Not applicable for MVP (Platform Admin Portal is English-only vendor tool). Error messages can be hardcoded English strings.
- **Audit**: All state-changing operations MUST create PlatformAuditLog entries (BR-PLT-017). Append-only, no update/delete (BR-PLT-016, NFR49).
- **Attachments/Notes/Tasks**: Not applicable to Platform API.

### Testing Strategy

- **Unit tests**: Auth service (credential verification, JWT generation, TOTP), audit service, response helpers
- **Integration tests**: Full route tests via `fastify.inject()` — no real HTTP, just Fastify's built-in test helper
- **Test database**: Use the existing Platform DB (`PLATFORM_DATABASE_URL`). Seed data from `prisma/seed.ts` provides a platform admin user (`admin@nexa-platform.local`).
- **Test patterns**: Follow ERP test patterns from `apps/api/src/core/auth/auth.routes.test.ts` — use `describe/it/expect` with `fastify.inject()`.
