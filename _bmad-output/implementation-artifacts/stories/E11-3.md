# Story 11.3: Task Notifications

Status: done

## Story

As a task assignee,
I want to receive notifications when I'm assigned a task, when a task's status changes, and when a task becomes overdue,
so that I stay informed about my responsibilities.

## Acceptance Criteria

1. GIVEN a TaskAssignee record is created WHEN the `task.assigned` event is emitted THEN each assignee receives an in-app + email notification (per their notification preferences)
2. GIVEN a task status changes to COMPLETED WHEN the `task.status_changed` event is emitted THEN the task creator and all assignees receive a notification
3. GIVEN a task with a due date WHEN the due date passes and status is OPEN or IN_PROGRESS THEN a `task.overdue` event is emitted and assignees are notified
4. GIVEN notification preferences WHEN a user has disabled task notifications for a channel THEN that channel is skipped (leverages E9 notification preferences system)
5. GIVEN the overdue check WHEN it runs THEN it executes daily as a BullMQ scheduled job

## Dev Notes

### Architecture Compliance

**E9 Notification Infrastructure (MANDATORY):**
This story builds on top of the E9 notification system. It does NOT implement its own notification delivery — it only:
1. Seeds notification templates (so `createNotificationsFromEvent` can match events to templates)
2. Registers events in `NOTIFICATION_TRIGGERING_EVENTS` (so the generic event loop subscribes to them)
3. Updates the target resolver to correctly identify task notification recipients
4. Adds a BullMQ cron job for overdue detection (emitting `task.overdue` events into the bus)

The existing E9 infrastructure handles:
- Template-based notification creation (`createNotificationsFromEvent`)
- Preference-based channel resolution (`resolveChannels` with `NotificationPreference`)
- Deduplication window (60s built into `createNotificationsFromEvent`)
- Async delivery via BullMQ dispatch queue (`enqueueNotificationDelivery`)
- IN_APP delivery via WebSocket push (`pushNotificationToUser`)
- EMAIL delivery via SMTP sender (`EmailSender`)

**BullMQ Cron Job Pattern:**
- Uses the same pattern as the notification dispatch plugin
- Queue + Worker + Fastify Plugin (lifecycle management)
- Graceful degradation: if REDIS_URL not set, plugin logs warning and skips (no crash)
- Worker processes are co-located in the API process (in-process for MVP, same as notification worker)

### Target Resolver Strategy for Task Events

The existing target resolver uses a priority chain of strategies. Task events need special handling:

| Event | Target Users | Actor (excluded) |
|-------|-------------|-----------------|
| `task.assigned` | `assigneeUserId` (direct) | `assignedBy` |
| `task.status_changed` (COMPLETED) | Task creator + all assignees (look up from DB) | `changedBy` |
| `task.overdue` | All users in `assigneeUserIds` array + `createdById` | None (system job) |

For `task.status_changed`, the payload doesn't include assignee list or creator ID. The target resolver needs to look up the Task record to get `createdById` and associated `TaskAssignee` records. This requires:
1. Adding `'Task'` to the `modelMap` in `lookupEntityCreator()`
2. Adding a new helper to look up all task assignees by taskId

Alternatively, enrich the `task.status_changed` event payload to include `createdById` and `assigneeUserIds` at emit time (in task.service.ts). This avoids additional DB lookups in the notification path. **Preferred approach:** enrich the event payload in task.service.ts.

### Overdue Deduplication Strategy (BR-TASK-007)

The cron job runs daily at 08:00 UTC. To avoid re-notifying the same overdue task every day:
- Before emitting `task.overdue`, check if a Notification record exists with:
  - `entityType = 'task.overdue'` (using the notification's `metadata` or derive from template eventName)
  - Created within the last 24 hours
  - For the same taskId
- If found, skip the task for this run

Implementation detail: Query `Notification` table:
```sql
SELECT id FROM notifications
WHERE "templateEventName" = 'task.overdue'
  AND metadata->>'taskId' = ?
  AND "createdAt" > NOW() - INTERVAL '24 hours'
LIMIT 1
```

If this Notification schema query is too complex, an alternative is to add a `lastOverdueNotifiedAt` column to the Task model — but this is over-engineering for MVP. A simpler approach: use the Notification table query above, or add a `taskOverdueNotifications` tracking table/set in Redis.

**Simplest approach (RECOMMENDED):** Track in a local Set within the worker run. On each cron execution, the worker queries ALL overdue tasks and ALL recent overdue notifications (last 24h), builds a "already notified" set, and only emits events for tasks not in that set. This avoids per-task queries.

- **companyId**: Overdue task query MUST scope by company (though for the cron job, it queries across all companies — each task has its own companyId in the event payload)

- **Non-blocking**: Notification failures must NEVER block the primary business operation. The event handler try/catch pattern from E9 ensures this.
