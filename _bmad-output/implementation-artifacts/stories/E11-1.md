# Story 11.1: Task Service (Backend)

Status: done

## Story

As a system,
I want a task CRUD service with assignment, status transitions, and polymorphic entity linking,
so that users can create and manage tasks from any business record.

## Acceptance Criteria

1. GIVEN a user creates a task WHEN they provide title, priority, and optional due date THEN a Task record is created with status OPEN, scoped to companyId
2. GIVEN a task with entityType and entityId WHEN created THEN the task is linked to that specific business record (polymorphic FK pattern matching Attachment/Note/RecordLink)
3. GIVEN a user assigns users to a task WHEN TaskAssignee records are created THEN a `task.assigned` event is emitted for each assignee (FR182)
4. GIVEN a task in OPEN status WHEN status changes to IN_PROGRESS THEN the transition is recorded
5. GIVEN a task WHEN status changes to COMPLETED THEN `completedAt` timestamp is set to now, and a `task.status_changed` event is emitted
6. GIVEN a task in COMPLETED or CANCELLED status WHEN a further status change is attempted THEN the request is rejected (terminal states, BR-TASK-006)
7. GIVEN a user WHEN they request `/tasks/my` THEN only tasks where they are a TaskAssignee are returned, scoped by companyId
8. GIVEN query params `entityType` and `entityId` WHEN filtering tasks THEN only tasks linked to that specific record are returned

## Dev Notes

### Architecture Compliance

**3-Layer Pattern (MANDATORY):**
- Routes (`task.routes.ts`) → Service (`task.service.ts`) → Prisma directly (no repository layer needed for cross-cutting — see note.service.ts pattern)
- The existing cross-cutting services (note, attachment, record-link) use **standalone exported async functions**, NOT classes
- Every function takes `(ctx: RequestContext, prisma: PrismaClient, ...)` as first params
- EventBus is passed as a parameter to functions that emit events (see access-groups.service.ts pattern)

**Key Difference from Note/Attachment:**
- Task has `companyId` directly on the model (unlike Note/Attachment which derive company scope from parent entity)
- Task has `deletedAt` for soft delete (same as Note)
- Task has `entityType`/`entityId` as OPTIONAL (nullable) — tasks can exist without a linked entity

### Exact Prisma Schema (from project-context.md §4)

```prisma
model Task {
  id            String       @id @default(uuid())
  companyId     String       @map("company_id")
  title         String       @db.VarChar(255)
  description   String?      @db.Text
  priority      TaskPriority @default(NORMAL)
  status        TaskStatus   @default(OPEN)
  dueDate       DateTime?    @map("due_date")
  entityType    String?      @map("entity_type")    // polymorphic FK
  entityId      String?      @map("entity_id")      // polymorphic FK
  createdById   String       @map("created_by_id")
  completedAt   DateTime?    @map("completed_at")
  deletedAt     DateTime?    @map("deleted_at")
  createdAt     DateTime     @default(now()) @map("created_at")
  updatedAt     DateTime     @updatedAt @map("updated_at")
  updatedBy     String       @map("updated_by")

  company       Company      @relation(fields: [companyId], references: [id])
  createdBy     User         @relation("TaskCreator", fields: [createdById], references: [id])
  assignees     TaskAssignee[]

  @@map("tasks")
  @@index([companyId, status], map: "idx_tasks_company_status")
  @@index([entityType, entityId], map: "idx_tasks_entity")
  @@index([dueDate], map: "idx_tasks_due_date")
  @@index([companyId, deletedAt], map: "idx_tasks_company_active")
}

model TaskAssignee {
  id       String @id @default(uuid())
  taskId   String @map("task_id")
  userId   String @map("user_id")
  task     Task   @relation(fields: [taskId], references: [id], onDelete: Cascade)
  user     User   @relation(fields: [userId], references: [id])
  @@map("task_assignees")
  @@unique([taskId, userId], map: "uq_task_assignees_task_user")
}

enum TaskPriority { LOW NORMAL HIGH URGENT }
enum TaskStatus { OPEN IN_PROGRESS COMPLETED CANCELLED }
```

**Added vs project-context.md:** `deletedAt` (soft delete, per Note pattern) and `updatedBy` (audit trail, per architecture §Timestamps).

### TaskStatus State Machine

```
OPEN ──────→ IN_PROGRESS ──────→ COMPLETED (terminal)
  │              │
  └──────────────┴──────────────→ CANCELLED (terminal)
```

| From | To | Guard | Side Effect | Event |
|------|----|-------|-------------|-------|
| OPEN | IN_PROGRESS | Any assignee or creator | — | `task.status_changed` |
| OPEN | COMPLETED | Any assignee or creator | Set `completedAt = now()` | `task.status_changed` |
| OPEN | CANCELLED | Creator or MANAGER+ | — | `task.status_changed` |
| IN_PROGRESS | COMPLETED | Any assignee or creator | Set `completedAt = now()` | `task.status_changed` |
| IN_PROGRESS | CANCELLED | Creator or MANAGER+ | — | `task.status_changed` |
| COMPLETED | * | BLOCKED | Throw 422 `TASK_TERMINAL_STATUS` | — |
| CANCELLED | * | BLOCKED | Throw 422 `TASK_TERMINAL_STATUS` | — |

### Business Rules (BR-TASK-001 to BR-TASK-008)

| Rule | Description |
|------|-------------|
| BR-TASK-001 | Title is mandatory on Task creation (min 1 char, max 255) |
| BR-TASK-002 | Task must be scoped to companyId (from ctx.companyId) |
| BR-TASK-003 | If entityType provided, it must be in VALID_ENTITY_TYPES registry AND entity must exist in same company (via `validateEntityExists`) |
| BR-TASK-004 | Assignee userId must reference a User that exists and has access to the same company |
| BR-TASK-005 | `task.assigned` event emitted for each new TaskAssignee created |
| BR-TASK-006 | COMPLETED and CANCELLED are terminal states — reject further status changes with 422 |
| BR-TASK-007 | Daily overdue check (E11.3 scope — NOT this story) |
| BR-TASK-008 | `completedAt` auto-set to current timestamp when status transitions to COMPLETED |

### Cross-Cutting Patterns (MANDATORY)

Every implementation MUST follow these patterns from project-context.md:
- **companyId**: Task has companyId directly — all queries MUST include `WHERE companyId = ?` AND `deletedAt IS NULL`
- **i18n**: Error messages use messageKey params (e.g., `'errors.task.notFound'`) — actual i18n keys for frontend labels are E11.2 scope
- **Audit**: Status changes and assignee changes emit typed events via eventBus. Full audit trail logging is inherited from the event bus infrastructure.
- **Attachments/Notes/Tasks**: Task itself IS the cross-cutting entity. It supports linking TO other entities, not the reverse (tasks don't need their own notes/attachments in MVP).
