# Story E5c.5: Automation Builder UI

Status: ready-for-dev

## Story

As an **admin**,
I want to create and configure AI automations (scheduled workflows with chained agent steps) through a visual builder,
so that I can set up autonomous AI workflows without coding.

## Acceptance Criteria

1. **AC-1: Automation list page** — GIVEN the Automation Builder list page WHEN rendered THEN it shows a T1 Entity List of all automations with columns: name (bold), trigger type (badge: "Scheduled" purple, "Event" blue, "Chain" amber, "Manual" grey), schedule (cron expression in human-readable form e.g. "Weekdays at 7:00 AM" or "—" for non-scheduled), step count (mono font), last run status (StatusBadge: green "Completed" / red "Failed" / amber "Running" / grey "Pending" / none "Never run"), last run time (relative format e.g. "2 hours ago"), active toggle (inline Switch with optimistic mutation). Supports search (on name, description), column sorting, and cursor-based pagination. The list page is navigable from the sidebar under AI Admin → Automations.

2. **AC-2: Create automation — basic config** — GIVEN the "Create Automation" action WHEN clicked THEN a builder page opens with sections:
   - **Header section:** name (required, text input), description (textarea, optional)
   - **Trigger section:** trigger type radio group (Scheduled, Event, Manual) with type-specific config:
     - **Scheduled:** cron builder component with individual fields (minute, hour, day-of-month, month, day-of-week) and a human-readable preview below (e.g. "Every weekday at 7:00 AM London time"), timezone selector (dropdown of IANA timezones, default "Europe/London"), isPaused toggle
     - **Event:** event type text input with helper text listing known event types (from Event Catalog — e.g. "invoice.overdue", "stock.reorder.triggered")
     - **Manual:** no additional config (just name, description, and steps)
   - **Budget section:** max token budget (number input, default 50,000), max duration (number input in seconds, default 300)
   - Form validates: name required, at least one step required before save, cron expression valid for scheduled type

3. **AC-3: Step builder** — GIVEN the automation editor's step builder section WHEN rendered THEN it shows an ordered list of steps as cards, each showing: step number (circle badge), agent selection dropdown (from AiAgent registry where `isActive=true`, showing display name + name in mono), goal text area (multi-line, with variable autocomplete on `{{`), input configuration panel (collapsible), output configuration panel (collapsible). Steps are visually connected with a vertical line/connector between them to show sequential flow.

4. **AC-4: Step add and reorder** — GIVEN the step builder WHEN an admin clicks "Add Step" THEN a new step card is appended at the end with empty agent/goal/config. Steps can be drag-reordered using dnd-kit (drag handle on left of each step card). Step numbers auto-update on reorder. Steps can be removed via a delete button (with confirmation if goal text is non-empty). Minimum 1 step is required to save the automation.

5. **AC-5: Step input/output configuration** — GIVEN a step's input configuration panel WHEN expanded THEN it shows a JSON editor (mono font, syntax validation) for `inputConfig` with helper text: "Define variable bindings for this step's input. Sources: DB_FIELD, DB_QUERY, SYSTEM, PREVIOUS_STEP, CONSTANT, EXPRESSION". For steps after the first, a "Use Previous Step Output" shortcut button auto-populates `inputConfig` with `{ "sources": [{ "type": "PREVIOUS_STEP", "stepOrder": N-1, "field": "*" }] }`. The output configuration panel shows a JSON editor for `outputConfig` with helper text: "Define what to capture from the agent's response".

6. **AC-6: Variable autocomplete in goal text** — GIVEN the goal text area within a step WHEN an admin types `{{` THEN a dropdown appears showing available variables grouped by source: System variables (`{{today}}`, `{{currentUser.name}}`, `{{company.name}}`, `{{company.baseCurrency}}`), Previous step outputs (`{{step1.output.*}}`, `{{step2.output.*}}`), Constants. The dropdown is populated from the `GET /ai/variables` endpoint. Selecting a variable inserts it at the cursor position. The autocomplete uses the same pattern as the prompt editor variable autocomplete from E5c-3.

7. **AC-7: Chain configuration** — GIVEN the automation editor's chain section WHEN an admin toggles "Chain to next automation" THEN a dropdown shows other active automations (excluding the current one) to chain to. The current automation's final step output is automatically mapped as the chained automation's input. If saving creates a circular chain (detected by backend with 422), the form shows an error: "Circular chain detected: [chain path]. Maximum chain depth is 10." Chain display shows the chain path as a visual breadcrumb.

8. **AC-8: Notification configuration** — GIVEN the automation editor's notification section WHEN an admin toggles "Notify on completion" THEN they can configure: notification recipients (multi-select of users), notification channels (checkboxes: in-app, email), notify on success (toggle, default true), notify on failure (toggle, default true). Configuration is stored as `notificationConfig` JSON on the automation.

9. **AC-9: Run Now** — GIVEN the "Run Now" button on the automation list (per row) or builder page WHEN clicked THEN a confirmation dialog appears. On confirm, the automation executes immediately via `POST /ai/automations/:id/run`. A toast shows "Automation started" with a link to the run detail. The list page refreshes to show the new run status.

10. **AC-10: Dashboard integration** — GIVEN the AI Configuration Dashboard WHEN rendered THEN it includes an Automations quick-nav card showing: total automation count, active count, paused count, last 24h run count (with success/fail breakdown). Clicking navigates to the Automation list page.

11. **AC-11: Visual design fidelity** — GIVEN all automation pages WHEN rendered THEN they match Concept D visual design: 12px card radius, purple-tinted hover shadows, `#7c3aed` primary buttons, `#f4f2ff` background, Plus Jakarta Sans headings, Inter body text, JetBrains Mono for code/config fields, StatusBadge with coloured dot + label, fadeInUp animation on page load.

## Technical Notes

### Backend Endpoints (All Exist — No Backend Work Required)

All API endpoints were implemented in E5c-1 and E5c-2. This story is **frontend-only**.

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/ai/automations` | List automations (paginated, filterable) |
| POST | `/ai/automations` | Create automation |
| GET | `/ai/automations/:id` | Get automation detail with steps |
| PATCH | `/ai/automations/:id` | Update automation |
| DELETE | `/ai/automations/:id` | Delete automation |
| POST | `/ai/automations/:id/run` | Manual trigger ("Run Now") |
| GET | `/ai/automations/:id/runs` | List runs for automation |
| GET | `/ai/automations/runs` | List all runs |
| GET | `/ai/automations/runs/:runId` | Run detail with step runs |
| POST | `/ai/automations/runs/:runId/retry` | Retry from failed step |
| GET | `/ai/variables` | List all prompt variables |

### Dependencies (npm packages)

- `@dnd-kit/core` + `@dnd-kit/sortable` — already in workspace for drag-reorder (used in E7 column reorder). If not present, install
- `cronstrue` — lightweight library for cron-to-human-readable conversion (or implement utility manually). Check if already available; prefer a small utility function over a new dependency

### Key Decisions

1. **No WebSocket for "Run Now" progress** — The epic mentions real-time step progress via WebSocket. For this story, we simplify: "Run Now" triggers the automation, shows a toast with a link to the run detail page. Real-time progress monitoring (WebSocket-based step-by-step updates) is deferred to E5c-6 (Automation Monitoring & Run History) which is the dedicated monitoring story.

2. **Variable autocomplete reuse** — The variable autocomplete should share the same pattern/component as the prompt editor from E5c-3. If E5c-3 built a reusable `VariableAutocomplete` component, import and extend it. If not, build a new one that can be shared.

3. **Cron builder vs raw input** — The cron builder provides both a visual field selector AND a raw text input. They sync bidirectionally. This accommodates both novice users (visual) and power users (raw cron).

4. **Chain cycle detection** — The frontend does NOT implement cycle detection. It relies on the backend returning a 422 error with a descriptive message. The form displays this error inline.
