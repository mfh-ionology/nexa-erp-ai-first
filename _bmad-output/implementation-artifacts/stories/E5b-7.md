# Story E5b-7: Inline Entity Mentions & Chat Autocomplete

**Epic:** E5b — AI Co-Pilot Intelligence — Memory, Skills & Dynamic Context
**Status:** done
**Dependencies:** E5b-2 (Skills Registry, Tool Framework & Dynamic Context — done), E5b-6 (E7 Skill Pack Validation — done), E6 (Frontend Shell — done)
**Complexity:** X-Large (12 tasks)

---

## User Story

As a user, I want the chat textbox to detect when I'm referencing ERP entities (contacts, invoices, customers) and show context-aware autocomplete suggestions, so that I can naturally reference entities without learning special commands or knowing exact IDs.

---

## Acceptance Criteria

### AC1: Trigger Word Detection (Local)
**GIVEN** the chat textbox
**WHEN** the user types a recognised entity trigger word (e.g., "contact", "invoice", "customer") followed by a space and at least 2 characters of partial text
**THEN** an autocomplete dropdown appears below the cursor showing matching entities from the relevant search endpoint
**AND** the trigger detection runs entirely in the browser — no server round-trip for trigger matching

### AC2: Context-Aware Scoping
**GIVEN** the entity trigger "contact" with `scopeBy: "customerId"`
**WHEN** the user's message already mentions a customer (e.g., "Send invoice 1042 to contact jo...")
**THEN** the contact search is scoped to that customer's contacts — NOT all contacts in the system
**AND** the scoping is resolved from other entity references (chips) already in the message

### AC3: Entity Chip Insertion
**GIVEN** the autocomplete dropdown
**WHEN** the user selects an entity (via click, Enter, or Tab)
**THEN** the trigger word + partial text is replaced with a styled entity chip showing the entity's display name (e.g., **John Smith**)
**AND** the underlying payload includes the structured reference `{contact:uuid-123}` that the AI receives
**AND** the chip is removable via its close button or Backspace when input is empty

### AC4: Structured Reference Resolution
**GIVEN** the user sends a message with entity chips
**WHEN** the AI processes it
**THEN** each chip is resolved to a structured reference with entity type and UUID in the message payload (e.g., `entityMentions: [{ type: "Contact", id: "uuid-123", name: "John Smith" }]`)
**AND** the AI can use these identifiers to call tools with exact entity references (no ambiguity, no fuzzy matching needed)

### AC5: Client-Side Trigger Registry Cache
**GIVEN** the entity trigger registry
**WHEN** loaded by the frontend
**THEN** triggers are fetched from `GET /ai/entity-triggers?isActive=true` and cached client-side via TanStack Query (1hr `staleTime`)
**AND** triggers are indexed by trigger word for O(1) lookup during detection

### AC6: Debounced Entity Search
**GIVEN** the entity search API call
**WHEN** triggered by autocomplete
**THEN** the request is debounced (300ms)
**AND** includes the user's partial text as `q` parameter
**AND** is scoped by companyId (from auth context) and any contextual entity (from message chips)
**AND** returns max 8 results with display name + subtitle (e.g., email for contacts, reference for invoices)

### AC7: No Trigger Conflicts
**GIVEN** entity types across modules
**WHEN** all trigger words are loaded
**THEN** there are no conflicts — each trigger word maps to exactly one entity type
**AND** if a module needs "order" for both Sales Orders and Purchase Orders, it uses distinct triggers ("sales order" vs "purchase order")
**AND** the `GET /ai/entity-triggers` endpoint returns triggers ordered by priority DESC so the highest-priority trigger wins

### AC8: Concept D Styling
**GIVEN** the autocomplete popup
**WHEN** rendered
**THEN** it follows Concept D styling:
  - Purple highlight (`bg-[#f5f3ff]`) on selected item
  - 12px border radius on popup, matching card shadow tokens
  - Entity icon in purple circle (`bg-[#ede9fe] text-[#6d28d9]`) for each result
  - Entity chips use purple pill style (`bg-[#ede9fe] text-[#6d28d9] rounded-full`)
**AND** supports keyboard navigation: ↑↓ to navigate, Enter to select, Tab to select first, Escape to dismiss

### AC9: Mobile Bottom Sheet
**GIVEN** a user on a mobile viewport (< 768px)
**WHEN** entity autocomplete triggers
**THEN** the results render as a bottom sheet (not a dropdown that gets hidden behind the virtual keyboard)
**AND** the bottom sheet is dismissible via swipe-down or Escape
**AND** results are tappable with adequate touch targets (min 44px)

### AC10: Backend Entity Search Proxy
**GIVEN** the `GET /ai/entity-search` endpoint
**WHEN** called with `type`, `q`, and optional `scopeBy` + `scopeValue` parameters
**THEN** it looks up the entity trigger by `type` to find the `searchEndpoint`
**AND** delegates to that module's search endpoint with companyId scoping + query + scope parameters
**AND** returns max 8 results formatted as `{ id, displayName, subtitle, entityType }`

### AC11: Message Display with Entity Chips
**GIVEN** a sent message containing entity mentions
**WHEN** rendered in the chat message list
**THEN** entity mentions display as inline chips within the message bubble
**AND** user message chips use `bg-white/20 text-white` (on purple bubble)
**AND** assistant message chips use `bg-[#ede9fe] text-[#6d28d9]` (on card bubble)
**AND** chips are non-removable in sent messages (display-only)

  - `entitySearchQuerySchema`: `{ type: z.string(), q: z.string().min(2).max(100), scopeBy: z.string().optional(), scopeValue: z.string().uuid().optional() }`
  - `entitySearchResultSchema`: `{ id: z.string(), displayName: z.string(), subtitle: z.string().nullable(), entityType: z.string() }`

- [x] 1.2 Create `apps/api/src/ai/entity-search.service.ts` with `EntitySearchService` class:

  - Method: `search(params: { type: string; q: string; companyId: string; scopeBy?: string; scopeValue?: string }): Promise<EntitySearchResult[]>`
  - Implementation: (1) look up the active entity trigger by `entityType === params.type`, (2) find the `searchEndpoint` and `displayField`/`subtitleField` from the trigger, (3) build a Prisma dynamic query against the entity's table based on the trigger's `entityType` mapping, (4) apply `companyId` scoping + `q` search (case-insensitive contains on display field), (5) if `scopeBy` and `scopeValue` provided, add scope filter, (6) limit to 8 results, (7) map results to `{ id, displayName, subtitle, entityType }`
  - Note: For the initial implementation, use a switch/map for known entity types (DataView, SavedView from E7). Future module epics will register their own search handlers
- [x] 1.3 Add `GET /entity-search` route to `entity-triggers.routes.ts`:
  - Path: `/entity-search` (registered under the `/ai` prefix, so full path is `GET /api/v1/ai/entity-search`)
  - Permission: `createPermissionGuard('ai.chat', 'view')` (same as entity triggers list)
  - Handler: calls `entitySearchService.search()` with query params + `request.user.companyId`
  - Response: `successEnvelope(entitySearchResponseSchema)`
- [x] 1.4 Wire `EntitySearchService` into `apps/api/src/ai/index.ts`:
  - Import and instantiate `EntitySearchService` in the AI plugin initialization
  - Decorate fastify instance: `fastify.decorate('aiEntitySearchService', entitySearchService)`
  - Add type augmentation for `aiEntitySearchService`

**Implementation notes:**
- The entity search proxy abstracts away module-specific search logic. For E7 entities (DataView, SavedView), it queries Prisma directly. Future modules will register custom search handlers
- companyId scoping is mandatory on all queries (DataView and SavedView are company-scoped)
- The proxy must handle the case where no trigger matches the requested `type` (return 404)

### Task 6: Frontend — useMentionDetection Hook
**AC:** #1, #2
**Files:**
- `apps/web/src/features/ai/entity-mentions/use-mention-detection.ts` (create)

**Subtasks:**
- [x] 6.1 Create `useMentionDetection(triggerMap: Map<string, EntityTrigger>, inputText: string)` hook:
  - Returns: `{ detected: { trigger: EntityTrigger; searchQuery: string; triggerStartIndex: number } | null }`
  - Algorithm:
    1. Split input text into words from the cursor position backwards
    2. For each position, check if the last N words (N=1,2 for multi-word triggers like "saved view") form a trigger word by looking up in `triggerMap`
    3. If a trigger is found, extract the text after the trigger word as the search query
    4. Only activate if the search query has >= 2 characters (prevents spurious searches)
    5. Return the matched trigger + search query + the index where the trigger word starts (needed for replacement in Task 8)
  - Multi-word trigger matching: Check 2-word combos first ("saved view"), then single words ("invoice"). Longer match wins
- [x] 6.2 The detection must handle:
  - Case-insensitive matching ("Invoice" matches "invoice" trigger)
  - Mid-sentence triggers ("Send to contact jo" detects "contact" trigger with query "jo")
  - Multiple triggers in same input — only the LAST (most recent) trigger is active
  - Trigger words that are part of longer words are NOT matched ("invoices" does NOT match "invoice" — require word boundary)

**Implementation notes:**
- This is a pure synchronous computation — no API calls, no debouncing. Runs on every input change
- Keep it performant — typical trigger list is <50 entries, typical input is <200 chars
- Use `useMemo` to avoid recomputation when inputs haven't changed
