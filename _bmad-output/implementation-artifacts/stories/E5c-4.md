# Story E5c.4: Agent & Skill Admin UI

Status: ready-for-dev

## Story

As an **admin**,
I want to configure AI agents (model, prompt, tools, guardrails) and manage skill packs (view, edit, test trigger phrases) through a web interface,
so that I can tune the AI behaviour for my organisation.

## Acceptance Criteria

1. **AC-1: Agent Configuration list** — GIVEN the Agent Configuration page WHEN rendered THEN it shows a T1 Entity List of all agents with columns: name (mono font, bold), display name, assigned model (display name or "Auto-routed" if `modelId` is null), assigned prompt (name, mono font), tool count (mono font), routing tags (as purple badges), max turns (mono font), active status (StatusBadge green "Active" / grey "Inactive"). Supports search (on name, displayName, description), column sorting, and cursor-based pagination.

2. **AC-2: Agent add/edit form** — GIVEN an agent WHEN an admin clicks to edit or "Add Agent" THEN a T2-style detail form opens with tabs:
   - **Main tab:** name (required, unique, mono font), display name (required), description (textarea, optional), model selection dropdown (from AiModel registry where `isActive=true`, with "Auto-route by tags" option for null `modelId`), prompt selection dropdown (from AiPrompt registry where `isActive=true`) with prompt name in mono font, routing tags (multi-select tag input: reasoning, standard, cheap, fast, vision, structured_output, briefing — purple pill badges), max turns (number input, range 1-50, default 10), active toggle (Switch).
   - **Tools tab:** JSON editor (textarea with mono font, syntax validation on blur) for `tools` array. Helper text: "Array of tool name strings this agent can invoke". Pre-populate with `[]` for new agents.
   - **Guardrails tab:** Structured form for `guardrails` JSON: `canRead` (multi-select tag input for entity names), `canWrite` (multi-select tag input), `requiresApproval` (Switch), `maxAmountWithoutApproval` (number input, only shown when `requiresApproval` is true), `blockedOperations` (tag input), `dataScope` (select dropdown: own / module / all).
   - **Triggers tab:** JSON editor (textarea with mono font, syntax validation on blur) for `triggerConfig` array. Helper text: "Array of AgentTrigger objects (chat_intent, ui_button, schedule, event, api)".
   - **Business rules:** (a) Cannot delete an agent referenced by `AiAutomationStep` — show error with step count. (b) Name must be unique — show 409 error on conflict. (c) `promptId` is required — form validation enforces this. (d) Cannot set `modelId` to a model that is `isActive=false`.

3. **AC-3: Skill Pack Manager list** — GIVEN the Skill Pack Manager page WHEN rendered THEN it shows skills grouped by `moduleKey` in accordion sections (collapsible, default expanded for first module). Each module section shows its module key as a heading badge. Within each section, skills are displayed as cards (not a table) showing: name (mono font, bold), display name, description (truncated), trigger phrases as blue-tinted pill tags, negative triggers as red-tinted pill tags, orchestration pattern badge (subtle purple), priority (mono font), active toggle (inline Switch that directly calls update mutation). Skills with no `moduleKey` appear under an "Unassigned" section at the bottom. A search bar at top filters across all modules. A "View as List" toggle switches to a flat T1 Entity List view.

4. **AC-4: Skill detail/edit form** — GIVEN a skill WHEN an admin clicks to edit THEN a T2-style detail form opens with tabs:
   - **Main tab:** name (required, unique, mono font), display name (required), description (textarea), category (select: document, analysis, communication, financial), module key (select from known modules: ar, ap, finance, sales, purchasing, inventory, crm, hr, manufacturing, reporting, views, system — or free text), pack key (text input), output type (select: pdf, json, markdown, email), priority (number input, default 100), version (read-only display), active toggle.
   - **Triggers tab:** trigger phrases (tag input with add/remove, blue pills), negative triggers (tag input, red pills), context required (tag input, grey pills with format hints like "screen:entity-list", "module:ar").
   - **Content tab:** skill content editor — large textarea (mono font, min-height 400px) for the full `skillContent` (SKILL.md instructions). Auto-resize with content.
   - **Schema tab:** input schema (JSON editor, mono font, syntax validation on blur), parameters (JSON editor), required tools (tag input for tool name strings), examples (JSON editor for example input/output pairs).
   - **Business rules:** (a) Skills use soft-delete only — toggling `isActive=false` rather than physical delete. A "Delete" action in overflow sets `isActive=false` with confirmation. (b) Name must be unique — 409 on conflict. (c) Orchestration pattern (select: SEQUENTIAL, PARALLEL, ITERATIVE, CONTEXT_AWARE, DOMAIN_INTELLIGENCE, or null/unset).

5. **AC-5: Test Trigger panel** — GIVEN the "Test Trigger" panel on the Skill Pack Manager page WHEN an admin types "show me overdue invoices" THEN the system runs the L0→L1→L2 routing chain simulation and shows:
   - **L0 result:** matched module (e.g., "ar") with confidence score
   - **L1 result:** matched skill within that module (e.g., "ar-overdue-analysis") with confidence score
   - **L2 result:** the selected skill's `requiredTools` list and first 200 chars of `skillContent`
   - If no match: "No matching skill found" with suggestions (list of active modules)
   - The simulation must respond within 3 seconds (NFR44)
   - The panel is accessible via a "Test Trigger" button at the top of the Skill Pack Manager page, opening a Sheet (slide-in from right)

6. **AC-6: Skill activation toggle** — GIVEN skill activation toggles in the Skill Pack Manager WHEN an admin toggles a skill off THEN it is set to `isActive=false` (excluded from the L1 module pack and never activated by the AI). When toggled back on, `isActive=true`. The toggle operates as an inline optimistic mutation with rollback on failure. A toast confirms the action: "Skill {name} deactivated" / "Skill {name} activated".

### Task 4: Create trigger testing service (AC: #5)

- [ ] 4.1 Create `apps/api/src/ai/admin/admin-trigger-test.service.ts`:
  - `testTrigger(phrase: string)` → simulate L0→L1→L2 routing:
    1. **L0 — Module routing:** Load all active skills grouped by moduleKey. For each module, compute a relevance score by checking if the phrase matches any `triggerPhrases` in that module's skills (case-insensitive substring match + Jaccard similarity on word tokens). Select the module with highest aggregate score. If no module scores above threshold (0.1) → return `{ noMatch: true, suggestions: [list of active moduleKeys] }`
    2. **L1 — Skill selection:** Within the matched module, score each active skill by: (a) `triggerPhrases` match (Jaccard similarity of word tokens), (b) `negativeTriggers` penalty (if phrase matches a negative trigger, subtract 0.5 from score), (c) `priority` weight (higher priority = slight boost). Select the skill with highest score
    3. **L2 — Skill details:** Return the selected skill's `id`, `name`, `displayName`, `requiredTools`, first 200 chars of `skillContent`, and the confidence scores from L0 and L1
  - Confidence score: 0.0 to 1.0, computed as normalised Jaccard similarity of word-level tokens between the input phrase and the trigger phrase set
  - Must complete within 3 seconds (NFR44) — pure in-memory computation on active skills (no external AI call)

### Key Design Decisions

1. **No Prisma migration needed**: AiAgent, AiSkill, AiSkillContext, AiSkillOverride models all exist from E5/E5b. This story is purely service layer + API + frontend.

2. **Extend existing admin.routes.ts**: Rather than creating a separate route plugin, add agent/skill handlers to the existing `adminRoutes` function in `admin.routes.ts`. This keeps all admin routes in one plugin under the `/ai/admin/` prefix.

3. **Agents are system-wide (no companyId)**: AiAgent has no `companyId` field. All agents are shared infrastructure. The CRUD is straightforward with no company scoping needed.

4. **Skills are system-wide with soft-delete**: AiSkill has no `companyId`. The delete action sets `isActive=false` rather than physical deletion, because skills may be referenced by `AiSkillOverride` records and are seeded by module epics.

5. **Skill Pack Manager uses accordion (not table)**: Skills are best viewed grouped by module. The accordion view with cards provides a better UX than a flat table for browsing skills by domain. A "View as List" toggle provides the flat table alternative.

6. **Guardrails as structured form (not raw JSON)**: Unlike `tools` and `triggerConfig` (which have variable structures), `guardrails` has a fixed `AgentGuardrails` interface. The form decomposes it into individual fields (tag inputs, switches, selects) for a better admin experience.

7. **Trigger testing is in-memory (no AI call)**: The Test Trigger simulation uses Jaccard similarity on word tokens — no external AI API call. This ensures sub-3-second response (NFR44) and zero token cost for testing.

8. **Per-tenant skill overrides are out of scope**: The `AiSkillOverride` table exists but managing per-tenant overrides is a separate admin feature. This story manages the system-level `isActive` flag only.
