# Finance Phase 2 Wave 1: Database Schema Changes

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Simulation, BudgetVersion, BudgetKey, and BudgetLineDimension models to the Prisma schema, add reverse relations to existing models, seed a SIMULATION number series (SIM-NNNNN), and seed 3 default budget keys. This creates the database foundation for Finance Phase 2 features F2 (Simulations), F3 (Budget Redesign), and F4 (Missing Settings).

**Architecture:** A single migration adding 5 new models (SimulationStatus enum, Simulation, SimulationLine, BudgetVersion, BudgetKey, BudgetLineDimension), modifying 5 existing models with new reverse relations and one new nullable FK, and seeding reference data. All changes follow the existing Prisma patterns from E0-E14 Wave 1.

**Tech Stack:** Prisma 7, PostgreSQL 17, TypeScript strict

**Depends on:** Finance Phase 1 (E14 Wave 1 -- complete). All 21 finance entities already exist in the schema.

**Blocked by this plan:** All Finance Phase 2 API stories (Simulations CRUD, Budget Version/Key CRUD, Dimension Management APIs)

```prisma
model Simulation {
  id              String           @id @default(uuid()) @map("id")
  companyId       String           @map("company_id")
  entryNumber     String           @map("entry_number") @db.VarChar(20)
  transactionDate DateTime         @map("transaction_date") @db.Date
  description     String           @map("description")
  reference       String?          @map("reference")
  status          SimulationStatus @default(ACTIVE) @map("status")
  periodId        String           @map("period_id")
  totalDebit      Decimal          @default(0) @map("total_debit") @db.Decimal(19, 4)
  totalCredit     Decimal          @default(0) @map("total_credit") @db.Decimal(19, 4)
  transferredToId String?          @map("transferred_to_id")
  createdAt       DateTime         @default(now()) @map("created_at")
  updatedAt       DateTime         @updatedAt @map("updated_at")
  createdBy       String           @map("created_by")
  updatedBy       String           @map("updated_by")

  period        FinancialPeriod @relation(fields: [periodId], references: [id])
  transferredTo JournalEntry?   @relation("SimulationTransfer", fields: [transferredToId], references: [id])
  lines         SimulationLine[]

  @@unique([companyId, entryNumber], name: "uq_simulation_company_number")
  @@index([companyId])
  @@index([companyId, status])
  @@map("simulations")
}
```

- [ ] **Step 4: Add the SimulationLine model**

Append immediately after `Simulation`:

```prisma
model SimulationLine {
  id              String   @id @default(uuid()) @map("id")
  simulationId    String   @map("simulation_id")
  lineNumber      Int      @map("line_number")
  accountCode     String   @map("account_code") @db.VarChar(20)
  companyId       String   @map("company_id")
  description     String?  @map("description")
  debit           Decimal  @default(0) @map("debit") @db.Decimal(19, 4)
  credit          Decimal  @default(0) @map("credit") @db.Decimal(19, 4)
  vatCode         String?  @map("vat_code") @db.VarChar(20)
  dimensionValues Json?    @map("dimension_values")
  createdAt       DateTime @default(now()) @map("created_at")
  updatedAt       DateTime @updatedAt @map("updated_at")

  simulation Simulation     @relation(fields: [simulationId], references: [id], onDelete: Cascade)
  account    ChartOfAccount @relation(fields: [companyId, accountCode], references: [companyId, code], onDelete: NoAction, onUpdate: NoAction)

  @@index([simulationId])
  @@index([companyId, accountCode])
  @@map("simulation_lines")
}
```

**Schema notes:**
- `SimulationLine.dimensionValues` uses JSON (not a junction table) because simulations are transient/analytical. When converting to a real journal, the JSON is read and proper `JournalLineDimension` records are created.
- `transferredToId` links to the `JournalEntry` created during conversion, providing an audit trail.

Find the section after `console.log('    NumberSeries JOURNAL seeded');` (line ~850). Add a parallel block for the SIMULATION series:

```typescript
  // 4b. Number Series for SIMULATION
  const existingSimSeries = await prisma.numberSeries.findFirst({
    where: {
      companyId: DEFAULT_COMPANY_ID,
      entityType: 'SIMULATION',
      validFrom: null,
    },
  });
  if (existingSimSeries) {
    await prisma.numberSeries.update({
      where: { id: existingSimSeries.id },
      data: { prefix: 'SIM-', padding: 5 },
    });
  } else {
    await prisma.numberSeries.create({
      data: {
        companyId: DEFAULT_COMPANY_ID,
        entityType: 'SIMULATION',
        prefix: 'SIM-',
        nextValue: 1,
        padding: 5,
        isActive: true,
      },
    });
  }
  console.log('    NumberSeries SIMULATION seeded');
```

This follows the exact same findFirst+create/update pattern used for the JOURNAL series, because NumberSeries uses partial unique indexes that Prisma's upsert cannot target.
